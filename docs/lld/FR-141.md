## Summary

FR-141 implements the **transactional outbox pattern** for reliable domain-event delivery.
Every module that changes application state inserts a row into `event_outbox` inside the **same
database transaction** as the state change (BRD §5.2.42, architecture §11.5, ADR-7).  A
background publisher worker polls `event_outbox` for `status = 'pending'` rows and relays each
event to Google Cloud Pub/Sub (`PUBSUB_TOPIC_EVENTS`).  The worker marks rows `published` on
success and `failed` after exhausting retries; the originating transaction is never blocked.

FR-141 owns two concerns:

| Concern | Owner class | Location |
|---|---|---|
| Transactional write helper | `OutboxService` | `apps/api/src/core/outbox/outbox.service.ts` |
| Background relay worker | `OutboxPublisherService` | `apps/api/src/core/outbox/outbox-publisher.service.ts` |

No public HTTP endpoint is exposed.  No automated credit-decisioning or automated LMS action
may consume these events in MVP — they are an **analytics / AI-readiness feed only**.

---

## Endpoints

FR-141 exposes **no public or internal HTTP endpoints**.  The observable surface is:

- `OutboxService.emit(code, aggregateType, aggregateId, payload, tx)` — called by all
  state-changing services inside their UnitOfWork transaction.
- `OutboxPublisherService` — NestJS `@Injectable` with a Cloud Scheduler-triggered HTTP
  handler or a self-scheduling interval.  The integration monitor (FR-140,
  `GET /admin/integrations`) surfaces outbox queue depth and failure rate.

## Data Operations (Kysely — exact column names from `schema.sql`)

### OutboxService.emit — INSERT (called within caller's tx)

```typescript
// Caller passes the active Kysely transaction handle (tx)
await tx
  .insertInto('event_outbox')
  .values({
    event_id:       generateUUID(),          // gen_random_uuid() equivalent
    org_id:         ORG_ID_DEFAULT,          // '00000000-0000-0000-0000-000000000001'
    event_code:     code,                    // EventCode enum value
    aggregate_type: aggregateType,           // e.g. 'Lead', 'Consent', 'Task'
    aggregate_id:   aggregateId,             // UUID of the aggregate root
    payload:        JSON.stringify(payload), // masked, schema-versioned JSONB
    schema_version: 1,                       // bump per schema-evolution policy
    status:         'pending',
    published_at:   null,
    // created_at / updated_at: DB default now()
  })
  .execute();
```

Indexes used: none (INSERT path).  No FK on `aggregate_id` (polymorphic).

### OutboxPublisherService.poll — SELECT pending rows

```typescript
const rows = await db
  .selectFrom('event_outbox')
  .select(['event_id', 'event_code', 'aggregate_type', 'aggregate_id',
           'payload', 'schema_version', 'org_id', 'created_at'])
  .where('status', '=', 'pending')
  .orderBy('created_at', 'asc')
  .limit(PUBLISHER_BATCH_SIZE)  // default 100, configurable; enforces NFR LIMIT
  .execute();
```

Index used: `ix_event_outbox_status (status, created_at)` — covers the WHERE + ORDER BY.

### OutboxPublisherService.markPublished — UPDATE after Pub/Sub ack

```typescript
await db
  .updateTable('event_outbox')
  .set({ status: 'published', published_at: new Date(), updated_at: new Date() })
  .where('event_id', '=', eventId)
  .where('status', '=', 'pending')   // guard: only advance from pending
  .execute();
```

### OutboxPublisherService.markFailed — UPDATE after exhausted retries

```typescript
await db
  .updateTable('event_outbox')
  .set({ status: 'failed', updated_at: new Date() })
  .where('event_id', '=', eventId)
  .where('status', '=', 'pending')
  .execute();
```

### Dead-letter query (operational)

```typescript
// Used by integration monitor / alerting
const failedCount = await db
  .selectFrom('event_outbox')
  .select(db.fn.countAll<number>().as('count'))
  .where('status', '=', 'failed')
  .executeTakeFirstOrThrow();
```

## Validation Logic (Zod — OutboxService.emit call-site)

`OutboxService.emit` is called internally, not via HTTP.  The following Zod schema validates
the arguments at the service boundary (before INSERT):

| Argument | Rule | Error raised if invalid |
|---|---|---|
| `code` | `z.nativeEnum(EventCode)` — must be a value in the `event_code` DB enum | `INTERNAL_ERROR` (500); logged with `correlation_id`; caller transaction is not blocked |
| `aggregateType` | `z.string().min(1).max(40)` — e.g. `'Lead'`, `'Consent'`, `'Task'` | `INTERNAL_ERROR` (500) |
| `aggregateId` | `z.string().uuid()` | `INTERNAL_ERROR` (500) |
| `payload` | `z.record(z.unknown())` — non-null object | `INTERNAL_ERROR` (500) |
| `payload` masking | All PII fields (mobile, pan_token, pan_masked, name, email, aadhaar_ref_token, ckyc_id, dob, address) must be masked or absent; enforced by `MaskingService.maskEventPayload(payload)` before insert | `INTERNAL_ERROR` (500) if masking fails |
| `schemaVersion` | `z.number().int().min(1)` (default 1) | `INTERNAL_ERROR` (500) |

Validation failures in `emit` log the full error with `correlation_id` and rethrow a typed
`InternalException`; they do **not** roll back the caller's transaction (emit is inside the tx
already — the caller decides whether to rollback on exception, which is the standard UoW pattern).

**Payload schema-versioning policy:** when a payload shape changes (field added, renamed, or
removed), `schema_version` is incremented.  Consumers must handle all schema versions they
subscribe to.  The current schema definitions are maintained in
`apps/api/src/core/outbox/schemas/` as versioned TypeScript types and are the reference for
analytics teams.

## Error Cases (error-taxonomy.md codes only)

| Scenario | Code | HTTP | detail.reason | Behaviour |
|---|---|---|---|---|
| Invalid `event_code` value passed to `emit` | `INTERNAL_ERROR` | 500 | — | Logged + alert; caller tx rolls back if uncaught |
| PII masking fails in `emit` | `INTERNAL_ERROR` | 500 | — | Logged + alert; event not inserted |
| DB INSERT in `emit` fails (transient) | `INTERNAL_ERROR` | 500 | — | Caller UoW tx rolls back atomically; entire state change rolls back |
| Pub/Sub publish call times out / 5xx | `UPSTREAM_UNAVAILABLE` | 503 | — | Row stays `pending`; next poll retry picks it up; Cloud Monitoring alert on repeated failure |
| Pub/Sub publish fails after `MAX_PUBLISH_RETRIES` | — | — | — | Row marked `failed`; dead-letter alert; originating transaction was already committed and is NOT rolled back (publish is post-commit) |
| Publisher batch SELECT returns 0 rows | — | — | — | No-op; publisher sleeps until next interval |

`UPSTREAM_UNAVAILABLE` is surfaced in the integration monitor (FR-140), not in any user API
response; FR-141 has no public endpoint.

## Backend Flow (Step-by-Step)

### Path A: Transactional outbox write (called by every state-changing FR)

```
State-changing use-case (e.g. LeadService.transitionStage)
  │
  ├─ 1. Opens UnitOfWork: uow.run(async (tx) => { … })
  │
  ├─ 2. Performs primary writes within tx:
  │       LeadService  → UPDATE leads  (status, version++)
  │       StageHistory → INSERT stage_history
  │       AuditAppender → emit audit intent (to AuditChainConsumer queue)
  │
  ├─ 3. OutboxService.emit(EventCode.LEAD_STAGE_CHANGED, 'Lead', leadId, maskedPayload, tx)
  │       ├─ MaskingService.maskEventPayload(rawPayload)  → masked payload
  │       ├─ Validate args (Zod)
  │       └─ Kysely INSERT INTO event_outbox (status='pending') — same tx connection
  │
  └─ 4. tx.commit()  — leads + stage_history + event_outbox all commit atomically
         (on failure: entire tx rolls back; no partial state; no orphaned outbox row)
```

**Rule:** `OutboxService.emit` must always receive an active tx handle.  Calling it outside a
transaction is a programming error (enforced by a type guard on the `tx` parameter type from
Kysely's `Transaction<DB>`).

### Path B: Background publisher worker (OutboxPublisherService)

```
OutboxPublisherService (triggered every N seconds by Cloud Scheduler or setInterval)
  │
  ├─ 1. Poll: SELECT pending rows ORDER BY created_at ASC LIMIT 100
  │        (ix_event_outbox_status covers this)
  │
  ├─ 2. For each row (can be parallelised up to PUBLISHER_CONCURRENCY):
  │
  │       ├─ 3. Build Pub/Sub message:
  │       │       messageId  = event_id (UUID)
  │       │       data        = JSON.stringify({ event_id, event_code, aggregate_type,
  │       │                                      aggregate_id, schema_version, payload,
  │       │                                      org_id, created_at })
  │       │       attributes  = { event_code, aggregate_type, schema_version }
  │       │
  │       ├─ 4. pubSubClient.topic(PUBSUB_TOPIC_EVENTS).publishMessage({ data, attributes })
  │       │       ├─ Success → step 5
  │       │       └─ Failure (timeout / 5xx):
  │       │               publish_attempt_count++
  │       │               if attempt_count < MAX_PUBLISH_RETRIES (default 5):
  │       │                 leave row as 'pending'; exponential backoff; retry on next poll
  │       │               else:
  │       │                 markFailed(event_id)  → status = 'failed'
  │       │                 emit Cloud Monitoring alert (outbox_dead_letter counter++)
  │       │
  │       └─ 5. markPublished(event_id): UPDATE event_outbox SET status='published',
  │                published_at=now() WHERE event_id=:id AND status='pending'
  │
  └─ 6. Log summary: {published: N, failed: M, correlation_id}
          Emit metric: outbox_lag_seconds = now() - min(created_at of oldest pending row)
```

**At-least-once guarantee:** the SELECT→publish→mark loop is not atomic.  A process crash
between publish and mark causes the message to be re-published on the next poll cycle.
Downstream consumers must be idempotent on `event_id` (use it as a deduplication key).

**Publisher instances:** the publisher worker runs inside the same Cloud Run instance as the
NestJS app (started via `OnApplicationBootstrap`).  It does NOT need concurrency=1 because it
does not write a hash chain; multiple instances may publish duplicate messages (at-least-once is
acceptable; consumers deduplicate on `event_id`).

## State Machine (EventOutbox.status)

Per `docs/contracts/state-machines.md` — "EventOutbox: pending → published / failed":

```
pending  ──(publish success)──►  published  (terminal — no further transitions)
   │
   └──(exhausted retries)──►  failed  (terminal — dead-letter queue; manual replay possible)
```

| Transition | Triggered by | Guard | Side effects |
|---|---|---|---|
| `pending → published` | `OutboxPublisherService.markPublished` | `status = 'pending'` | `published_at = now()` |
| `pending → failed` | `OutboxPublisherService.markFailed` | `status = 'pending'` AND `attempt_count >= MAX_PUBLISH_RETRIES` | Cloud Monitoring dead-letter alert |

Invalid transitions (not implemented; enforced by WHERE guards on UPDATE):
- `published → pending` (no replay via DB; must insert a new row)
- `failed → pending` (same; operational replay creates a new row)

## Transaction Boundaries via UnitOfWork

```
UnitOfWork.run(async (tx) => {
  // --- all writes below use the same tx ---
  await LeadService.transitionStage(leadId, toStage, reason, expectedVersion, tx);
    // internally writes:
    //   UPDATE leads
    //   INSERT stage_history
    //   (AuditAppender queues audit intent — NOT in this tx; single-writer async)
  await OutboxService.emit(EventCode.LEAD_STAGE_CHANGED, 'Lead', leadId, maskedPayload, tx);
    // INSERT event_outbox (status='pending')
});
// ↑ single commit; on any failure: full rollback
// ↑ post-commit: publisher worker eventually reads and relays — never blocks here
```

The publish step (Path B) runs entirely **outside** any UnitOfWork transaction.  It reads
committed `pending` rows and issues independent DB UPDATEs per row.

## External Service Calls via OutboxPublisherService

| Service | Port / Client | Call | Failure handling |
|---|---|---|---|
| Google Cloud Pub/Sub | `@google-cloud/pubsub` (`PubSub` client) — **no IntegrationGateway** (Pub/Sub is infrastructure, not a business integration; it does not write `integration_logs`) | `topic.publishMessage({ data, attributes })` | Retry with exponential backoff up to `MAX_PUBLISH_RETRIES`; then `markFailed`; Cloud Monitoring alert |

Note: `IntegrationGateway` (FR-140) is **not** used here.  `IntegrationGateway` wraps LOS/KYC
business ports and writes `integration_logs` per call.  The outbox publisher is infrastructure
plumbing; it uses the Pub/Sub client directly and its failure state is tracked in
`event_outbox.status`, surfaced via the integration monitor metric.

## Assumptions

1. The Pub/Sub topic `PUBSUB_TOPIC_EVENTS` (env `PUBSUB_TOPIC_EVENTS`) is pre-created in the
   GCP project and the Cloud Run service account has `roles/pubsub.publisher`.
2. `MAX_PUBLISH_RETRIES` (default 5) and `PUBLISHER_BATCH_SIZE` (default 100) are runtime
   config constants in the publisher — not env vars (no BRD requirement to make them configurable).
3. The publisher is triggered on a fixed interval (e.g. every 5 s) via `setInterval` inside
   `OnApplicationBootstrap`, or optionally via a lightweight Cloud Scheduler endpoint.  The BRD
   does not specify the interval; 5 s is a reasonable default.
4. `attempt_count` is tracked in-memory per publisher run cycle (not persisted).  A process
   restart resets the count, giving the row additional publish attempts — acceptable for
   at-least-once.
5. Operational manual replay (re-queuing `failed` rows) is done by inserting a new
   `event_outbox` row with `status='pending'`; no endpoint is defined for this in MVP.

## Summary

FR-102 allows an RM or BM to log the outcome (disposition) of a call or field visit against a
task on a lead. For visit tasks the payload optionally carries a geotag (latitude, longitude,
accuracy) captured by the mobile PWA. A `CommunicationLog` row of channel `in_app` is written
to record the interaction. CTI click-to-call and disposition sync (Phase 1.5) flows through
`TelephonyPort` (IntegrationGateway); the MVP path is manual disposition only.

The endpoint is shared with FR-100 (`PATCH /tasks/{id}`); FR-102 adds the disposition and geo
fields to the request body and the side-effect of writing a `CommunicationLog` row.

---

## Endpoint

### `PATCH /api/v1/tasks/{id}`

Shared with FR-100. When `disposition` (and optionally `geo`) are present in the body, FR-102
logic runs.

#### Request

```http
PATCH /api/v1/tasks/{id}
Authorization: Bearer <jwt>
Content-Type: application/json
```

**Path parameter**

| Name | Type   | Required | Description    |
|------|--------|----------|----------------|
| id   | UUID   | Yes      | `tasks.task_id` |

**Body** (Zod-validated; only fields accepted for this FR — other PATCH fields owned by FR-100)

```json
{
  "disposition": "visited",
  "result_note": "Customer present, income docs promised next week.",
  "next_action_at": "2026-06-17T10:00:00+05:30",
  "geo": {
    "lat": 19.0760,
    "lng": 72.8777,
    "accuracy_m": 12
  }
}
```

| Field          | Type              | Required | Notes                                              |
|----------------|-------------------|----------|----------------------------------------------------|
| `disposition`  | `disposition` enum | Yes (for FR-102 flow) | One of: `connected`, `no_answer`, `wrong_number`, `not_interested`, `visited`, `rescheduled`, `callback_requested`, `docs_promised` |
| `result_note`  | string (max 1000) | No       | Free text; must not contain raw PII in logs        |
| `next_action_at` | ISO 8601 string | No       | Future datetime; required when `disposition` = `rescheduled` or `callback_requested` |
| `geo`          | object            | No       | Permitted only when `tasks.type IN ('visit')`. Omitted when location permission not granted. |
| `geo.lat`      | number            | Yes (if geo) | `-90` to `90` |
| `geo.lng`      | number            | Yes (if geo) | `-180` to `180` |
| `geo.accuracy_m` | number          | Yes (if geo) | `> 0` |

#### Response — 200 OK

```json
{
  "data": {
    "task_id": "uuid",
    "lead_id": "uuid",
    "type": "visit",
    "status": "done",
    "disposition": "visited",
    "result_note": "Customer present, income docs promised next week.",
    "geo": { "lat": 19.076, "lng": 72.8777, "accuracy_m": 12 },
    "next_action_at": null,
    "updated_at": "2026-06-10T09:00:00.000Z"
  },
  "meta": { "correlation_id": "corr_…" },
  "error": null
}
```

Note: `geo` is returned as stored; `result_note` is returned as stored but is **not logged** by
the pino logger (may contain customer details). Masking interceptor does not touch these fields
(not PII-classified), but `result_note` must never be written to `audit_logs.detail` raw.

## Auth Check

1. **Authentication:** `JwtAuthGuard` (global) validates the JWT. Missing or expired token →
   `AUTH_REQUIRED` (401).

2. **Authorisation:**
   ```typescript
   @Requires('edit_lead', (req) => resolveTaskScope(req.task))
   ```
   `AbacGuard` calls `EntitlementService.can(user, 'edit_lead', task)`.
   - RM scope `O`: passes only if `task.owner_id === user.user_id`.
   - BM scope `B`: passes only if `task.lead.branch_id === user.branch_id`.
   - Other roles (SM, HEAD, KYC, PARTNER, CUSTOMER) do not have `edit_lead` on `tasks` (per
     auth-matrix); → `FORBIDDEN` (403).

3. **Ownership check:** `resolveTaskScope` loads the task (with its `lead_id` and `lead.branch_id`)
   before the guard runs; a non-existent task → `NOT_FOUND` (404) before scope evaluation.

## Data Operations

All Kysely queries use parameterised bindings. Column names are exact from `schema.sql`.

### 1. Load and scope-check the task

```typescript
const task = await db
  .selectFrom('tasks')
  .innerJoin('leads', 'leads.lead_id', 'tasks.lead_id')
  .select([
    'tasks.task_id',
    'tasks.org_id',
    'tasks.lead_id',
    'tasks.type',
    'tasks.status',
    'tasks.owner_id',
    'tasks.disposition',
    'leads.branch_id',
    'leads.owner_id as lead_owner_id',
  ])
  .where('tasks.task_id', '=', taskId)
  .where('tasks.org_id', '=', user.org_id)
  .executeTakeFirst();
```

Returns `undefined` → `NOT_FOUND` (404).

### 2. Update task (within UnitOfWork transaction)

```typescript
await uow.run(async (tx) => {
  // 2a. Update tasks row
  const updated = await tx
    .updateTable('tasks')
    .set({
      disposition: dto.disposition,
      result_note: dto.result_note ?? null,
      geo: dto.geo ? JSON.stringify(dto.geo) : undefined,
      next_action_at: dto.next_action_at ? new Date(dto.next_action_at) : undefined,
      status: 'done',
      updated_at: new Date(),
      updated_by: user.user_id,
    })
    .where('task_id', '=', taskId)
    .where('org_id', '=', user.org_id)
    .where('status', '!=', 'done')    // guard: idempotent re-log rejected
    .returningAll()
    .executeTakeFirst();

  if (!updated) {
    // task already done → CONFLICT (409)
    throw new ConflictException('CONFLICT', 'Task already completed.');
  }

  // 2b. Insert CommunicationLog row
  await tx
    .insertInto('communication_logs')
    .values({
      org_id: user.org_id,
      lead_id: updated.lead_id,
      template_id: null,
      channel: 'in_app',
      recipient: updated.owner_id,         // the RM/BM who made contact
      consent_basis: null,                  // internal activity log, not a customer message
      status: 'sent',
      provider_ref: null,
      failure_reason: null,
      sent_at: new Date(),
      created_by: user.user_id,
      updated_by: user.user_id,
    })
    .execute();

  // 2c. Append audit log via AuditAppender
  await auditAppender.append(tx, {
    actor_id: user.user_id,
    action: 'lead_update',
    entity_type: 'tasks',
    entity_id: taskId,
    lead_id: updated.lead_id ?? undefined,
    detail: {
      event: 'disposition_logged',
      disposition: dto.disposition,
      task_type: updated.type,
      // result_note intentionally excluded from audit detail (may contain free text)
      has_geo: dto.geo != null,
    },
  });

  // 2d. Emit outbox event (contactability metric downstream, FR-121)
  await outboxService.emit(
    tx,
    'LEAD_STAGE_CHANGED',    // closest available event_code for field activity signals
    'tasks',
    taskId,
    {
      lead_id: updated.lead_id,
      task_type: updated.type,
      disposition: dto.disposition,
      actor_id: user.user_id,
    },
  );

  return updated;
});
```

### CTI Phase 1.5 — TelephonyPort call

When `CTI_ENABLED=true` and `task.type = 'call'`, after the manual log path above, the service
additionally records an `IntegrationLog` via `IntegrationGateway`:

```typescript
await integrationGateway.call(
  'TelephonyPort',
  { action: 'log_disposition', task_id: taskId, disposition: dto.disposition },
  { idempotencyKey: `cti-${taskId}-${dto.disposition}` },
);
```

Failure → `UPSTREAM_UNAVAILABLE` (503). The manual disposition row is already committed (the CTI
call is post-commit); a CTI failure does NOT roll back the disposition. The `IntegrationLog` row
records the failure for retry.

## Validation Logic

Implemented as a Zod schema applied by the global `ValidationPipe` at the controller boundary.

| Field              | Rule                                                                   | Error message                                             |
|--------------------|------------------------------------------------------------------------|-----------------------------------------------------------|
| `id` (path)        | UUID format                                                            | "Invalid task ID format."                                 |
| `disposition`      | Must be a value from `disposition` enum (`@shared/enums`)              | "disposition must be one of: connected, no_answer, wrong_number, not_interested, visited, rescheduled, callback_requested, docs_promised." |
| `result_note`      | String, max 1000 characters                                            | "result_note must not exceed 1000 characters."            |
| `next_action_at`   | ISO 8601 datetime string; if present must be in the future             | "next_action_at must be a future datetime."               |
| `next_action_at`   | Required when `disposition` is `rescheduled` or `callback_requested`   | "next_action_at is required for rescheduled or callback_requested dispositions." |
| `geo`              | Object with `lat`, `lng`, `accuracy_m` (if provided)                  | "geo must include lat, lng, and accuracy_m."              |
| `geo.lat`          | `number`, `-90 <= lat <= 90`                                           | "geo.lat must be between -90 and 90."                     |
| `geo.lng`          | `number`, `-180 <= lng <= 180`                                         | "geo.lng must be between -180 and 180."                   |
| `geo.accuracy_m`   | `number`, `> 0`                                                        | "geo.accuracy_m must be a positive number."               |
| `geo`              | Only allowed when task `type` is `call` or `visit`                     | "geo is only permitted on call or visit tasks."           |

All validation failures produce `VALIDATION_ERROR` (400) with `fields[]` in the error envelope.

## Backend Flow (Step-by-Step)

```
1. HTTP PATCH /api/v1/tasks/{id}
   CorrelationMiddleware → assigns/reads X-Correlation-Id

2. JwtAuthGuard
   → validates Bearer token → loads user (user_id, role, org_id, branch_id, team_id, owner_id)
   → fail: AUTH_REQUIRED (401)

3. AbacGuard  (@Requires('edit_lead', resolveTaskScope))
   → loads task row (see Data Operations §1) to determine scope
   → task absent: NOT_FOUND (404)
   → EntitlementService.can(user, 'edit_lead', task) — checks role + scope
   → fail: FORBIDDEN (403)

4. ValidationPipe (Zod)
   → parses + validates request body (UpdateTaskDispositionDto)
   → fail: VALIDATION_ERROR (400) with fields[]

5. EngagementController.patchTask()
   → delegates to TaskService.logDisposition(taskId, dto, user)

6. TaskService.logDisposition()
   a. Guard: task.status must be 'open' or 'in_progress'
      → 'done'/'cancelled': CONFLICT (409)
   b. Guard: if geo provided, task.type must be 'call' or 'visit'
      → VALIDATION_ERROR (400)
   c. Guard: if disposition = 'rescheduled'|'callback_requested', next_action_at must be set
      → VALIDATION_ERROR (400)

7. UnitOfWork.run(tx => ...)  [single atomic transaction]
   a. UPDATE tasks SET disposition, result_note, geo, next_action_at, status='done',
      updated_at, updated_by WHERE task_id AND org_id AND status != 'done'
      → returns updated row; if no row returned → CONFLICT (409)
   b. INSERT INTO communication_logs (channel='in_app', status='sent', ...)
   c. AuditAppender.append(tx, { action: 'lead_update', entity_type: 'tasks', ... })
   d. OutboxService.emit(tx, 'LEAD_STAGE_CHANGED', 'tasks', taskId, payload)

8. [Phase 1.5 only, post-commit, non-blocking]
   IF CTI_ENABLED AND task.type = 'call':
     IntegrationGateway.call('TelephonyPort', ..., { idempotencyKey })
     → failure: log IntegrationLog(status=failed); respond UPSTREAM_UNAVAILABLE (503)
     → success: IntegrationLog(status=success)

9. ResponseEnvelopeInterceptor
   → wraps updated task in { data, meta, error:null }

10. Respond 200 OK
```

## Task State Machine for Disposition Logging

From `docs/contracts/state-machines.md`:

```
Task state:  open → in_progress → done
                  └→ cancelled
open/in_progress → overdue (sweep)
```

FR-102 drives the `→ done` transition when a disposition is logged.

Valid pre-states for this FR: `open`, `in_progress`.
Invalid pre-states: `done`, `cancelled`, `overdue` (overdue tasks can still be dispositioned —
treat `overdue` as equivalent to `in_progress` for this transition).

**Note:** The `overdue` state is set by a background sweep (FR-104 / FR-100); an RM may log a
disposition on an overdue task. The UPDATE query in step 7a uses `WHERE status != 'done'` to
allow this (also covers `cancelled` — a cancelled task returning zero rows triggers CONFLICT).

---

## Transaction Boundaries

All writes in step 7 execute inside a single `UnitOfWork` transaction:

```
BEGIN
  UPDATE tasks (disposition, geo, status → done)
  INSERT communication_logs
  INSERT audit_logs         [via AuditAppender, single-writer]
  INSERT event_outbox       [via OutboxService]
COMMIT
```

If any step fails, the entire transaction rolls back. The CTI integration call (Phase 1.5) is
**post-commit** and does not participate in this transaction; its failure is handled separately.

## External Service Call (Phase 1.5 — CTI)

| Aspect             | Detail                                                              |
|--------------------|---------------------------------------------------------------------|
| Port               | `TelephonyPort` in `apps/api/src/core/integration/ports/`          |
| Adapter (prod)     | `TelephonyHttpAdapter` (vendor TBD, OD-08)                         |
| Adapter (test)     | `TelephonyMockAdapter` (`CTI_MOCK=true`)                           |
| Env vars           | `CTI_BASE_URL`, `CTI_API_KEY`, `CTI_MOCK`, `CTI_ENABLED`           |
| Gateway call       | `IntegrationGateway.call('TelephonyPort', payload, {idempotencyKey})` |
| IdempotencyKey     | `cti-{task_id}-{disposition}`                                      |
| Failure handling   | Provider error/timeout → `UPSTREAM_UNAVAILABLE` (503) + `IntegrationLog(status=failed/retrying)` + Cloud Tasks retry (exp backoff, max 3) |
| Circuit breaker    | Opens on repeated failure; manual disposition always available     |
| Recording consent  | Recording only sent/started where `CTI_RECORDING_ENABLED=true` AND the lead has a granted `consent_purpose = 'lead_contact'` consent record; otherwise recording is not initiated |

## Assumptions

1. The `LEAD_STAGE_CHANGED` event code is the closest available in the schema enum for
   contactability signals; no dedicated `TASK_COMPLETED` event code exists in the schema. The
   event payload carries `task_type` and `disposition` to differentiate downstream.

2. `communication_logs.recipient` stores the actor's `user_id` (UUID) as a VARCHAR(255) for
   internal activity (not a customer mobile/email), because there is no separate contact-log
   table for internal field activity. This differs from FR-101's use of the same table where
   `recipient` is a masked customer contact.

3. The `overdue` task status (set by the FR-104 sweep) is treated as eligible for disposition
   logging; the UPDATE guard uses `WHERE status != 'done'` (not a restrictive list of pre-states).

4. CTI recording policy enforcement (`CTI_RECORDING_ENABLED` env + consent check) is delegated
   to the `TelephonyPort` adapter implementation; the LLD specifies the contract but not the
   adapter internals (OD-08).

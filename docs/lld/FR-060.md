**Tier: 3** (Complex — multi-entity transactional writes across `customer_links`, `documents`, `consent_records`, and `audit_logs`; OTP step-up authentication via a dedicated guard; file upload pipeline with async virus-scan; external communication dispatch via `NotificationChannelPort`; stateful `link_status` machine with revocation; customer-facing public micro-site UI; token lifecycle with invalidation on resend; rate-limited OTP; multiple public endpoints each with distinct validation and audit paths.)

## Summary

FR-060 provides a **tokenised, expiring, revocable** self-service link that allows a customer to upload documents, grant consent, and take limited actions on their own lead — without installing an app or holding a user account. A staff user (RM/BM/KYC) creates the link via `POST /leads/{id}/customer-link`; the system generates a cryptographically random opaque token (raw, URL-safe), stores its SHA-256 hash in `customer_links.token_hash`, and dispatches the URL via the communication layer (FR-101). All subsequent customer-facing endpoints under `/c/{token}` are `@Public()` and protected by `CustomerLinkGuard` (token validity + OTP step-up gate). Resend issues a new token and revokes the previous one. Every open/action is audited. File uploads are virus-scanned before becoming available; infected files are rejected and not stored.

## Auth Check

### Internal endpoint (`POST /leads/{id}/customer-link`)

```
JwtAuthGuard (global, validates JWT access token)
  → AbacGuard → @Requires('customer_comm', scopeResolver)
      → EntitlementService.can(user, 'customer_comm', lead)
          → RM: scope O (must be lead.owner_id === user.user_id)
          → BM/KYC: scope B (must be lead.branch_id === user.branch_id)
          → FORBIDDEN (403) on any mismatch
```

The `lead_id` path parameter is resolved first; if the lead does not exist or is out of scope → `NOT_FOUND` (404).

### Public token endpoints (`/c/{token}/*`)

```
@Public() — JwtAuthGuard skipped
CustomerLinkGuard (in modules/self-service/):
  1. SHA-256 hash the raw token from the path
  2. SELECT customer_links WHERE token_hash = $hash AND status = 'active'
  3. If no row → NOT_FOUND (404)   [hides existence — token invalid/expired/revoked looks identical]
  4. If expires_at < now() → mark status='expired', return NOT_FOUND (404) with friendly hint
  5. Set req.customerLink = row (lead_id, purpose, customer_link_id, otp_verified_at)
  OTP gate (for /documents and /consent only):
  6. Check Redis key clsession:{customer_link_id} → if absent/expired → AUTH_REQUIRED (401)
     with message "OTP verification required."
```

**Note:** `GET /c/{token}` and `POST /c/{token}/otp` are exempt from the OTP gate check (they are the landing and OTP entry steps themselves).

### OTP verify (`POST /c/{token}/otp`)

```typescript
// OTP value stored in Redis: key otp:{customer_link_id}, TTL 10 min
// The OTP was dispatched (via NotificationChannelPort) when the link was created
// (or on the landing page request if otp_verified_at is null — see step 2 of customerOpen)

const stored = await redis.get(`otp:${customerId}`);
if (!stored || stored !== dto.otp) {
  // Increment attempt counter; if > 10 within window → RATE_LIMITED
  throw new RateLimitedOrUnauthorised();
}

// Mark OTP verified in DB
await db.updateTable('customer_links')
  .set({ otp_verified_at: sql`now()`, updated_at: sql`now()`, updated_by: SYSTEM_USER_ID })
  .where('customer_link_id', '=', customerId)
  .execute();

// Set session key in Redis (TTL = remaining link validity, max 30 min idle)
await redis.set(`clsession:${customerId}`, '1', 'EX', SESSION_TTL_SECONDS);
await redis.del(`otp:${customerId}`);
```

## Error Cases

| Scenario | Code | HTTP | `detail.reason` | Notes |
|---|---|---|---|---|
| Lead not found / out of scope for staff | `NOT_FOUND` | 404 | — | Hides existence for out-of-scope |
| Staff lacks `customer_comm` capability | `FORBIDDEN` | 403 | — | AbacGuard deny |
| Token not found / revoked / already used | `NOT_FOUND` | 404 | — | Identical response to hide timing |
| Token expired (detected at guard time) | `NOT_FOUND` | 404 | — | Mark `status='expired'` then 404 |
| OTP wrong or absent for upload/consent | `AUTH_REQUIRED` | 401 | — | "OTP verification required." |
| OTP rate limit exceeded (>10 in 10 min) | `RATE_LIMITED` | 429 | — | Retry-After header set |
| File too large (>10 MB) | `PAYLOAD_TOO_LARGE` | 413 | — | Before GCS upload |
| Disallowed MIME type | `UNSUPPORTED_MEDIA` | 415 | — | After magic-byte inspection |
| Infected file (post-scan callback) | `VALIDATION_ERROR` | 400 | — | Document soft-deleted; not stored in clean bucket |
| Action not in link's `purpose` array | `FORBIDDEN` | 403 | — | e.g. upload attempted but purpose=['status'] |
| Notification send failure | `UPSTREAM_UNAVAILABLE` | 503 | — | Link row already created; retry dispatch only |
| Unhandled server error | `INTERNAL_ERROR` | 500 | — | No stack trace in response |

## Backend Flow — Step by Step

### A. Create customer link (staff-initiated)

1. `POST /leads/{id}/customer-link` → `CorrelationMiddleware` assigns `X-Correlation-Id`.
2. `JwtAuthGuard` validates JWT; loads `user` + ABAC attributes onto request context.
3. `AbacGuard` / `@Requires('customer_comm', scope)` → `EntitlementService.can(user, 'customer_comm', lead)` — `FORBIDDEN` (403) on deny.
4. `ValidationPipe` (Zod) validates `CreateCustomerLinkDto` — `VALIDATION_ERROR` (400) on fail.
5. `CustomerLinkService.create(leadId, dto, actorUser)`:
   a. Load lead — verify exists and in scope; else `NOT_FOUND` (404).
   b. Generate `rawToken = crypto.randomBytes(32).toString('base64url')`.
   c. Compute `tokenHash = sha256(rawToken)`.
   d. `UnitOfWork.run(tx => ...)`:
      - Revoke previous active link for this lead (UPDATE … status='revoked').
      - INSERT into `customer_links` with `tokenHash`, `purpose`, `status='active'`, `expires_at`.
      - `AuditAppender.append(tx, { action: 'link_create', ... })`.
      - `OutboxService.emit(tx, { event_code: 'DOC_REQUEST', ... })`.
   e. Tx commits.
   f. **Post-commit** (non-blocking): `NotificationDispatchService.send(lead, 'customer_link_send', channel, { url: buildCustomerUrl(rawToken) })` — dispatches via `NotificationChannelPort`; failure → logs + retries via Cloud Tasks; does **not** roll back the link creation.
6. Return 201 with link metadata (token never returned).

### B. Customer opens the link (`GET /c/{token}`)

1. `CustomerLinkGuard` hashes token, looks up `customer_links` — `NOT_FOUND` (404) on any miss.
2. If `expires_at < now()` → update `status='expired'` + audit → `NOT_FOUND` (404).
3. Set `req.customerLink`; record `opened_at` (idempotent — no-op if already set).
4. Read `leads.stage` + `product_code` for display label (no PII returned).
5. Return 200 with `{ purpose, otp_required: true, otp_verified: false, lead_display }`.

### C. OTP step-up (`POST /c/{token}/otp`)

1. `CustomerLinkGuard` validates token (no OTP check at this stage).
2. `ValidationPipe` checks `VerifyOtpDto`.
3. `OtpService.verify(customerId, dto.otp)`:
   a. Check Redis rate-limit key `otplimit:{customerId}` — increment; if > 10 within window → `RATE_LIMITED` (429).
   b. Fetch Redis key `otp:{customerId}`; compare (constant-time). On mismatch: increment fail counter; `AUTH_REQUIRED` (401).
   c. On match: DELETE OTP key; SET `clsession:{customerId}` with TTL; UPDATE `customer_links.otp_verified_at`.
4. Return 200 `{ otp_verified: true, session_expires_at }`.

**Note:** the OTP itself is dispatched when the link is first sent (step A.f above) and again on demand if the customer requests a resend via the landing page (separate endpoint `POST /c/{token}/otp/resend` — out of scope for this FR, listed as ambiguity). OTP TTL: 10 minutes.

### D. Customer uploads document (`POST /c/{token}/documents`)

1. `CustomerLinkGuard`: token valid + OTP gate (`clsession` Redis key present); else `AUTH_REQUIRED` (401).
2. Guard verifies `'upload' in req.customerLink.purpose`; else `FORBIDDEN` (403).
3. `ValidationPipe` validates multipart fields.
4. MIME content-inspection (magic bytes) — `UNSUPPORTED_MEDIA` (415) on unknown type.
5. File size check — `PAYLOAD_TOO_LARGE` (413) if > `MAX_UPLOAD_BYTES`.
6. GCS: upload to staging bucket (not public-access). Path: `staging/{org_id}/{lead_id}/{uuid}_{original_name_sanitised}`.
7. `UnitOfWork.run(tx => ...)`:
   - INSERT `documents` row: `status='uploaded'`, `virus_scan_status='pending'`, `uploaded_via='customer_link'`.
   - `AuditAppender.append(tx, { action: 'doc_upload', ... })`.
   - `OutboxService.emit(tx, { event_code: 'DOC_UPLOADED', ... })`.
8. Tx commits.
9. **Post-commit**: enqueue Cloud Tasks job → `VirusScanWorker`:
   - On clean: UPDATE `documents` SET `virus_scan_status='clean'`, `status='under_review'`, move GCS object to clean bucket.
   - On infected: UPDATE `documents` SET `virus_scan_status='infected'`, `deleted_at=now()` (soft-delete); notify KYC queue; DO NOT move to clean bucket.
10. Return 201 with `{ document_id, status:'uploaded', virus_scan_status:'pending' }`.

### E. Customer grants consent (`POST /c/{token}/consent`)

1. `CustomerLinkGuard`: token valid + OTP gate; `'consent' in purpose`; else `FORBIDDEN`.
2. `ValidationPipe` validates `CustomerConsentDto`.
3. `UnitOfWork.run(tx => ...)`:
   - INSERT `consent_records` (append-only; no UPDATE/DELETE).
   - `AuditAppender.append(tx, { action: 'consent_grant', ... })`.
4. Tx commits.
5. Return 201 with consent record summary.

## State Machine — `CustomerLink.link_status`

Reference: `docs/contracts/state-machines.md` (compact table: CustomerLink).

```
active  ──(resend / new link created)──►  revoked
active  ──(expires_at < now())──────────►  expired   [auto, at guard time]
active  ──(staff revoke via separate admin endpoint, FR-062 or future)──►  revoked
active  ──(link_used = all purpose actions complete, optional)──►  used
```

Valid transitions enforced by `CustomerLinkService`:
- `active → revoked`: only by a staff user with `customer_comm` capability on the same lead.
- `active → expired`: detected by `CustomerLinkGuard` when `expires_at < now()`; UPDATE performed before returning 404.
- `active → used`: set when all declared `purpose` actions have been completed (optional optimisation; not a hard guard).

**Invalid transitions (→ `CONFLICT` 409):**
- Any write to a `revoked`, `expired`, or `used` link — `CustomerLinkGuard` returns `NOT_FOUND` (same as invalid token).

### Transaction Boundary (UnitOfWork)

Every write that combines two or more tables runs inside `UnitOfWork.run(async (tx) => { ... })`. The scope for each operation:

| Operation | Tables in single tx | Post-commit (non-blocking) |
|---|---|---|
| Create link | `customer_links` (revoke old + insert new) + `audit_logs` + `event_outbox` | `NotificationChannelPort` send |
| Open link | `customer_links` (set `opened_at`) + `audit_logs` | — |
| Upload document | `documents` (insert) + `audit_logs` + `event_outbox` | Cloud Tasks virus-scan job |
| Grant consent | `consent_records` (insert) + `audit_logs` | — |

If any step inside `UnitOfWork.run` throws, the entire transaction rolls back — no partial state persists.

## External Service Calls

All calls via `IntegrationGateway.call(port, req, { idempotencyKey })`.

### `NotificationChannelPort` — send customer link URL

| Aspect | Detail |
|---|---|
| Port | `NotificationChannelPort` → `SmsAdapter` / `WhatsappAdapter` / `EmailAdapter` |
| When | Post-commit after link creation |
| Idempotency key | `customer_link:{customer_link_id}:send` |
| Failure handling | `UPSTREAM_UNAVAILABLE` (503) logged; Cloud Tasks retry (exp backoff, max 3); link already in DB — resend does not recreate the link |
| India gating | SMS: TRAI DLT template required; WhatsApp: pre-approved Meta template; no free-form |
| What is sent | `{ channel, recipient: lead_mobile/email, template: 'customer_link_send', vars: { url, expires_at } }` |
| PII in logs | Never log `recipient` value — only `masked_recipient`; never log raw token |

### `VirusScanWorker` (Cloud Tasks async, post-commit)

| Aspect | Detail |
|---|---|
| Trigger | Cloud Tasks job enqueued post-commit |
| Input | `{ document_id, gcs_staging_path }` |
| On clean | UPDATE `documents.virus_scan_status='clean'`, `status='under_review'`; GCS move staging → clean bucket |
| On infected | UPDATE `documents.virus_scan_status='infected'`, `deleted_at=now()`; DO NOT move; emit `DOC_MISMATCH` outbox event |
| On scan provider error | Retry via Cloud Tasks (max 3); after max retries: `virus_scan_status` stays `'pending'`, alert raised |

## Assumptions

1. OTP generation and delivery for the initial link send is handled by `OtpService` (within M7), which calls `NotificationChannelPort`. The OTP is generated at link-creation time (not lazily). If the link is opened and OTP has expired (TTL 10 min), the micro-site can trigger an OTP resend — the mechanism for this resend (`POST /c/{token}/otp/resend`) is not specified in the BRD FR table and is listed as an ambiguity below.
2. The `SYSTEM_USER_ID` (a well-known UUID from the seed bootstrap) is used as `actor_id` / `created_by` / `updated_by` for all customer-facing writes (no JWT user for public endpoints). This satisfies the FK constraint on `audit_logs.actor_id` and `documents.created_by`.
3. The virus-scan provider is accessed via the existing `IntegrationGateway` infrastructure (not a new direct SDK call), using an appropriate port (e.g. `VirusScanPort`) to be added to the `integration-map.md` in Stage 5 refresh or by the implementing agent with the appropriate ambiguity note.
4. `CUSTOMER_LINK_SEND` is not in the `event_code` enum; the closest is `DOC_REQUEST` (used when a document request is made alongside the link). The outbox event uses `DOC_REQUEST` for the initial link send.

## Ambiguities

| # | Gap | Impact | Best-effort resolution |
|---|---|---|---|
| AMB-1 | OTP resend endpoint (`POST /c/{token}/otp/resend`) not defined in `api-contract.yaml` or BRD FR table | Customer cannot get a new OTP if the 10-min TTL expires before they submit | Implement as a thin wrapper on `OtpService.dispatch`; use same rate-limit key; do not add a new DB row — update `otp_verified_at` only after success |
| AMB-2 | `CUSTOMER_LINK_CREATED` / `CUSTOMER_LINK_SEND` event codes not present in `event_code` enum in schema.sql | Cannot emit a typed outbox event for link creation | Use `DOC_REQUEST` (already in enum, semantically closest); document this choice in implementing PR |
| AMB-3 | `VirusScanPort` not listed in `integration-map.md` | Cannot wire the virus-scan call through IntegrationGateway without a port | Implementing agent must add `VirusScanPort` to `integration-map.md` or raise an AMBIGUITY.md if blocked |
| AMB-4 | Whether `link_status` should transition to `'used'` automatically after all purpose actions complete, or remain `'active'` until expiry | Affects whether the guard allows re-entry | Default: leave `'active'`; mark `'used'` only on explicit staff revoke or expiry. BRD says "used" is a valid state but does not mandate the trigger condition |

## Summary

FR-113 provides a managed registry of Digital Lending App (DLA), Loan Service Provider (LSP), and partner entities that the NBFC must evidence and disclose under RBI digital-lending guidelines. The registry stores mandatory disclosure metadata — name, type, owner, URL, grievance officer, enabled products, data collected, storage location, and operational status.

The FR delivers three capabilities:

1. **List** (`GET /compliance/dla`) — paginated registry view for DPO/ADMIN. Optionally filtered by `type` and `status`. Inactive (retired) entries are visible to internal actors but hidden from the customer-facing disclosure block (BRD AC-3).
2. **Create** (`POST /compliance/dla`) — add a new DLA/LSP/partner entry. Active entries require all mandatory disclosure fields to be non-null; draft entries are permitted with incomplete fields.
3. **Update** (`PATCH /compliance/dla`) — update fields or transition status (`draft → active`, `active → retired`). Activating an entry triggers re-validation of mandatory disclosure fields.

No stage transitions on `leads` occur. No external service calls. A DLA/LSP audit record is written via `AuditAppender` on create and update. The registry content drives customer-facing disclosures rendered by FR-060 (Lead 360 / customer micro-site), and feeds compliance exports initiated by FR-122.

The `PATCH` endpoint does not carry an `id` path param in `api-contract.yaml` (`PATCH /compliance/dla`); the `dla_registry_id` must therefore be carried in the request body. See Ambiguities section.

## Auth Check

| Guard | Detail |
|-------|--------|
| `JwtAuthGuard` | Global; applied automatically to all three handlers |
| `AbacGuard` | `@Requires('configuration', scopeResolver)` |
| `EntitlementService.can(user, 'configuration', resource)` | Roles with `configuration`: DPO (scope B), ADMIN (scope A). All other roles are denied with `FORBIDDEN` (403). |

Per `auth-matrix.json` `resource_governance`, `dla_registry` is governed by the `configuration` capability and owned by M14 (maker-checker), but FR-113 specifies DPO/ADMIN as the authorised roles. The `configuration` capability is the correct gate; no lead-scope resolution is needed since the registry is org-scoped (not lead-scoped). `scopeResolver` always resolves the org_id from the authenticated user's session and filters the query accordingly.

ADMIN has no standing lead-content access (capability_conditions), but `dla_registry` is configuration/master data, not lead content — ADMIN access to this resource is correct.

## Data Operations

All queries use Kysely parameterised builders. Column names are exact as per `schema.sql`.

### 4.1 List DLA/LSP entries

```typescript
// DlaRegistryRepository.list(orgId, filters, pagination)
db.selectFrom('dla_registry')
  .selectAll()
  .where('org_id', '=', orgId)
  .$if(filters.type !== undefined, (qb) =>
    qb.where('type', '=', filters.type)
  )
  .$if(filters.status !== undefined, (qb) =>
    qb.where('status', '=', filters.status)
  )
  .orderBy(allowedSortColumn, sortDirection)
  .limit(pagination.limit)
  .offset((pagination.page - 1) * pagination.limit)
  .execute();

// Count for pagination.total
db.selectFrom('dla_registry')
  .select(db.fn.countAll<number>().as('total'))
  .where('org_id', '=', orgId)
  .$if(filters.type !== undefined, (qb) => qb.where('type', '=', filters.type))
  .$if(filters.status !== undefined, (qb) => qb.where('status', '=', filters.status))
  .executeTakeFirstOrThrow();
```

### 4.2 Create entry

```typescript
// DlaRegistryRepository.create(dto, actorId, orgId)
db.insertInto('dla_registry')
  .values({
    org_id:            orgId,
    name:              dto.name,
    type:              dto.type,
    owner:             dto.owner ?? null,
    url:               dto.url ?? null,
    grievance_officer: dto.grievance_officer ? JSON.stringify(dto.grievance_officer) : null,
    enabled_products:  dto.enabled_products ? JSON.stringify(dto.enabled_products) : null,
    data_collected:    dto.data_collected ? JSON.stringify(dto.data_collected) : null,
    storage_location:  dto.storage_location ?? null,
    status:            dto.status ?? 'draft',
    created_by:        actorId,
    updated_by:        actorId,
  })
  .returningAll()
  .executeTakeFirstOrThrow();
```

Unique constraint on `(org_id, name)` does not exist in the DDL. The service enforces name-uniqueness via a pre-insert lookup (`findByName`); if found, throw `CONFLICT`. See Ambiguities.

### 4.3 Update entry

```typescript

  .set({

    ...(patch.url              !== undefined && { url: patch.url }),
    ...(patch.grievance_officer !== undefined && {

    }),
    ...(patch.enabled_products !== undefined && {
      enabled_products: JSON.stringify(patch.enabled_products),
    }),
    ...(patch.data_collected   !== undefined && {
      data_collected: JSON.stringify(patch.data_collected),
    }),
    ...(patch.storage_location !== undefined && { storage_location: patch.storage_location }),
    ...(patch.status           !== undefined && { status: patch.status }),
    updated_by: actorId,
    updated_at: new Date(),
  })
  .where('dla_registry_id', '=', id)
  .where('org_id', '=', orgId)
  .returningAll()
```


**Business-rule validation (service layer — cannot be expressed in Zod alone):**

| Rule | Error |
|------|-------|
| `status = active` AND `owner` is null or empty | `VALIDATION_ERROR`, `fields: [{ field: 'owner', message: 'owner is required for active entries' }]` |
| `status = active` AND `url` is null or empty | `VALIDATION_ERROR`, `fields: [{ field: 'url', message: 'url is required for active entries' }]` |
| `status = active` AND `grievance_officer` is null | `VALIDATION_ERROR`, `fields: [{ field: 'grievance_officer', message: 'grievance_officer is required for active entries' }]` |
| `status = active` AND `storage_location` is null or empty | `VALIDATION_ERROR`, `fields: [{ field: 'storage_location', message: 'storage_location is required for active entries' }]` |
| Entry with the same `name` already exists for this `org_id` | `CONFLICT` (409) |

## Backend Flow

### Flow A — List entries (GET /compliance/dla)

```
1. JwtAuthGuard validates JWT → user principal resolved (user.id, user.org_id, user.roles).
2. AbacGuard: EntitlementService.can(user, 'configuration', 'dla_registry') → FORBIDDEN (403) if denied.
3. Controller parses query params through PaginationParams + ListDlaFiltersDto (Zod).
   Field errors → VALIDATION_ERROR (400) with fields[].
4. DlaRegistryService.list(user.org_id, filters, pagination):
   a. DlaRegistryRepository.list(orgId, filters, pagination) → paginated rows + count.
   b. No masking required (no PII fields in dla_registry).
5. Return 200 with data[] + meta.pagination.
```

### Flow B — Create entry (POST /compliance/dla)

```
1. JwtAuthGuard validates JWT.
2. AbacGuard: EntitlementService.can(user, 'configuration', 'dla_registry') → FORBIDDEN (403) if denied.
3. Controller parses body through CreateDlaDto (Zod). Zod errors → VALIDATION_ERROR (400) with fields[].
4. DlaRegistryService.create(dto, user.id, user.org_id):
   a. If dto.status === 'active': validateMandatoryDisclosureFields(dto) → VALIDATION_ERROR (400)
      listing each missing mandatory field.
   b. DlaRegistryRepository.findByNameAndOrg(dto.name, orgId):
      - If found → throw CONFLICT (409).
   c. Open UnitOfWork:
      i.  DlaRegistryRepository.create(dto, actorId, orgId, tx) → created entry.
      ii. AuditAppender.emit({ action: 'config_change', entity_type: 'dla_registry',
            entity_id: entry.dla_registry_id, detail: { name, type, status } }, tx).
5. Return 201 with the created entry wrapped in { data, meta, error: null }.
```

### Flow C — Update entry (PATCH /compliance/dla)

```
1. JwtAuthGuard validates JWT.
2. AbacGuard: EntitlementService.can(user, 'configuration', 'dla_registry') → FORBIDDEN (403) if denied.
3. Controller parses body through UpdateDlaDto (Zod). Zod errors → VALIDATION_ERROR (400) with fields[].
4. DlaRegistryService.update(dto, user.id, user.org_id):
   a. DlaRegistryRepository.findById(dto.dla_registry_id, orgId):
      - If not found → throw NOT_FOUND (404).
   b. If dto.status is present: validateStatusTransition(existing.status, dto.status):
      - 'draft' → 'active': allowed.
      - 'active' → 'retired': allowed.
      - 'retired' → any: throw CONFLICT (409).
      - Any other jump (e.g. 'draft' → 'retired'): throw CONFLICT (409).
   c. Compute mergedEntry = { ...existing, ...dto } (patch merge).
   d. If mergedEntry.status === 'active': validateMandatoryDisclosureFields(mergedEntry)
      → VALIDATION_ERROR (400) listing each missing mandatory field.
   e. Open UnitOfWork:
      i.  DlaRegistryRepository.update(dto.dla_registry_id, orgId, patch, actorId, tx) → updated entry.
          - If returns undefined → throw NOT_FOUND (should not occur after step a, guards race).
      ii. AuditAppender.emit({ action: 'config_change', entity_type: 'dla_registry',
            entity_id: dto.dla_registry_id,
            detail: { changed_fields, previous_status: existing.status, new_status: mergedEntry.status } }, tx).
5. Return 200 with the updated entry wrapped in { data, meta, error: null }.
```

## State Machine — DLARegistry.status (config_status)

Per `state-machines.md` compact machines: `draft → active → retired`.

| From | To | Trigger | Who | Side effects |
|------|----|---------|-----|--------------|
| `draft` | `active` | PATCH with `status: 'active'` | DPO / ADMIN | Mandatory disclosure field validation; audit `config_change` |
| `active` | `retired` | PATCH with `status: 'retired'` | DPO / ADMIN | Entry hidden from customer-facing disclosure block (FR-060); audit `config_change` |

**Invalid transitions (→ 409 CONFLICT):**
- `retired → active` (no reinstatement; create a new entry if needed)
- `retired → draft`
- `draft → retired` (must go through `active` first — see Ambiguities)
- Any backward transition

**No outbox event** is emitted for DLA status changes. Customer-facing disclosure is derived at read time by filtering `status = 'active'` entries, so no push notification is required.

## Transaction Boundaries via UnitOfWork

| Sub-flow | Entities written | UnitOfWork required |
|----------|-----------------|---------------------|
| Create entry | `dla_registry` (insert) + `audit_logs` (append) | Yes — `uow.run(async (tx) => { repo.create(..., tx); audit.emit(..., tx) })` |
| Update entry | `dla_registry` (update) + `audit_logs` (append) | Yes — `uow.run(async (tx) => { repo.update(..., tx); audit.emit(..., tx) })` |
| List entries | `dla_registry` (read only) | No |

`dla_registry` is not `leads` — `LeadService` is not involved. No `stage_history`, `event_outbox`, or `LeadService.transitionStage` is invoked.

```
      Button "Save" (disabled while isSubmitting; shows spinner)
    Toast (success / error — maps VALIDATION_ERROR.fields to inline errors per field)
    ConfirmDialog (when transitioning to 'retired': "This will hide the entry from customer disclosures. Confirm?")
```

### B. Customer-Facing Disclosure Block (read-only, consumed by FR-060)

The DLA/LSP registry is queried by FR-060 (Lead 360 / customer micro-site) to populate the lender/LSP disclosure section. FR-113 exposes a **read query** scoped to `status = 'active'` entries; the UI component itself is owned by FR-060. From FR-113's perspective, this is a shared data dependency, not a separate endpoint.

The customer-facing display rule from BRD AC-3 — "inactive DLA hidden from customer-facing display" — is enforced by filtering `status = 'active'` in the repository query used by FR-060.

## Ambiguities

1. **PATCH path has no `{id}` segment.** `api-contract.yaml` defines `PATCH /compliance/dla` without an `{id}` path parameter. All other update endpoints in the contract use `PATCH /<resource>/{id}`. This LLD resolves this by placing `dla_registry_id` in the request body. If the contract is corrected to `PATCH /compliance/dla/{id}`, the controller signature changes to a `@Param('id')` path parameter and the body DTO drops the `dla_registry_id` field.

2. **Name-uniqueness constraint.** The `dla_registry` DDL has no unique index on `(org_id, name)`. This LLD enforces uniqueness via a pre-insert service-layer lookup. If multiple concurrent `POST` calls race, a second attempt will succeed without a DB-level constraint. If the team wants strong deduplication, a `UNIQUE (org_id, name)` partial index should be added via a Flyway migration.

3. **`draft → retired` transition.** The state-machines.md compact machine shows `draft → active → retired`. It is ambiguous whether `draft → retired` should be permitted (to discard a draft without activating it). This LLD blocks `draft → retired` and requires going through `active` first. If the team decides discard-without-activate is valid, the transition table should be updated.

4. **Customer-facing disclosure endpoint.** BRD AC-3 requires customer-facing lender/LSP/grievance info where required by config. The customer-facing read of active DLA entries is owned by FR-060. This LLD does not create a separate public endpoint — FR-060 should call `DlaRegistryService.listActive(orgId)` directly. If a dedicated public endpoint is required, it should be added to `api-contract.yaml` and a separate LLD section written.

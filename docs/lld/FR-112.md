## Summary

FR-112 implements the data-principal rights workflow required under India's digital lending and data-protection regulations (BRD §A.1/A.3). It allows customers to raise access, correction, update, erasure, withdrawal, or grievance requests against their data, and enables the DPO to review, disposition, and process those requests.

**Customer path:** A customer raises a request via the customer link (`POST /api/v1/c/{token}/data-rights`) or the Compliance Console. The request is inserted into `data_rights_requests` with `status='open'` and a calculated `due_at` (business SLA). A `DATA_RIGHT_REQUEST` outbox event is emitted in the same transaction to notify the DPO.

**DPO path:** The DPO lists all requests (`GET /api/v1/data-rights`), opens a specific one, and processes it via `PATCH /api/v1/data-rights/{id}`. Processing includes:
- Setting `status` to `in_review`, `fulfilled`, or `rejected_retained`.
- For an `erasure` request: consulting `retention_policies` and checking for active legal holds. If a legal hold exists, the transition to `fulfilled` is blocked and the response carries `CONFLICT` with `detail.reason=LEGAL_HOLD`; the request is moved to `rejected_retained` with a disposition explaining the basis.
- For a `fulfilled` erasure: the DPO confirms the anonymisation/purge was handled by FR-115's `RetentionEngine`. FR-112 does not directly call the engine; it records the approved disposition and emits an outbox event; the RetentionEngine acts asynchronously.

This FR is Tier 3 because it involves a state machine with conditional branching (legal-hold check), a multi-entity transactional write (`data_rights_requests` + `audit_logs` + `event_outbox`), SLA calculation via `SlaEngine`/`BusinessCalendarService`, and a customer-link authenticated intake path in addition to the JWT-authenticated DPO path.

### 3. PATCH /api/v1/data-rights/{id}

**Summary:** Process a data-rights request (DPO only). Transitions status, sets disposition, checks legal hold for erasure.

**Request headers**

| Header | Required | Notes |
|---|---|---|
| `Authorization` | Yes | Bearer JWT (DPO role required) |

**Path parameter:** `id` — UUID of the `data_rights_request`.

**Request body**

```json
{
  "status": "fulfilled",
  "disposition": "Data anonymised per RetentionPolicy pii/rejected/365 — no legal hold.",
  "owner_id": "uuid"
}
```

| Field | Type | Required | Notes |
|---|---|---|---|
| `status` | `rights_status` enum | Yes | Must be a valid forward transition (see state machine) |
| `disposition` | string (max 500) | Yes when status is `fulfilled` or `rejected_retained` | Reason/basis for decision |
| `owner_id` | UUID | No | Assign DPO owner; defaults to caller if omitted |

**Response 200 — success**

```json
{
  "data": {
    "data_rights_request_id": "uuid",
    "status": "fulfilled",
    "disposition": "Data anonymised per RetentionPolicy...",
    "owner_id": "uuid",
    "updated_at": "2026-06-10T12:00:00.000Z"
  },
  "meta": { "correlation_id": "corr_..." },
  "error": null
}
```

**Response 409 — legal hold blocks erasure fulfillment**

```json
{
  "data": null,
  "meta": { "correlation_id": "corr_..." },
  "error": {
    "code": "CONFLICT",
    "message": "This action conflicts with the current state. Refresh and retry.",

    "detail": {
      "reason": "LEGAL_HOLD",
      "explanation": "One or more active retention policies have legal_hold=true for this data category."
```

## Data Operations

All queries use Kysely parameterised builders. Table and column names are exact from `docs/data-model/schema.sql`.

### List requests (GET /data-rights)

```typescript
// DataRightsRepository.findAll
const rows = await db
  .selectFrom('data_rights_requests as drr')
  .selectAll('drr')
  .where('drr.org_id', '=', orgId)
  .$if(status !== undefined, (qb) => qb.where('drr.status', '=', status!))
  .$if(requestType !== undefined, (qb) => qb.where('drr.request_type', '=', requestType!))
  .$if(customerProfileId !== undefined, (qb) => qb.where('drr.customer_profile_id', '=', customerProfileId!))
  .$if(dueBefore !== undefined, (qb) => qb.where('drr.due_at', '<=', dueBefore!))
  .orderBy('drr.due_at', 'asc')
  .limit(limit)   // max 100; default 25
  .offset((page - 1) * limit)
  .execute();
```

Count query for pagination `total` uses the same filters with `select(db.fn.count('drr.data_rights_request_id').as('count'))`.

### Insert new request (POST /data-rights)

```typescript
// DataRightsRepository.create — called inside UnitOfWork
const record = await db
  .insertInto('data_rights_requests')
  .values({
    org_id: orgId,
    customer_profile_id: dto.customerProfileId,
    lead_id: dto.leadId ?? null,
    request_type: dto.requestType,
    status: 'open',
    owner_id: null,
    due_at: slaEngine.calculateDue('data_rights', now, calendarId),
    created_by: actorId,
    updated_by: actorId,
  })
  .returningAll()
  .executeTakeFirstOrThrow();
```

`due_at` is computed by `SlaEngine` using the org's default `BusinessCalendar`. There is no dedicated `sla_target` enum value for data-rights SLA (see Ambiguities); the SLA threshold is resolved from the `sla_policies` table by `applies_to='grievance'` as a proxy, or from a hard-coded default of 30 calendar days if no matching policy exists (see Ambiguities).

### Update request (PATCH /data-rights/{id})

```typescript
// DataRightsRepository.update — called inside UnitOfWork
const updated = await db
  .updateTable('data_rights_requests')
  .set({
    status: dto.status,
    disposition: dto.disposition ?? existing.disposition,
    owner_id: dto.ownerId ?? existing.owner_id,
    updated_by: actorId,
    // updated_at handled by trg_data_rights_requests_updated_at trigger
  })
  .where('data_rights_request_id', '=', requestId)
  .where('org_id', '=', orgId)
  .returningAll()
  .executeTakeFirstOrThrow();
```

### Legal-hold check (for `erasure` fulfillment)

Before allowing `status='fulfilled'` on an `erasure` request, the service queries:

```typescript
// DataRightsService — legalHoldCheck
const holdPolicies = await db
  .selectFrom('retention_policies')
  .select(['retention_policy_id', 'data_category', 'legal_hold'])
  .where('org_id', '=', orgId)
  .where('legal_hold', '=', true)
  .where('is_active', '=', true)
  .limit(1)
  .executeTakeFirst();

if (holdPolicies) {
  throw new ConflictException({
    code: 'CONFLICT',
    detail: { reason: 'LEGAL_HOLD', explanation: '...' },
  });
}
```

If the customer's data is linked to a lead whose outcome matches a legal-hold retention policy, the fulfilment is blocked. The DPO must instead transition to `rejected_retained` and document the basis in `disposition`.

## Error Cases

All errors use the uniform envelope `{ data: null, meta: { correlation_id }, error: { code, message, retryable, fields?, detail? } }`.

| Scenario | HTTP | Code | detail.reason | Notes |
|---|---|---|---|---|
| Missing/invalid JWT | 401 | `AUTH_REQUIRED` | — | Global `JwtAuthGuard` |
| Non-DPO calls PATCH | 403 | `FORBIDDEN` | — | `AbacGuard` denies consent_ledger|
| Invalid `request_type` | 400 | `VALIDATION_ERROR` | — | `fields: [{ field: "request_type", issue: "..." }]` |
| `disposition` omitted when finalising | 400 | `VALIDATION_ERROR` | — | `fields: [{ field: "disposition", issue: "..." }]` |
| `data_rights_request_id` not found | 404 | `NOT_FOUND` | — | Record absent or wrong org |
| Erasure blocked by legal hold | 409 | `CONFLICT` | `LEGAL_HOLD` | Active `retention_policies.legal_hold=true` |
| Invalid state transition (e.g. `fulfilled → open`) | 409 | `CONFLICT` | — | State machine guard |
| Replayed `Idempotency-Key` on POST | 200 | (original result) | `IDEMPOTENT_REPLAY` | Returns original 201 data; no duplicate row |
| Rate limit exceeded | 429 | `RATE_LIMITED` | — | Mutation limit 60/min/user |
| DB failure / unhandled exception | 500 | `INTERNAL_ERROR` | — | Full error logged internally; no stack in response |

## Backend Flow

### Create flow (POST /data-rights or POST /c/{token}/data-rights)

1. **Guard layer:** `JwtAuthGuard` (or `CustomerLinkGuard` for customer path) validates the principal.
2. **ABAC check:** `AbacGuard` calls `EntitlementService.can(user, 'consent_ledger', { orgId })`. DPO passes with scope A; a non-DPO non-customer call returns 403.
3. **Idempotency check:** `DataRightsService.create` checks Redis for the `Idempotency-Key`. If already present, returns the cached 201 result with `IDEMPOTENT_REPLAY`.
4. **Validation:** Zod `CreateDataRightsDto` validation (by global `ValidationPipe`); returns 400 with `fields[]` on failure.
5. **Customer-link scope check:** If the path is `/c/{token}/data-rights`, verify `lead_id` (if supplied) matches the token's bound `lead_id`; 400 `VALIDATION_ERROR` if mismatch.
6. **SLA calculation:** `SlaEngine.calculateDue('data_rights', now, calendarId)` — computes `due_at`. Falls back to now + 30 calendar days if no dedicated policy entry exists (see Ambiguities).
7. **Transaction (UnitOfWork):**
   ```
   await uow.run(async (tx) => {
     a. DataRightsRepository.create(dto, actorId, orgId, dueAt, tx)
        → INSERT INTO data_rights_requests ... RETURNING *
     b. AuditAppender.emit({ action: 'consent_grant', entityType: 'DataRightsRequest', entityId, actorId, detail: { request_type }, ... }, tx)
        // NOTE: 'consent_grant' is the closest available audit_action enum value (see Ambiguity #4).
        // The schema's audit_action enum does not include a data-rights-specific value.
     c. OutboxService.emit('DATA_RIGHT_REQUEST', { dataRightsRequestId, requestType, customerId }, tx)
   });
   ```
8. **Post-commit:** Cache `Idempotency-Key` in Redis (TTL 24 h) with the response body. `NotificationDispatchService` sends an in-app/email notification to DPO (fire-and-forget, handled by outbox relay).
9. **Response:** 201 Created with the inserted record in the uniform envelope.

### Process flow (PATCH /data-rights/{id})

1. **Guard layer:** `JwtAuthGuard`.
2. **ABAC check:** `EntitlementService.can(user, 'consent_ledger', { orgId })` — DPO only; 403 otherwise.
3. **Load record:** `DataRightsRepository.findById(id, orgId)` — 404 if absent.
4. **State machine validation:** `DataRightsStateMachine.validateTransition(current.status, dto.status)` — 409 `CONFLICT` if invalid (e.g. `fulfilled → open`).
5. **Validation:** Zod `UpdateDataRightsDto` — 400 if `disposition` missing when status finalises.
6. **Legal-hold check (erasure only):** If `existing.request_type === 'erasure'` AND `dto.status === 'fulfilled'`:
   - Query `retention_policies` for any active `legal_hold=true` row applicable to the customer's data.
   - If found: throw 409 `CONFLICT` with `detail.reason=LEGAL_HOLD`. The DPO must transition to `rejected_retained` instead.
7. **Transaction (UnitOfWork):**
   ```
   await uow.run(async (tx) => {
     a. DataRightsRepository.update({ id, status, disposition, ownerId, updatedBy }, tx)
        → UPDATE data_rights_requests SET status=?, disposition=?, owner_id=?, updated_by=? WHERE data_rights_request_id=? AND org_id=?
     b. AuditAppender.emit({ action: 'consent_grant', entityType: 'DataRightsRequest', entityId, actorId, detail: { from_status, to_status } }, tx)
        // NOTE: same audit_action caveat as above (see Ambiguity #4).
     c. If status === 'fulfilled' AND request_type === 'erasure':
        OutboxService.emit('DATA_RIGHT_REQUEST', { dataRightsRequestId, customerId, leadId, subType: 'ERASURE_APPROVED' }, tx)
        // 'DATA_RIGHT_REQUEST' is the only event_code in the schema enum for this entity (see Ambiguity #5).
        // FR-115 RetentionEngine filters by payload.subType='ERASURE_APPROVED'.
     d. Else:
        OutboxService.emit('DATA_RIGHT_REQUEST', { dataRightsRequestId, status, subType: 'UPDATED' }, tx)
   });
   ```
8. **Response:** 200 OK with updated record.

### List flow (GET /data-rights)

1. **Guard:** `JwtAuthGuard` + `AbacGuard` (`consent_ledger`, scope A for DPO).
2. **Query:** `DataRightsRepository.findAll(filters, pagination)` with all filters applied and LIMIT enforced (max 100).
3. **Response:** 200 with paginated list.

## State Machine

From `docs/contracts/state-machines.md` — Compact machines:

```
DataRightsRequest — rights_status

open → in_review        (DPO takes ownership; sets owner_id)
in_review → fulfilled   (DPO completes; erasure requires no legal hold)
in_review → rejected_retained  (erasure blocked by legal hold or business reason; disposition required)
open → rejected_retained  (DPO may immediately reject if legal hold obvious)

Invalid transitions (→ 409 CONFLICT):
  fulfilled → any
  rejected_retained → any
  in_review → open   (backwards)
```

Transition validation is performed by `DataRightsStateMachine.validateTransition(from, to)` before any DB write.

---

## Transaction Boundaries

Every mutating operation uses `UnitOfWork.run(async (tx) => { ... })`. The tx boundary covers:

**POST (create):**
- `INSERT` into `data_rights_requests`
- `AuditAppender.emit(...)` — intent queued in same tx
- `OutboxService.emit('DATA_RIGHT_REQUEST', ..., tx)` — `INSERT` into `event_outbox`

**PATCH (process):**
- `UPDATE` on `data_rights_requests`
- `AuditAppender.emit(...)` intent
- `OutboxService.emit(event, ..., tx)` — `INSERT` into `event_outbox`

If any step fails, the entire transaction rolls back. No partial state (e.g. a request marked `fulfilled` without an outbox event) can persist.

The `AuditChainConsumer` (single-writer, `max-instances=1`) reads from the audit intent queue and appends the hash chain row to `audit_logs` post-commit. This preserves the tamper-evident chain without concurrent-write risk (architecture §11.4).

## External Service Calls via IntegrationGateway

FR-112 does not call any external provider directly. The only infrastructure service used is:

- **`SlaEngine` + `BusinessCalendarService`** (internal, `core/sla/`) — to compute `due_at` for new requests.
- **`OutboxService`** (internal, `core/outbox/`) — the `DATA_RIGHT_ERASURE_APPROVED` outbox event is the signal to FR-115's `RetentionEngine`. FR-115 acts asynchronously; FR-112 does not block on it.

No `IntegrationGateway` port calls are made within this FR.

## Assumptions

1. The `sla_policies` table has no `sla_target` value for `data_rights` (the enum only covers `first_contact`, `document`, `kyc_exception`, `grievance`, `handoff_retry`). FR-112 uses `grievance` policy if one is configured, otherwise falls back to a hardcoded 30-calendar-day default. This assumption must be confirmed by the DPO/product team; if a dedicated SLA policy is required, a new `sla_target` enum value must be added via a Flyway migration.

2. The legal-hold check queries `retention_policies` for any `legal_hold=true` and `is_active=true` row in the org. A more granular per-customer-profile or per-lead-outcome check is not specified in the BRD; the conservative approach (any active legal-hold policy blocks erasure) is used. Product/compliance to confirm scope.

3. FR-115's `RetentionEngine` is the sole executor of the actual anonymisation/purge. FR-112 signals it via the `DATA_RIGHT_ERASURE_APPROVED` outbox event. The interface between FR-112 and FR-115 is the outbox payload; the RetentionEngine implementation is out of scope for this FR.

4. The customer-link path (`POST /c/{token}/data-rights`) is listed in `auth-matrix.json`'s `public_endpoints` as `POST /api/v1/c/{token}/callback` and related paths. The data-rights path is not explicitly listed; it should follow the same pattern (CustomerLinkGuard, OTP step-up). If this path must be added to `auth-matrix.json#public_endpoints`, that change must be made before this FR is implemented.

## Ambiguities

1. **SLA target for data-rights requests:** The `sla_target` enum in `schema.sql` does not include a `data_rights` value. The LLD assumes a fallback of 30 calendar days if no matching `sla_policies` row exists. This must be resolved before implementation.

2. **Legal-hold granularity:** BRD says "erasure under legal hold → rejected_retained". The schema stores legal hold at the `retention_policies` (data_category × lead_outcome) level, not at the individual customer-profile level. Whether the check is org-wide or customer/lead-scoped needs product/compliance confirmation.

3. **Customer self-raise via customer link:** The `public_endpoints` list in `auth-matrix.json` does not include `POST /api/v1/c/{token}/data-rights`. The BRD mentions customer-link intake for rights requests (BRD §2312). This endpoint should be added to `auth-matrix.json` before implementation.

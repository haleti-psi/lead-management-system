## Summary

FR-052 implements the Pipeline Board — a Kanban-style view of leads grouped by `lead_stage`. It has two backend surfaces:

1. **Board load** — the frontend reuses `GET /api/v1/leads` (FR-050) with `filter[stage]=<value>` per column; no new GET endpoint is needed.
2. **Stage transition** — `PATCH /api/v1/leads/{id}/stage` moves a lead from its current stage to a caller-specified `toStage`, enforcing §10.3 guards via `StageGuardService`, then writing `leads` + `stage_history` + `audit_logs` + `event_outbox` atomically via `UnitOfWork`.

The frontend renders a multi-column Kanban board (desktop) and a stage-selector sheet (mobile). Card fields include ageing, product, requested amount, source, owner, hot flag, consent/KYC status, and next action. Drag-and-drop triggers the PATCH; guard failures snap the card back with a `Toast`. Concurrent moves use optimistic locking: stale `expectedVersion` → `CONFLICT`.

No external service is called directly; post-commit side effects (SLA timer reset, notifications) are dispatched by the owning services that subscribe to the `LEAD_STAGE_CHANGED` outbox event.

## Endpoints

### 1. Board load — GET /api/v1/leads (reuse of FR-050)

The Kanban board loads each column by calling `GET /api/v1/leads` with `filter[stage]=<stageName>`, `limit=25`, `page=1`. Pagination per column is supported via the standard `page`/`limit` parameters. No dedicated board-fetch endpoint is introduced; the board is a UI view over the scope-filtered lead list.

**Query parameters used by the board:**

| Parameter | Example | Notes |

### 2. Stage transition — PATCH /api/v1/leads/{id}/stage

**Request:**

```
PATCH /api/v1/leads/{id}/stage
Authorization: Bearer <jwt>
Content-Type: application/json

{
  "toStage": "qualified",
  "expectedVersion": 3,
  "reason": "Customer confirmed product interest"
}
```

| Field | Type | Required | Rule |
|---|---|---|---|
| `toStage` | `lead_stage` enum | yes | Must be a valid `lead_stage` value from `@shared/enums` |
| `expectedVersion` | `integer >= 1` | yes | Optimistic lock; must match `leads.version` |
| `reason` | `string` | no | Max 500 chars; required for `rejected` and `dormant` transitions per §10.3 |

**Response (200) — success:**

```json
{
  "data": {
    "leadId": "uuid",
    "leadCode": "LD-2026-000042",
    "stage": "qualified",
    "version": 4,
    "updatedAt": "2026-06-09T07:15:00Z"
  },
  "meta": { "correlation_id": "corr_xyz" },
  "error": null
}
```

**Response (400) — STAGE_GUARD_FAILED:**

```json
{
  "data": null,
  "meta": { "correlation_id": "corr_xyz" },
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Stage transition blocked by one or more guards.",
    "retryable": false,
    "detail": {
      "reason": "STAGE_GUARD_FAILED",
      "failed_guards": ["kyc_signoff", "mandatory_docs"]
    }
  }
}
```

**Response (403) — FORBIDDEN (out of scope / capability missing):**

```json
{
  "data": null,
  "meta": { "correlation_id": "corr_xyz" },
  "error": { "code": "FORBIDDEN", "message": "You don't have access to this.", "retryable": false }
}
```

**Response (409) — CONFLICT (stale optimistic lock):**

```json
{
  "data": null,
  "meta": { "correlation_id": "corr_xyz" },
  "error": {
    "code": "CONFLICT",
    "message": "This action conflicts with the current state. Refresh and retry.",
    "retryable": false
  }
}
```

**Response (404) — NOT_FOUND:**

```json
{
  "data": null,
  "meta": { "correlation_id": "corr_xyz" },
  "error": { "code": "NOT_FOUND", "message": "We couldn't find that item.", "retryable": false }
}
```

## Auth Check

| Layer | Guard / Decorator | Detail |
|---|---|---|
| Authentication | `JwtAuthGuard` (global) | Validates JWT; loads user + ABAC attributes onto request context. Missing/expired JWT → `AUTH_REQUIRED` (401). |
| Authorization | `@Requires('move_stage', scopeResolver)` + `AbacGuard` | Calls `EntitlementService.can(user, 'move_stage', lead)`. Denied → `FORBIDDEN` (403) + audit. |
| Scope resolution | `scopeResolver` | RM: `scope=O` — only leads where `leads.owner_id = user.userId`. BM: `scope=B` — `leads.branch_id = user.branchId`. SM: `scope=T` — `leads.team_id = user.teamId`. HEAD does not have `move_stage` capability in the auth-matrix. |
| Role check per transition | `StageGuardService` | Validates `allowedRoles` for the specific `fromStage → toStage` pair per §10.3. A role that is not permitted for a specific transition receives `VALIDATION_ERROR` + `STAGE_GUARD_FAILED`. |

**Board load (GET /leads) auth:** governed by `view_lead` capability (FR-050). Same scope rules apply. The board is read-only from an auth perspective; the PATCH triggers the authorization check for `move_stage`.

## Data Operations

### Board load — SELECT (Kysely)

The board calls the existing `LeadRepository.list(filters, pagination, tx)` from FR-050. No separate query is written. The service receives `filter.stage` from the query params and adds:

```typescript
// apps/api/src/modules/workspace/pipeline-board.service.ts
// Calls LeadRepository.list — exact Kysely fragment inside the repository:
db.selectFrom('leads as l')
  .innerJoin('lead_identities as li', 'li.lead_identity_id', 'l.lead_identity_id')
  .innerJoin('users as u', 'u.user_id', 'l.owner_id')
  .leftJoin('tasks as t', (join) =>
    join
      .onRef('t.lead_id', '=', 'l.lead_id')
      .on('t.status', '=', 'open')
  )
  .select([
    'l.lead_id', 'l.lead_code', 'l.stage', 'l.product_code',
    'l.requested_amount', 'l.is_hot', 'l.consent_status', 'l.kyc_status',
    'l.owner_id', 'l.sla_first_contact_due_at', 'l.created_at', 'l.version',
    'li.name as customer_name',
    'u.full_name as owner_name',
    db.fn.min('t.next_action_at').as('next_action_at'),
  ])
  .where('l.org_id', '=', orgId)
  .where('l.stage', '=', filters.stage)             // parameterised
  .where('l.deleted_at', 'is', null)
  // scope filter injected by EntitlementService:
  // RM: .where('l.owner_id', '=', userId)
  // BM: .where('l.branch_id', '=', branchId)
  // SM: .where('l.team_id', '=', teamId)
  .groupBy(['l.lead_id', 'li.lead_identity_id', 'u.user_id'])
  .orderBy('l.created_at', 'desc')
  .limit(pagination.limit)   // max 100 enforced by PaginationParams schema
  .offset((pagination.page - 1) * pagination.limit)
```

### Stage transition — UPDATE + 3 INSERTs (Kysely, one UnitOfWork transaction)

All four writes happen inside a single `UnitOfWork.run(async (tx) => { ... })`.

**Step 1 — Lock and load lead:**

```typescript
const lead = await db
  .selectFrom('leads')
  .select(['lead_id', 'stage', 'version', 'org_id', 'owner_id', 'branch_id', 'team_id'])
  .where('lead_id', '=', leadId)
  .where('org_id', '=', orgId)
  .where('deleted_at', 'is', null)
  .forUpdate()           // row-level lock within the transaction
  .executeTakeFirst();
// If not found → NOT_FOUND (404)
// If out of scope → FORBIDDEN (403)
```

**Step 2 — Guard evaluation (outside the DB, inside the tx boundary):**

```typescript
const guardResult = await stageGuardService.evaluate({
  fromStage: lead.stage,
  toStage: dto.toStage,
  lead,
  actor: requestUser,
  tx,
});
// If guardResult.failed.length > 0 → VALIDATION_ERROR + STAGE_GUARD_FAILED
```

**Step 3 — Update leads (optimistic lock):**

```typescript
const updated = await db
  .updateTable('leads')
  .set({
    stage: dto.toStage,
    version: sql`version + 1`,
    updated_at: new Date(),
    updated_by: requestUser.userId,
  })
  .where('lead_id', '=', leadId)
  .where('version', '=', dto.expectedVersion)   // optimistic lock predicate
  .returning(['lead_id', 'lead_code', 'stage', 'version', 'updated_at'])
  .executeTakeFirst();
// If updated is undefined → CONFLICT (409) — stale version
```

**Step 4 — Insert stage_history (append-only):**

```typescript
await db
  .insertInto('stage_history')
  .values({
    stage_history_id: generateUuid(),
    org_id: orgId,
    lead_id: leadId,
    from_stage: lead.stage,
    to_stage: dto.toStage,
    actor_id: requestUser.userId,
    reason: dto.reason ?? null,
    occurred_at: new Date(),
    created_at: new Date(),
    updated_at: new Date(),
  })
  .execute();
```

**Step 5 — Emit audit via AuditAppender:**

```typescript
await auditAppender.append({
  action: 'stage_transition',
  entityType: 'leads',
  entityId: leadId,
  leadId,
  detail: { fromStage: lead.stage, toStage: dto.toStage, reason: dto.reason },
  actorId: requestUser.userId,
  ipDevice: requestContext.ipDevice,
}, tx);
```

**Step 6 — Enqueue outbox event via OutboxService:**

```typescript
await outboxService.emit({
  eventCode: 'LEAD_STAGE_CHANGED',
  aggregateType: 'leads',
  aggregateId: leadId,
  payload: {
    leadId,
    fromStage: lead.stage,
    toStage: dto.toStage,
    actorId: requestUser.userId,
    occurredAt: new Date().toISOString(),
  },
}, tx);
```

All six steps execute within the same `tx`. If any step throws, the entire transaction rolls back — no partial state.

## Validation Logic

Validated with Zod at the `ValidationPipe` boundary before reaching the service.

### PATCH /leads/{id}/stage — StageTransitionDto

| Field | Zod rule | Error message |
|---|---|---|
| `id` (path) | `z.string().uuid()` | "Invalid lead ID format." |
| `toStage` | `z.nativeEnum(LeadStage)` (from `@shared/enums`) | "toStage must be a valid lead stage value." |
| `expectedVersion` | `z.number().int().min(1)` | "expectedVersion must be a positive integer." |
| `reason` | `z.string().max(500).optional()` | "reason must not exceed 500 characters." |
| `reason` required when | `toStage === 'rejected'` or `toStage === 'dormant'` — enforced via Zod `.refine()` | "reason is required for rejected and dormant transitions." |

All Zod failures produce `VALIDATION_ERROR` (400) with `error.fields[]` listing each failing field.

### Board filter — BoardColumnQueryDto

| Field | Zod rule |
|---|---|
| `stage` | `z.nativeEnum(LeadStage).optional()` |
| `page` | `z.coerce.number().int().min(1).default(1)` |
| `limit` | `z.coerce.number().int().min(1).max(100).default(25)` |

## Error Cases

| Error code | HTTP | Condition | detail.reason |
|---|---|---|---|
| `AUTH_REQUIRED` | 401 | Missing or expired JWT on any board/PATCH request | — |
| `FORBIDDEN` | 403 | `EntitlementService.can()` returns false (wrong scope, missing capability) | — |
| `NOT_FOUND` | 404 | `lead_id` does not exist or `deleted_at IS NOT NULL` for this org | — |
| `VALIDATION_ERROR` | 400 | Zod field validation failure (bad `toStage` enum, missing `reason`, non-integer `expectedVersion`) | — (uses `fields[]`) |
| `VALIDATION_ERROR` | 400 | `StageGuardService` evaluation returns one or more failing guards | `STAGE_GUARD_FAILED` in `error.detail.reason`; `failed_guards[]` lists names |
| `CONFLICT` | 409 | `UPDATE ... WHERE version = expectedVersion` matches zero rows (optimistic lock — concurrent edit) | — |
| `INTERNAL_ERROR` | 500 | Unhandled exception in any step (DB down, unexpected error) | — |

**No `UPSTREAM_UNAVAILABLE` (503)** is raised by FR-052 directly. Post-commit notifications and SLA-timer resets are asynchronous (Pub/Sub consumers) and do not block or fail the PATCH response.

## Backend Flow (Step-by-Step)

```
PATCH /api/v1/leads/{id}/stage
  │
  ├─ 1. CorrelationMiddleware  → assigns / propagates X-Correlation-Id
  ├─ 2. JwtAuthGuard           → validates JWT; populates req.user (userId, role, branchId, teamId, partnerId)
  ├─ 3. AbacGuard              → EntitlementService.can(user, 'move_stage', {leadId})
  │       └─ scope filter resolved; FORBIDDEN (403) on deny
  ├─ 4. ValidationPipe (Zod)   → validates path UUID + StageTransitionDto body
  │       └─ VALIDATION_ERROR (400) with fields[] on failure
  │
  ├─ 5. WorkspaceController.transitionStage(id, dto, req)
  │       └─ delegates to PipelineBoardService.transitionStage(id, dto, requestUser)
  │
  ├─ 6. PipelineBoardService.transitionStage
  │   ├─ 6a. UnitOfWork.run(async (tx) => {
  │   │   ├─ 6b. LeadRepository.findByIdForUpdate(id, orgId, tx)
  │   │   │       → SELECT ... FOR UPDATE
  │   │   │       → NOT_FOUND (404) if no row
  │   │   │       → FORBIDDEN (403) if lead is out of actor's scope
  │   │   │
  │   │   ├─ 6c. StageGuardService.evaluate({ fromStage, toStage, lead, actor, tx })
  │   │   │       → evaluates §10.3 guard matrix
  │   │   │       → VALIDATION_ERROR + STAGE_GUARD_FAILED (400) if any guard fails
  │   │   │         (error includes failed_guards[])
  │   │   │
  │   │   ├─ 6d. LeadService.transitionStage(leadId, toStage, guardCtx, expectedVersion, tx)
  │   │   │   ├─ UPDATE leads SET stage=:toStage, version=version+1 WHERE lead_id=:id AND version=:v
  │   │   │   │   → CONFLICT (409) if 0 rows updated (optimistic lock miss)
  │   │   │   ├─ INSERT INTO stage_history (...)
  │   │   │   ├─ AuditAppender.append(stage_transition, ...)
  │   │   │   └─ OutboxService.emit(LEAD_STAGE_CHANGED, ...)
  │   │   │
  │   │   └─ 6e. tx commits (all 4 writes in one DB transaction)
  │   │       → INTERNAL_ERROR (500) on DB error (full rollback)
  │   │
  │   └─ 6f. Return updated lead projection
  │
  ├─ 7. ResponseEnvelopeInterceptor → wraps result in { data, meta, error:null }
  └─ 8. Return HTTP 200
```

**Post-commit (async, non-blocking):**
- `LEAD_STAGE_CHANGED` outbox row picked up by outbox publisher → Pub/Sub topic.
- SLA engine consumer resets / starts SLA timers per §10.3 side effects (FR-104).
- `NotificationDispatchService` sends in-app / email notifications (FR-101) per §11.1 event matrix.

## State Machine and Transaction Boundaries

FR-052 drives the `Lead` state machine (`leads.stage`, `lead_stage` enum). The full valid-transition table is in `docs/contracts/state-machines.md §Lead`.

**Permitted transitions (roles in parentheses):**

| From | To | Allowed roles | Guard names |
|---|---|---|---|
| `captured` | `assigned` | system, BM, SM | `valid_branch_product_source` |
| `assigned` | `contacted` | RM, BM | `contact_logged` |
| `contacted` | `qualified` | RM, BM | `intent_captured`, `progressive_fields` |
| `qualified` | `documents_pending` | RM, BM | `checklist_generated` |
| `documents_pending` | `kyc_in_progress` | RM, KYC, BM | `mandatory_docs_or_waiver` |
| `kyc_in_progress` | `eligibility_requested` | RM, BM, KYC | `kyc_sufficient`, `consent_eligibility` |
| `eligibility_requested` | `ready_for_handoff` | system, BM, KYC | `eligibility_received`, `docs_kyc_ready` |
| `ready_for_handoff` | `handed_off` | BM, KYC, RM (delegated) | `consent_present`, `duplicate_clear`, `mandatory_docs_verified`, `kyc_signoff`, `valid_payload` |
| any active | `rejected` | RM, BM, SM | `rejection_reason_provided` |
| `rejected` | prior active stage | BM, SM, RM | `within_reopen_window`, `reason_provided` |
| any active | `dormant` | RM, BM, SM | `nurture_reason`, `next_followup_date` |
| `dormant` | `assigned`/`contacted` | RM, BM, SM, system | `followup_due_or_reactivation` |

**Invalid transitions → `CONFLICT` (409):**
- `handed_off` → any (terminal state).
- Skip-ahead (e.g. `captured` → `qualified`).
- `rejected` → active outside the reopen window.

**Transaction boundary:**

All four writes (`leads` UPDATE, `stage_history` INSERT, `audit_logs` INSERT via AuditAppender, `event_outbox` INSERT via OutboxService) execute inside a **single `UnitOfWork` transaction**. If any write fails, the entire transaction rolls back. No partial state can persist.

The `AuditAppender` inserts the audit intent row within the transaction; the `AuditChainConsumer` (single-writer, `max-instances=1`) computes the hash chain asynchronously — it does not block the PATCH response.

### Component responsibilities

**`KanbanBoard`**
- Manages per-column TanStack Query queries (`useQuery` per stage, keyed by `['leads', { stage }]`).
- Provides the `DndContext` (drag-and-drop context) with `prefers-reduced-motion` awareness.
- On drag-start: records source column + card version.
- On drag-end: calls `handleStageDrop(leadId, fromStage, toStage, expectedVersion)`.

**`KanbanColumn`**
- Renders the stage label, lead count badge, and the vertical list of `LeadCard` components.
- Acts as a drop-target; accepts cards from other columns only if the `toStage` is a valid transition from the card's `fromStage` (checked client-side before the API call for instant feedback).

**`LeadCard`**
- Displays: `leadCode`, masked `customerName` via `<MaskedField>`, `productCode`, `requestedAmount` (INR), `ageingDays` badge, `isHot` flame icon, `<StatusChip consentStatus>`, `<StatusChip kycStatus>`, `ownerName`, `nextActionAt`.
- SLA breach highlight: if `slaFirstContactDueAt` is in the past, the card border turns `--destructive`.
- Drag handle with `aria-label="Drag lead {leadCode} to another stage"`.

**`MobileStageSelectorSheet`**
- Triggered from a "Move Stage" button on the `LeadCard` (mobile only).
- Presents a bottom sheet with a `<Select>` of allowed next stages (computed client-side from the state machine map).
- On confirm: calls the same `transitionStage` mutation as the desktop drag-and-drop path.

**`StageGuardErrorToast`**
- On `VALIDATION_ERROR` + `STAGE_GUARD_FAILED`: renders a `<Toast>` listing the `failed_guards` in human-readable form.
- The `KanbanBoard` reverts the optimistic card position (TanStack Query `onError` callback cancels the mutation and restores prior query cache state).

**Optimistic updates:**
- On drag start, TanStack Query `useMutation` performs an optimistic cache update (moves the card in the cache immediately).
- `onError`: cache is rolled back to the pre-mutation snapshot; `Toast` shows the error.
- `onSuccess`: cache is invalidated for both the source and destination column queries to ensure the server state is reflected.

# FR-103: Notification Preference & Opt-Out Centre

**Tier: 2**

```yaml

module: M11 (Engagement)
depends_on:
  data_model_tables:
    - notification_preferences
    - audit_logs
    - users
    - customer_profiles
  contracts_endpoints:
    - "PUT /api/v1/preferences"
  auth_matrix:
    roles: [RM, BM, CUSTOMER]
    capability: customer_comm
    scopes: [O, B, C]
  state_machines: []
last_updated: "2026-06-09"
version: 1
```

## Summary

FR-103 implements the notification preference and opt-out centre for Module M11 (Engagement). It allows:

- **RM / BM** (internal staff): upsert notification preferences on behalf of a customer or for themselves as a `user` subject.
- **CUSTOMER** (self-service via customer link token): upsert their own preferences through the `/c/{token}/preferences` customer-link route.

The core operation is a single `PUT /api/v1/preferences` endpoint (and its customer-link mirror) that performs an upsert on `notification_preferences` using the `UNIQUE (subject_type, subject_ref, channel, purpose)` constraint. Changes are immediately effective — `NotificationDispatchService` (FR-101) reads this table before every send.

Every write is audited via `AuditAppender`. There are no external service calls and no lead-stage transitions. The FR does not write to `leads`; LeadService is not involved.

The preference centre has two surfaces: an internal view (RM/BM manages on behalf of a subject) and a customer-facing panel accessible from the customer link micro-site.

## Endpoints

### 1. PUT /api/v1/preferences

**Summary:** Upsert one or more notification preferences for a given subject (batch upsert).

**Auth:** `JwtAuthGuard` (global) + `AbacGuard` + `@Requires('customer_comm', scopeResolver)`.

**Request**

| Parameter | In | Type | Required | Notes |
|---|---|---|---|---|
| `subject_type` | body | `subject_type` enum | Yes | `user` or `customer` |
| `subject_ref` | body | UUID | Yes | `user_id` (if `user`) or `customer_profile_id` (if `customer`) |
| `preferences` | body | array of preference items | Yes | 1–50 items per request |
| `preferences[].channel` | body | `comm_channel` enum | Yes | `in_app`, `email`, `sms`, `whatsapp` |
| `preferences[].purpose` | body | `consent_purpose` enum | Yes | See enum values |
| `preferences[].opted_in` | body | boolean | Yes | `true` = opted in; `false` = opted out |

**Request body example**

```json
{
  "subject_type": "customer",
  "subject_ref": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "preferences": [
    { "channel": "whatsapp", "purpose": "marketing", "opted_in": false },
    { "channel": "sms",      "purpose": "document_processing", "opted_in": true }
  ]
}
```

**Response 200**

```json
{
  "data": {
    "subject_type": "customer",
    "subject_ref": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
    "preferences": [
      {
        "notification_preference_id": "uuid",
        "channel": "whatsapp",
        "purpose": "marketing",
        "opted_in": false,
        "updated_at": "2026-06-10T09:00:00+05:30"
      },
      {
        "notification_preference_id": "uuid",
        "channel": "sms",
        "purpose": "document_processing",
        "opted_in": true,
        "updated_at": "2026-06-10T09:00:00+05:30"
      }
    ]
  },
  "meta": { "correlation_id": "corr_abc123" },
  "error": null
}
```

**Error responses**

| HTTP | Code | Condition |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Enum value invalid, array empty or > 50 items, UUID malformed |
| 401 | `AUTH_REQUIRED` | Missing or expired JWT |
| 403 | `FORBIDDEN` | Caller does not have `customer_comm` capability, or scope check fails |
| 429 | `RATE_LIMITED` | Mutations > 60/min per user |
| 500 | `INTERNAL_ERROR` | Unhandled error |

### 2. PUT /api/v1/c/{token}/preferences  (customer self-service)

**Summary:** Customer upserts their own notification preferences via an opaque link token.

**Auth:** `@Public()` + `CustomerLinkGuard` (opaque token + OTP step-up). This route is NOT protected by `JwtAuthGuard`. The guard resolves the `customer_profile_id` associated with the token and enforces that the `subject_ref` in the body matches.

**Request / Response:** Identical body and response shape to the internal endpoint. `subject_type` must be `customer`; `subject_ref` must equal the `customer_profile_id` bound to the token (guard rejects mismatches with `FORBIDDEN`).

**Additional error**

| HTTP | Code | Condition |
|---|---|---|
| 403 | `FORBIDDEN` | Token expired/revoked/used, OTP step-up not completed, or `subject_ref` does not match the token's lead/customer |
| 429 | `RATE_LIMITED` | Customer-link endpoints: 10/min per IP |

## Auth Check

### Internal callers (JWT)

1. `JwtAuthGuard` validates the Bearer JWT. Missing/expired → `AUTH_REQUIRED` (401).
2. `AbacGuard` with `@Requires('customer_comm', scopeResolver)`:
   - `EntitlementService.can(user, 'customer_comm', resource)` evaluates the caller's role against `auth-matrix.json`.
   - RM: scope `O` — may manage preferences only for customers on their own assigned leads.
   - BM: scope `B` — may manage preferences for customers in their branch.
   - SM: scope `T` — may manage preferences for customers in their team.
   - HEAD/ADMIN/DPO: `customer_comm` capability per matrix.
   - If the caller attempts to set preferences for a `subject_ref` outside their scope → `FORBIDDEN` (403).
3. `subject_type=user` (managing an internal user's own preferences): the caller must be that user themselves, or a BM/ADMIN managing within scope.

### Customer-link callers

1. Route is decorated `@Public()` — `JwtAuthGuard` skips.
2. `CustomerLinkGuard` validates the opaque token:
   - Token must exist in `customer_links`, `status='active'`, not expired.
   - OTP step-up must be complete for this session.
   - Resolves `customer_profile_id` from the link.
3. Guard enforces `subject_type='customer'` and `subject_ref == customer_profile_id`. Mismatch → `FORBIDDEN` (403).

## Data Operations

### Upsert — `notification_preferences`

For each item in `preferences[]`, execute a Kysely upsert on the unique constraint `uq_notif_pref (subject_type, subject_ref, channel, purpose)`:

```typescript
// Executed inside UnitOfWork transaction for the whole batch
for (const item of dto.preferences) {
  await tx
    .insertInto('notification_preferences')
    .values({
      org_id:       caller.org_id,
      subject_type: dto.subject_type,
      subject_ref:  dto.subject_ref,
      channel:      item.channel,
      purpose:      item.purpose,
      opted_in:     item.opted_in,
      created_by:   actorUserId,
      updated_by:   actorUserId,
    })
    .onConflict((oc) =>
      oc.constraint('uq_notif_pref').doUpdateSet({
        opted_in:   item.opted_in,
        updated_by: actorUserId,
        updated_at: sql`now()`,
      })
    )
    .returning(['notification_preference_id', 'channel', 'purpose', 'opted_in', 'updated_at'])
    .executeTakeFirstOrThrow();
}
```

All items in a single request run in one `UnitOfWork` transaction — if any item fails validation or DB write, the whole batch rolls back.

### Default values at first insert

- `opted_in` defaults:
  - `purpose = 'marketing'` → `false` (default out per BRD AC-2).
  - All other purposes → `true` (default in per BRD AC-2).
- These defaults are applied in the **service layer before upsert** when a record does not yet exist and the caller has not supplied an explicit value. (The DTO always requires `opted_in` explicitly, so this note applies only to implicit initialisation by the dispatch service when reading a missing row.)

### Audit write

After the batch upsert succeeds (still within the same transaction), call:

```typescript
await auditAppender.append(tx, {
  actor_id:    actorUserId,
  action:      'lead_update',          // closest available audit_action; see Ambiguities
  entity_type: 'notification_preferences',
  entity_id:   null,                   // batch — no single entity_id
  lead_id:     null,                   // preferences are not lead-scoped
  detail: {
    subject_type: dto.subject_type,
    subject_ref:  dto.subject_ref,
    changes: dto.preferences.map(p => ({
      channel:  p.channel,
      purpose:  p.purpose,
      opted_in: p.opted_in,
    })),
    // NOTE: subject_ref is a UUID — no PII value is logged (name/mobile/email not included)
  },
});
```

### Read — `notification_preferences`

```typescript
await db
  .selectFrom('notification_preferences')
  .selectAll()
  .where('org_id',       '=', caller.org_id)
  .where('subject_type', '=', subjectType)
  .where('subject_ref',  '=', subjectRef)
  .limit(100)   // a subject can have at most 4 channels × 11 purposes = 44 rows; 100 is safe
  .execute();
```

## Validation Logic

Implemented as a Zod schema on the request DTO; validated by `ValidationPipe` at the controller boundary before the service is called.

| Field | Rule | Error message |
|---|---|---|
| `subject_type` | Required; must be `subject_type` enum (`user` or `customer`) | "subject_type must be 'user' or 'customer'" |
| `subject_ref` | Required; valid UUID v4 | "subject_ref must be a valid UUID" |
| `preferences` | Required; non-empty array; length ≤ 50 | "preferences must contain 1 to 50 items" |
| `preferences[].channel` | Required; `comm_channel` enum value (`in_app`, `email`, `sms`, `whatsapp`) | "channel must be one of: in_app, email, sms, whatsapp" |
| `preferences[].purpose` | Required; `consent_purpose` enum value | "purpose must be a valid consent purpose" |
| `preferences[].opted_in` | Required; boolean | "opted_in must be a boolean" |
| Duplicate `(channel, purpose)` pairs within a single request | Deduplicated before upsert (last-write wins within the batch) | No error; last value for the pair is applied |

**Transactional KYC/document opt-out warning:** If any item being set to `opted_in=false` has `purpose` in `['kyc', 'document_processing', 'communication']`, the response includes a non-blocking warning in `meta.warnings[]`:

```json
"meta": {
  "correlation_id": "...",
  "warnings": [
    { "field": "preferences[0]", "message": "Opting out of KYC/document reminders may delay your application processing." }
  ]
}
```

This is informational only and does not block the save (per BRD edge case AC).

## Backend Flow (Step-by-Step)

### PUT /api/v1/preferences

1. `CorrelationMiddleware` assigns/propagates `X-Correlation-Id`.
2. `JwtAuthGuard` validates the JWT; extracts `user_id`, `org_id`, `role`.
3. `AbacGuard` (`@Requires('customer_comm', scopeResolver)`) calls `EntitlementService.can(user, 'customer_comm', { subject_type, subject_ref })`. Fails → `FORBIDDEN` (403).
4. `ValidationPipe` runs the Zod DTO schema. Fails → `VALIDATION_ERROR` (400) with `fields[]`.
5. `PreferenceService.upsertBatch(dto, actor)` is called.
6. Deduplicate `preferences[]` by `(channel, purpose)` within the request (last item wins).
7. `UnitOfWork.run(async (tx) => { ... })`:
   a. For each item, execute the Kysely `INSERT … ON CONFLICT … DO UPDATE` query on `notification_preferences`.
   b. Collect the returned rows.
   c. Build the audit `detail` payload (channel/purpose/opted_in changes only — no PII values).
   d. Call `AuditAppender.append(tx, auditPayload)`.
8. Transaction commits. If any step inside the `UnitOfWork` throws, the whole batch rolls back → `INTERNAL_ERROR` (500).
9. Check each returned row: if `purpose` is `kyc`, `document_processing`, or `communication` and `opted_in=false`, add a warning entry to `meta.warnings`.
10. `ResponseEnvelopeInterceptor` wraps the result in `{ data, meta, error: null }`.
11. Return HTTP 200.

### PUT /api/v1/c/{token}/preferences

1. `CorrelationMiddleware` assigns/propagates `X-Correlation-Id`.
2. `JwtAuthGuard` skips (route is `@Public()`).
3. `CustomerLinkGuard` validates the token: checks `customer_links` table for `status='active'`, not expired, OTP step-up complete. Resolves `customer_profile_id`. Enforces `subject_type='customer'` and `subject_ref == customer_profile_id`. Fails → `FORBIDDEN` (403).
4. `ThrottlerGuard` enforces the customer-link rate tier (10/min per IP). Exceeds → `RATE_LIMITED` (429).
5. `ValidationPipe` runs the Zod DTO schema. Fails → `VALIDATION_ERROR` (400).
6. `PreferenceService.upsertBatch(dto, actor)` (same service method; actor is the system user representing the customer link session).
7. Steps 6–11 identical to the internal flow above.

## Transaction Boundaries

This FR uses `UnitOfWork` for the batch write.

```
UnitOfWork.run(tx => [
  INSERT/UPDATE notification_preferences (N rows, one per preference item)
  INSERT audit_logs (1 row, batch summary)
])
```

If any row fails, the entire transaction rolls back — no partial state persists. The `UnitOfWork` runs in request scope (nestjs-cls).

No `event_outbox` write: `notification_preferences` is not a domain event aggregate. The `event_code` enum does not contain a preference-changed event (see Ambiguities). Dispatch-side effects (FR-101) are immediate via table read, not event-driven.

## Ambiguities

1. **No customer-link preferences endpoint in `api-contract.yaml`:** The `public_endpoints` list in `auth-matrix.json` does not include `PUT /api/v1/c/{token}/preferences` or `GET /api/v1/c/{token}/preferences`, and `api-contract.yaml` has no `x-frs: [FR-103]` entry for these paths. The BRD AC states "customer self via link". This LLD assumes these endpoints exist and are protected by `CustomerLinkGuard` (consistent with the `c/{token}/*` pattern). The contracts package should be updated to add these endpoints before coding begins.

2. **No `NOTIFICATION_PREF_CHANGED` event in `event_code` enum:** The `event_code` enum in `schema.sql` does not include a preference-change event. The BRD AC states "changes audited and reflected immediately" — immediate reflection is achieved via table read (no event needed for dispatch). The audit is written to `audit_logs` only. No outbox event is emitted. If a domain event is required for downstream consumers, the enum must be extended in a Flyway migration first.

3. **`audit_action` enum has no `pref_change` value:** The `audit_action` enum in `schema.sql` does not include a preference-specific action. This LLD uses `lead_update` as the closest available action. The schema should be extended with a `pref_change` value in a Flyway migration, or the team should confirm `lead_update` is acceptable for this purpose.

4. **`notification_preferences` resource governance marked `system_managed` in `auth-matrix.json`:** The resource governance entry lists `notification_preferences` as system-managed (`writer: owning service/jobs`). The BRD AC grants customer and RM/BM write access. This LLD treats the ABAC capability `customer_comm` (which RM, BM, SM, and CUSTOMER all hold) as the governing capability, consistent with the BRD intent. The auth-matrix resource_governance entry should be corrected to reference `customer_comm`.

5. **No `GET /api/v1/preferences` endpoint in `api-contract.yaml`:** The contract only declares `PUT /preferences`. A read endpoint is required to populate the UI. This LLD adds `GET /preferences` (and its customer-link mirror) under the same capability. The contracts package should be updated.

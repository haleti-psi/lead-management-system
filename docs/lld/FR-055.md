## Summary

FR-055 implements the senior-approver gate that sits between eligibility and LOS hand-off. After a lead reaches `eligibility_requested` and the existing guards are satisfied (`eligibility_received`, `docs_kyc_ready`), the system moves it to the new `pending_approval` stage instead of directly to `ready_for_handoff`. A BM, SM, or HEAD approver then calls `POST /leads/{id}/approval` with `{ decision, reason? }`.

`ApprovalService.decide` executes one `UnitOfWork` that: (1) inserts a `lead_approvals` row (owned by `ApprovalRepository`); (2) delegates to `LeadService.recordApprovalDecision` — the sole writer of `leads` — which moves the stage, sets `approval_status`, writes `stage_history`, emits an audit log via `AuditAppender`, and calls `OutboxService` within the same transaction. The existing FR-081 hand-off (`ready_for_handoff → handed_off`) is completely unchanged.

## Auth Check

| Layer | Guard / Decorator | Detail |
|---|---|---|
| Authentication | `JwtAuthGuard` (global) | Validates JWT; loads user + ABAC attributes onto request context. Missing/expired JWT → `AUTH_REQUIRED` (401). |
| Authorization | `@Requires(Capability.APPROVE_LEAD, scopeResolver)` + `AbacGuard` | Calls `EntitlementService.can(user, 'approve_lead', lead)`. Denied → `FORBIDDEN` (403) + audit. |
| Scope resolution | `scopeResolver` (mirror FR-030 / FR-052 resolver) | BM: `scope=B` — `leads.branch_id = user.branchId`. SM: `scope=T` — `leads.team_id = user.teamId`. HEAD: `scope=A` — all. RM and PARTNER do NOT hold `approve_lead` → 403. |

## Validation Logic

Validated with Zod at the `ValidationPipe` boundary before reaching the service.

### POST /leads/{id}/approval — ApprovalDto

| Field | Zod rule | Error |
|---|---|---|
| `id` (path) | `z.string().uuid()` | "Invalid lead ID format." |
| `decision` | `z.nativeEnum(ApprovalDecision)` (from `@shared/enums`) | "decision must be 'approve' or 'reject'." |
| `reason` | `z.string().min(5).max(500).optional()` + `superRefine` | "reason is required when rejecting (min 5, max 500 characters)." |

`superRefine` rejects the DTO when `decision === 'reject'` and `reason` is absent or blank. All Zod failures produce `VALIDATION_ERROR` (400) with `error.detail.fields[]`.

---

## Error Cases

| Error code | HTTP | Condition |
|---|---|---|
| `AUTH_REQUIRED` | 401 | Missing or expired JWT |
| `FORBIDDEN` | 403 | `EntitlementService.can()` returns false (wrong scope, missing `approve_lead` capability) |
| `NOT_FOUND` | 404 | `lead_id` does not exist or `deleted_at IS NOT NULL` for this org |
| `VALIDATION_ERROR` | 400 | Zod field failure (bad enum value, reason absent on reject, reason too short/long) |
| `CONFLICT` | 409 | Lead's current `stage !== 'pending_approval'` (already approved/rejected or not yet at gate) |
| `INTERNAL_ERROR` | 500 | Unhandled exception (DB down, etc.) |

No `UPSTREAM_UNAVAILABLE` is raised; FR-055 makes no external service calls.

## Backend Flow (Step-by-Step)

```
POST /api/v1/leads/{id}/approval
  │
  ├─ 1. CorrelationMiddleware    → assigns / propagates X-Correlation-Id
  ├─ 2. JwtAuthGuard             → validates JWT; populates req.user
  ├─ 3. AbacGuard                → EntitlementService.can(user, 'approve_lead', {leadId})
  │       └─ scope filter resolved; FORBIDDEN (403) on deny
  ├─ 4. ValidationPipe (Zod)     → validates path UUID + ApprovalDto body
  │       └─ VALIDATION_ERROR (400) with fields[] on failure
  │
  ├─ 5. ApprovalController.decide(id, dto, req)
  │       └─ delegates to ApprovalService.decide(id, dto, requestUser)
  │
  ├─ 6. ApprovalService.decide
  │   └─ UnitOfWork.run(async (tx) => {
  │       │
  │       ├─ 6a. LeadRepository.findByIdForUpdate(id, orgId, tx)
  │       │       → SELECT ... FOR UPDATE
  │       │       → NOT_FOUND (404) if no row or deleted_at IS NOT NULL
  │       │       → FORBIDDEN (403) if lead is out of actor's scope
  │       │
  │       ├─ 6b. Assert lead.stage === 'pending_approval'
  │       │       → CONFLICT (409) if not
  │       │
  │       ├─ 6c. ApprovalRepository.insert({ leadId, decision, reason, decidedBy, orgId }, tx)
  │       │       → INSERT INTO lead_approvals (...)
  │       │       (ApprovalRepository OWNS this table)
  │       │
  │       ├─ 6d. LeadService.recordApprovalDecision(leadId, decision, reason, actorId, tx)
  │       │   ├─ UPDATE leads SET stage=<toStage>, approval_status=<status>,
  │       │   │                   version=version+1, updated_at, updated_by
  │       │   │   WHERE lead_id=:id AND deleted_at IS NULL
  │       │   │   RETURNING lead_id, lead_code, stage, approval_status, version
  │       │   ├─ INSERT INTO stage_history (from_stage='pending_approval', to_stage, actor_id, reason, occurred_at)
  │       │   ├─ AuditAppender.append({ action: 'stage_transition', entityType: 'leads',
  │       │   │                         entityId: leadId, detail: { fromStage, toStage, reason } }, tx)
  │       │   └─ OutboxService.emit({ eventCode: 'LEAD_APPROVED'|'LEAD_REJECTED',
  │       │                           aggregateType: 'leads', aggregateId: leadId,
  │       │                           payload: { leadId, decision, decidedBy, occurredAt } }, tx)
  │       │
  │       └─ tx commits (all writes in one DB transaction)
  │           → INTERNAL_ERROR (500) on DB error (full rollback)
  │
  ├─ 7. ResponseEnvelopeInterceptor → wraps result in { data, meta, error:null }
  └─ 8. Return HTTP 200
```

**Post-commit (async, non-blocking):**
- `LEAD_APPROVED` / `LEAD_REJECTED` outbox row picked up by outbox publisher → Pub/Sub topic.
- `NotificationDispatchService` sends in-app / email to lead owner and relevant manager.
- On approve: FR-081 hand-off becomes eligible immediately (no additional trigger from FR-055).

## Data Operations

### INSERT lead_approvals (ApprovalRepository — owns this table)

```typescript
await db
  .insertInto('lead_approvals')
  .values({
    approval_id:  generateUuid(),
    org_id:       orgId,
    lead_id:      leadId,
    decision:     dto.decision,          // 'approved' | 'rejected'  (stored enum)
    reason:       dto.reason ?? null,
    decided_by:   actor.userId,
    decided_at:   new Date(),
    created_at:   new Date(),
    updated_at:   new Date(),
    created_by:   actor.userId,
    updated_by:   actor.userId,
  })
  .execute();
```

### UPDATE leads — inside LeadService.recordApprovalDecision (sole writer of leads)

```typescript
const toStage       = dto.decision === 'approve' ? LeadStage.READY_FOR_HANDOFF : LeadStage.REJECTED;
const approvalStatus = dto.decision === 'approve' ? ApprovalStatus.APPROVED : ApprovalStatus.REJECTED;

const updated = await db
  .updateTable('leads')
  .set({
    stage:           toStage,
    approval_status: approvalStatus,
    version:         sql`version + 1`,
    updated_at:      new Date(),
    updated_by:      actorId,
  })
  .where('lead_id',    '=', leadId)
  .where('deleted_at', 'is', null)
  .returning(['lead_id', 'lead_code', 'stage', 'approval_status', 'version'])
  .executeTakeFirst();

if (!updated) throw new NotFoundException('NOT_FOUND');
```

No optimistic-lock version predicate on this path — stage equality was asserted in step 6b with a `FOR UPDATE` lock, so no concurrent approval race is possible.

### INSERT stage_history (append-only, inside LeadService.recordApprovalDecision)

```typescript
await db
  .insertInto('stage_history')
  .values({
    stage_history_id: generateUuid(),
    org_id:           orgId,
    lead_id:          leadId,
    from_stage:       'pending_approval',
    to_stage:         toStage,
    actor_id:         actorId,
    reason:           dto.reason ?? null,
    occurred_at:      new Date(),
    created_at:       new Date(),
    updated_at:       new Date(),
  })
  .execute();
```

### AuditAppender.append (inside LeadService.recordApprovalDecision)

```typescript
await auditAppender.append({
  action:     'stage_transition',
  entityType: 'leads',
  entityId:   leadId,
  leadId,
  detail: {
    fromStage:  'pending_approval',
    toStage,
    decision:   dto.decision,
    reason:     dto.reason ?? null,
  },
  actorId,
  ipDevice: requestContext.ipDevice,
}, tx);
```

### OutboxService.emit (inside LeadService.recordApprovalDecision)

```typescript
await outboxService.emit({
  eventCode:     dto.decision === 'approve' ? EventCode.LEAD_APPROVED : EventCode.LEAD_REJECTED,
  aggregateType: 'leads',
  aggregateId:   leadId,
  payload: {
    leadId,
    decision:   dto.decision,
    decidedBy:  actorId,
    occurredAt: new Date().toISOString(),
  },
}, tx);
```

All four writes (lead_approvals INSERT, leads UPDATE, stage_history INSERT, event_outbox INSERT) execute within the same `tx`. Any failure rolls back the entire transaction — no partial state.

## State Machine Touchpoints

FR-055 makes the following changes to `docs/contracts/state-machines.md §Lead`:

| Change | Detail |
|---|---|
| **Re-point transition** | `eligibility_requested → ready_for_handoff` becomes `eligibility_requested → pending_approval` (same guards: `eligibility_received`, `docs_kyc_ready`; same allowed roles: system/BM/KYC) |
| **ADD transition** | `pending_approval → ready_for_handoff` — trigger: APPROVE; who: BM/SM/HEAD; guards: none |
| **ADD transition** | `pending_approval → rejected` — trigger: REJECT; who: BM/SM/HEAD; guards: `rejection_reason_provided` |
| **ADD to ACTIVE_STAGES** | `LeadStage.PENDING_APPROVAL` — eligible for the generic `any active → dormant` path |

`StageGuardService` changes:
- Re-point `ELIGIBILITY_REQUESTED:READY_FOR_HANDOFF` entry to `ELIGIBILITY_REQUESTED:PENDING_APPROVAL` (same `allowedRoles`, same guards).
- ADD `PENDING_APPROVAL:READY_FOR_HANDOFF` entry — `allowedRoles: [BM, SM, HEAD]`, guards: `[]`.
- `PENDING_APPROVAL:REJECTED` is handled by the existing generic "any active → rejected" path (`rejection_reason_provided` guard is satisfied by the `reason` field from `ApprovalDto`).
- Add `LeadStage.PENDING_APPROVAL` to `ACTIVE_STAGES` set.

## Data Model Touchpoints

Schema additions belong in `docs/data-model/schema.sql` and `docs/data-model/migrations/V6__add_lead_approval.sql`.

**New enum additions:**
- `ALTER TYPE lead_stage ADD VALUE IF NOT EXISTS 'pending_approval'`
- `ALTER TYPE event_code ADD VALUE IF NOT EXISTS 'LEAD_APPROVED'`
- `ALTER TYPE event_code ADD VALUE IF NOT EXISTS 'LEAD_REJECTED'`
- `CREATE TYPE approval_status AS ENUM ('not_required','pending','approved','rejected')`
- `CREATE TYPE approval_decision AS ENUM ('approved','rejected')` — stored outcome

**Column addition:**
- `ALTER TABLE leads ADD COLUMN approval_status approval_status NOT NULL DEFAULT 'not_required'`

**New table `lead_approvals`:**

```sql
CREATE TABLE lead_approvals (
  approval_id  UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id       UUID         NOT NULL DEFAULT '00000000-0000-0000-0000-000000000001' REFERENCES orgs(id),
  lead_id      UUID         NOT NULL,
  decision     approval_decision NOT NULL,
  reason       VARCHAR(500),
  decided_by   UUID         NOT NULL,
  decided_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),
  created_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),
  updated_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),
  created_by   UUID         NOT NULL,
  updated_by   UUID         NOT NULL,
  CONSTRAINT fk_lead_approvals_lead_id  FOREIGN KEY (lead_id)    REFERENCES leads(lead_id) ON DELETE CASCADE,
  CONSTRAINT fk_lead_approvals_decided  FOREIGN KEY (decided_by) REFERENCES users(user_id),
  CONSTRAINT ck_lead_approvals_reject_reason CHECK (decision <> 'rejected' OR reason IS NOT NULL)
);
CREATE INDEX ix_lead_approvals_lead       ON lead_approvals (lead_id);
CREATE INDEX ix_lead_approvals_decided_at ON lead_approvals (org_id, decided_at DESC);
```

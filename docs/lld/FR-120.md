## Summary

FR-120 delivers the **core report pack** for M13 Reporting: four parameterised aggregate reports that give management the data to run the sales operation.

| Report code | Name | Primary metric |
|---|---|---|
| `funnel_conversion` | Funnel / Conversion | Stage-by-stage counts + overall conversion rate |
| `source_performance` | Source Performance | Leads captured, handed-off, conversion % by source/sub-source |
| `rm_performance` | RM / Team Performance | Captured, contacted, qualified, handed-off, rejection rate per RM/team |
| `rejection_summary` | Rejection Summary | Rejections by primary/sub reason, period, scope |

All four share a single endpoint `GET /api/v1/reports/{code}`. Reports are **scope-filtered** per auth-matrix (§3.9 reconciliation rules §12.5). They are read-only aggregate queries with no side-effects. Heavy ad-hoc exports are out of scope here (see FR-122 ExportJob).

Timeout guard: if a report query runs longer than 8 s (configurable via `REPORT_TIMEOUT_MS`) the controller returns `INTERNAL_ERROR` with an advisory to use FR-122 async export. This protects the read replica from unbounded queries.

## Endpoint

```

Query params (all optional):
  from          ISO 8601 date string (IST)   — start of period, inclusive
```

## Auth Check

**Step 1 — Authentication:** `JwtAuthGuard` (global) validates the Bearer token. Returns `AUTH_REQUIRED` (401) if absent/expired.

**Step 2 — Capability:** `@Requires('reports', scopeResolver)` triggers `AbacGuard`. `EntitlementService.can(user, 'reports', null)` checks the capability_matrix:

| Role | `reports` capability | Scope |
|---|---|---|
| RM | yes | O — own assigned leads only |
| BM | yes | B — own branch |
| SM | yes | T — own team |
| HEAD | yes | A — all org |
| KYC | yes | B — own branch |
| DPO | yes | M — masked compliance view |
| PARTNER | yes | P — own submitted leads only |
| ADMIN | no `reports` in matrix | blocked (403) |
| CUSTOMER | no `reports` in matrix | blocked (403) |

**Step 3 — Scope enforcement in query:** The resolved scope is injected into every aggregate query as a `WHERE` clause — no role may broaden its scope by passing query parameters. Specifically:

- `RM` (O): query is `AND l.owner_id = :userId`. Any `owner_id` param that is not the caller's own UUID is rejected with `FORBIDDEN`.
- `BM`/`KYC` (B): `AND l.branch_id = :userBranchId`. Any `branch_id` param for a different branch is rejected with `FORBIDDEN`.
- `SM` (T): `AND l.team_id = :userTeamId`. Similarly enforced.
- `HEAD` (A): no forced scope restriction; all optional filters accepted.
- `DPO` (M): scope is `A` for aggregates (DPO may see org-wide counts); individual lead-level data is masked — not exposed in this endpoint.
- `PARTNER` (P): `AND sa.partner_id = :userPartnerId`. A PARTNER may not pass a different `partner_id`.

## Data Operations

All four report queries are **parameterised Kysely aggregate reads** on the Postgres read replica (when available). No writes. LIMIT clause applied to paginated result sets (max 100). No unbounded queries.

### Report: `funnel_conversion`

Groups by `product_code`. For each product, counts leads that ever reached each stage (stage-reached funnel using `stage_history`).

```typescript
// Pseudocode — real implementation uses Kysely eb() and ref()
db.selectFrom('leads as l')
  .innerJoin('stage_history as sh', 'sh.lead_id', 'l.lead_id')
  .where('l.org_id', '=', orgId)
  .where('l.deleted_at', 'is', null)
  // + scope WHERE clauses (branch_id / owner_id / team_id)
  // + period filter: l.created_at BETWEEN :from AND :to
  .select([
    'l.product_code',
    db.fn.count('l.lead_id').distinct().as('captured'),
    db.fn.countAll()  // reached assigned — via FILTER(WHERE sh.to_stage='assigned')
    // ... per stage using conditional aggregates
  ])
  .groupBy('l.product_code')
  .orderBy('l.product_code')
  .limit(limit).offset((page - 1) * limit)
```

Exact column references:
- `leads.lead_id`, `leads.product_code`, `leads.stage`, `leads.branch_id`, `leads.owner_id`, `leads.team_id`, `leads.created_at`, `leads.deleted_at`, `leads.org_id`
- `stage_history.lead_id`, `stage_history.to_stage`, `stage_history.occurred_at`

Active pipeline = captured - handed_off - rejected (within scope/period). §12.5: counts are monotonic stage-reached (a lead entering `kyc_in_progress` is also counted in `captured`, `contacted`, `qualified`, `documents_pending`).

### Report: `source_performance`

Groups by `source_attributions.source` (and optionally `sub_source` or `partner_id`).

```typescript
db.selectFrom('leads as l')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .where('l.org_id', '=', orgId)
  .where('l.deleted_at', 'is', null)
  // + scope/period/product filters
  .select([
    'sa.source',
    db.fn.count('l.lead_id').distinct().as('captured'),
    // handed_off count: WHERE l.stage = 'handed_off'
    // conversion_pct: computed post-query (avoid division in SQL for zero-denominator safety)
  ])
  .groupBy('sa.source')
  .orderBy('sa.source')
  .limit(limit).offset((page - 1) * limit)
```

Exact column references:
- `source_attributions.source_attribution_id`, `source_attributions.source`, `source_attributions.sub_source`, `source_attributions.partner_id`

### Report: `rm_performance`

Groups by `leads.owner_id`, joined to `users` for name.

```typescript
db.selectFrom('leads as l')
  .innerJoin('users as u', 'u.user_id', 'l.owner_id')
  .where('l.org_id', '=', orgId)
  .where('l.deleted_at', 'is', null)
  // + scope/period/product/team/branch filters
  .select([
    'l.owner_id',
    'u.full_name as owner_name',
    db.fn.count('l.lead_id').distinct().as('captured'),
    // contacted, qualified, handed_off via stage_history stage-reached counts
    // rejection_rate: rejected / captured
  ])
  .groupBy(['l.owner_id', 'u.full_name'])
  .orderBy('captured', 'desc')
  .limit(limit).offset((page - 1) * limit)
```

Exact column references:
- `users.user_id`, `users.full_name`
- `leads.owner_id`, `leads.team_id`

RM scope (O) forces `l.owner_id = caller.userId`; result is always a single row.

### Report: `rejection_summary`

Groups by `rejection_reasons.primary_reason` (and optionally `sub_reason`).

```typescript
db.selectFrom('leads as l')
  .innerJoin('rejection_reasons as rr', 'rr.rejection_reason_id', 'l.rejection_reason_id')
  .innerJoin('stage_history as sh', (join) =>
    join.onRef('sh.lead_id', '=', 'l.lead_id').on('sh.to_stage', '=', 'rejected')
  )
  .where('l.org_id', '=', orgId)
  .where('l.deleted_at', 'is', null)
  .where('l.stage', '=', 'rejected')
  // + scope/period/product filters
  .select([
    'rr.primary_reason',
    'rr.sub_reason',
    db.fn.count('l.lead_id').distinct().as('rejected_count'),
    // rejection_rate vs captured computed post-query
  ])
  .groupBy(['rr.primary_reason', 'rr.sub_reason'])
  .orderBy('rejected_count', 'desc')
  .limit(limit).offset((page - 1) * limit)
```

Exact column references:
- `rejection_reasons.rejection_reason_id`, `rejection_reasons.primary_reason`, `rejection_reasons.sub_reason`
- `leads.rejection_reason_id`, `leads.stage`

## Validation Logic

Validated by Zod at the controller layer (`GetReportQueryDto`) before the service is called. `ValidationPipe` is globally applied; failures return `VALIDATION_ERROR` (400) with `fields[]`.

| Field | Rule | Error message |
|---|---|---|
| `code` (path) | Must be one of `funnel_conversion`, `source_performance`, `rm_performance`, `rejection_summary` | "Invalid report code. Allowed: funnel_conversion, source_performance, rm_performance, rejection_summary." |
| `from` | ISO 8601 date string if present; must be `<= to` | "from must be a valid ISO date and not later than to." |
| `to` | ISO 8601 date string if present; must be `>= from` | "to must be a valid ISO date and not earlier than from." |
| `branch_id` | UUID v4 if present | "branch_id must be a valid UUID." |
| `team_id` | UUID v4 if present | "team_id must be a valid UUID." |
| `owner_id` | UUID v4 if present | "owner_id must be a valid UUID." |
| `product_code` | Must be a value in `@shared/enums` `product_code` enum if present | "product_code is not a valid product code." |
| `source` | Must be a value in `@shared/enums` `lead_source` enum if present | "source is not a valid lead source." |
| `partner_id` | UUID v4 if present | "partner_id must be a valid UUID." |
| `page` | Integer >= 1; default 1 | "page must be a positive integer." |
| `limit` | Integer 1–100; default 25 | "limit must be between 1 and 100." |

Post-Zod scope checks (raise `FORBIDDEN`, not `VALIDATION_ERROR`):
- `owner_id` param present and caller is RM: must equal `caller.userId`.
- `branch_id` param present and caller is BM/KYC: must equal `caller.branchId`.
- `partner_id` param present and caller is PARTNER: must equal `caller.partnerId`.

## Error Cases

| Scenario | Code | HTTP | `detail.reason` |
|---|---|---|---|
| Missing / expired JWT | `AUTH_REQUIRED` | 401 | — |
| Role has no `reports` capability (ADMIN, CUSTOMER) | `FORBIDDEN` | 403 | — |
| `branch_id` param targets a different branch than caller's | `FORBIDDEN` | 403 | — |
| `owner_id` param is a different RM than the caller (RM role) | `FORBIDDEN` | 403 | — |
| `partner_id` param targets a different partner | `FORBIDDEN` | 403 | — |
| `code` path param not in allowed set | `VALIDATION_ERROR` | 400 | — |
| `from` > `to` | `VALIDATION_ERROR` | 400 | — |
| Invalid enum value in `product_code` or `source` | `VALIDATION_ERROR` | 400 | — |
| Aggregate query exceeds `REPORT_TIMEOUT_MS` | `INTERNAL_ERROR` | 500 | — |
| Unhandled DB error | `INTERNAL_ERROR` | 500 | — |

No `CONFLICT` (409), `NOT_FOUND` (404), or `UPSTREAM_UNAVAILABLE` (503) are raised by this FR; it is a pure read with no external service calls and no status-touching writes.

## Backend Flow (Step-by-Step)

1. **Request enters** `ReportController.getReport(code, query, req)`.
2. `JwtAuthGuard` (global) validates JWT; on failure → `AUTH_REQUIRED` (401).
3. `AbacGuard` calls `EntitlementService.can(user, 'reports', null)`. If false → `FORBIDDEN` (403).
4. **Zod validation** of `GetReportQueryDto` via global `ValidationPipe`. Failures → `VALIDATION_ERROR` (400) with `fields[]`.
5. **Scope resolution**: `ReportService.resolveScope(user, query)` computes the mandatory `WHERE` additions and validates that any explicit filter params are within the caller's scope. Out-of-scope param → `FORBIDDEN` (403).
6. **Dispatch** to the specific report builder method in `ReportService`:
   - `buildFunnelConversionReport(scope, filters, pagination)`
   - `buildSourcePerformanceReport(scope, filters, pagination)`
   - `buildRmPerformanceReport(scope, filters, pagination)`
   - `buildRejectionSummaryReport(scope, filters, pagination)`
7. Each builder executes a **Kysely aggregate query** against the read replica (primary if replica unavailable). Query is wrapped in a `Promise.race` against a timeout of `REPORT_TIMEOUT_MS` ms. If timeout fires → log `warn` + throw `INTERNAL_ERROR`.
8. **Post-query computation**: percentage fields are calculated in TypeScript (not SQL) to apply the zero-denominator rule: `denominator === 0 → "–"`, otherwise `(numerator / denominator * 100).toFixed(1)`.
9. **Response assembly**: `ReportService` returns `{ rows, total }`. Controller wraps in the uniform envelope via `ResponseEnvelopeInterceptor`.
10. Response sent; `CorrelationMiddleware` attaches `X-Correlation-Id`.

No transaction is needed (read-only). No `AuditAppender`, no `OutboxService`, no `UnitOfWork`. No external service calls.

- Low-bandwidth mode: charts disabled; DataTable only.
- WCAG 2.1 AA: all filter controls have visible labels; `aria-required` where applicable; DataTable headers use `<th scope="col">`.

## Assumptions

1. The `REPORT_TIMEOUT_MS` environment variable is defined in `docs/contracts/environment-contract.md` (or a reasonable default of 8000 ms is used if not found — to be confirmed by the environment-contract owner).
2. Read replica routing (connection string `DATABASE_REPLICA_URL`) is out of scope for this FR's implementation; the service calls `Db` (primary) and the infrastructure team configures replica routing at the proxy layer. If `DATABASE_REPLICA_URL` is available the `DbModule` exposes a `DbReplica` token — the repository will inject whichever is available.
3. The `product_code` enum values are as generated in `@shared/enums` from the BRD §5.5 catalog; this FR does not add new enum values.
4. The `rm_performance` report returns `"–"` for percentage fields when the calling user has scope O and has no captured leads in the period (not an error).

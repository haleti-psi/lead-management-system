## Summary

FR-081 implements the guarded, idempotent hand-off of a lead to the external LOS. It is the sole write path that transitions a lead from `ready_for_handoff` to `handed_off` (a terminal stage). The handler evaluates a mandatory six-point guard checklist, performs deduplication using `Idempotency-Key` / `integration_logs.idempotency_key`, calls the LOS via `IntegrationGateway` → `LosPort`, and — on success — commits a single transaction that updates `leads`, writes `integration_logs`, `data_sharing_logs`, the initial `los_application_mirrors` row, and emits `LEAD_HANDED_OFF` through the transactional outbox. LOS errors produce `UPSTREAM_UNAVAILABLE` (503) and enqueue a Cloud Tasks retry without creating a duplicate LOS application.

This FR is **Tier 3** because it:
- touches six tables in one atomic transaction via `UnitOfWork`;
- performs a state transition with six guarded pre-conditions;
- calls an external service via `IntegrationGateway` with full failure-handling;
- requires idempotency, optimistic locking, and outbox events.

## Auth Check

**Authentication:** `JwtAuthGuard` (global) — validates JWT, loads user + ABAC attributes onto request context. Returns `AUTH_REQUIRED` (401) if missing/expired.

**Authorization:** `AbacGuard` + `@Requires('hand_off', scopeResolver)` → `EntitlementService.can(user, 'hand_off', lead)`

Scope resolution:
- `BM` → scope `B` (branch): `lead.branch_id === user.branch_id`
- `KYC` → scope `B` (branch): same branch check
- `RM` → scope `O` (own): `lead.owner_id === user.user_id` (configurable delegation OD-03)
- `DPO` → scope `M` (masked): capability present but data masked

Deny (scope mismatch) → `FORBIDDEN` (403), audit `action='handoff_attempt'` with outcome `denied`.

**Ownership note:** BM is the default hand-off actor. RM/KYC delegation is a configurable org-level setting checked by `EntitlementService` before reaching the handler.

## Validation Logic

Zod schema for the path parameter:

```typescript
const HandoffPathSchema = z.object({
  id: z.string().uuid({ message: 'Lead ID must be a valid UUID' }),
});

const IdempotencyKeySchema = z.object({
  'idempotency-key': z.string().min(1).max(120, {
    message: 'Idempotency-Key must be 1–120 characters',
  }),
});
```

There is no request body. All business validation occurs in the guard checklist (see Backend Flow). Zod `VALIDATION_ERROR` (400) with `fields[]` is returned for malformed path/header before guards are evaluated.

**Guard checklist evaluated by `StageGuardService.evaluate('ready_for_handoff', 'handed_off', lead)`:**

| Guard | Condition | Failure code |
|-------|-----------|--------------|
| `stage_valid` | `lead.stage === 'ready_for_handoff'` | `VALIDATION_ERROR` / `STAGE_GUARD_FAILED` (lists guard) |
| `consent_present` | `consent_records` has `purpose='los_handoff'`, `state='granted'` | `FORBIDDEN` / `CONSENT_MISSING` |
| `duplicate_clear` | `lead.duplicate_status` in `('none','linked','merged')` | `CONFLICT` / `DUPLICATE_BLOCKED` |
| `docs_complete` | No unverified/un-waived mandatory documents | `VALIDATION_ERROR` / `STAGE_GUARD_FAILED` |
| `kyc_signed_off` | No `kyc_verifications` with `status='exception'`; lead `kyc_status` in `('verified','waived')` | `CONFLICT` / `KYC_EXCEPTION_OPEN` |
| `product_payload_valid` | `product_config_id` live; product-required fields populated | `VALIDATION_ERROR` / `STAGE_GUARD_FAILED` |

`STAGE_GUARD_FAILED` responses include `error.detail.failed_guards: string[]` listing every failing guard so the UI can display a specific checklist.

## Backend Flow (Step-by-Step)

```
POST /api/v1/leads/{id}/handoff
  Header: Idempotency-Key: <key>
```

**Step 1 — Correlation & parsing**
`CorrelationMiddleware` assigns or forwards `X-Correlation-Id`. `HandoffController` parses + validates `{id}` (UUID), `Idempotency-Key` header via Zod. Malformed → `VALIDATION_ERROR` (400).

**Step 2 — Authentication**
`JwtAuthGuard` (global) validates the JWT, loads `user` (including `role`, `branch_id`, `team_id`, `partner_id`) onto the request. Missing/expired token → `AUTH_REQUIRED` (401).

**Step 3 — Authorization**
`AbacGuard` + `@Requires('hand_off', scopeResolver)` calls `EntitlementService.can(user, 'hand_off', { lead_id: id })`. Reads the lead's `branch_id` / `owner_id` to determine scope. Deny → `FORBIDDEN` (403) + audit `handoff_attempt` with outcome `denied`.

**Step 4 — Load lead**
`LosService` fetches the lead from `leads` WHERE `lead_id = $1 AND deleted_at IS NULL`. Not found → `NOT_FOUND` (404).

**Step 5 — Idempotency check**
Query `integration_logs` for `idempotency_key = $idempotencyKey AND integration = 'los_handoff'`.
- Row found, `status = 'success'`: return the stored `request_ref` (= `los_application_id`) as the original 200 response. No further processing. Log `IDEMPOTENT_REPLAY`.
- Row found, `status IN ('pending','retrying')`: return `UPSTREAM_UNAVAILABLE` (503) — retry in progress; do not double-submit.
- No row: proceed.

**Step 6 — Guard evaluation (delegated to `StageGuardService`)**
`StageGuardService.evaluate('ready_for_handoff', 'handed_off', lead)` runs all six guards (see Validation Logic). Collects all failures; returns the combined `failed_guards[]` list. Any consent failure → `FORBIDDEN` (403) / `CONSENT_MISSING`. KYC exception → `CONFLICT` (409) / `KYC_EXCEPTION_OPEN`. Duplicate blocked → `CONFLICT` (409) / `DUPLICATE_BLOCKED`. Remaining guard failures → `VALIDATION_ERROR` (400) / `STAGE_GUARD_FAILED`.

Emit `HANDOFF_READY` outbox event immediately before the LOS call (step 7a) if all guards pass. (Note: `HANDOFF_READY` was originally emitted when the lead reached `ready_for_handoff`; here we emit it again immediately prior to the LOS submission as notification to stakeholders.)

**Step 7 — Build LOS payload**
`LosHandoffPayloadBuilder.build(lead)` assembles the product-specific payload from `leads`, `lead_identities`, `customer_profiles`, `lead_product_details`, `documents` (verified/waived), `eligibility_snapshots`. No PII is logged. Payload mapped to the `LosPort.HandoffRequest` interface.

**Step 8 — Record pending `integration_logs` row**
Outside the UoW transaction (to avoid holding the DB connection during the HTTP call), insert `integration_logs` with `status = 'pending'`, `idempotency_key`, `integration = 'los_handoff'`. The unique partial index `uq_integration_idempotency` prevents a race-condition double-insert.

**Step 9a — Call LOS (`IntegrationGateway` → `LosPort`)**
```
IntegrationGateway.call(LosPort, losPayload, { idempotencyKey, correlationId })
```
- `LosPort` in production → `LosHttpAdapter` (POST to `LOS_BASE_URL/applications`, header `x-api-key: LOS_API_KEY`).
- In test/dev → `LosMockAdapter` (`LOS_MOCK=true`).
- Response carries `los_application_id`.
- Timeout / 5xx → `UPSTREAM_UNAVAILABLE` (503); go to Step 9b.

**Step 9b — LOS failure path**
On LOS error:
1. Update `integration_logs` SET `status = 'failed'`, `http_status`, `error_code`, `completed_at`.
2. Emit `HANDOFF_FAILED` outbox event (outside UoW — fire-and-forget post-error).
3. Cloud Tasks enqueues an exponential-backoff retry task (max 3 retries). The retry re-enters at Step 5 (idempotency check short-circuits if the original key is still pending).
4. Emit audit `handoff_failure` via `AuditAppender`.
5. Return `UPSTREAM_UNAVAILABLE` (503) to caller.

**Step 9c — LOS success: open UoW transaction**
```
UnitOfWork.run(async (tx) => {
  a. LeadService.markHandedOff(leadId, losApplicationId, expectedVersion, tx)
     → leads UPDATE (stage='handed_off', los_application_id, version++)
     → stage_history INSERT
     → AuditAppender.emit('handoff_success', ...) — deferred to AuditChainConsumer
     → OutboxService.emit('LEAD_HANDED_OFF', ..., tx)
  b. integration_logs UPDATE (status='success', request_ref=losApplicationId, completed_at, http_status)
  c. data_sharing_logs INSERT (purpose='los_handoff', consent_id, recipient='LOS')
  d. los_application_mirrors INSERT (status='submitted', los_application_id)
})
```
If any step inside the UoW throws, the entire transaction rolls back. The `integration_logs` row with `status='pending'` (written in Step 8 outside the tx) is then cleaned up or left for the retry sweeper. In practice, the pending row is updated to `failed` in a catch handler before returning 503.

**Step 10 — Post-commit (non-transactional)**
`OutboxService` publisher relay will pick up `LEAD_HANDED_OFF` and publish to Pub/Sub (at-least-once). `NotificationDispatchService` (FR-101) handles `LEAD_HANDED_OFF` events to send customer/RM notifications (post-commit, does not roll back the transition).

**Step 11 — Response**
Return 200 with `{ leadId, stage: 'handed_off', losApplicationId, handedOffAt }` in the standard envelope.

## State Machine & Transaction Boundaries

**State transition (per `state-machines.md`):**
```
ready_for_handoff  →  handed_off   (terminal)
```

Trigger: guards pass + LOS success  
Who: BM / KYC / RM (delegated)  
Invalid from: any other stage → `CONFLICT` (409)  
Invalid to: any stage from `handed_off` → `CONFLICT` (409) — terminal  

**One-transaction boundary (UoW):**
```
BEGIN
  UPDATE leads  SET stage='handed_off', los_application_id=?, version=version+1  WHERE lead_id=? AND version=?
  INSERT INTO stage_history  (from='ready_for_handoff', to='handed_off')
  -- AuditAppender emits intent; AuditChainConsumer writes audit_logs serially (§11.4)
  INSERT INTO event_outbox   (event_code='LEAD_HANDED_OFF')
  UPDATE integration_logs    SET status='success', request_ref=los_application_id
  INSERT INTO data_sharing_logs  (purpose='los_handoff')
  INSERT INTO los_application_mirrors  (status='submitted')
COMMIT
```

All six writes are atomic. If the DB commit fails after the LOS call succeeded, the integration_logs row remains `pending`; the Cloud Tasks retry will re-enter at Step 5 and the idempotency check (idempotency_key match + status='pending') will wait for a resolution or trigger a fresh attempt after expiry.

**Optimistic lock:** `LeadService.markHandedOff` uses `WHERE version = :expectedVersion`. Stale → 0 rows affected → `CONFLICT` (409). Caller must refresh and retry with the new version.

## External Service Calls via IntegrationGateway

**Port:** `LosPort`  
**Adapter (prod):** `LosHttpAdapter` → `POST ${LOS_BASE_URL}/applications`  
**Adapter (test/dev):** `LosMockAdapter` (`LOS_MOCK=true`)  
**Auth:** `x-api-key: ${LOS_API_KEY}` (from Secret Manager)

**`LosPort.HandoffRequest` interface (illustrative — exact fields per LOS wire contract):**
```typescript
interface LosHandoffRequest {
  idempotencyKey:   string;           // mirrors Idempotency-Key header
  correlationId:    string;
  leadCode:         string;           // leads.lead_code (non-PII reference)
  productCode:      string;           // leads.product_code
  applicantRef:     string;           // leads.lead_identity_id (opaque ref)
  requestedAmount:  string;           // NUMERIC as decimal string (INR)
  branchCode:       string;           // branches.branch_code
  eligibilityRef?:  string;           // eligibility_snapshots.request_ref if present
}
```
No raw PII (name, mobile, PAN, Aadhaar) is transmitted in the payload passed through the application tier — the LOS pulls KYC artefacts via secure document references if required. The exact LOS wire contract is defined in `los-mock.openapi.yaml` (ADR-4).

**Failure handling:**

| Failure type | Detection | Action |
|---|---|---|
| LOS timeout (> configured N s) | `fetch` AbortController | Mark `integration_logs.status='failed'`; emit `HANDOFF_FAILED` outbox; enqueue Cloud Tasks retry (exp backoff, max 3); return `UPSTREAM_UNAVAILABLE` (503) |
| LOS 5xx | HTTP status 5xx | Same as timeout |
| LOS 4xx (mapping error) | HTTP 400/422 from LOS | Mark `status='failed'`, `error_code=los_mapping_error`; return `VALIDATION_ERROR` (400) — config issue, no retry |
| LOS duplicate (idempotent replay) | LOS returns existing `los_application_id` | Accept as success; complete the UoW (same outcome as first call) |
| Circuit breaker open | IntegrationGateway threshold reached | Immediately return `UPSTREAM_UNAVAILABLE` (503); no LOS call attempted |

## Assumptions

1. `StageGuardService.evaluate` is the single owner of the guard matrix and returns `{ passed: boolean; failedGuards: string[] }`. FR-081 does not duplicate guard logic.
2. The exact LOS wire contract (request/response fields) is defined in `los-mock.openapi.yaml` (ADR-4). The `LosHandoffRequest` interface above is illustrative; implement against the versioned mock spec.
3. `HANDOFF_READY` outbox event is emitted when the lead first reaches `ready_for_handoff` (by the stage-transition in FR-052/080). FR-081 does not re-emit it; it only emits `LEAD_HANDED_OFF` (success) or `HANDOFF_FAILED` (failure).
4. The Cloud Tasks retry task calls the same endpoint with the original `Idempotency-Key`, ensuring no duplicate LOS application. The retry task is created by the `IntegrationGateway` (FR-140) — not by FR-081 directly.
5. `los_application_mirrors` initial row uses `received_via = 'webhook'` as a placeholder. FR-082 will upsert the real status when the LOS sends a webhook or poll response.
6. The `integration_logs` insert in Step 8 occurs outside the UoW to avoid holding a DB connection during the HTTP call. On LOS failure, the pre-inserted pending row is updated to `failed` in the catch handler before returning 503.

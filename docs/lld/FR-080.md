## Summary

FR-080 allows an RM or BM to trigger an LOS eligibility check for a lead and store the response as a read-only snapshot. The system builds a product-specific payload from `ProductConfig.eligibility_mapping` and `LeadProductDetail.attributes`, checks that a valid `product_eligibility` consent is present, then calls the LOS via `IntegrationGateway` + `LosPort`. On success the snapshot is stored with status `received` and the lead transitions to `eligibility_requested`. On timeout the snapshot stays `pending` and a Cloud Tasks retry is enqueued — the lead workflow is never blocked. The response card displayed in Lead 360 is read-only and always labelled indicative/preliminary unless LOS returns `response_basis = 'final'`.

---

## Endpoint

### POST /api/v1/leads/{id}/eligibility

Trigger an LOS eligibility check and return the resulting snapshot. Idempotent by `Idempotency-Key` header: if a `pending` or `received` snapshot already exists for the same idempotency key, the existing snapshot is returned with `IDEMPOTENT_REPLAY` in `error.detail.reason` and HTTP 200.

**Path parameter:** `id` — UUID of the lead.

**Request headers:**
| Header | Required | Description |
|---|---|---|
| `Authorization` | Yes | `Bearer <jwt>` |
| `Idempotency-Key` | Recommended | Opaque string (max 120 chars); deduplicates concurrent/retry calls |
| `X-Correlation-Id` | No | Auto-generated if absent |

**Request body:** None (all required data is sourced from the lead and product config).

**Success response — HTTP 200**
```json
{
  "data": {
    "eligibilitySnapshotId": "uuid",
    "leadId": "uuid",
    "requestRef": "string",
    "status": "received",
    "indicativeAmount": "500000.00",
    "tenureMonths": 36,
    "rateRange": "10.5-12.0",
    "conditions": { "note": "Subject to final KYC verification" },
    "validityUntil": "2026-07-09T00:00:00.000Z",
    "responseBasis": "indicative",
    "createdAt": "2026-06-09T10:00:00.000Z"
  },
  "meta": { "correlation_id": "corr_abc123" },
  "error": null
}
```

**Pending response — HTTP 200** (LOS timeout; retry enqueued)
```json
{
  "data": {
    "eligibilitySnapshotId": "uuid",
    "leadId": "uuid",
    "requestRef": "string",
    "status": "pending",
    "indicativeAmount": null,
    "tenureMonths": null,
    "rateRange": null,
    "conditions": null,
    "validityUntil": null,
    "responseBasis": null,
    "createdAt": "2026-06-09T10:00:00.000Z"
  },
  "meta": { "correlation_id": "corr_abc123" },
  "error": null
}
```

**Error responses:**
| HTTP | Code | `error.detail.reason` | When |
|---|---|---|---|
| 400 | `VALIDATION_ERROR` | — | `eligibility_mapping` missing or incomplete on product config |
| 400 | `VALIDATION_ERROR` | `STAGE_GUARD_FAILED` | Lead stage is not `kyc_in_progress` (invalid transition) |
| 401 | `AUTH_REQUIRED` | — | Missing or expired JWT |
| 403 | `FORBIDDEN` | — | Lead is out of caller's scope |
| 403 | `FORBIDDEN` | `CONSENT_MISSING` | No `granted` `product_eligibility` consent on the lead |
| 404 | `NOT_FOUND` | — | Lead ID does not exist |
| 503 | `UPSTREAM_UNAVAILABLE` | — | LOS returned 5xx (non-timeout); retry queued |

## Auth Check

1. **Authentication:** `JwtAuthGuard` (global) — validates JWT, loads user + ABAC attributes. Returns `AUTH_REQUIRED` (401) if token is absent, invalid, or expired.
2. **Authorization:** `@Requires('move_stage', scopeResolver)` with `AbacGuard` calling `EntitlementService.can(user, 'move_stage', lead)`.
   - Scope resolution: RM — `lead.owner_id === user.user_id`; BM — `lead.branch_id === user.branch_id`; KYC — `lead.branch_id === user.branch_id`. Any other role is denied.
   - Out-of-scope leads return `FORBIDDEN` (403), not `NOT_FOUND` (do not hide existence for authed users in scope).
3. **Consent gate (service layer):** before building the LOS payload, verify that `consent_records` has a row with `lead_id = :leadId`, `purpose = 'product_eligibility'`, `state = 'granted'`, and `expires_at IS NULL OR expires_at > now()`. No match → `FORBIDDEN` + `detail.reason = 'CONSENT_MISSING'`.

## Data Operations

All queries use the injected Kysely `Db` instance with parameterised bindings. Every list query applies a `LIMIT`.

### 1. Load and validate the lead

```typescript
const lead = await db
  .selectFrom('leads')
  .selectAll()
  .where('lead_id', '=', leadId)
  .where('org_id', '=', orgId)
  .where('deleted_at', 'is', null)
  .executeTakeFirst();
// → NOT_FOUND if absent
// → FORBIDDEN if outside caller's scope (AbacGuard already checked; double-check here)
```

### 2. Consent check

```typescript
const consent = await db
  .selectFrom('consent_records')
  .select(['consent_id', 'state', 'expires_at'])
  .where('lead_id', '=', leadId)
  .where('org_id', '=', orgId)
  .where('purpose', '=', 'product_eligibility')
  .where('state', '=', 'granted')
  .where((eb) =>
    eb.or([eb('expires_at', 'is', null), eb('expires_at', '>', new Date())])
  )
  .orderBy('created_at', 'desc')
  .limit(1)
  .executeTakeFirst();
// → FORBIDDEN + CONSENT_MISSING if absent
```

### 3. Load product config (eligibility_mapping)

```typescript
const productConfig = await db
  .selectFrom('product_configs')
  .select(['product_config_id', 'eligibility_mapping', 'product_code'])
  .where('product_config_id', '=', lead.product_config_id)
  .where('org_id', '=', orgId)
  .executeTakeFirstOrThrow();
// → VALIDATION_ERROR if eligibility_mapping is null or incomplete (validated by EligibilityMappingValidator)
```

### 4. Load lead product detail (attributes)

```typescript
const lpd = await db
  .selectFrom('lead_product_details')
  .select(['attributes', 'validation_status'])
  .where('lead_id', '=', leadId)
  .where('org_id', '=', orgId)
  .executeTakeFirst();
```

### 5. Idempotency check (before inserting)

```typescript
// If Idempotency-Key header present, check integration_logs for an existing key
const existing = await db
  .selectFrom('integration_logs')
  .select(['integration_log_id', 'status', 'request_ref'])
  .where('idempotency_key', '=', idempotencyKey)
  .where('integration', '=', 'los_eligibility')
  .where('org_id', '=', orgId)
  .limit(1)
  .executeTakeFirst();
// If found with status success|pending → load snapshot and return IDEMPOTENT_REPLAY (HTTP 200)
```

### 6. Atomic transaction (UnitOfWork)

Inside `uow.run(async (tx) => { ... })`:

**6a. Insert EligibilitySnapshot (status = pending)**

```typescript
await db
  .insertInto('eligibility_snapshots')
  .values({
    org_id: orgId,
    lead_id: leadId,
    request_ref: requestRef,          // generated: `ELIG-{leadCode}-{timestamp}`
    status: 'pending',
    created_by: userId,
    updated_by: userId,
  })
  .execute();
```

**6b. Insert IntegrationLog (status = pending)**

```typescript
await db
  .insertInto('integration_logs')
  .values({
    org_id: orgId,
    integration: 'los_eligibility',
    direction: 'outbound',
    lead_id: leadId,
    correlation_id: correlationId,
    idempotency_key: idempotencyKey ?? null,
    request_ref: requestRef,
    status: 'pending',
    created_by: userId,
    updated_by: userId,
  })
  .execute();
```

**6c. Transition lead stage via LeadService**

```typescript
await leadService.transitionStage(
  leadId,
  'eligibility_requested',
  guardCtx,          // { consentId, productConfigId }
  lead.version,      // expectedVersion for optimistic lock
  tx,
);
// LeadService internally: UPDATE leads SET stage='eligibility_requested', version=version+1
//   WHERE lead_id=:id AND org_id=:org AND version=:v AND deleted_at IS NULL
// + INSERT stage_history
// + AuditAppender.emit(stage_transition, ...)
// + OutboxService.emit('LEAD_STAGE_CHANGED', ...)
// → CONFLICT (409) if version mismatch
// → VALIDATION_ERROR + STAGE_GUARD_FAILED if stage guard fails
```

**6d. Insert DataSharingLog (append-only sink; within UoW tx)**

```typescript
await db
  .insertInto('data_sharing_logs')
  .values({
    org_id: orgId,
    lead_id: leadId,
    recipient: 'LOS',
    purpose: 'product_eligibility',
    data_category: 'financial',
    consent_id: consent.consent_id,
    status: 'shared',
    created_by: userId,
    updated_by: userId,
  })
  .execute();
```

The transaction commits here. The LOS call happens **after** commit to avoid holding the DB connection open during a network I/O.

### 7. Call LOS via IntegrationGateway (post-commit)

```typescript
const losResponse = await integrationGateway.call(
  LosPort,
  {
    method: 'POST',
    path: '/eligibility',
    body: losPayload,   // built by EligibilityPayloadBuilder (see Validation Logic)
  },
  { idempotencyKey: requestRef, integrationLogId },
);
```

`IntegrationGateway` updates `integration_logs` (status/http_status/completed_at) internally.

### 8. Update EligibilitySnapshot (success path)

If `losResponse.status === 'success'`:

```typescript
await db
  .updateTable('eligibility_snapshots')
  .set({
    status: 'received',
    indicative_amount: losResponse.data.indicativeAmount ?? null,
    tenure_months: losResponse.data.tenureMonths ?? null,
    rate_range: losResponse.data.rateRange ?? null,
    conditions: losResponse.data.conditions ?? null,
    validity_until: losResponse.data.validityUntil ?? null,
    response_basis: losResponse.data.responseBasis ?? 'indicative',
    updated_by: userId,
    updated_at: new Date(),
  })
  .where('eligibility_snapshot_id', '=', snapshotId)
  .where('org_id', '=', orgId)
  .execute();
```

Then emit `ELIGIBILITY_RECEIVED` outbox event (standalone tx):

```typescript
await outboxService.emit(
  'ELIGIBILITY_RECEIVED',
  'eligibility_snapshot',
  snapshotId,
  { leadId, snapshotId, status: 'received' },
);
```

### 9. Timeout / failure path

If `losResponse.status === 'timeout'`:
- Snapshot stays `status = 'pending'` (no further DB write needed).
- `IntegrationGateway` enqueues a Cloud Tasks retry (exponential backoff, max 3 attempts).
- Return the `pending` snapshot to the caller — never block the lead workflow.

If `losResponse.status === 'failed'` (5xx, not timeout):
- Update snapshot `status = 'failed'`.
- Return `UPSTREAM_UNAVAILABLE` (503).

## Validation Logic

All validation uses Zod. The `ValidationPipe` runs at the controller boundary.

### Path param
| Field | Rule | Error field |
|---|---|---|
| `id` (lead UUID) | Must be a valid UUID v4 | `id` |

### Request header
| Field | Rule | Error field |
|---|---|---|
| `Idempotency-Key` | Optional; if present, max 120 chars, printable ASCII | `Idempotency-Key` |

### Service-layer validation (EligibilityMappingValidator)

After loading `ProductConfig.eligibility_mapping`, validate that the mapping is present and complete:

| Check | Rule | Error |
|---|---|---|
| `eligibility_mapping` present | Not null on ProductConfig | `VALIDATION_ERROR` — "Product configuration has no eligibility mapping. Contact IT." |
| Required mapping keys present | All keys required by LosPort payload schema exist in `eligibility_mapping` | `VALIDATION_ERROR` — "Eligibility mapping is incomplete for product {productCode}." |
| `LeadProductDetail` attributes cover mapped fields | Every LMS field referenced in the mapping has a non-null value in `lead_product_details.attributes` | `VALIDATION_ERROR` — "Required attribute '{field}' is missing from lead product details." |
| Lead stage guard | Stage must be `kyc_in_progress`; only valid transition is → `eligibility_requested` | `VALIDATION_ERROR` + `STAGE_GUARD_FAILED` |
| Consent present | Active `product_eligibility` grant exists | `FORBIDDEN` + `CONSENT_MISSING` |

### LOS payload builder (EligibilityPayloadBuilder)

Constructs the outbound LOS payload by applying `ProductConfig.eligibility_mapping` (a JSON object mapping LMS field paths to LOS field paths) to `LeadProductDetail.attributes` and supplementary lead fields (`lead_code`, `product_code`, source channel, KYC/doc status, consent ref). The payload is never logged at the debug level if it contains PII field values.

## Error Cases

| Scenario | Code | HTTP | `detail.reason` | `retryable` |
|---|---|---|---|---|
| Lead not found | `NOT_FOUND` | 404 | — | false |
| Lead outside caller scope | `FORBIDDEN` | 403 | — | false |
| No granted `product_eligibility` consent | `FORBIDDEN` | 403 | `CONSENT_MISSING` | false |
| `eligibility_mapping` null or incomplete | `VALIDATION_ERROR` | 400 | — | false |
| Lead not in `kyc_in_progress` stage | `VALIDATION_ERROR` | 400 | `STAGE_GUARD_FAILED` | false |
| Optimistic lock stale (`leads.version` mismatch) | `CONFLICT` | 409 | — | false |
| LOS HTTP 5xx (non-timeout) | `UPSTREAM_UNAVAILABLE` | 503 | — | true |
| LOS timeout | — (HTTP 200, snapshot `pending`) | 200 | — | n/a — retry enqueued |
| Replayed Idempotency-Key (existing pending/received) | — (HTTP 200, original snapshot) | 200 | `IDEMPOTENT_REPLAY` | — |
| Unauthenticated | `AUTH_REQUIRED` | 401 | — | false |
| Unhandled server error | `INTERNAL_ERROR` | 500 | — | false |

## Backend Flow (Step-by-Step)

```
POST /api/v1/leads/{id}/eligibility
│
├─ 1. CorrelationMiddleware: generate/propagate X-Correlation-Id
├─ 2. JwtAuthGuard: validate JWT, load user + ABAC attributes → 401 if invalid
├─ 3. AbacGuard @Requires('move_stage'): EntitlementService.can(user, 'move_stage', lead) → 403 if denied
├─ 4. ValidationPipe(Zod): validate path param {id} → 400 if invalid UUID
│
└─ EligibilityController.requestEligibility(id, headers, user)
   │
   ├─ 5. Load lead: db.selectFrom('leads')...where(lead_id, org_id, not deleted) → 404 if missing
   ├─ 6. Scope re-check (service layer defensive): owner_id / branch_id vs user → 403 if mismatch
   ├─ 7. Consent check: consent_records where purpose='product_eligibility', state='granted', not expired → 403+CONSENT_MISSING
   ├─ 8. Load ProductConfig: select eligibility_mapping → 400 if mapping null/incomplete
   ├─ 9. Load LeadProductDetail: select attributes → 400 if required attributes missing
   ├─ 10. Idempotency check: integration_logs where idempotency_key and integration='los_eligibility'
   │      → if pending/success snapshot exists: return snapshot + IDEMPOTENT_REPLAY (HTTP 200)
   │
   ├─ 11. EligibilityPayloadBuilder.build(lead, productConfig, lpd, consentId) → losPayload
   │
   ├─ 12. uow.run(tx):
   │      ├─ 12a. INSERT eligibility_snapshots (status='pending')
   │      ├─ 12b. INSERT integration_logs (status='pending', idempotency_key)
   │      ├─ 12c. LeadService.transitionStage(leadId, 'eligibility_requested', guardCtx, expectedVersion, tx)
   │      │       ├─ StageGuardService.evaluate('kyc_in_progress' → 'eligibility_requested', guardCtx)
   │      │       │   → VALIDATION_ERROR+STAGE_GUARD_FAILED if guard fails
   │      │       ├─ UPDATE leads SET stage='eligibility_requested', version=version+1 WHERE version=:v
   │      │       │   → CONFLICT(409) if version mismatch
   │      │       ├─ INSERT stage_history
   │      │       ├─ AuditAppender.emit('stage_transition', ...)
   │      │       └─ OutboxService.emit('LEAD_STAGE_CHANGED', ...)
   │      └─ 12d. INSERT data_sharing_logs (recipient='LOS', purpose='product_eligibility', consent_id)
   │         COMMIT (DB connection released here)
   │
   ├─ 13. integrationGateway.call(LosPort, { POST /eligibility, losPayload }, { idempotencyKey: requestRef })
   │      IntegrationGateway:
   │        ├─ Updates integration_logs.status='pending' (already done at insert)
   │        ├─ Calls LosPort.checkEligibility(losPayload) with timeout=LOS_TIMEOUT_MS
   │        └─ On result updates integration_logs: status, http_status, completed_at
   │
   ├─ 14a. SUCCESS path (losResponse.ok):
   │       ├─ UPDATE eligibility_snapshots SET status='received', indicative_amount, tenure_months, ...
   │       └─ OutboxService.emit('ELIGIBILITY_RECEIVED', snapshotId, ...)   [standalone tx]
   │
   ├─ 14b. TIMEOUT path:
   │       ├─ IntegrationGateway enqueues Cloud Tasks retry (backoff, max 3)
   │       └─ Return snapshot with status='pending' (HTTP 200)
   │
   └─ 14c. FAILURE path (5xx):
           ├─ UPDATE eligibility_snapshots SET status='failed'
           └─ Throw UPSTREAM_UNAVAILABLE (503) → AllExceptionsFilter → envelope response
```

## State Machine & Transaction Boundaries

### Lead stage transition

```
kyc_in_progress  ──[POST /eligibility + guards pass]──>  eligibility_requested
```

- **Pre-condition:** `lead.stage = 'kyc_in_progress'`; active `product_eligibility` consent; complete `eligibility_mapping`; sufficient KYC.
- **Invalid:** Any stage other than `kyc_in_progress` → `VALIDATION_ERROR` + `STAGE_GUARD_FAILED`.
- **Terminal guard:** `handed_off` is terminal; transition is rejected.

### EligibilitySnapshot status machine (BRD §10 / state-machines.md compact machines)

```
pending  ──[LOS responds OK]──>  received
pending  ──[LOS 5xx]──────────>  failed
```

Snapshot is read-only once `received`. UI never shows editable fields.

### Transaction boundary (architecture §11.1)

```
┌─ UnitOfWork tx (single DB connection) ─────────────────────────────┐
│  INSERT eligibility_snapshots (status=pending)                      │
│  INSERT integration_logs (status=pending)                           │
│  UPDATE leads (stage transition, version++)                         │
│  INSERT stage_history                                               │
│  AuditAppender.emit (async to single-writer consumer)               │
│  OutboxService.emit LEAD_STAGE_CHANGED                              │
│  INSERT data_sharing_logs                                           │
└─────────────────────────────────────── COMMIT ─────────────────────┘
           ↓ (post-commit, no DB connection held)
┌─ IntegrationGateway call ──────────────────────────────────────────┐
│  LosPort.checkEligibility(losPayload)                               │
│  UPDATE integration_logs (status/http_status/completed_at)          │
└─────────────────────────────────────────────────────────────────────┘
           ↓ (success only)
┌─ Snapshot update tx ───────────────────────────────────────────────┐
│  UPDATE eligibility_snapshots (status=received, response fields)    │
│  OutboxService.emit ELIGIBILITY_RECEIVED                            │
└─────────────────────────────────────────────────────────────────────┘
```

**Rollback behaviour:** if the UnitOfWork transaction fails at any step (e.g. optimistic lock on leads, DB error on integration_logs insert), all four writes roll back atomically — no orphaned snapshot, no stale stage, no partial data sharing log.

## External Service Calls via IntegrationGateway

### LosPort — eligibility check

**Port:** `LosPort` (interface in `core/integration/ports/los.port.ts`)
**Adapters:** `LosHttpAdapter` (prod), `LosMockAdapter` (dev/test, `LOS_MOCK=true`)
**Method:** `LosPort.checkEligibility(payload: LosEligibilityPayload): Promise<LosEligibilityResponse>`

**Env vars required:**
- `LOS_BASE_URL` — base URL of the LOS REST API
- `LOS_API_KEY` — API key for outbound auth (from Secret Manager)
- `LOS_MOCK` — if `'true'`, routes to `LosMockAdapter`
- `LOS_TIMEOUT_MS` — request timeout in milliseconds (default 10000)

**Payload shape (built from `eligibility_mapping`):**
```typescript
interface LosEligibilityPayload {
  leadCode: string;          // lead.lead_code
  productCode: string;       // lead.product_code
  sourceChannel: string;     // lead.channel_created_by
  consentRef: string;        // consent_records.consent_id (UUID)
  kycStatus: string;         // lead.kyc_status
  attributes: Record<string, unknown>;  // mapped from lead_product_details.attributes
  idempotencyKey: string;    // request_ref (ELIG-{leadCode}-{timestamp})
}
```

**Response shape:**
```typescript
interface LosEligibilityResponse {
  requestRef: string;
  indicativeAmount: string | null;   // decimal string, NUMERIC(15,2)
  tenureMonths: number | null;
  rateRange: string | null;          // e.g. "10.5-12.0"
  conditions: Record<string, unknown> | null;
  validityUntil: string | null;      // ISO8601 UTC
  responseBasis: 'indicative' | 'preliminary' | 'final';
}
```

**Failure handling (per integration-map.md):**
| Failure | Detection | Response |
|---|---|---|
| Timeout (`>LOS_TIMEOUT_MS`) | Promise timeout | Snapshot stays `pending`; IntegrationGateway enqueues Cloud Tasks retry (max 3, exp backoff); HTTP 200 returned to caller |
| LOS 5xx | HTTP status ≥ 500 | Snapshot updated to `failed`; `UPSTREAM_UNAVAILABLE` (503) returned; retry queued |
| LOS 4xx (bad payload) | HTTP status 400–499 | `VALIDATION_ERROR` (400) logged internally; treat as mapping error; do not retry |
| Circuit breaker open | Repeated failures | `UPSTREAM_UNAVAILABLE` (503) immediately; operations queue alerted |
| Mapping missing (pre-call) | EligibilityMappingValidator | `VALIDATION_ERROR` (400); LOS never called |

**No raw PII** (name, mobile, pan_token, aadhaar_ref_token) is included in the LOS payload. Only the attributes defined in `eligibility_mapping` (IT-approved, BRD §5.9) are sent.

## UI Component Tree

The Eligibility card lives within the Lead 360 view (`apps/web/src/components/lead360/EligibilityCard.tsx`). It is read-only at all times — no inline editing.

```
Lead360Page
└── EligibilityCard
    ├── PageHeader (title="Eligibility", status chip)
    ├── [status='pending']  → LoadingSkeleton + "Awaiting LOS response"
    ├── [status='received'] → EligibilityDetails
    │   ├── StatusChip (received, indicative/preliminary/final label)
    │   ├── IndeterminateLabel   — "Indicative" badge always shown unless responseBasis='final'
    │   ├── AmountDisplay        — INR formatted (MaskedField if role requires)
    │   ├── TenureDisplay        — "{n} months"
    │   ├── RateRangeDisplay     — "{lo}–{hi} % p.a."
    │   ├── ConditionsList       — read-only JSON conditions rendered as key-value pairs
    │   └── ValidityDisplay      — "Valid until dd-MM-yyyy" (date-fns IST)
    ├── [status='failed']   → ErrorState (icon + "LOS unavailable. Retry below.")
    │   └── RetryButton (calls POST /eligibility again, disabled while isSubmitting)
    ├── [no snapshot yet]   → EmptyState + RequestEligibilityButton
    │   └── RequestEligibilityButton
    │       ├── triggers POST /api/v1/leads/{id}/eligibility
    │       ├── disabled if lead.stage !== 'kyc_in_progress' or consent missing
    │       ├── shows spinner + disabled while isSubmitting
    │       └── Toast on success/error (non-blocking)
    └── DisabledOverlay (if lead.stage is terminal: 'handed_off'/'rejected')
```

**Data fetching:** TanStack Query `useQuery` for snapshot (polled every 15 s while `status='pending'`). Mutation via `useMutation` for the POST. Errors map `UPSTREAM_UNAVAILABLE` → "Service temporarily unavailable. Retrying shortly." and `CONSENT_MISSING` → "Consent for product eligibility check has not been captured."

**Accessibility:** `StatusChip` has `role="status"` + `aria-live="polite"` for pending → received transitions. All buttons have descriptive `aria-label`. `ConfirmDialog` is not required here (no destructive action). Eligibility card is keyboard-navigable; retry button reachable via Tab.

**i18n:** all labels via `t('los.eligibility.*')`.

## Assumptions

1. The `LosPort.checkEligibility` method signature and the LOS API's response shape (specifically field names for `indicativeAmount`, `tenureMonths`, `rateRange`, `responseBasis`) are defined in `los-mock.openapi.yaml` (integration-map.md). This LLD uses field names consistent with the mock spec; if the real LOS wire contract differs, only the `LosHttpAdapter` changes — the service and types are unaffected (ADR-4).
2. `LOS_TIMEOUT_MS` is set via environment (environment-contract.md, Stage 5). Default 10 000 ms used here.
3. `ProductConfig.eligibility_mapping` is a flat or nested JSON object whose structure is IT-approved (BRD §5.9). The exact shape of this mapping is not specified beyond "LMS field → LOS payload field". `EligibilityPayloadBuilder` treats it as a key-value map where keys are dot-notation paths in `LeadProductDetail.attributes` and values are paths in the LOS payload.
4. The `GET /leads/{id}/eligibility` read endpoint (to surface the latest snapshot without re-triggering the LOS call) is referenced in BRD §2 (Lead 360 reads) but is not listed as a separate FR-080 entry in `api-contract.yaml`. It is implemented as a service method used internally by Lead 360 (FR-051), not as a standalone FR-080 endpoint.
5. The `NotificationDispatchService` for `ELIGIBILITY_RECEIVED` (BRD §14 notification catalog) is invoked by the outbox consumer, not directly by this service. This FR emits the outbox event; the notification is a downstream concern (FR-101).

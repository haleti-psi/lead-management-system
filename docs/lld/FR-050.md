# FR-050 — Lead List & Saved Work Queues (Low-Level Design)

**Tier: 2 (Moderate)** — multi-table scope-filtered read (`leads` + joins) with allow-listed filter/sort, pagination clamp, role-based masking, and audited bulk actions; plus single-entity CRUD on `saved_views`. No state-machine transition, no external service call, no cross-entity transaction → not Tier 3; richer than a single happy path → not Tier 1. (Matches the `testing-contract.md` exemplar list, which names FR-050 under *Moderate*.)

## Summary

FR-050 gives internal staff (RM, BM, SM, HEAD, KYC) a filterable, server-paginated **lead list** with named **saved work queues** ("My Leads", "Hot", "SLA Breached", etc.) and a **saved-view** persistence facility. Every list read is hard scope-filtered by the caller's ABAC scope (FR-002) and masked per role; filter/sort fields are allow-listed; page size is clamped to ≤ 100. Saved queues are filter presets translated into the same allow-listed filter grammar; user-defined saved views are stored in `saved_views` and may be shared within the owner's scope.

This FR is **read-only on `leads`** — it never writes the `leads` table. The "bulk actions" in AC-4 are *fan-outs* to existing per-lead mutators (FR-030 reassign, FR-052 stage move, etc.), each already owned by its FR; FR-050 owns only (a) the **bulk-action authorisation gate** (`bulk_action` capability + scope check before dispatch) and (b) writing **one `audit_logs` intent row per bulk action** via `AuditAppender`. FR-050 does **not** re-implement those mutators.

## Tier 2 Specification

### Endpoint 1 — `GET /leads` (operationId `listLeads`)

Scope-filtered, masked, paginated lead list. Backs both the lead list and saved-queue chips (a saved queue is just a preset `filter`).

**Request (query string):**

| Param | Type | Default | Rule |
|---|---|---|---|
| `page` | integer | `1` | `≥ 1` (`PaginationParams`) |
| `limit` | integer | `25` | `1 ≤ limit ≤ 100`; over-limit → **clamped to 100** (not an error — BRD §FR-050 edge case "over-limit clamped") |
| `sort` | string | `created_at:desc` | `^(<allowed_field>):(asc\|desc)$`; field must be in the **sort allow-list** below |
| `filter` | object (bracketed query, e.g. `filter[stage]=docs_pending`) | `{}` | each key in the **filter allow-list**; values validated by per-key Zod schema |
| `q` | string | — | optional free-text search (name / mobile / lead_code / pan_masked / partner / gstin / los_application_id), min length 2 |

**Response `200` (`LeadList`, uniform envelope):**
```json
{
  "data": [
    {
      "lead_id": "uuid",
      "lead_code": "LD-2026-000123",
      "stage": "docs_pending",
      "product_code": "CV",
      "is_hot": true,
      "score": 78,
      "consent_status": "granted",
      "kyc_status": "in_progress",
      "name_masked": "Ra***** K****",
      "mobile_masked": "98xxxxxx10"
    }
  ],
  "meta": { "correlation_id": "…", "pagination": { "page": 1, "limit": 25, "total": 137 } },
  "error": null
}
```
- `data[]` shape is exactly the contract `Lead` schema (`api-contract.yaml#/components/schemas/Lead`). Raw `name`/`mobile`/`pan`/`gstin` are **never** serialized — only `*_masked` projections (masking interceptor; see Auth & Masking).
- `meta.pagination.total` is the scope-filtered count (a second `count(*)` with the same WHERE, no `LIMIT`).

**Errors:** `401 AUTH_REQUIRED`, `403 FORBIDDEN`, `400 VALIDATION_ERROR` (bad/unknown filter or sort field).

### Endpoint 2 — `GET /saved-views` (operationId `listSavedViews`)

Lists saved views visible to the caller: own (`owner_id = user`) **plus** shared views whose `scope` the caller is inside (`is_shared = true AND <scope predicate>`), within `org_id`.

**Response `200` (`Ok` envelope):**
```json
{
  "data": [
    { "saved_view_id": "uuid", "name": "Hot CV — North", "filter_json": { "stage": ["docs_pending"], "is_hot": true },
      "is_shared": true, "scope": "B", "owner_id": "uuid", "created_at": "…", "updated_at": "…" }
  ],
  "meta": { "correlation_id": "…", "pagination": { "page": 1, "limit": 25, "total": 6 } },
  "error": null
}
```
**Errors:** `401 AUTH_REQUIRED`. (No `403`: an authenticated user with `view_lead` may always list their own/shared views; rows outside scope are simply not returned.)

### Endpoint 3 — `POST /saved-views` (operationId `createSavedView`)

Persists a filter preset.

**Request body:**
```json
{ "name": "Hot CV — North", "filter_json": { "stage": ["docs_pending"], "is_hot": true, "product_code": ["CV"] },
  "is_shared": true, "scope": "B" }
```

**Response `201` (`Created` envelope):** `data` = the created saved-view row (same shape as list item).

**Errors:** `401 AUTH_REQUIRED`, `400 VALIDATION_ERROR` (name empty/too long, `filter_json` contains a non-allow-listed key or invalid value, `scope` not one of the enum, or `is_shared=true` with a `scope` broader than the caller's own scope).

### Auth Check (exact)

Global **`JwtAuthGuard`** authenticates (→ `401 AUTH_REQUIRED` on missing/invalid token). Authorisation via **`AbacGuard` + `@Requires(...)` → `EntitlementService.can(user, capability, resource)`**:

| Endpoint | `@Requires` | Scope enforcement |
|---|---|---|
| `GET /leads` | `@Requires('view_lead', leadListScopeResolver)` | `EntitlementService.can(user,'view_lead','leads')` gates the call; the **same scope** (`O/T/B/A/M`) is then compiled into the SQL `WHERE` by `LeadScopeService.applyScope(qb, user)` so no out-of-scope row is ever read. DPO (`M`) is read-only + force-masked. PARTNER/CUSTOMER are **not** FR-050 roles → resolver denies (`403`). |
| `GET /saved-views` | `@Requires('view_lead', orgScopeResolver)` | Visibility predicate (own ∪ in-scope shared) applied in SQL. |
| `POST /saved-views` | `@Requires('view_lead', orgScopeResolver)` | Caller may only create `is_shared=true` views with `scope ⊆` caller's own `view_lead` scope (else `400 VALIDATION_ERROR`, field `scope`). |
| Bulk-action gate (before dispatch) | `@Requires('bulk_action', leadListScopeResolver)` on the bulk endpoint that fans out | RM has **no** `bulk_action` → `403 FORBIDDEN`. BM=B, SM=T, KYC=B, HEAD=A. Each target lead is re-checked individually inside its owning mutator. |

**Owner-writes note:** FR-050 never writes `leads`. Bulk actions delegate to the owning mutator (`LeadService.assignOwner` for reassign, `LeadService.transitionStage` for stage moves) exactly as those FRs specify. `saved_views` is owned by M6 (this FR) — FR-050 is its sole writer.

### Data Operations (Kysely — typed, parameterised; no raw SQL)

**`LeadListRepository.list(user, params)`** (`apps/api/src/modules/workspace/lead-list.repository.ts`):
```ts
// 1) Base query — active leads only, joined for search/masking/source filter.
let qb = db
  .selectFrom('leads as l')
  .innerJoin('lead_identities as li', 'li.lead_identity_id', 'l.lead_identity_id')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .where('l.org_id', '=', user.orgId)
  .where('l.deleted_at', 'is', null);

// 2) ABAC scope — compiled from the caller's view_lead scope. NEVER skipped.
qb = leadScopeService.applyScope(qb, user); // O: l.owner_id = user.id
                                            // T: l.team_id   = ANY(user.teamIds)
                                            // B: l.branch_id = ANY(user.branchIds)
                                            // A: (no extra predicate)  M(DPO): A-rows but forced-masked

// 3) Allow-listed filters (each branch added only if key present & validated)
//    e.g. filter[stage], filter[product_code], filter[branch_id], filter[team_id],
//    filter[owner_id], filter[priority], filter[consent_status], filter[kyc_status],
//    filter[is_hot], filter[score_band], filter[sla_state], filter[partner], filter[date_from/date_to]
for (const [field, col, value] of compiledFilters) qb = qb.where(col, value.op, value.val);

// 4) Free-text q (parameterised; trigram-backed on li.name)
if (q) qb = qb.where((eb) => eb.or([
  eb('l.lead_code', 'ilike', `%${q}%`),
  eb('li.name', 'ilike', `%${q}%`),       // ix_lead_identities_name_trgm (GIN gin_trgm_ops)
  eb('li.mobile', '=', q),
  eb('li.pan_masked', 'ilike', `%${q}%`),
  eb('li.gstin', '=', q.toUpperCase()),
  eb('l.los_application_id', '=', q),
]));

// 5) Sort (allow-listed column + direction) then paginate
qb = qb.orderBy(sortColumn, sortDir)
       .limit(limit)              // clamped ≤ 100
       .offset((page - 1) * limit);

// 6) total = same WHERE, count(*) (no limit/offset)
```
**Selected columns only** (never `select *`): `l.lead_id, l.lead_code, l.stage, l.product_code, l.is_hot, l.score, l.consent_status, l.kyc_status, li.name, li.mobile, li.pan_masked`. Raw `name`/`mobile` are fetched solely so the masking interceptor can project `name_masked`/`mobile_masked`; they are stripped before serialization.

**`score_band`** mapping (FR-050 AC-3 "score band"; `leads.score` is `0..100`): `hot ≥ 75`, `warm 50–74`, `cold < 50`, `unscored = score IS NULL` → translated to a range predicate on `l.score`.

**`sla_state`** mapping (`leads.sla_first_contact_due_at`): `breached = due_at < now()`, `due_soon = due_at BETWEEN now() AND now()+interval`, `ok = due_at >= …`, `none = due_at IS NULL` (business-time comparison uses `now()`; SLA *recompute* is FR-104's job — FR-050 only reads the stored due-at).

**`SavedViewRepository`** (`saved-view.repository.ts`):
```ts
// list: own ∪ in-scope shared
db.selectFrom('saved_views')
  .where('org_id', '=', user.orgId)
  .where((eb) => eb.or([ eb('owner_id', '=', user.id),
                         eb.and([ eb('is_shared', '=', true), savedViewScopePredicate(eb, user) ]) ]))
  .selectAll().orderBy('updated_at', 'desc').limit(limit).offset((page-1)*limit);

// create: owner_id/created_by/updated_by = user.id; org_id = user.orgId
db.insertInto('saved_views')
  .values({ org_id: user.orgId, owner_id: user.id, name, filter_json, is_shared, scope,
            created_by: user.id, updated_by: user.id })
  .returningAll().executeTakeFirstOrThrow();
```
All saved-view writes are single-row, single-entity → **no `UnitOfWork`** required (architecture §11.1 only mandates it for multi-entity writes). `updated_at` is maintained by the existing `trg_saved_views_updated_at` trigger.

### Validation Logic (Zod; field → rule → message)

`ListLeadsQuerySchema` (`apps/api/src/modules/workspace/dto/list-leads.dto.ts`):

| Field | Rule | Message (→ `VALIDATION_ERROR.fields[]`) |
|---|---|---|
| `page` | int ≥ 1, default 1 | "page must be a positive integer" |
| `limit` | int 1..100; **>100 coerced to 100** (transform, not reject) | (no error; clamped) |
| `sort` | `^(lead_code\|created_at\|updated_at\|score\|stage\|priority\|sla_first_contact_due_at):(asc\|desc)$` | "sort field is not allowed" |
| `filter` keys | each key ∈ `FILTER_ALLOWLIST` | "unknown filter '<key>'" |
| `filter[stage]` | each ∈ `lead_stage` enum (`@shared/enums`) | "invalid stage value" |
| `filter[product_code]` | each ∈ `product_code` enum | "invalid product_code value" |
| `filter[priority]` | ∈ `priority` enum | "invalid priority value" |
| `filter[consent_status]` | ∈ `consent_status` enum | "invalid consent_status value" |
| `filter[kyc_status]` | ∈ `kyc_status` enum | "invalid kyc_status value" |
| `filter[is_hot]` | boolean | "is_hot must be true/false" |
| `filter[score_band]` | ∈ `{hot,warm,cold,unscored}` | "invalid score_band" |
| `filter[sla_state]` | ∈ `{breached,due_soon,ok,none}` | "invalid sla_state" |
| `filter[branch_id\|team_id\|owner_id\|partner]` | uuid (`partner` = partner code string) | "must be a valid id" |
| `filter[date_from\|date_to]` | ISO date; `from ≤ to` | "date_from must be ≤ date_to" |
| `q` | string, length ≥ 2 when present | "search needs at least 2 characters" |

`CreateSavedViewSchema`:

| Field | Rule | Message |
|---|---|---|
| `name` | string, 1..120 (matches `saved_views.name VARCHAR(120)`) | "name is required (max 120 chars)" |
| `filter_json` | object; every key ∈ `FILTER_ALLOWLIST` and each value passes the same per-key schema as above | "saved view contains an unsupported filter" |
| `is_shared` | boolean, default false | — |
| `scope` | ∈ `data_scope` enum (`O/T/B/R/A/...`) | "invalid scope" |
| `scope` (cross-field) | when `is_shared=true`, `scope` must be **⊆** caller's own `view_lead` scope | "you cannot share a view wider than your own scope" |

`FILTER_ALLOWLIST` (single source of truth, shared by list + saved-view validation): `product_code, stage, branch_id, team_id, owner_id, source, partner, priority, consent_status, kyc_status, is_hot, score_band, sla_state, date_from, date_to`. (Directly enumerates FR-050 AC-3 filters.)

### Backend Flow (`GET /leads`)

1. `JwtAuthGuard` validates the access token, loads `user` + ABAC attributes (`role`, `orgId`, `owner_id`, `teamIds[]`, `branchIds[]`, scope letter) onto the request.
2. `AbacGuard` runs `@Requires('view_lead', leadListScopeResolver)` → `EntitlementService.can(user,'view_lead','leads')`. Deny → `403 FORBIDDEN` (+ audit).
3. Controller parses query via `ListLeadsQuerySchema` (Zod). Invalid filter/sort → `400 VALIDATION_ERROR`. `limit` is transform-clamped to ≤ 100 here (no error).
4. `LeadListService.list(user, params)` calls `LeadListRepository.list(...)`:
   a. builds the base query (org + `deleted_at IS NULL`);
   b. `LeadScopeService.applyScope(qb, user)` injects the scope predicate **unconditionally**;
   c. appends compiled allow-listed filters + optional `q`;
   d. applies allow-listed `sort`, `limit`, `offset`;
   e. runs the row query and a parallel scope-identical `count(*)` for `total`.
5. `ResponseEnvelopeInterceptor` wraps `{ data, meta:{correlation_id, pagination}, error:null }`.
6. `MaskingService` interceptor projects `name → name_masked`, `mobile → mobile_masked`, drops `pan_masked`/raw PII not in the `Lead` schema, applying **strictest** masking for DPO (`M`).
7. Response returned (read; NFR p95 ≤ 500 ms — covered by `ix_leads_*` and `ix_lead_identities_name_trgm`).

### Backend Flow (bulk action — FR-050's slice only)

1. `JwtAuthGuard` + `AbacGuard @Requires('bulk_action', leadListScopeResolver)`. RM → `403`.
2. The set of `lead_id`s is **re-scoped**: any id outside the caller's `view_lead` scope is dropped before dispatch (defence in depth — a stale client list cannot act cross-scope).
3. For each in-scope lead, delegate to the owning mutator **inside its own `uow.run` transaction** (e.g. `LeadService.assignOwner` for reassign — owned by FR-030). FR-050 does **not** open a transaction across leads; each lead's mutation is its own atomic unit, so one failure does not roll back the others.
4. FR-050 emits **one `audit_logs` intent per bulk action** (action, actor, target count, filter/selection) via `AuditAppender` (append-only sink; single-writer chain per architecture §11.4). It does **not** write `leads`.
5. Return a per-item result list (succeeded / skipped-out-of-scope / failed-with-reason).

> **Note (best-effort):** the api-contract exposes no dedicated `/leads/bulk-*` path; AC-4 bulk actions are realised as a **client-side fan-out** over the existing per-lead endpoints (`POST /leads/{id}/reassign`, `PATCH /leads/{id}/stage`), each already authz/audit-complete in its FR. FR-050 backend therefore contributes the `bulk_action` capability gate + the audit intent, not a new endpoint. See Ambiguities.

### State Machine

**Not applicable.** FR-050 performs no `lead_stage` transition (it is read-only on `leads`; `saved_views` has no status column). Stage moves triggered *via* a bulk action are governed by FR-052 / `state-machines.md`, not by this FR.

### Transaction Boundaries

- `GET /leads`, `GET /saved-views`: read-only, no transaction.
- `POST /saved-views`: a single-row INSERT — no `UnitOfWork` (only multi-entity writes need it, §11.1).
- Bulk actions: each delegated mutation runs in **its own** `uow.run(tx => …)` owned by the delegated FR; FR-050 does not span a transaction across leads.

### External Service Calls

**None.** No `IntegrationGateway`/port usage. (Search is local SQL; no LOS/KYC/comms calls.)

## Assumptions & Ambiguities (resolved best-effort; logged for write-back)

1. **No dedicated bulk endpoint in the contract.** `api-contract.yaml` lists no `/leads/bulk-action` path; AC-4 ("bulk actions respect role/scope and write audit") is implemented as a **client fan-out** over existing per-lead mutator endpoints, with FR-050 owning the `bulk_action` capability gate + one `audit_logs` intent per action. If a single transactional bulk endpoint is later required, it must be added to the contract first (do not invent a path).
2. **`q` free-text param is not enumerated in the contract for `/leads`** (the contract `GET /leads` lists `page/limit/filter/sort`). AC-1 explicitly requires search by name/mobile/lead_code/masked-PAN/partner/asset-id/GSTIN/LOS-app-id, so `q` is added as an optional query param consistent with the FR-054 search semantics. Flagged for contract write-back.
3. **"vehicle/asset ID" search (AC-1)** maps to `lead_product_details.attributes` (JSONB, `ix_lpd_attributes` GIN) rather than a first-class column; included as an extension of `q` only if `lead_product_details` is joined. Treated as out-of-core-scope for the initial list (asset-ID search may instead live in FR-054 global search); noted so it is not silently dropped.
4. **Built-in saved-queue definitions** (the 13 named queues) are not stored as rows; they are code-level filter presets in `saved-view-chips.tsx` mapped onto `FILTER_ALLOWLIST`. The mapping (e.g. "SLA Breached" → `filter[sla_state]=breached`, "Hot" → `filter[is_hot]=true`, "Customer Upload Received" → a stage/flag filter) follows the §10 stages; any preset whose mapping is ambiguous against the canonical stage set should be confirmed against `state-machines.md` before shipping.

## Summary

FR-122 governs the lifecycle of every data export from the LMS. When a user requests a report export (from FR-120/121 report views or the DLA registry), the system creates an `export_jobs` row, evaluates whether the export exceeds the approval threshold or requests unmasked PII, and either queues it immediately (`queued`) or halts it pending a manager/DPO approval (`awaiting_approval`). Approved/queued jobs are processed asynchronously: data is fetched with scope-enforced, role-appropriate masking, written to a watermarked CSV/XLSX file, uploaded to GCS, and the `artefact_ref` is stored. The requester is notified on completion. Every creation, approval decision, and download is appended to `audit_logs`. Files are served only via short-lived signed URLs and only to the original requester (or approver). This FR is Tier 3 because it involves multi-entity transactional writes, an asynchronous background job, GCS integration, approval workflow, role-based masking, and a non-trivial state machine.

### 1. POST /api/v1/exports — Create export job

**Request**
```json
{
  "report_code": "first_contact_sla",
  "filters": {
    "date_from": "2026-05-01",
    "date_to":   "2026-05-31",
    "branch_id": "uuid-branch-1"
  },
  "scope":         "B",
  "masking_level": "partial",
  "purpose":       "monthly_compliance_review"
}
```

**Headers**
- `Authorization: Bearer <jwt>` (required)
- `Idempotency-Key: <client-uuid>` (optional; replayed key returns original `ExportJob` response without re-creating)

**Response — 202 Accepted (job queued or awaiting approval)**
```json
{
  "data": {
    "export_job_id": "ej-uuid",
    "status":        "queued",
    "report_code":   "first_contact_sla",
    "masking_level": "partial",
    "scope":         "B",
    "created_at":    "2026-05-31T10:00:00Z"
  },
  "meta": { "correlation_id": "corr_…" },
  "error": null
}
```

When approval is required, `status` = `"awaiting_approval"` and HTTP is still `202`.

**Response — 409 Conflict (approval required — same shape, different status)**
Per `api-contract.yaml` comment `# EXPORT_APPROVAL_REQUIRED`, the contract returns `409` with sub-reason. However, because the job row is _created_ in `awaiting_approval` state (not rejected), the implementation must distinguish:

- If the NBFC config has `EXPORT_REQUIRE_APPROVAL_ON_THRESHOLD=true` and the estimated row count exceeds `EXPORT_APPROVAL_ROW_THRESHOLD` OR masking_level = `unmasked`:
  - Row is persisted with `status = awaiting_approval`.
  - HTTP **409** with body:
    ```json
    {
      "data": null,
      "meta": { "correlation_id": "corr_…" },
      "error": {
        "code":      "CONFLICT",
        "message":   "This action conflicts with the current state. Refresh and retry.",
        "retryable": false,
        "detail":    { "reason": "EXPORT_APPROVAL_REQUIRED", "export_job_id": "ej-uuid" }
      }
    }
    ```
  The `export_job_id` in `detail` allows the client to poll `GET /exports/{id}` for approval resolution.

### 4. POST /api/v1/exports/{id}/approve — Approve awaiting export

This endpoint is _not_ listed in `api-contract.yaml` as a distinct path for FR-122; however, the state machine requires an approval transition. See **Ambiguities** section. Best-effort design pending contract clarification: implement as a separate action endpoint on the same resource.

> **AMBIGUITY-1**: `api-contract.yaml` does not define a `POST /exports/{id}/approve` path. The BRD AC#2 requires approval for high-volume/sensitive exports; the state machine lists `awaiting_approval` as a valid status. Approval must be triggered by _some_ endpoint. Until contracts are updated, this LLD implements it as `POST /api/v1/exports/{id}/approve` with the same auth guard patterns and logs the ambiguity.

**Request body** — empty `{}`

**Roles permitted:** BM (scope B), SM (scope T), HEAD (scope A), DPO (scope M/A) — i.e., any user who holds `export` capability at scope ≥ requester's scope.

**Response — 200 OK**
```json
{
  "data": { "export_job_id": "ej-uuid", "status": "queued" },
  "meta": { "correlation_id": "corr_…" },
  "error": null
}
```

## Auth Check

### Guard stack (per architecture §5)
```
JwtAuthGuard (global) → AbacGuard(@Requires('export', scopeResolver)) → handler
```

### Capability matrix

| Endpoint | Capability | Scope enforcement |
|---|---|---|
| POST /exports | `export` | Actor's scope from `capability_matrix`; masking_level must be ≥ role minimum |
| GET /exports | `export` | Own jobs only (RM/KYC/PARTNER); or scoped-all (BM=B, SM=T, HEAD/DPO/ADMIN=A) |
| GET /exports/{id} | `export` | Must be `requested_by = actor.user_id` OR actor has scope-A export (HEAD/DPO/ADMIN) |
| POST /exports/{id}/approve | `export` | approver scope must be ≥ requester's scope; approver ≠ requester |

### Masking enforcement (auth-matrix capability_conditions)
> `"*.export": "applies strictest masking for the actor's scope; >threshold or unmasked PII -> EXPORT_APPROVAL_REQUIRED (FR-122)"`

The request field `masking_level` is validated against a role minimum table (enforced in `ExportService`):

| Role | Minimum masking_level allowed |
|---|---|
| RM | `full` (most restrictive; cannot reduce) |
| KYC, PARTNER | `full` |
| BM, SM | `partial` |
| HEAD | `partial` |
| DPO | `unmasked` (still triggers approval gate) |
| ADMIN | `partial` (system logs only; no lead-content access) |

If `masking_level` in the request is less restrictive than the role minimum, respond with `VALIDATION_ERROR` (400) + `fields: [{ field: "masking_level", issue: "masking level not permitted for your role" }]`.

### Scope cross-check
`scope` field in the request must be ≤ the actor's entitlement scope (e.g., an RM with scope O cannot request scope B). Violation → `FORBIDDEN` (403).

## Data Operations

All Kysely queries use parameterised bindings (no string interpolation). Table and column names are exact from `docs/data-model/schema.sql`.

### INSERT export_jobs (POST /exports)
```typescript
// ExportRepository.create(db | tx, dto, actorId)
await db
  .insertInto('export_jobs')
  .values({
    export_job_id:  generateUuid(),
    org_id:         actor.org_id,
    requested_by:   actorId,
    report_code:    dto.reportCode,
    filters:        JSON.stringify(dto.filters),
    scope:          dto.scope,
    masking_level:  dto.maskingLevel,
    status:         initialStatus,   // 'queued' | 'awaiting_approval'
    approver_id:    null,
    artefact_ref:   null,
    row_count:      null,
    created_by:     actorId,
    updated_by:     actorId,
  })
  .returningAll()
  .executeTakeFirstOrThrow();
```

### SELECT export_jobs for list (GET /exports)
```typescript
// scope-filtered, always LIMIT applied
let query = db
  .selectFrom('export_jobs')
  .selectAll()
  .where('org_id', '=', actor.org_id)
  .orderBy('created_at', 'desc')
  .limit(params.limit)       // max 100; default 25
  .offset((params.page - 1) * params.limit);

if (!actorCanSeeAll) {
  // RM/KYC/PARTNER see only their own
  query = query.where('requested_by', '=', actor.user_id);
} else if (actorScope === 'B') {
  // BM: filter by requested_by users in actor's branch
  // (join users on branch_id — see note below)
  query = query.where('requested_by', 'in',
    db.selectFrom('users').select('user_id').where('branch_id', '=', actor.branch_id)
  );
}
// HEAD/DPO/ADMIN with scope A: no additional filter (org_id already applied)
```

> **Note:** The `export_jobs` table does not store `branch_id`. BM-scope filtering requires a subquery join to `users`. This is the correct design per owner-writes (the export job is scoped to a requesting user, and that user carries branch membership).

### SELECT export_jobs by ID (GET /exports/{id})
```typescript
await db
  .selectFrom('export_jobs')
  .selectAll()
  .where('export_job_id', '=', jobId)
  .where('org_id', '=', actor.org_id)
  .executeTakeFirst();
// returns undefined → 404; access check in service layer
```

### UPDATE export_jobs (job state transitions — service only)
```typescript
// ExportRepository.updateStatus(db | tx, jobId, patch)
await db
  .updateTable('export_jobs')
  .set({ status: newStatus, updated_by: actorId, ...optionalFields })
  .where('export_job_id', '=', jobId)
  .where('org_id', '=', actor.org_id)
  .execute();
```
Updates are done only by `ExportService` (the owning module M13 service). No other service writes `export_jobs`.

### AuditLog writes (via AuditAppender — append-only)
Two audit actions are defined in `schema.sql` `audit_action` enum:
- `export_generate` — written when job is first created
- `export_download` — written when the signed URL is generated and served to the requester

```typescript
// Within the same tx as export_jobs INSERT
await auditAppender.emit({
  actor_id:    actorId,
  action:      'export_generate',
  entity_type: 'ExportJob',
  entity_id:   newJob.export_job_id,
  detail: {
    report_code:   dto.reportCode,
    scope:         dto.scope,
    masking_level: dto.maskingLevel,
    filters:       dto.filters,   // filter keys only, no PII values
    status:        initialStatus,
  },
}, tx);
```

### EventOutbox (via OutboxService — written in same tx as INSERT)
```typescript
await outboxService.emit(
  { event_code: 'EXPORT_COMPLETED', aggregate_type: 'ExportJob', aggregate_id: jobId, payload: { report_code, masking_level, row_count } },
  tx
);
// Only emitted when status transitions TO 'completed' (in the async worker, not on initial POST)
```

## Validation Logic (Zod)

### CreateExportDto
| Field | Rule | Error message |
|---|---|---|
| `report_code` | `z.string().min(1).max(60)` + must be in allow-list of valid report codes (FR-120/121 codes) | "report_code is required and must be a recognised report code" |
| `filters` | `z.record(z.unknown())` (JSONB); no arbitrary keys — validated against a per-report filter schema | "invalid filter key for report {report_code}" |
| `scope` | `z.enum(['O','T','B','R','A','P'])` | "scope must be a valid data scope code" |
| `masking_level` | `z.enum(['full','partial','unmasked'])` | "masking_level must be full, partial, or unmasked" |
| `purpose` | `z.string().min(1).max(255)` | "purpose is required" |

`date_from` / `date_to` inside filters: ISO-8601 date strings; `date_from` ≤ `date_to`; range ≤ 366 days (configurable).

### Approval threshold check (service logic, not Zod)
Before persisting with `status = queued`, `ExportService` runs a COUNT query using the same parameterised filters to get an estimated row count. If the count exceeds `EXPORT_APPROVAL_ROW_THRESHOLD` (env, default `5000`) OR `masking_level = 'unmasked'`, `status` is set to `awaiting_approval` and the response is HTTP 409 with sub-reason `EXPORT_APPROVAL_REQUIRED`.

## Error Cases

| Scenario | Code | HTTP | detail.reason | Triggering condition |
|---|---|---|---|---|
| Missing/invalid fields | `VALIDATION_ERROR` | 400 | — | Zod schema failure; also masking_level below role minimum; scope exceeds entitlement |
| Unauthenticated | `AUTH_REQUIRED` | 401 | — | Missing or expired JWT |
| Requesting outside entitlement scope | `FORBIDDEN` | 403 | — | `EntitlementService.can` returns false for `export` capability |
| Job not found / not in org | `NOT_FOUND` | 404 | — | `export_job_id` absent or `org_id` mismatch |
| Download by non-requester without scope-A | `FORBIDDEN` | 403 | — | Requester is not `requested_by` user and lacks A-scope export |
| Approval threshold / unmasked PII | `CONFLICT` | 409 | `EXPORT_APPROVAL_REQUIRED` | Row count ≥ threshold OR masking_level = unmasked |
| GCS upload failure during async generation | `INTERNAL_ERROR` | 500 (job status → `failed`) | — | GCS SDK error; job row updated; alert fired |
| Rate limit exceeded | `RATE_LIMITED` | 429 | — | >60 mutations/min per user (ThrottlerGuard) |
| Self-approval attempt | `FORBIDDEN` | 403 | — | `approver_id = requested_by` |
| Approving a non-awaiting_approval job | `CONFLICT` | 409 | — | Job status ≠ `awaiting_approval` |

## Backend Flow (Step by Step)

### POST /exports — create

1. **CorrelationMiddleware** stamps `X-Correlation-Id`.
2. **JwtAuthGuard** validates Bearer JWT; extracts `actor` (user_id, role, scope, branch_id, org_id).
3. **AbacGuard** calls `EntitlementService.can(actor, 'export', null)` — raises `FORBIDDEN` (403) on failure.
4. **ValidationPipe (Zod)** validates `CreateExportDto`; raises `VALIDATION_ERROR` (400) on failure.
5. **ExportService.create(dto, actor)**:
   a. Check `masking_level` ≥ role minimum; reject with `VALIDATION_ERROR` (400) if not.
   b. Check requested `scope` ≤ actor's entitlement scope; reject with `FORBIDDEN` (403) if not.
   c. Run parameterised COUNT estimate query (read replica if available) using `dto.filters` and `dto.scope` to get `estimatedRows`.
   d. Determine `initialStatus`:
      - `awaiting_approval` if `estimatedRows >= EXPORT_APPROVAL_ROW_THRESHOLD` OR `masking_level === 'unmasked'`
      - `queued` otherwise
   e. Open `UnitOfWork.run(async (tx) => { ... })`:
      - INSERT into `export_jobs` (step above).
      - `AuditAppender.emit({ action: 'export_generate', ... }, tx)`.
      - If `initialStatus === 'queued'`: `OutboxService.emit(EXPORT_COMPLETED` _pre-enqueue_, NOT the final complete event — we use a job-specific event or skip outbox until completion; see note below).
      - Commit.
   f. If `initialStatus === 'queued'`: enqueue async Cloud Tasks job (`ExportGenerationTask`) with `export_job_id`.
   g. If `initialStatus === 'awaiting_approval'`: do NOT enqueue yet; notify approver pool via `NotificationDispatchService` (out-of-band, post-commit).
6. Return HTTP 202 (queued) or HTTP 409 (awaiting_approval) with the job object.

> Note on outbox: `EXPORT_COMPLETED` is emitted by the async worker when generation finishes, not on initial creation. The POST endpoint does not emit `EXPORT_COMPLETED` into `event_outbox`.

### Async generation worker (Cloud Tasks / ExportGenerationTask)

1. Load `ExportJob` by `export_job_id`; validate `status = 'queued'`.
2. Update `status → 'running'`.
3. Build the data query for `report_code` with `filters` and `scope` constraints — exactly the same parameterised query used by the corresponding FR-120/FR-121 report endpoint, but unbounded (all rows matching scope/filter, chunked by cursor for memory safety).
4. Apply `MaskingService` at the row level using the job's `masking_level`. For `masking_level = 'partial'`: PAN → `ABCxxxx1F`, mobile → `98xxxxxx10`. For `masking_level = 'full'`: all PII columns replaced with `***`. For `masking_level = 'unmasked'`: no masking applied (only allowed post-approval).
5. Serialize rows to CSV (UTF-8 BOM for Excel compatibility).
6. Inject watermark row at the top of the file:
   ```
   # LMS Export | Generated by: {user_display_name} ({user_id}) | At: {ISO_timestamp} | Report: {report_code} | Masking: {masking_level}
   ```
7. Upload to GCS: `gs://{GCS_BUCKET}/exports/{org_id}/{export_job_id}.csv` using `@google-cloud/storage`.
8. Within `UnitOfWork`:
   - UPDATE `export_jobs` SET `status = 'completed'`, `row_count = N`, `artefact_ref = 'gs://...'`, `updated_by = system_uuid`.
   - `AuditAppender.emit({ action: 'export_generate', detail: { completed: true, row_count: N } })` — append a completion audit entry.
   - `OutboxService.emit('EXPORT_COMPLETED', 'ExportJob', export_job_id, { report_code, row_count, masking_level })`.
   - Commit.
9. Post-commit: `NotificationDispatchService` sends in-app notification to `requested_by` user.
10. On GCS failure: rollback; UPDATE `export_jobs` SET `status = 'failed'`; log error with `correlation_id`; alert via Cloud Monitoring.

### POST /exports/{id}/approve

1. Auth guards as above.
2. Load job; verify `status = 'awaiting_approval'`; raise `CONFLICT` (409) if not.
3. Verify `approver.user_id ≠ job.requested_by`; raise `FORBIDDEN` (403) if same (no self-approval).
4. Verify `EntitlementService.can(approver, 'export', ...)` at scope ≥ requester's scope; raise `FORBIDDEN` if not.
5. `UnitOfWork.run`:
   - UPDATE `export_jobs` SET `status = 'queued'`, `approver_id = approver.user_id`.
   - `AuditAppender.emit({ action: 'export_generate', detail: { event: 'approved', approver_id } })`.
   - Commit.
6. Enqueue `ExportGenerationTask`.
7. Return HTTP 200 with updated job.

### GET /exports/{id} — download URL generation

1. Auth + EntitlementService check.
2. Load job; 404 if absent.
3. Scope check: must be `requested_by = actor.user_id` OR `actor` holds A-scope export; else `FORBIDDEN`.
4. If `status = 'completed'` and `artefact_ref` set:
   - Generate a GCS signed URL (V4, TTL = `GCS_SIGNED_URL_TTL` seconds) for `artefact_ref`.
   - `AuditAppender.emit({ action: 'export_download', entity_id: jobId, detail: { masking_level, row_count } })` (no transaction needed — audit write via `AuditChainConsumer` queue).
5. Return job + `download_url` + `download_url_expires_at`.

## State Machine + Transaction Boundaries

```
queued  ──(worker picks up)──►  running  ──(GCS uploaded)──►  completed
                                    │                              │
                            (GCS fail)                   (user downloads)
                                    ▼                              │
                                  failed                   audit: export_download

awaiting_approval  ──(approver approves)──►  queued  ──► (above flow)
```

Valid transitions:

| From | To | Trigger | Actor | Side effects (within UnitOfWork tx) |
|---|---|---|---|---|
| `queued` | `running` | worker start | system | UPDATE export_jobs |
| `running` | `completed` | GCS success | system | UPDATE export_jobs (row_count, artefact_ref) + AuditLog + EventOutbox |
| `running` | `failed` | GCS error | system | UPDATE export_jobs + log error |
| `awaiting_approval` | `queued` | approver action | BM/SM/HEAD/DPO | UPDATE export_jobs (approver_id) + AuditLog |
| `failed` | `queued` | manual retry (future) | admin | Not in MVP scope; job can be re-submitted as new request |

Invalid transitions (→ 409 CONFLICT):
- `completed → any` (terminal; file already generated; re-export requires new job)
- `queued → awaiting_approval` (threshold check is at creation only)
- Self-approval (`approver_id = requested_by`)

### Transaction boundaries (UnitOfWork)

| Operation | Transaction scope |
|---|---|
| POST /exports | INSERT export_jobs + AuditLog(export_generate) in one tx |
| POST /exports/{id}/approve | UPDATE export_jobs + AuditLog(approval event) in one tx |
| Async worker completion | UPDATE export_jobs + AuditLog(completion) + EventOutbox in one tx |
| GET /exports/{id} download | AuditLog(export_download) via AuditChainConsumer queue (no tx needed) |

## External Service Calls via IntegrationGateway

FR-122 does not call LOS or KYC providers. The only external service call is GCS.

### GCS upload (async worker)
- **Client:** `@google-cloud/storage` (approved, `dependency-register.md`).
- **Not** routed through `IntegrationGateway` (GCS is a GCP-native service without an `IntegrationLog` pattern; it is used directly, similar to how M8 uploads documents).
- **Failure handling:** `StorageException` → catch, UPDATE `export_jobs.status = 'failed'`, log structured error with `correlation_id` and `export_job_id`. Alert fires via Cloud Monitoring on `INTERNAL_ERROR`.
- **No retry via Cloud Tasks** for GCS failures in MVP — the user re-submits a new export job. (Cloud Tasks retry is reserved for external provider calls via `IntegrationGateway`.)

### GCS signed URL (GET /exports/{id})
- Generated on-demand using `Storage.bucket(GCS_BUCKET).file(artefact_ref).getSignedUrl(...)`.
- TTL: `GCS_SIGNED_URL_TTL` seconds (default 600).
- The `artefact_ref` GCS path is never returned in API responses; only the signed URL is.

## Ambiguities

1. **AMBIGUITY-1 (Approval endpoint):** `api-contract.yaml` lists only `GET/POST /exports` and `GET /exports/{id}` for FR-122. No `POST /exports/{id}/approve` path is specified. The BRD AC#2 requires approval for threshold/unmasked exports. This LLD implements the approval as `POST /api/v1/exports/{id}/approve` (best effort). The contracts package must be updated to add this endpoint before implementation begins.

2. **AMBIGUITY-2 (Export GCS bucket):** `environment-contract.md` defines only `GCS_BUCKET` (used by M8 for documents). FR-122 generates export files and stores them in GCS (`artefact_ref`). It is ambiguous whether exports should share `GCS_BUCKET` with documents or use a separate bucket with different IAM/lifecycle policies. Best-effort: use `GCS_BUCKET` with an `exports/` prefix. The environment contract should be updated with a dedicated `GCS_EXPORT_BUCKET` variable if NBFC compliance requires separation.

3. **AMBIGUITY-3 (Export approval threshold config):** The BRD says "if configured" but `environment-contract.md` has no `EXPORT_APPROVAL_ROW_THRESHOLD` variable. Best-effort: default 5000 rows; `masking_level = unmasked` always requires approval regardless. The environment contract should add `EXPORT_APPROVAL_ROW_THRESHOLD` (default `5000`) and `EXPORT_REQUIRE_APPROVAL_ON_THRESHOLD` (default `true`).

4. **AMBIGUITY-4 (report_code allow-list):** The BRD defines report codes informally (`first_contact_sla`, `consent_ops`, etc.) but no canonical enum exists in `schema.sql` or `@shared/enums`. Best-effort: define a `ReportCode` Zod enum in `packages/shared/src/enums/report-code.ts` derived from FR-120/FR-121 report names. This should be formalised in the data model before implementation.

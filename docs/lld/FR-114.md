## Summary

FR-114 implements the full end-to-end grievance lifecycle for the NBFC Compliance Console (M12). It allows any internal role or a customer (via tokenised link) to raise a grievance, assigns an owner, starts an SLA timer via `SlaEngine`, and supports the state machine: `open → in_progress → escalated → resolved → closed`. Closing requires a `closure_proof_ref`. No/late response triggers escalation prompts via a scheduled sweep. The `GRIEVANCE_CREATED` event is emitted to the transactional outbox. Every status transition writes `grievances` + `audit_logs` + `event_outbox` in one UnitOfWork transaction. Duplicate/multi-channel grievances may be linked (via `lead_id`).

This is a **Tier 3** FR: it involves multi-entity writes, a status machine with guard-enforced transitions, SLA computation via `BusinessCalendarService` + `SlaEngine`, a scheduled escalation sweep, and a full UI in the Compliance Console.

### 3. `PATCH /api/v1/grievances/{id}` — Update / resolve / close grievance

**Auth:** `JwtAuthGuard` + `AbacGuard` + `@Requires('consent_ledger', ownershipOrDpoScopeResolver)`

Ownership check: the caller must be the `grievances.owner_id` OR hold `consent_ledger` scope `A` (DPO/HEAD). BM may act on branch-scoped grievances.

**Request body (all fields optional; only provided fields are updated):**
```json
{
  "status": "resolved",
  "response": "Apology sent; RM re-contacted customer on 2026-06-11.",
  "closureProofRef": "gcs://docs-bucket/grievance-closures/GRV-2026-000001.pdf",
  "ownerId": "uuid"
}
```

**Allowed transitions (enforced by `GrievanceService`):**

| From | To | Guard |
|---|---|---|
| `open` | `in_progress` | `owner_id` must be set |
| `in_progress` | `escalated` | automatic via sweep; manual allowed |
| `in_progress` / `escalated` | `resolved` | `response` must be non-empty |
| `resolved` | `closed` | `closure_proof_ref` must be non-empty |
| `closed` | any | Forbidden → `CONFLICT` (409); reopen creates a linked grievance |

**Response `200 OK`:** same shape as GET single grievance (full resource).

**Error `400 VALIDATION_ERROR`:** missing required field for the target transition (e.g. closing without `closure_proof_ref`).

## Auth Check

### Internal endpoints (`GET /grievances`, `POST /grievances`, `PATCH /grievances/{id}`)

```
JwtAuthGuard       → validates JWT, loads user + ABAC attributes onto request context
AbacGuard          → calls EntitlementService.can(user, 'consent_ledger', resource)
scopeResolver      → RM: owner_id = self; SM: team; BM: branch; HEAD/DPO: all org (scope A);
                      PARTNER: own partner's leads only (grievances linked to partner's leads)
```

Deny → `FORBIDDEN` (403), no resource detail exposed.

For PATCH specifically: the resolver additionally checks `grievances.owner_id === caller.userId` OR caller has scope `A`. Out-of-scope PATCH → `FORBIDDEN`.

### Customer-link endpoint (`POST /c/{token}/grievance`)

```
@Public() on JwtAuthGuard
CustomerLinkGuard  → validates opaque token + OTP step-up (M7 / self-service module)
                      scoped to the single lead the token was issued for
```

Customer intake is handled in `modules/self-service/` (FR-061). That handler calls `GrievanceService.create(...)` in M12 to persist the grievance with `source = 'customer_link'`. The `lead_id` is derived from the token's lead context.

## Data Operations

All queries use Kysely parameterised builders on `apps/api/src/core/db/` types generated by `kysely-codegen`. Column names are exact matches to `docs/data-model/schema.sql`.

### List grievances (scoped)

```typescript
db.selectFrom('grievances')
  .selectAll()
  .where('org_id', '=', orgId)
  .$if(scopeFilter.type === 'owner', qb => qb.where('owner_id', '=', callerId))
  .$if(scopeFilter.type === 'branch', qb =>
    qb.where('owner_id', 'in',
      db.selectFrom('users').select('user_id').where('branch_id', '=', branchId)
    )
  )
  .$if(statusFilter != null, qb => qb.where('status', '=', statusFilter!))
  .$if(categoryFilter != null, qb => qb.where('category', '=', categoryFilter!))
  .$if(ownerFilter != null, qb => qb.where('owner_id', '=', ownerFilter!))
  .$if(leadIdFilter != null, qb => qb.where('lead_id', '=', leadIdFilter!))
  .$if(fromFilter != null, qb => qb.where('created_at', '>=', fromFilter!))
  .$if(toFilter != null, qb => qb.where('created_at', '<=', toFilter!))
  .orderBy(sortColumn, sortDir)
  .limit(limit)          // max 100; NFR-17
  .offset((page - 1) * limit)
  .execute()
```

Count query uses the same WHERE clause with `.select(db.fn.count('grievance_id').as('total'))`.

### Insert grievance

```typescript
db.insertInto('grievances')
  .values({
    grievance_id:       generateUuid(),
    org_id:             orgId,
    grievance_no:       generatedNo,          // from CodeGenerator
    lead_id:            dto.leadId ?? null,
    source:             dto.source,
    category:           dto.category,
    description:        dto.description,
    owner_id:           dto.ownerId ?? null,
    sla_due_at:         slaDueAt,             // from SlaEngine.computeDue(...)
    status:             'open',
    response:           null,
    closure_proof_ref:  null,
    created_by:         callerId,
    updated_by:         callerId,
  })
  .returningAll()
  .executeTakeFirstOrThrow()
```

### Update grievance (transition)

```typescript
db.updateTable('grievances')
  .set({
    status:            newStatus,
    response:          dto.response ?? existingRow.response,
    closure_proof_ref: dto.closureProofRef ?? existingRow.closure_proof_ref,
    owner_id:          dto.ownerId ?? existingRow.owner_id,
    updated_by:        callerId,
    updated_at:        new Date(),
  })
  .where('grievance_id', '=', grievanceId)
  .where('org_id', '=', orgId)
  .executeTakeFirstOrThrow()
```

### Escalation sweep (scheduled job)

```typescript
db.selectFrom('grievances')
  .selectAll()
  .where('org_id', '=', orgId)
  .where('status', 'in', ['open', 'in_progress'])
  .where('sla_due_at', '<', new Date())
  .limit(100)
  .execute()
// For each: update status to 'escalated', emit audit, emit escalation notification
```

## Validation Logic

Implemented as a Zod schema at the controller boundary (NestJS `ValidationPipe`).

### `CreateGrievanceDto`

| Field | Rule | Error message |
|---|---|---|
| `leadId` | optional UUID v4; if present must exist in `leads` for this `org_id` | "Lead not found." |
| `source` | required; enum `grievance_source` | "Source must be one of: customer_link, rm, branch, call_centre, partner, admin." |
| `category` | required; enum `grievance_category` | "Category must be one of: service_delay, mis_selling, data_privacy, document_issue, staff_conduct, other." |
| `description` | required string; min 10 chars; max 2000 chars | "Description must be between 10 and 2000 characters." |
| `ownerId` | optional UUID v4; if present must reference an active user in this org | "Owner not found." |

### `UpdateGrievanceDto`

| Field | Rule | Error message |
|---|---|---|
| `status` | optional; enum `grievance_status`; must be a valid transition from current status | "Invalid status transition from `{current}` to `{target}`." |
| `response` | required when `status` transitions to `resolved`; max 2000 chars | "Response is required to resolve a grievance." |
| `closureProofRef` | required when `status` transitions to `closed`; max 255 chars; non-empty string | "Closure proof reference is required to close a grievance." |
| `ownerId` | optional UUID v4; if present must reference an active user | "Owner not found." |

At least one field must be present in the PATCH body (otherwise `VALIDATION_ERROR`).

## Backend Flow (Step-by-Step)

### Create Grievance (`POST /grievances`)

1. `CorrelationMiddleware` attaches `X-Correlation-Id` to the request context.
2. `JwtAuthGuard` validates the access token; loads user (with role, branch, scope attributes) onto request.
3. `AbacGuard` calls `EntitlementService.can(user, 'consent_ledger', null)`. Deny → `FORBIDDEN`.
4. `ThrottlerGuard` enforces mutation rate limit (60/min per user); over limit → `RATE_LIMITED`.
5. If `Idempotency-Key` header is present, check Redis for a cached response. On cache hit → return original result with `detail.reason = 'IDEMPOTENT_REPLAY'`.
6. `ValidationPipe(Zod)` validates `CreateGrievanceDto`. Fail → `VALIDATION_ERROR` with `fields[]`.
7. If `leadId` is provided, `GrievanceRepository.findLeadOrThrow(leadId, orgId)` verifies the lead exists in this org (Kysely `selectFrom('leads').where(...)` — scoped). Not found → `NOT_FOUND`.
8. If `ownerId` is provided, verify the user is active in this org. Not found → `VALIDATION_ERROR`.
9. Generate `grievance_no` via `CodeGenerator.nextGrievanceNo(orgId, tx)` (sequential, format `GRV-{YYYY}-{seq6}`).
10. Compute SLA due timestamp: `SlaEngine.computeDue({ target: 'grievance', orgId, branchId })` → queries `sla_policies` where `applies_to = 'grievance'` and `is_active = true`; feeds `threshold_minutes` through `BusinessCalendarService.addBusinessMinutes(now, threshold)` → returns `slaDueAt`.
11. Open `UnitOfWork.run(async (tx) => { ... })`:
    - a. `db.insertInto('grievances').values(...).returningAll()` — insert the new row.
    - b. `AuditAppender.emit({ action: 'lead_update', entityType: 'grievance', entityId: grievanceId, leadId: dto.leadId ?? null, actorId: callerId, detail: { event: 'GRIEVANCE_CREATED', category, source } }, tx)` — audit intent enqueued.
    - c. `OutboxService.emit({ eventCode: 'GRIEVANCE_CREATED', aggregateType: 'grievance', aggregateId: grievanceId, payload: { grievanceNo, category, source, ownerId, slaDueAt } }, tx)` — outbox row inserted in same tx.
12. If `Idempotency-Key` was present, cache the created response in Redis (TTL 24 h).
13. Return `201 Created` with the full grievance resource inside the uniform envelope.

### Update/Resolve/Close Grievance (`PATCH /grievances/{id}`)

1–4. Same global pipeline (CorrelationMiddleware, JwtAuthGuard, AbacGuard with ownership/scope check, ThrottlerGuard).
5. `ValidationPipe(Zod)` validates `UpdateGrievanceDto`. Fail → `VALIDATION_ERROR`.
6. `GrievanceRepository.findByIdOrThrow(id, orgId)` fetches current row. Not found → `NOT_FOUND`. Out-of-scope (owner check failed in AbacGuard step) → `FORBIDDEN`.
7. **State machine guard** (`GrievanceService.validateTransition(current.status, dto.status)`):
   - If `dto.status` is absent, no transition — skip guard.
   - Invalid transition → `CONFLICT` (409).
   - Transition to `resolved` without `response` → `VALIDATION_ERROR`.
   - Transition to `closed` without `closure_proof_ref` → `VALIDATION_ERROR`.
8. Open `UnitOfWork.run(async (tx) => { ... })`:
   - a. `db.updateTable('grievances').set({ ... }).where('grievance_id', '=', id).where('org_id', '=', orgId)`.
   - b. `AuditAppender.emit({ action: 'lead_update', entityType: 'grievance', entityId: id, actorId: callerId, detail: { transition: { from: current.status, to: newStatus } } }, tx)`.
   - c. If `dto.status === 'resolved'`, emit `GRIEVANCE_CREATED` event is not re-emitted; instead emit a generic compliance event via `OutboxService` if downstream consumers need resolution events (see Ambiguities).
9. Return `200 OK` with the updated resource.

### Escalation Sweep (Scheduled — Cloud Scheduler → Cloud Tasks)

A Cloud Tasks handler (`GrievanceEscalationJob`) runs on a configurable schedule (e.g. every 30 minutes):

1. Query `grievances` where `status IN ('open', 'in_progress')` AND `sla_due_at < now()` AND `org_id = orgId`. LIMIT 100.
2. For each breached grievance, open `UnitOfWork.run`:
   - a. Update `status` to `'escalated'`, `updated_by = SYSTEM_USER_ID`.
   - b. Append to `audit_logs` via `AuditAppender`.
   - c. Dispatch escalation notification via `NotificationDispatchService.send(...)` (post-commit, does not roll back the transition). Notification targets come from `sla_policies.escalation_chain`.
3. Log a structured summary: `{ module: 'compliance', job: 'grievance-escalation', escalatedCount, correlationId }`.

## State Machine

Reference: `docs/contracts/state-machines.md` §Grievance.

```
open ──[owner assigned]──► in_progress
in_progress ──[sla breach / manual]──► escalated
in_progress / escalated ──[response provided]──► resolved
resolved ──[closure_proof_ref provided]──► closed

closed ──► any   FORBIDDEN — CONFLICT (409); reopen = create a linked new grievance
```

| Transition | Guard | Side effects (same UnitOfWork tx) |
|---|---|---|
| `open → in_progress` | `owner_id` must be set | `grievances` update + audit |
| `in_progress → escalated` | SLA breached OR manual by owner/DPO | `grievances` update + audit + escalation notification (post-commit) |
| `in_progress/escalated → resolved` | `response` non-empty | `grievances` update + audit |
| `resolved → closed` | `closure_proof_ref` non-empty | `grievances` update + audit |
| `closed → *` | blocked | `CONFLICT` (409) |

## Transaction Boundaries (UnitOfWork)

Every write that changes `grievances.status` or creates a new grievance opens a single `UnitOfWork` transaction that atomically writes:

1. `grievances` (INSERT or UPDATE)
2. `audit_logs` intent via `AuditAppender` (inserted into the outbox/queue by the single-writer consumer)
3. `event_outbox` row (INSERT) via `OutboxService.emit(..., tx)`

If any step fails, the entire transaction rolls back. No partial state persists. Notification/escalation dispatch is post-commit and retryable; its failure does not revert the transition.

---

## External Service Calls

FR-114 does **not** call any external provider directly. All external communication (escalation notifications) routes through `NotificationDispatchService` → `NotificationChannelPort` (M11). That service checks `NotificationPreference` + `consent_basis` before dispatch (FR-103/101).

SLA computation uses `SlaEngine` + `BusinessCalendarService` — both local services (`core/sla/`). No `IntegrationGateway` call is required for this FR.

## Assumptions

1. `CodeGenerator.nextGrievanceNo` uses a database sequence or atomic counter scoped to `(org_id, year)` to produce `GRV-{YYYY}-{seq6}` without gaps (consistent with the `lead_code` pattern documented in `shared-utilities.md`). Format is assumed to be `GRV-{YYYY}-{seq6}` by analogy; the exact format is not specified in any contract.
2. The `sla_policies` table has at least one active row with `applies_to = 'grievance'` seeded by M14 (FR-131). If no active grievance SLA policy exists, `sla_due_at` is set to `null` and logged as a warning (not a hard error).
3. The `audit_action` enum value `lead_update` is reused for grievance lifecycle events (no dedicated `grievance_*` action exists in the schema's `audit_action` enum). `entity_type = 'grievance'` disambiguates.
4. Escalation notification recipients are derived from `sla_policies.escalation_chain` JSONB (array of user IDs or role codes). The exact shape of `escalation_chain` is not specified in the contracts; this LLD assumes `[{ "userId": "uuid" }]` or `[{ "role": "DPO" }]` entries.
5. No outbox `event_code` for grievance resolution/closure exists in the schema's `event_code` enum (`GRIEVANCE_CREATED` is the only grievance event). Resolution/closure transitions emit only an `audit_logs` entry. If downstream consumers need a resolution event, a new enum value must be added and reflected in the data model.

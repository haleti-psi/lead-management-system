## Summary

FR-042 introduces two distinct sub-flows:

1. **Scheme administration** (Product Ops / ADMIN): create and list scheme master records (`schemes` table) via `POST /admin/schemes` and `GET /admin/schemes`. A scheme captures festival/dealer campaign metadata — code, name, product affinity, subvention flag, and validity window. Schemes are org-scoped and must have `is_active = true` to be attached to leads.

2. **Scheme attachment** (RM / BM): an authorised RM or BM attaches an active, product-matched scheme to a lead by setting `lead_product_details.attributes.scheme_code`. The BRD requires that an expired or inactive scheme cannot be attached. Scheme metadata is later included in the LOS eligibility payload (FR-080) if `product_configs.eligibility_mapping` contains a scheme mapping.

No stage transition is triggered by scheme attachment; it is a profile update on `lead_product_details`. LMS never computes pricing or sanction — scheme metadata is pass-through to LOS.

### 2. Create a scheme

```
POST /api/v1/admin/schemes
```

**Request body**

```json
{
  "code": "DEALER-TW-Q3",
  "name": "Dealer TW Quarter 3",
  "product_code": "TW",
  "subvention_flag": true,
  "valid_from": "2026-07-01",
  "valid_to": "2026-09-30"
}
```

| Field | Type | Required | Constraints |
|-------|------|----------|-------------|
| `code` | string | Yes | 1–40 chars; alphanumeric + hyphens; unique per org |
| `name` | string | Yes | 1–120 chars |
| `product_code` | enum | Yes | one of `CV\|CAR\|TRACTOR\|CE\|TW\|SBL\|HRM`; nullable means all-product |
| `subvention_flag` | boolean | No | default `false` |
| `valid_from` | date (ISO) | Yes | `YYYY-MM-DD`; must be ≤ `valid_to` |
| `valid_to` | date (ISO) | Yes | `YYYY-MM-DD`; must be ≥ `valid_from` |

**Response — 201 Created**

```json
{
  "data": {
    "scheme_id": "uuid",
    "code": "DEALER-TW-Q3",
    "name": "Dealer TW Quarter 3",
    "product_code": "TW",
    "subvention_flag": true,
    "valid_from": "2026-07-01",
    "valid_to": "2026-09-30",
    "is_active": true,
    "created_at": "2026-06-09T08:30:00Z"
  },
  "meta": { "correlation_id": "uuid" },
  "error": null
}
```

**Response — error cases**

| HTTP | Code | Condition |
|------|------|-----------|
| 400 | `VALIDATION_ERROR` | Any field fails validation (carries `fields[]`) |
| 401 | `AUTH_REQUIRED` | No valid JWT |
| 403 | `FORBIDDEN` | Caller lacks `configuration` capability |
| 409 | `CONFLICT` | Scheme `code` already exists for this `org_id` |

### 3. Attach scheme to a lead

This uses the existing `PATCH /api/v1/leads/{id}` endpoint (FR-050/FR-011). Scheme attachment is a partial update to `lead_product_details.attributes`.

```
PATCH /api/v1/leads/{id}
```

**Request body (partial — scheme only)**

```json
{
  "product_detail": {
    "scheme_code": "DEALER-TW-Q3"
  }
}
```

| Field | Type | Required | Constraints |
|-------|------|----------|-------------|
| `product_detail.scheme_code` | string | Yes (for attach) | must match `schemes.code`; scheme must be active and product-matched |

To detach a scheme pass `"scheme_code": null`.

**Response — 200 OK**

```json
{
  "data": {
    "lead_id": "uuid",
    "lead_product_detail": {
      "lead_product_detail_id": "uuid",
      "attributes": { "scheme_code": "DEALER-TW-Q3" },
      "validation_status": "valid"
    }
  },
  "meta": { "correlation_id": "uuid" },
  "error": null
}
```

**Response — error cases**

| HTTP | Code | Condition |
|------|------|-----------|
| 400 | `VALIDATION_ERROR` | `scheme_code` not found, or scheme is inactive, or scheme product does not match the lead's `product_code` |
| 401 | `AUTH_REQUIRED` | No valid JWT |
| 403 | `FORBIDDEN` | Caller lacks `edit_lead` or is out of scope |
| 404 | `NOT_FOUND` | `lead_id` does not exist (or out of scope) |
| 409 | `CONFLICT` | Optimistic-lock mismatch on `leads.version` |

## Auth Check

### Scheme administration (GET / POST /admin/schemes)

| Guard | Detail |
|-------|--------|
| `JwtAuthGuard` | Global; applied automatically |
| `AbacGuard` | `@Requires('configuration', scopeResolver)` |
| `EntitlementService.can(user, 'configuration', resource)` | Roles with `configuration`: ADMIN (A), BM (B), HEAD (A), KYC (B) |

BRD §3.1 maps "Product Ops" to the `ADMIN` role for configuration purposes. Only ADMIN / HEAD have org-wide scope `A`; BM/KYC are branch-scoped `B` (schemes are global/org-level, so effectively only ADMIN/HEAD can create; BM/KYC can list).

For `GET /admin/schemes` used as a scheme picker by RMs, the controller additionally accepts `view_lead` capability as sufficient for read-only access (scheme list is non-sensitive configuration data needed by capture flows).

### Scheme attachment (PATCH /leads/{id})

| Guard | Detail |
|-------|--------|
| `JwtAuthGuard` | Global |
| `AbacGuard` | `@Requires('edit_lead', scopeResolver)` where `scopeResolver` checks `lead.owner_id` (RM=O, BM=B, SM=T, HEAD=A) |
| `EntitlementService.can(user, 'edit_lead', lead)` | RM: scope O (own leads); BM: scope B; SM/HEAD: wider |

PARTNER role is excluded — partners do not attach schemes. ADMIN has no standing lead-content access.

## Data Operations

All queries use Kysely parameterised builders. Column names are exact as per `schema.sql`.

### 3.1 List schemes

```typescript
// SchemeRepository.list(orgId, filters, pagination)
db.selectFrom('schemes')
  .selectAll()
  .where('org_id', '=', orgId)
  .$if(filters.product_code !== undefined, (qb) =>
    qb.where('product_code', '=', filters.product_code)
  )
  .$if(filters.is_active !== undefined, (qb) =>
    qb.where('is_active', '=', filters.is_active)
  )
  .orderBy('valid_from', 'desc')
  .limit(pagination.limit)
  .offset((pagination.page - 1) * pagination.limit)
  .execute();

// Count query for pagination.total
db.selectFrom('schemes')
  .select(db.fn.countAll<number>().as('total'))
  .where('org_id', '=', orgId)
  .$if(...)  // same filters
  .executeTakeFirstOrThrow();
```

### 3.2 Create scheme

```typescript
// SchemeRepository.create(dto, actorId, orgId, tx?)
db.insertInto('schemes')
  .values({
    org_id: orgId,
    code: dto.code,
    name: dto.name,
    product_code: dto.product_code ?? null,
    subvention_flag: dto.subvention_flag ?? false,
    valid_from: dto.valid_from,
    valid_to: dto.valid_to,
    is_active: true,
    created_by: actorId,
    updated_by: actorId,
  })
  .returningAll()
  .executeTakeFirstOrThrow();
```

Unique constraint `uq_schemes_code` on `(org_id, code)` catches duplicates — catch `UniqueViolationError` from Kysely/pg and rethrow as `CONFLICT`.

### 3.3 Resolve active scheme for attachment

```typescript
// SchemeRepository.findActiveByCode(code, orgId)
db.selectFrom('schemes')
  .select(['scheme_id', 'code', 'product_code', 'valid_from', 'valid_to', 'is_active'])
  .where('org_id', '=', orgId)
  .where('code', '=', code)
  .executeTakeFirst();
// Returns undefined if not found → NOT_FOUND or VALIDATION_ERROR depending on context
```

### 3.4 Update LeadProductDetail attributes (scheme attach)

```typescript
// LeadProductDetailRepository.upsertAttributes(leadId, patch, actorId, tx)
// Uses a Kysely JSONB merge so other attributes are preserved
db.updateTable('lead_product_details')
  .set({
    attributes: db.raw(
      `attributes || ?::jsonb`,
      [JSON.stringify({ scheme_code: patch.scheme_code })]
    ),
    updated_by: actorId,
    updated_at: new Date(),
  })
  .where('lead_id', '=', leadId)
  .executeTakeFirstOrThrow();
```

Note: `lead_product_details` has a `UNIQUE (lead_id)` constraint — one row per lead. `attributes` is JSONB and merges the new key without clobbering existing fields.

### 3.5 Audit log (config_change for scheme create; lead_update for scheme attach)

Emitted via `AuditAppender` (intent only — single-writer `AuditChainConsumer` writes the chain):

```typescript
// For scheme creation
await auditAppender.emit({
  action: 'config_change',
  entity_type: 'scheme',
  entity_id: scheme.scheme_id,
  lead_id: null,
  detail: { code: scheme.code, product_code: scheme.product_code },
  actor_id: actorId,
}, tx);

// For scheme attachment to lead
await auditAppender.emit({
  action: 'lead_update',
  entity_type: 'lead_product_detail',
  entity_id: lpd.lead_product_detail_id,
  lead_id: leadId,
  detail: { scheme_code: newSchemeCode, previous_scheme_code: oldSchemeCode },
  actor_id: actorId,
}, tx);
```

## Validation Logic (Zod)

### CreateSchemeDto

```typescript
const CreateSchemeDto = z.object({
  code: z
    .string()
    .min(1, 'Scheme code is required')
    .max(40, 'Scheme code must not exceed 40 characters')
    .regex(/^[A-Z0-9][A-Z0-9\-]*$/, 'Scheme code must be uppercase alphanumeric with hyphens'),
  name: z
    .string()
    .min(1, 'Scheme name is required')
    .max(120, 'Scheme name must not exceed 120 characters'),
  product_code: z
    .enum(['CV', 'CAR', 'TRACTOR', 'CE', 'TW', 'SBL', 'HRM'])
    .nullable()
    .optional()
    .default(null),
  subvention_flag: z.boolean().optional().default(false),
  valid_from: z
    .string()
    .regex(/^\d{4}-\d{2}-\d{2}$/, 'valid_from must be a date in YYYY-MM-DD format'),
  valid_to: z
    .string()
    .regex(/^\d{4}-\d{2}-\d{2}$/, 'valid_to must be a date in YYYY-MM-DD format'),
}).refine(
  (d) => d.valid_to >= d.valid_from,
  { message: 'valid_to must be on or after valid_from', path: ['valid_to'] }
);
```

### AttachSchemeDto (nested in the lead PATCH payload)

```typescript
const AttachSchemeDto = z.object({
  scheme_code: z.string().min(1).max(40).nullable(),
});
```

Business-rule validation (cannot be expressed in Zod alone — validated in the service layer):

| Rule | Error |
|------|-------|
| Scheme not found for the org | `VALIDATION_ERROR`, `fields: [{ field: 'scheme_code', message: 'Scheme not found' }]` |
| `scheme.is_active === false` | `VALIDATION_ERROR`, `fields: [{ field: 'scheme_code', message: 'Scheme is inactive' }]` |
| `valid_to < today` (expired) | `VALIDATION_ERROR`, `fields: [{ field: 'scheme_code', message: 'Scheme has expired' }]` |
| `scheme.product_code !== null` and `scheme.product_code !== lead.product_code` | `VALIDATION_ERROR`, `fields: [{ field: 'scheme_code', message: 'Scheme is not available for this product' }]` |

## Backend Flow

### Flow A — Create scheme (POST /admin/schemes)

```
1. JwtAuthGuard validates JWT → user principal resolved.
2. AbacGuard: EntitlementService.can(user, 'configuration', 'schemes') → FORBIDDEN (403) if denied.
3. Controller parses body through CreateSchemeDto (Zod). Field errors → VALIDATION_ERROR (400) with fields[].
4. SchemeService.create(dto, user.id, user.org_id):
   a. Validate valid_from ≤ valid_to (Zod refinement already does this — service re-asserts).
   b. Call SchemeRepository.create(dto, actorId, orgId).
      - Catches UniqueViolationError (pg error code 23505) on uq_schemes_code → throw CONFLICT (409).
   c. Call AuditAppender.emit({ action: 'config_change', entity_type: 'scheme', ... }).
      (No UnitOfWork needed — single-entity write; audit is append-only and can be in the same or separate tx.)
5. Return 201 with the created scheme row wrapped in { data, meta, error:null }.
```

### Flow B — List schemes (GET /admin/schemes)

```
1. JwtAuthGuard validates JWT.
2. AbacGuard: caller must have 'configuration' OR 'view_lead' capability.
3. Controller parses query params (PaginationParams + optional product_code / is_active filters).
4. SchemeService.list(orgId, filters, pagination):
   a. SchemeRepository.list(orgId, filters, pagination) → paginated rows + count.
5. Return 200 with pagination metadata.
```

### Flow C — Attach scheme to lead (PATCH /leads/{id} with product_detail.scheme_code)

```
1. JwtAuthGuard validates JWT.
2. AbacGuard: EntitlementService.can(user, 'edit_lead', lead) — scope check (O/B/T/A).
3. Load lead (LeadRepository.findById): NOT_FOUND (404) if absent or out of scope.
4. Controller passes product_detail.scheme_code through AttachSchemeDto (Zod).
5. SchemeService.validateAndResolveScheme(schemeCode, lead.product_code, orgId, today):
   a. Query: SchemeRepository.findActiveByCode(schemeCode, orgId).
   b. If not found → VALIDATION_ERROR with fields: [scheme_code → 'Scheme not found'].
   c. If scheme.is_active === false → VALIDATION_ERROR with fields: [scheme_code → 'Scheme is inactive'].
   d. If scheme.valid_to < today (UTC date) → VALIDATION_ERROR with fields: [scheme_code → 'Scheme has expired'].
   e. If scheme.product_code !== null AND scheme.product_code !== lead.product_code
      → VALIDATION_ERROR with fields: [scheme_code → 'Scheme is not available for this product'].
6. Open UnitOfWork transaction:
   a. LeadProductDetailRepository.upsertAttributes(lead.lead_id, { scheme_code: schemeCode }, userId, tx).
      - JSONB merge preserves all other attributes.
   b. AuditAppender.emit({ action: 'lead_update', entity_type: 'lead_product_detail',
        lead_id, detail: { scheme_code: newCode, previous_scheme_code: oldCode } }, tx).
   c. (No outbox event needed — scheme attachment is a profile edit, not a stage transition.)
7. Return 200 with the updated LeadProductDetail.
```

## State Machine — LeadProductDetail.validation_status

Scheme attachment does not change `lead_product_details.validation_status` directly. The status is re-evaluated by the existing `ProductConfig.field_schema` validation job (FR-040). However:

- If after attachment all required `field_schema` fields are satisfied, the service may set `validation_status = 'valid'`.
- If scheme attachment introduces a schema violation (e.g., scheme field is required by the field_schema), `validation_status` transitions to `'invalid'`.

This re-evaluation is synchronous within the attach tx if the service calls `LeadProductDetailService.revalidate(leadId, tx)`, otherwise it is left as `'incomplete'` for the next save.

The lead `stage` is not affected. No `stage_history`, `event_outbox(LEAD_STAGE_CHANGED)`, or `LeadService.transitionStage` is invoked.

## Transaction Boundaries via UnitOfWork

| Sub-flow | Entities written | UnitOfWork required |
|----------|-----------------|---------------------|
| Create scheme | `schemes` (insert) + `audit_logs` (append) | No (single-table write; audit is a separate append-only sink) |
| Attach scheme to lead | `lead_product_details` (update) + `audit_logs` (append) | Yes — `uow.run(async (tx) => { lpd.upsert(..., tx); audit.emit(..., tx) })` |

The `leads` table is NOT written during scheme attachment (scheme is stored on `lead_product_details`, not on `leads`). `LeadService` is therefore not invoked, and optimistic locking on `leads.version` is not triggered unless the caller's broader lead PATCH also modifies `leads` fields.

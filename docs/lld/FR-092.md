## Summary

`GET /partners/{id}/quality` computes and returns the §12.4 partner quality score plus a full
factor breakdown for a single partner. The endpoint serves two access patterns:

- **PARTNER** users may only access their **own** quality profile (`partner_id` from
  `users.partner_id` must equal the path `{id}`). They see a transparent score with a factor
  breakdown but no payout-engine data.
- **BM / SM / HEAD** users may access any partner within their data scope (branch/team/org)
  for coaching and prioritisation.

The service aggregates lead-level data from five tables
(`leads`, `source_attributions`, `stage_history`, `documents`, `duplicate_matches`) using
scope-bounded Kysely queries, evaluates the six factors per §12.4, clamps the final score to
`[0, 100]`, and **writes** the result back to `partners.quality_score` as a cached value. If
the partner has fewer than the minimum-volume threshold (configurable; default **10** submitted
leads), the endpoint returns `insufficient_data: true` and a `null` score — factors with a
zero denominator render as `null`, never `0%` (BRD §12.5).

No external provider is called; this is a pure aggregation read + one lightweight cache write.

## Endpoint

### `GET /partners/{id}/quality` — `partnerQuality`

Path prefix: `/api/v1`.

**Path parameter:** `id` — UUID of the target partner.

**Query parameters (optional):**

| Parameter | Type | Default | Constraint | Purpose |
|---|---|---|---|---|
| `from` | ISO-8601 date (YYYY-MM-DD) | 30 days ago | must be < `to` | Start of scoring window |
| `to`   | ISO-8601 date (YYYY-MM-DD) | today (IST) | must be >= `from` | End of scoring window |

If neither `from` nor `to` is supplied the service uses a rolling 30-day window (today − 29 days
to today, IST). The window is passed to all aggregate queries as `leads.created_at BETWEEN
from_ts AND to_ts` (both inclusive, converted to `TIMESTAMPTZ` at IST midnight).

**Response `200 OK`:**

## Auth Check

Global `JwtAuthGuard` is applied (no `@Public()`). Authorization via
`AbacGuard` + `@Requires('reports', qualityScope)` → `EntitlementService.can(user, 'reports', resource)`.

| Role | Capability | Scope | Ownership rule |
|---|---|---|---|
| `PARTNER` | `reports` | `P` | `user.partner_id === pathParam.id`; any other `id` → `FORBIDDEN` |
| `BM`      | `reports` | `B` | partner's `branch_id` must be in `user.branch_ids` |
| `SM`      | `reports` | `T` | partner's `mapped_rm_id`'s team must be in `user.team_ids` |
| `HEAD`    | `reports` | `A` | any partner in `org_id` |
| `RM`, `KYC`, `DPO`, `ADMIN`, `CUSTOMER` | no `reports` capability on `partners` | — | → `FORBIDDEN` |

The `qualityScope` resolver passed to `@Requires` receives the path `{id}` and the authenticated
user and returns `{ partnerId: id }`. `EntitlementService.can` then evaluates:

1. Checks `capability_matrix[user.role].reports` exists; absent → `FORBIDDEN`.
2. For scope `P`: asserts `user.partner_id === id`; mismatch → `FORBIDDEN`.
3. For scope `B`: loads `partners.branch_id` for the target `id` and checks it is in
   `user.branch_ids`; outside → `FORBIDDEN`.
4. For scope `T`: loads `partners.mapped_rm_id`, resolves that user's `team_id`, checks it is
   in `user.team_ids`; outside → `FORBIDDEN`.
5. For scope `A`: checks `partners.org_id === user.org_id`; outside → `FORBIDDEN`.

If the partner row does not exist at all → `NOT_FOUND` (404) before any scope check (existence
is hidden-by-default; returning 404 for out-of-scope also acceptable but NOT_FOUND is used here
because the endpoint is scoped — not a secret existence endpoint).

## Data Operations

All queries are **Kysely** typed parameterised builders against the exact schema column names.
Every query carries `org_id`. The following five queries run **in parallel** (`Promise.all`)
after the partner is verified and the scope gate passes.

### A. Resolve & gate partner (serial, before aggregation)

```ts
const partner = await db
  .selectFrom('partners')
  .select([
    'partner_id', 'partner_code', 'legal_name', 'type', 'status',
    'branch_id', 'mapped_rm_id', 'quality_score'
  ])
  .where('org_id', '=', user.org_id)
  .where('partner_id', '=', partnerId)
  .executeTakeFirst();
// null -> NOT_FOUND 404
// scope check (per Auth Check above) -> FORBIDDEN 403 if fails
```

### B. Lead counts aggregate (parallel query 1)

Counts total, contactable, duplicate, rejected, handed-off leads for the partner within the
window.

```ts
const leadCounts = await db
  .selectFrom('leads as l')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .select(eb => [
    eb.fn.count<string>('l.lead_id').as('total_leads'),
    eb.fn.count<string>('l.lead_id')
      .filterWhere('l.stage', 'in', [
        'contacted','qualified','documents_pending','kyc_in_progress',
        'eligibility_requested','ready_for_handoff','handed_off'
      ])
      .as('contactable_leads'),
    eb.fn.count<string>('l.lead_id')
      .filterWhere('l.duplicate_status', 'in', ['flagged','linked','merged'])
      .as('duplicate_leads'),
    eb.fn.count<string>('l.lead_id')
      .filterWhere('l.stage', '=', 'rejected')
      .as('rejected_leads'),
    eb.fn.count<string>('l.lead_id')
      .filterWhere('l.stage', '=', 'handed_off')
      .as('handed_off_leads'),
  ])
  .where('sa.org_id', '=', user.org_id)
  .where('sa.partner_id', '=', partnerId)
  .where('l.created_at', '>=', fromTs)
  .where('l.created_at', '<=', toTs)
  .where('l.deleted_at', 'is', null)
  .executeTakeFirstOrThrow();
```

> **Contactability definition (BRD AC1):** a lead is "contactable" when it has progressed to
> `contacted` or any later stage. This is determined from `leads.stage` (current stage) rather
> than `stage_history` to avoid double-counting leads that regressed (e.g., `contacted →
> dormant`). See Ambiguities §1.

### C. Document quality aggregate (parallel query 2)

```ts
const docCounts = await db
  .selectFrom('documents as d')
  .innerJoin('leads as l', 'l.lead_id', 'd.lead_id')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .select(eb => [
    eb.fn.count<string>('d.document_id')
      .filterWhere('d.status', 'not in', ['pending'])
      .as('uploaded_docs'),
    // "verified first time" = status is 'verified' AND there is no prior 'mismatch' status
    // for the same document (version = 1 when first verified with no mismatch history).
    // Approximation: version = 1 AND status = 'verified'.
    eb.fn.count<string>('d.document_id')
      .filterWhere('d.status', '=', 'verified')
      .filterWhere('d.version', '=', 1)
      .as('verified_docs_first_time'),
  ])
  .where('sa.org_id', '=', user.org_id)
  .where('sa.partner_id', '=', partnerId)
  .where('l.created_at', '>=', fromTs)
  .where('l.created_at', '<=', toTs)
  .where('l.deleted_at', 'is', null)
  .executeTakeFirstOrThrow();
```

### D. KYC mismatch count (parallel query 3)

```ts
const kycMismatch = await db
  .selectFrom('kyc_verifications as kv')
  .innerJoin('leads as l', 'l.lead_id', 'kv.lead_id')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .select(eb =>
    eb.fn.count<string>('l.lead_id').distinct().as('kyc_mismatch_leads')
  )
  .where('sa.org_id', '=', user.org_id)
  .where('sa.partner_id', '=', partnerId)
  .where('kv.status', '=', 'failed')
  .where('l.created_at', '>=', fromTs)
  .where('l.created_at', '<=', toTs)
  .where('l.deleted_at', 'is', null)
  .executeTakeFirstOrThrow();
```

### E. Speed index — partner median document TAT (parallel query 4)

The §12.4 `speed_index` formula requires: `min(all-partner median doc TAT) / this-partner median
doc TAT * 100`. This is computed across two sub-queries:

- Sub-query E1: compute this partner's median time from `leads.created_at` to first document
  reaching `status = 'uploaded'` (from `documents.updated_at` at first upload), expressed in
  hours.
- Sub-query E2: compute the same median for every other active partner in the org within the
  same window, then take the minimum.

```ts
// E1: this partner's median doc TAT in fractional hours
// doc TAT = earliest doc upload timestamp - lead created_at, per lead, for leads with >= 1 doc
const thisPartnerTat = await db
  .selectFrom('leads as l')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .innerJoin('documents as d', 'd.lead_id', 'l.lead_id')
  .select(eb =>
    eb.fn.avg(
      eb.raw<number>(
        `EXTRACT(EPOCH FROM (MIN(d.updated_at) - l.created_at)) / 3600`
      )
    ).as('median_tat_hours')
  )
  .where('sa.org_id', '=', user.org_id)
  .where('sa.partner_id', '=', partnerId)
  .where('d.status', 'not in', ['pending'])
  .where('l.created_at', '>=', fromTs)
  .where('l.created_at', '<=', toTs)
  .where('l.deleted_at', 'is', null)
  .groupBy('l.lead_id')
  .executeTakeFirst();

// E2: minimum median TAT across all partners in org (for denominator)
const allPartnersMinTat = await db
  .selectFrom('leads as l')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .innerJoin('documents as d', 'd.lead_id', 'l.lead_id')
  .select(eb =>
    eb.fn.min(
      eb.fn.avg(
        eb.raw<number>(`EXTRACT(EPOCH FROM (MIN(d.updated_at) - l.created_at)) / 3600`)
      )
    ).as('min_median_tat_hours')
  )
  .where('sa.org_id', '=', user.org_id)
  .where('d.status', 'not in', ['pending'])
  .where('l.created_at', '>=', fromTs)
  .where('l.created_at', '<=', toTs)
  .where('l.deleted_at', 'is', null)
  .groupBy(['sa.partner_id', 'l.lead_id'])
  .executeTakeFirst();
```

> See Ambiguities §2 for the median vs. average approximation and the Kysely limitation with
> window functions.

### F. Cache write (serial, after computation; lightweight UPDATE, no UnitOfWork needed)

Once the score is computed:

```ts
await db
  .updateTable('partners')
  .set({ quality_score: computedScore, updated_at: new Date(), updated_by: user.user_id })
  .where('partner_id', '=', partnerId)
  .where('org_id', '=', user.org_id)
  .execute();
```

This is a best-effort cache write; a failure is **logged** but does not fail the API response
(the response carries the freshly computed score regardless). No `UnitOfWork` is needed because
this is a single-table update with no downstream audit/outbox chain — it is a mutable cache
scalar, not a state transition.

## Backend Flow

1. `JwtAuthGuard` authenticates the request → `req.user` (`role`, `user_id`, `org_id`,
   `partner_id`).
2. `ThrottlerGuard` (read tier, 300/min/user) checks rate.
3. Controller parses and validates `PartnerQualityQueryDto` (Zod: path `id`, query `from`/`to`).
   Failure → `VALIDATION_ERROR` (400) with `fields[]`.
4. `AbacGuard` + `@Requires('reports', qualityScope)` → `EntitlementService.can(user, 'reports',
   { partnerId: id })`. No `reports` capability → `FORBIDDEN` (403).
5. `PartnerQualityService.compute(user, partnerId, from, to)`:

   a. **Resolve partner** (op A): load partner row; null → `NOT_FOUND` (404). Ownership scope
      check (PARTNER-own / BM-branch / SM-team / HEAD-all); fail → `FORBIDDEN` (403).

   b. **Compute window:** if `from`/`to` supplied, convert to `TIMESTAMPTZ` at IST midnight.
      Default: rolling 30-day window (`now() - 29 days` to `now()`).

   c. **Run aggregation queries in parallel** (`Promise.all`): ops B (lead counts), C (doc
      quality), D (KYC mismatch), E (TAT sub-queries).

   d. **Insufficient-data check:** if `total_leads < MIN_VOLUME` (default 10, configurable via
      `PARTNER_QUALITY_MIN_VOLUME` env var), return `insufficient_data: true`, `quality_score:
      null`, all `factors.*: null`.

   e. **Compute factors** per §12.4 formula:
      - `contactability_index = contactable / total * 100` (null if total = 0)
      - `duplicate_penalty = duplicate / total * 100` (null if total = 0)
      - `rejection_penalty = rejected / total * 100` (null if total = 0)
      - `handoff_index = handed_off / total * 100` (null if total = 0)
      - `document_quality_index = verified_first / uploaded * 100` (null if uploaded = 0)
      - `speed_index = min_all_median_tat / this_partner_median_tat * 100` (null if no TAT data,
        capped at 100)

   f. **Compute score:**
      ```
      raw_score = 0.25 * (contactability_index ?? 0)
               + 0.30 * (handoff_index ?? 0)
               + 0.20 * (document_quality_index ?? 0)
               + 0.15 * (speed_index ?? 0)
               - 0.05 * (duplicate_penalty ?? 0)
               - 0.05 * (rejection_penalty ?? 0)
      quality_score = Math.round(Math.max(0, Math.min(100, raw_score)))
      ```
      Any factor that is `null` contributes `0` to the weighted sum (no data = neutral for that
      factor) but is still returned as `null` in the `factors` object so the UI can display "–".

   g. **Cache write** (op F): `UPDATE partners SET quality_score = computedScore ...`. Fire-and-
      forget; failure logged at `warn` level, does not reject the request.

6. `ResponseEnvelopeInterceptor` wraps the result → `200` envelope. No masking needed on the
   quality payload (no PII fields).

## State Machine & Transaction Boundaries

FR-092 does **not** perform a `lead_stage` transition. The only write is the cache update to
`partners.quality_score` (op F), which is a single-table UPDATE — no UnitOfWork, no
stage_history, no audit_logs chain, no outbox event. Quality score is a mutable operational
cache scalar (not an append-only audit artefact).

**No `UnitOfWork` is used.** The cache write failure is non-fatal (best-effort), so wrapping
it in a transaction that could roll back the read response would be incorrect.

## Assumptions

1. **Contactability definition:** a lead is "contactable" when `leads.stage` is `contacted` or
   any stage that follows it in the standard pipeline (`qualified`, `documents_pending`,
   `kyc_in_progress`, `eligibility_requested`, `ready_for_handoff`, `handed_off`). Using current
   `stage` rather than `stage_history` avoids double-counting but may undercount if a lead
   regressed. If the BRD intends "ever reached contacted" (stage_history), the query must be
   adapted to use `stage_history` with `to_stage IN (...)`.

2. **Speed-index TAT approximation:** the §12.4 formula specifies `partner_median_doc_tat`;
   Kysely does not support `PERCENTILE_CONT` natively. This LLD approximates median with
   `AVG` per lead and takes the per-partner average of those lead averages, then takes `MIN`
   across partners. If the product team requires true statistical median, a raw SQL expression
   via `eb.raw` or a pre-computed `MATERIALIZED VIEW` with `PERCENTILE_CONT(0.5)` should be
   added. Document here and back-write when confirmed.

3. **`verified_docs_first_time` approximation:** the BRD AC1 mentions "verified docs first
   time." The schema does not have an explicit "first_time_verify" column. This LLD approximates
   it as `documents.status = 'verified' AND documents.version = 1` (no mismatch before
   verification). If the schema is extended to track mismatch history, update this query.

4. **Score cache write actor:** `partners.updated_by` is set to `user.user_id` (the requesting
   user) for the cache write. This is a system-driven update, not a user edit, but the schema
   requires `updated_by NOT NULL`. Alternatively, a system user UUID (seeded at bootstrap) may
   be used. Best-effort: use requesting user's ID until a system-user convention is established.

5. **Min-volume threshold:** the minimum lead count for a valid score is **10**, configurable
   via the `PARTNER_QUALITY_MIN_VOLUME` environment variable (see
   `docs/contracts/environment-contract.md`). If this env var is absent, the service defaults
   to 10.

6. **Weights not configurable at runtime:** BRD §12.4 says weights are "configurable" but no
   configuration table is defined in the schema for partner quality weights. This LLD hard-codes
   the §12.4 weights as constants in `partner-quality.service.ts`. If runtime configurability
   is required, a new `configuration_versions` row or a partner-quality-config table must be
   added to the schema and specified before Stage 7 coding.

## Ambiguities

1. **Contactability via current stage vs. stage_history.** BRD AC1 says "contactable %"; the
   exact definition (current stage >= contacted, or ever reached contacted) is not resolved.
   Best-effort: use current `stage`. If the product team requires "ever reached," the query
   changes to a `stage_history` subquery.

2. **Median TAT (PERCENTILE_CONT vs. AVG).** True statistical median requires a PostgreSQL
   ordered-set aggregate not natively supported in Kysely's expression builder. Best-effort:
   use AVG approximation or `eb.raw`. Confirm with the team before Stage 7 coding; if median
   is required, add a read-replica view.

3. **`doc_tat` definition.** "Partner median doc TAT" could mean: (a) lead created_at to
   first doc uploaded, (b) lead created_at to first doc verified, or (c) doc uploaded_at to
   verified_at. This LLD uses (a): `first doc upload timestamp - lead.created_at`. Confirm with
   product before coding.

4. **Weight configurability.** BRD §12.4 says weights are "configurable" but no config table
   exists in the schema. Weights are hard-coded in this LLD. A schema extension is needed to
   make them runtime-configurable.

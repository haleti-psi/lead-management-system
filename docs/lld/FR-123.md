## Summary

FR-123 implements a searchable, paginated audit trail explorer restricted to DPO and ADMIN roles. It reads records from the append-only, hash-chained `audit_logs` table, applies role-appropriate masking to JSONB `detail` fields, and verifies tamper-evidence by checking `prev_audit_hash` chain continuity across the returned page window. Evidence export is delegated to the existing FR-122 governed-export pathway — the frontend initiates a `POST /exports` with `report_code: 'audit_export'` and the current filter state; this FR does not manage `ExportJob` lifecycle.

Key design constraints:

- `audit_logs` is append-only and written exclusively by `AuditChainConsumer` (architecture §11.4 / `core/audit/`). FR-123 only reads; it never inserts, updates, or deletes any row.
- **DPO** receives all-org audit records. PII fields inside `detail` JSONB (`name`, `mobile`, `email`, `pan_token`, `aadhaar_ref_token`, `ckyc_id`) are masked by `MaskingService` unless an active `BreakGlassGrant` covers the relevant scope. `ip_device` is never returned in the API response (stored for internal forensics only).
- **ADMIN** receives only system/config-scoped audit actions (`config_change`, `user_change`, `role_change`, `break_glass_access`, `login`, `logout`, `login_failed`, `mfa_failed`, `export_generate`, `export_download`). A mandatory server-side filter is applied regardless of user input. `lead_id` is zeroed to `null` in all ADMIN responses. ADMIN cannot supply a `lead_id` query parameter — doing so returns `FORBIDDEN` (403).
- **Integrity check** runs on the current page window after fetching: rows are reversed to oldest-first, then each consecutive pair is checked for `curr.prev_audit_hash === prev.after_hash`. A mismatch sets `integrity_badge: 'broken'`, records the `audit_id` of the first broken link in `meta.integrity_break_at`, and emits a `warn`-level structured log event. The response is still returned (evidence must not be silently withheld); 200 is the HTTP status.
- **No writes** occur in this FR. There is no `UnitOfWork`, no `LeadService` call, no `OutboxService` call.

## Auth Check

| Layer | Detail |
|---|---|
| Authentication | Global `JwtAuthGuard` — validates the JWT, loads `user` (user_id, role_code, org_id, scope, ABAC attrs) onto request context. Returns `AUTH_REQUIRED` (401) if token is missing, invalid, or expired. |
| Authorization guard | `AbacGuard` + `@Requires('audit_trail', auditScopeResolver)` |
| Entitlement call | `EntitlementService.can(user, 'audit_trail', { org_id: user.org_id })` |
| DPO | `capability_matrix.DPO.audit_trail = 'A'` — all-org. Lead-content PII masked (scope `M`) unless an active `BreakGlassGrant` covers the relevant scope. |
| ADMIN | `capability_matrix.ADMIN.audit_trail = 'A'` — system/config action rows only. Lead content never exposed without a break-glass grant; enforced at the service layer by the mandatory ADMIN action-filter and `lead_id` zeroing. |
| All other roles | All other role_codes (RM, BM, SM, HEAD, KYC, PARTNER, CUSTOMER) — `auditScopeResolver` returns `false`, causing `AbacGuard` to return `FORBIDDEN` (403). The `audit_trail` capability exists for those roles in the matrix but access to the `/audit` explorer endpoint is restricted to DPO and ADMIN only. |
| `lead_id` filter — ADMIN | Validated in `AuditExplorerService.search()` before the repository call: if `user.role_code === 'ADMIN'` and `dto.lead_id` is present, throw `ForbiddenException` → `FORBIDDEN` (403). |

**`auditScopeResolver`** passed to `@Requires`: a resolver function that returns `false` for any `role_code` that is not `'DPO'` or `'ADMIN'`, triggering `FORBIDDEN` in `AbacGuard`.

## Data Operations

All queries use the Kysely `Db` instance with parameterised bindings. String-interpolated SQL is prohibited (security.md / coding.md).

### Primary list query

```typescript
// apps/api/src/modules/reporting/audit-explorer.repository.ts
import { sql } from 'kysely';

// AuditRepository.search(filters, user, pagination): Promise<AuditLogRow[]>
let query = db
  .selectFrom('audit_logs as a')
  .innerJoin('users as u', 'u.user_id', 'a.actor_id')
  .select([
    'a.audit_id',
    'a.actor_id',
    sql<string>`concat(u.first_name, ' ', u.last_name, ' · ', u.role_code)`.as('actor_display'),
    'a.action',
    'a.entity_type',
    'a.entity_id',
    'a.lead_id',
    'a.before_hash',
    'a.after_hash',
    'a.prev_audit_hash',
    'a.detail',
    // ip_device intentionally excluded — never returned in the API
    'a.created_at',
  ])
  .where('a.org_id', '=', user.org_id)
  .orderBy('a.created_at', 'desc')
  .orderBy('a.audit_id', 'desc'); // stable tie-break for integrity window

// Apply optional filters (all parameterised bindings)
if (filters.lead_id)     query = query.where('a.lead_id',     '=',  filters.lead_id);
if (filters.actor_id)    query = query.where('a.actor_id',    '=',  filters.actor_id);
if (filters.action)      query = query.where('a.action',      '=',  filters.action);
if (filters.entity_type) query = query.where('a.entity_type', '=',  filters.entity_type);
if (filters.from)        query = query.where('a.created_at',  '>=', filters.from);
if (filters.to)          query = query.where('a.created_at',  '<=', filters.to);

// Pagination — LIMIT enforced to max 100 (NFR-17)
const offset = (pagination.page - 1) * pagination.limit;
const rows = await query.limit(pagination.limit).offset(offset).execute();
```

### ADMIN mandatory scope filter

Applied by `AuditExplorerService.search()` before the repository call, whenever `user.role_code === 'ADMIN'`:

```typescript
// Restrict ADMIN to non-lead-content actions (auth-matrix.json capability_conditions)
const ADMIN_ALLOWED_ACTIONS: readonly AuditAction[] = [
  'config_change', 'user_change', 'role_change', 'break_glass_access',
  'login', 'logout', 'login_failed', 'mfa_failed',
  'export_generate', 'export_download',
] as const;

// Enforced as an additional filter; cannot be overridden by user input
// (merged into the filters object passed to AuditRepository.search)
dto.actionIn = ADMIN_ALLOWED_ACTIONS;

// In result mapping: every item.lead_id is zeroed regardless of DB value
rows.forEach(r => { r.lead_id = null; });
```

The repository applies `actionIn` as `.where('a.action', 'in', dto.actionIn)` when present.

### Break-glass check (DPO unmasked access)

```typescript
// AuditExplorerService checks for an active BreakGlassGrant when user.role_code === 'DPO'
const grant = await db
  .selectFrom('break_glass_grants')
  .select(['grant_id', 'scope_type'])
  .where('grantee_id',  '=', user.user_id)
  .where('status',      '=', 'active')
  .where('valid_until', '>',  new Date())
  .limit(1)
  .executeTakeFirst();

const hasBreakGlass = grant != null;
// Passed to MaskingService to determine masking level per row
```

### Hash-chain integrity verification

```typescript
// AuditExplorerService.verifyChainIntegrity(rows: AuditLogRow[])
// rows are ordered newest-first from DB; reverse for oldest→newest walk
const sorted = [...rows].reverse();
let integrityBroken = false;
let breakAt: string | null = null;

for (let i = 1; i < sorted.length; i++) {
  const prev = sorted[i - 1];
  const curr = sorted[i];
  // AuditChainConsumer sets curr.prev_audit_hash = SHA-256 hash of prev record
  if (curr.prev_audit_hash !== prev.after_hash) {
    integrityBroken = true;
    breakAt = curr.audit_id;
    break;
  }
}

if (integrityBroken) {
  logger.warn({
    module: 'reporting',
    event: 'audit_chain_break',
    break_at: breakAt,
    user_id: user.user_id,
    correlation_id,
  });
}

const badge: 'intact' | 'broken' | 'not_checked' =
  rows.length <= 1 ? 'not_checked' :
  integrityBroken  ? 'broken'      : 'intact';

return { badge, breakAt, checkedCount: rows.length };
```

**Scope note:** Integrity is verified across the current page window only. A full-history chain audit is an offline operational concern (not on the request path). The explorer badge is a real-time per-page indicator for the DPO.

## Validation Logic

All validation via Zod in `AuditExplorerQueryDto`, applied at the controller boundary before the service is called.

| Field | Rule | Error message (in `fields[]`) |
|---|---|---|
| `lead_id` | Optional. If present: `z.string().uuid()`. ADMIN role: checked in service before repo; if present → `FORBIDDEN` (403) — not a Zod error. | `"lead_id must be a valid UUID."` |
| `actor_id` | Optional. If present: `z.string().uuid()`. | `"actor_id must be a valid UUID."` |
| `action` | Optional. If present: `z.nativeEnum(AuditAction)` (from `@shared/enums`). | `"Invalid audit_action value."` |
| `entity_type` | Optional. If present: `z.string().max(50)` then `.refine(v => ENTITY_TYPE_ALLOWLIST.includes(v))`. Allowlist: `leads`, `users`, `roles`, `consent_records`, `stage_history`, `documents`, `kyc_verifications`, `export_jobs`, `configuration_versions`, `break_glass_grants`, `partners`, `tasks`, `grievances`, `data_rights_requests`, `import_jobs`, `communication_logs`, `dla_registry`. Values outside the list → `VALIDATION_ERROR`. | `"entity_type is not a recognised entity."` |
| `from` | Optional. If present: `z.coerce.date()`. If both `from` and `to` present: Zod superRefine checks `from <= to`. | `"from must be a valid date."` / `"from must not be after to."` |
| `to` | Optional. If present: `z.coerce.date()`. | `"to must be a valid date."` / `"to must not be before from."` |
| `page` | `z.coerce.number().int().min(1).default(1)` | `"page must be a positive integer."` |
| `limit` | `z.coerce.number().int().min(1).max(100).default(25)` — values above 100 rejected with VALIDATION_ERROR by the max() constraint (do not silently clamp in Zod; clamp only in the service if needed for defence-in-depth). | `"limit must be between 1 and 100."` |

The `entity_type` allowlist prevents probing for non-canonical table names or schema discovery.

**Integrity break** is NOT a 4xx/5xx. The API returns 200 with `integrity_badge: "broken"` and `meta.integrity_break_at` set to the offending `audit_id`. A `warn`-level structured log event is emitted. Evidence must not be withheld even when the chain is broken — the DPO needs to see both the data and the alert.

## Backend Flow (Step-by-Step)

1. **Request arrives** — `CorrelationMiddleware` attaches `X-Correlation-Id` to the request context.
2. **Authentication** — `JwtAuthGuard` validates the JWT, loads `user` (user_id, role_code, org_id, scope, branch_id, ABAC attrs) onto the request. Returns `AUTH_REQUIRED` (401) if token is missing, invalid, or expired.
3. **Rate limiting** — `ThrottlerGuard` (Redis-backed) checks the read tier (300 req/min per user). Over limit → `RATE_LIMITED` (429) with `Retry-After` header.
4. **Authorization** — `AbacGuard` + `@Requires('audit_trail', auditScopeResolver)` calls `EntitlementService.can(user, 'audit_trail', { org_id: user.org_id })`. `auditScopeResolver` returns `false` for any role_code other than `'DPO'` or `'ADMIN'` → `FORBIDDEN` (403). ABAC deny is logged at `warn` level.
5. **DTO validation** — `ValidationPipe` applies the `AuditExplorerQueryDto` Zod schema to the query string. Invalid inputs → `VALIDATION_ERROR` (400) with `fields[]` listing each offending field.
6. **ADMIN lead_id guard** — `AuditExplorerService.search()` checks: if `user.role_code === 'ADMIN'` and `dto.lead_id` is present → throw `ForbiddenException` → `FORBIDDEN` (403).
7. **Scope strategy** — Service determines query strategy:
   - **DPO:** optional `lead_id` filter permitted; check for active `BreakGlassGrant` (one parameterised query on `break_glass_grants`).
   - **ADMIN:** `dto.actionIn` forcibly set to `ADMIN_ALLOWED_ACTIONS` before repository call; `lead_id` parameter stripped.
8. **Repository query** — `AuditRepository.search(filters, user, pagination)` executes the Kysely SELECT with all parameterised filters. `ip_device` is excluded from the SELECT clause. LIMIT capped to `min(pagination.limit, 100)` (NFR-17). Ordered by `created_at DESC, audit_id DESC`.
9. **Count query** — `AuditRepository.count(filters, user)` executes in parallel with the list query for `meta.total`.
10. **ADMIN lead_id zeroing** — For ADMIN role: every `row.lead_id` is set to `null` in-memory before masking or serialization.
11. **Masking** — `MaskingService.maskAuditDetail(row, { user, hasBreakGlass })` called per row:
    - `ip_device` key is never included (excluded at SELECT).
    - For DPO without `hasBreakGlass`: masks `detail` JSONB keys `name`, `mobile`, `email`, `pan_token`, `aadhaar_ref_token`, `ckyc_id` per masking patterns (mobile → `98xxxxxx10`, PAN → `ABCxxxx1F`).
    - For DPO with `hasBreakGlass`: PII keys returned unmasked.
    - For ADMIN: `detail` returned as-is for allowed action rows (non-lead-content).
12. **Integrity check** — `AuditExplorerService.verifyChainIntegrity(rows)` walks the page window. If broken: `logger.warn` event, sets `badge = 'broken'`, `breakAt = <audit_id>`.
13. **Response assembly** — `ResponseEnvelopeInterceptor` wraps result in the uniform `{ data, meta, error }` envelope.

## No State Machine / Transaction Boundary

FR-123 is **read-only**. No `UnitOfWork`, no `LeadService` call, no `AuditAppender` call, no `OutboxService.emit()`. The only side effects are:

- Pino structured logs (correlation, timing, integrity break events).
- Redis rate-limiter counter increment via `ThrottlerGuard`.

Evidence export is not managed by this FR. The frontend initiates `POST /exports` (FR-122) with `report_code: 'audit_export'` and the current filter state serialised as the `filters` payload. FR-122 governs the `ExportJob` lifecycle, masking level, approval threshold, GCS artefact, and download.

## Summary

FR-130 implements full lifecycle management for the four identity-infrastructure entities — **users**, **roles** (with their capability/scope permission sets), **teams**, and **branches** (read + deactivate). Only the ADMIN role can perform write operations (capability `user_mgmt`, scope `A`).

Key behaviours:
- **User create/edit:** assign role, branch, team, region, product skills, partner mapping, reporting manager. Unique username and email within org.
- **User deactivate:** blocked if the user has open (non-terminal) leads unless an explicit `reassign_to` UUID is supplied; reassignment is performed atomically in the same transaction as the status change.
- **Role CRUD with permission assignment:** each role carries a set of `role_permissions` rows (capability + max_scope); adding/removing permissions is included.
- **Team CRUD:** team is scoped to a branch; a manager user may be assigned.
- **Branch deactivate:** branch with active teams/users cannot be deleted — only deactivated (sets `is_active = false`).
- Every write emits an `audit_logs` row (`user_change` or `role_change` action) inside the same transaction.
- No external service calls; no `event_outbox` emission (user/role is infrastructure, not a domain event code in the `event_code` enum).

> **Assumption A-1:** `event_code` enum does not contain a `USER_CHANGED` value (confirmed: only lead/config/compliance events). Audit is satisfied by `audit_logs` alone with `action = 'user_change'` / `action = 'role_change'`.
>
> **Assumption A-2:** "Open leads" means leads where `stage NOT IN ('handed_off', 'rejected')` AND `deleted_at IS NULL` AND `owner_id = :userId`.
>
> **Assumption A-3:** Branches are not creatable via FR-130 (that is FR-131 master-data); FR-130 exposes branch list (read) and branch deactivation only, both via the `/admin/users` context for branch assignment. The `GET/PATCH /admin/{masterResource}` endpoints (FR-131) own branch CRUD — FR-130 only references branches as a foreign key on users/teams.
>
> **Assumption A-4:** Role `code` is a fixed enum (`role_code`) from the schema; ADMIN cannot create new role codes — only configure the permission set for existing role codes.

## Auth Check

| Step | Guard | Detail |
|---|---|---|
| 1. Authentication | Global `JwtAuthGuard` | Validates JWT; populates `req.user` with `{ user_id, org_id, role_code, ... }`. Returns `AUTH_REQUIRED` (401) if invalid/expired. |
| 2. Authorisation | `AbacGuard` + `@Requires('user_mgmt', () => 'A')` | Calls `EntitlementService.can(user, 'user_mgmt', resource)`. Only `ADMIN` has `user_mgmt: A` in `capability_matrix`. Non-ADMIN → `FORBIDDEN` (403). |
| 3. Org isolation | Service layer | Every Kysely query includes `.where('org_id', '=', req.user.org_id)`. |

No ownership check beyond role — the ADMIN has scope `A` (all-org) for `user_mgmt`. There is no partial-scope version of this capability.

## Data Operations

All Kysely queries use the `Db` instance from `core/db/`. Column names are exact from `schema.sql`.

### List users (paginated)

```typescript
// UserRepository.listUsers(orgId, filters, pagination)
db.selectFrom('users as u')
  .leftJoin('roles as r', 'r.role_id', 'u.role_id')
  .select([
    'u.user_id', 'u.username', 'u.full_name', 'u.email', 'u.mobile',
    'u.role_id', 'r.code as role_code', 'u.branch_id', 'u.team_id',
    'u.region_id', 'u.partner_id', 'u.product_skills', 'u.mfa_enabled',
    'u.status', 'u.reporting_manager_id', 'u.last_login_at', 'u.created_at'
  ])
  .where('u.org_id', '=', orgId)
  .$if(filters.status != null, (q) => q.where('u.status', '=', filters.status!))
  .$if(filters.role_id != null, (q) => q.where('u.role_id', '=', filters.role_id!))
  .$if(filters.branch_id != null, (q) => q.where('u.branch_id', '=', filters.branch_id!))
  .$if(filters.team_id != null, (q) => q.where('u.team_id', '=', filters.team_id!))
  .orderBy(allowedSort(pagination.sort, 'u.created_at'), 'desc')
  .limit(pagination.limit)   // max 100
  .offset((pagination.page - 1) * pagination.limit)
```

Count query runs in the same transaction for `total`.

### Insert user

```typescript
// UserRepository.createUser(orgId, dto, actorId, tx)
db.insertInto('users')
  .values({
    org_id: orgId,
    username: dto.username,
    email: dto.email,
    full_name: dto.full_name,
    mobile: dto.mobile ?? null,
    password_hash: await argon2.hash(temporaryPassword),
    role_id: dto.role_id,
    branch_id: dto.branch_id ?? null,
    team_id: dto.team_id ?? null,
    region_id: dto.region_id ?? null,
    partner_id: dto.partner_id ?? null,
    product_skills: dto.product_skills ? JSON.stringify(dto.product_skills) : null,
    mfa_enabled: dto.mfa_enabled ?? false,
    status: 'active',
    reporting_manager_id: dto.reporting_manager_id ?? null,
    created_by: actorId,
    updated_by: actorId,

  .returningAll()

```

Unique constraint violation (`uq_users_username`, `uq_users_email`) is caught and rethrown as `CONFLICT` (409).

### Update user (partial patch)

```typescript
// UserRepository.updateUser(orgId, userId, patch, actorId, tx)
db.updateTable('users')
  .set({ ...patch, updated_by: actorId, updated_at: new Date() })
  .where('user_id', '=', userId)
  .where('org_id', '=', orgId)
  .returningAll()
  .executeTakeFirstOrThrow()
```

### Count open leads for a user

```typescript
// UserRepository.countOpenLeads(orgId, userId, tx)
db.selectFrom('leads')
  .select(db.fn.count('lead_id').as('open_count'))
  .where('org_id', '=', orgId)
  .where('owner_id', '=', userId)
  .where('deleted_at', 'is', null)
  .where('stage', 'not in', ['handed_off', 'rejected'])
  .executeTakeFirstOrThrow()
```

### Bulk reassign leads (used in deactivation flow)

```typescript
// UserRepository.reassignLeads(orgId, fromUserId, toUserId, actorId, tx)
db.updateTable('leads')
  .set({ owner_id: toUserId, updated_by: actorId, updated_at: new Date() })
  .where('org_id', '=', orgId)
  .where('owner_id', '=', fromUserId)
  .where('deleted_at', 'is', null)
  .where('stage', 'not in', ['handed_off', 'rejected'])
```

> Owner-writes caveat: `leads` is normally written only by `LeadService`. The reassign here is an exception: it is a bulk update of `owner_id` only (not a stage transition, not a version-bumping mutator). Per the CLAUDE.md note on §11, this type of administrative reassignment is owned by the admin module's `UnitOfWork` call but must be logged as `reassign` in `audit_logs`. This assumption is recorded here — if a strict reading of §11 requires routing through `LeadService.assignOwner()`, each lead would require its `expectedVersion`; with potentially hundreds of leads, the admin service instead performs a bulk SQL update under a single transaction and logs a `reassign` audit entry per lead or as a single bulk entry.
>
> **Assumption A-5:** Bulk lead reassignment on deactivation uses a direct Kysely `updateTable('leads')` call (owner_id only, no version increment) because it is not a domain stage transition; a single aggregate `audit_logs` row with `action='reassign'` and `detail.bulk=true` is written instead of per-lead rows.

### Insert audit log row

```typescript
// AuditAppender.append(action, entityType, entityId, detail, actorId, tx)
db.insertInto('audit_logs')
  .values({
    org_id: orgId,
    actor_id: actorId,
    action: action,           // 'user_change' | 'role_change' | 'reassign'
    entity_type: entityType,  // 'user' | 'role'
    entity_id: entityId,
    lead_id: null,
    detail: JSON.stringify(detail),  // masked — no raw PII
    created_at: new Date(),
    updated_at: new Date(),
  })
```

### Role permission replacement (PATCH /admin/roles/{id})

```typescript
// RoleRepository.replacePermissions(orgId, roleId, permissions, actorId, tx)
// Step 1: delete existing permissions for the role
db.deleteFrom('role_permissions')
  .where('role_id', '=', roleId)
  .where('org_id', '=', orgId)
// Step 2: insert new permissions
db.insertInto('role_permissions')
  .values(permissions.map(p => ({
    org_id: orgId,
    role_id: roleId,
    capability: p.capability,
    max_scope: p.max_scope,
    created_by: actorId,
    updated_by: actorId,
  })))
```

Both steps run inside the caller's `UnitOfWork` transaction.

## Backend Flow

### Create User (POST /admin/users)

```
1. JwtAuthGuard validates JWT → populates req.user.
2. AbacGuard checks EntitlementService.can(user, 'user_mgmt', 'users') → FORBIDDEN if not ADMIN.
3. ValidationPipe runs CreateUserDto Zod schema → VALIDATION_ERROR (400) on failure.
4. AdminService.createUser(orgId, dto, actorId) enters UnitOfWork.run(async (tx) => {
     a. UserRepository.findByUsernameOrEmail(orgId, username, email, tx)
        → if found: throw CONFLICT (409).
     b. Validate role_id, branch_id, team_id, region_id, partner_id exist in org
        (single SELECT per FK; throw NOT_FOUND if absent).
     c. UserRepository.createUser(orgId, dto, actorId, tx) → inserts users row.
     d. AuditAppender.append('user_change', 'user', newUserId,
          { action: 'create', username, role_id }, actorId, tx).
   })
5. ResponseEnvelopeInterceptor wraps result → 201.
```

### Deactivate User with Reassignment (PATCH /admin/users/{id}, status='inactive')

```
1–3. Auth + validation (UpdateUserDto).
4. AdminService.updateUser(orgId, userId, patch, actorId) enters UnitOfWork.run(async (tx) => {
     a. UserRepository.findById(orgId, userId, tx) → NOT_FOUND if absent.
     b. If patch.status === 'inactive':
        i.  Count open leads: UserRepository.countOpenLeads(orgId, userId, tx).
        ii. If open_count > 0 AND patch.reassign_to is null → throw CONFLICT (409) with open_lead_count.
        iii.If open_count > 0 AND patch.reassign_to provided:
            - Validate reassign_to user exists, is active, same org → NOT_FOUND / VALIDATION_ERROR.
            - UserRepository.reassignLeads(orgId, userId, reassign_to, actorId, tx).
            - AuditAppender.append('reassign', 'user', userId,
                { bulk: true, reassigned_to: reassign_to, count: open_count }, actorId, tx).
     c. Build update patch (only fields present in request body).
     d. UserRepository.updateUser(orgId, userId, patch, actorId, tx).
     e. If patch includes role_id change:
        - AuditAppender.append('role_change', 'user', userId,
            { from: oldRoleId, to: patch.role_id }, actorId, tx).
        else:
        - AuditAppender.append('user_change', 'user', userId,
            { changed_fields: Object.keys(patch) }, actorId, tx).
   })
5. Return 200 with updated user.
```

### Update Role Permissions (PATCH /admin/roles/{id})

```
1–3. Auth + validation (UpdateRoleDto).
4. AdminService.updateRole(orgId, roleId, dto, actorId) enters UnitOfWork.run(async (tx) => {
     a. RoleRepository.findById(orgId, roleId, tx) → NOT_FOUND if absent.
     b. If dto.permissions provided:
        - Validate each capability ∈ capability enum, max_scope ∈ data_scope enum (already done by Zod).
        - RoleRepository.replacePermissions(orgId, roleId, dto.permissions, actorId, tx):
          DELETE existing role_permissions WHERE role_id = roleId AND org_id = orgId.
          INSERT new role_permissions rows.
     c. If dto.name or dto.default_scope present:
        - RoleRepository.updateRole(orgId, roleId, { name, default_scope }, actorId, tx).
     d. AuditAppender.append('role_change', 'role', roleId,
          { changed_fields: Object.keys(dto), permission_count: dto.permissions?.length }, actorId, tx).
   })
5. Return 200 with updated role + permissions.
```

### Create Team (POST /admin/teams)

```
1–3. Auth + validation (CreateTeamDto).
4. AdminService.createTeam(orgId, dto, actorId) enters UnitOfWork.run(async (tx) => {
     a. Validate branch_id exists and is_active in org → NOT_FOUND.
     b. If manager_id provided: validate user exists and is active → NOT_FOUND.
     c. TeamRepository.createTeam(orgId, dto, actorId, tx) → inserts teams row.
     d. AuditAppender.append('user_change', 'team', newTeamId,
          { action: 'create', name: dto.name, branch_id: dto.branch_id }, actorId, tx).
   })
5. Return 201 with new team.
```

## State Machine

Per `docs/contracts/state-machines.md` (User compact machine):

```
active  <-->  inactive
  |               |
  +---> locked ---+
           |
         active (unlock)
```

**Transitions allowed via FR-130 PATCH:**
| From | To | Trigger | Condition | Side effect |
|---|---|---|---|---|
| `active` | `inactive` | ADMIN PATCH `status=inactive` | open_count == 0 OR reassign_to provided | reassign leads (if any) + audit |
| `inactive` | `active` | ADMIN PATCH `status=active` | none | audit `user_change` |
| `locked` | `active` | ADMIN PATCH `status=active` | — | audit `user_change` (unlock) |

**Invalid via this FR:**
- `inactive → locked` (lockout is system-only, via auth module FR-001).
- `locked → inactive` (must go through unlock first).

These invalid transitions return `VALIDATION_ERROR` (400) with `fields[{ field: 'status', issue: 'invalid transition from current state' }]`.

## Transaction Boundaries

Every write operation uses `UnitOfWork.run(async (tx) => { ... })`. The following writes are atomic per operation:

| Operation | Tables written atomically |
|---|---|
| Create user | `users` (INSERT) + `audit_logs` (INSERT) |
| Update user (fields only) | `users` (UPDATE) + `audit_logs` (INSERT) |
| Deactivate user without open leads | `users` (UPDATE) + `audit_logs` (INSERT) |
| Deactivate user with reassignment | `leads` (UPDATE bulk) + `users` (UPDATE) + `audit_logs` (INSERT ×2: reassign + user_change) |
| Update role permissions | `role_permissions` (DELETE + INSERT) + `roles` (UPDATE if name/scope) + `audit_logs` (INSERT) |
| Create team | `teams` (INSERT) + `audit_logs` (INSERT) |
| Update team | `teams` (UPDATE) + `audit_logs` (INSERT) |

If any step in a `UnitOfWork` block throws, the entire transaction is rolled back. No partial state persists.

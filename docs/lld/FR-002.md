# FR-002 — Attribute-Based Access Control (ABAC) (Low-Level Design)

**Tier: 2 (Moderate)** — significant data logic (multi-attribute evaluation, scope predicate injection, role-based masking), a shared UI component (`MaskedField`), and audited deny paths; no external service call, no lead-stage state machine transition, and no cross-entity transactional write → not Tier 3. Richer than a trivial CRUD — cross-cutting middleware backed by a non-trivial evaluation algorithm — → not Tier 1. (See `docs/contracts/testing-contract.md` Tier-2 exemplar list.)

## Summary

FR-002 implements the single ABAC decision point for the entire platform. It is realised as:

1. **`EntitlementService.can(user, capability, resource)`** — the pure evaluation function used by every protected operation.
2. **`AbacGuard`** — the NestJS guard that calls `EntitlementService.can()`, injects the resolved scope predicate into the request context, and returns `FORBIDDEN` (403) + an audit event on deny.
3. **`MaskedField`** — the shared React component that renders PAN/mobile/Aadhaar in masked form based on the authenticated user's role.
4. **`MaskingService` / `MaskingInterceptor`** — server-side serialisation layer that applies role-based masking to outbound JSON; exports apply the strictest applicable masking.

FR-002 has **no dedicated HTTP endpoint**. It is a cross-cutting infrastructure that every protected endpoint in the system enlists. The LLD specifies the internal contract (data operations, evaluation algorithm, masking rules, deny handling, audit) that `EntitlementService` and `AbacGuard` must implement so all other FR agents can rely on them as a black box.

## Tier 2 Specification

### Internal API Contract (no dedicated external endpoint)

FR-002 exposes two internal interfaces consumed by all other FRs:

#### Interface 1 — `EntitlementService.can(user, capability, resource)`

**Input:**

| Parameter | TypeScript type | Description |
|---|---|---|
| `user` | `AuthenticatedUser` | Loaded by `JwtAuthGuard`; contains `user_id`, `org_id`, `role_id`, `role_code`, `branch_id`, `team_id`, `region_id`, `partner_id`, `product_skills`, `status`, `activeBreakGlassGrant?` |
| `capability` | `Capability` (from `@shared/enums`) | The action being attempted (e.g. `view_lead`, `edit_lead`) |
| `resource` | `AbacResource` | `{ resourceType: ResourceType, ownerId?: string, branchId?: string, partnerId?: string, dataClassification?: DataClassification }` |

**Output:**

```typescript
type EntitlementResult =
  | { granted: true; scope: DataScope; scopePredicate: ScopePredicate }
  | { granted: false; reason: 'NO_CAPABILITY' | 'OUT_OF_SCOPE' | 'SUSPENDED_USER' | 'ADMIN_LEAD_BLOCKED' | 'PARTNER_CROSS_ACCESS' }
```

**`ScopePredicate`** is a serialisable filter descriptor injected into Kysely queries:

```typescript
type ScopePredicate =
  | { type: 'own';     userId: string }
  | { type: 'team';    userIds: string[] }
  | { type: 'branch';  branchId: string }
  | { type: 'region';  branchIds: string[] }
  | { type: 'all';     orgId: string }
  | { type: 'partner'; partnerId: string }
  | { type: 'masked';  orgId: string }
  | { type: 'customer_token'; leadId: string }
```

#### Interface 2 — `@Requires(capability, scopeResolver?)` + `AbacGuard`

`@Requires` is a NestJS decorator applied to controller methods:

```typescript
@Requires('view_lead', (req) => ({
  resourceType: 'leads',
  ownerId: req.lead?.owner_id,
  branchId: req.lead?.branch_id,
  partnerId: req.lead?.partner_id,
}))
```

`AbacGuard` (executed after `JwtAuthGuard`) calls `EntitlementService.can()`, and:
- On **grant**: attaches `req.scopePredicate` and `req.effectiveScope` to the request context for the handler to use.
- On **deny**: emits an audit event via `AuditAppender` (action `break_glass_access` for unmask, or a generic deny log) and throws `ForbiddenException` → `FORBIDDEN` (403).

---

### Auth Check

FR-002 **is** the authorisation layer; the "auth check" here describes how the layer itself is protected and bootstrapped:

- `JwtAuthGuard` (global) loads the calling user from the JWT, reads `users` + `roles` + `role_permissions` from Postgres (or Redis cache — see Data Operations below), and attaches `AuthenticatedUser` to `req.user`.
- Expired/missing token → `AUTH_REQUIRED` (401) before `AbacGuard` runs.
- Suspended user (`users.status = 'inactive' | 'locked'`) → `FORBIDDEN` (403), reason `SUSPENDED_USER`.
- ADMIN/DPO attempting a lead-content capability without an active `break_glass_grant` → deny, reason `ADMIN_LEAD_BLOCKED` (see `auth-matrix.json` `capability_conditions`).

### Data Operations

All queries use the **Kysely** typed query builder. Column and table names are taken verbatim from `docs/data-model/schema.sql`.

#### Read 1 — Load actor's entitlement record (performed by `JwtAuthGuard` on every request)

```
SELECT
  u.user_id, u.org_id, u.status,
  u.role_id, u.branch_id, u.team_id, u.region_id, u.partner_id,
  u.product_skills,
  r.code        AS role_code,
  r.default_scope,
  rp.capability, rp.max_scope, rp.conditions
FROM users u
JOIN roles r          ON r.role_id = u.role_id
JOIN role_permissions rp ON rp.role_id = u.role_id
WHERE u.user_id = $userId
  AND u.org_id  = $orgId
  AND u.status  = 'active'
LIMIT 50;     -- a user's role will never have > 18 capabilities; 50 is a safety ceiling
```

Result is **cached in Redis** (key `abac:{orgId}:{userId}`, TTL 60 s) to meet the p95 ≤ 500 ms NFR. Cache is invalidated on `role_permissions` write, user role/status change, or explicit `CACHE_ABAC_TTL` expiry.

#### Read 2 — Resolve team member list for scope `T` (team scope)

```
SELECT user_id
FROM users
WHERE team_id = $teamId
  AND org_id  = $orgId
  AND status  = 'active'
LIMIT 100;
```

Cached with key `team_members:{teamId}`, TTL 120 s.

#### Read 3 — Resolve branch list for scope `R` (region scope)

```
SELECT branch_id
FROM branches
WHERE region_id = $regionId
  AND org_id    = $orgId
  AND is_active = true
LIMIT 100;
```

Cached with key `region_branches:{regionId}`, TTL 300 s.

#### Read 4 — Check active break-glass grant for ADMIN/DPO (only for those roles)

```
SELECT grant_id, scope_type, lead_id, valid_until
FROM break_glass_grants
WHERE grantee_id = $userId
  AND status     = 'active'
  AND valid_until > now()
LIMIT 1;
```

Not cached (must be authoritative; TTL would be the window itself).

#### Scope predicate injection (consumed by repository layers of other FRs)

After `AbacGuard` resolves the `ScopePredicate`, the downstream repository applies it to every `leads` list/detail query. Repositories accept `scopePredicate: ScopePredicate` and apply it via Kysely:

```typescript
// example — scope O (own)
qb = qb.where('leads.owner_id', '=', scopePredicate.userId);

// scope T (team)
qb = qb.where('leads.owner_id', 'in', scopePredicate.userIds);

// scope B (branch)
qb = qb.where('leads.branch_id', '=', scopePredicate.branchId);

// scope R (region)
qb = qb.where('leads.branch_id', 'in', scopePredicate.branchIds);

// scope A (all)
qb = qb.where('leads.org_id', '=', scopePredicate.orgId);

// scope P (partner)
qb = qb.where('leads.partner_id', '=', scopePredicate.partnerId);

// scope M (DPO masked — same as A but MaskingInterceptor applies PII masking)
qb = qb.where('leads.org_id', '=', scopePredicate.orgId);
// MaskingInterceptor sees effectiveScope='M' and masks all PII fields

// scope C (customer token) — handled entirely by CustomerLinkGuard; leads by lead_id
qb = qb.where('leads.lead_id', '=', scopePredicate.leadId);
```

All queries include `leads.deleted_at IS NULL` (soft-delete) and `leads.org_id = $orgId`.

### Evaluation Algorithm (EntitlementService.can)

```
1. If user.status ∈ {inactive, locked}
     → deny (SUSPENDED_USER)

2. Look up role_permission for (user.role_id, capability).
     If not found
     → deny (NO_CAPABILITY)

3. effective_scope = min(role_permission.max_scope, user's attribute scope)
   Scope ordering (most→least permissive): A > R > B > T > O > P > C > M > X
   "User's attribute scope" = narrowest scope the user's attributes can supply:
     - has branch_id → can supply up to B
     - has team_id   → can supply up to T
     - has partner_id → P only
     - DPO role      → M
     - ADMIN role without break-glass → X for lead-content capabilities

4. Apply capability_conditions (auth-matrix.json):
     - ADMIN.* (lead content): require active break_glass_grant
         if none → deny (ADMIN_LEAD_BLOCKED)
     - PARTNER.*: require resource.partnerId == user.partner_id
         if mismatch → deny (PARTNER_CROSS_ACCESS)
     - DPO.view_lead (scope M): allow but tag response for masking
     - KYC.edit_lead: restrict editable fields to KYC subset (enforced in service layer)
     - *.export: tag response for strictest masking; check export threshold (FR-122)

5. Resource ownership check (for scoped capabilities):
     - scope O: resource.ownerId must equal user.user_id
     - scope T: resource.ownerId must be in team member list
     - scope B: resource.branchId must equal user.branch_id
     - scope R: resource.branchId must be in region's branch list
     - scope P: resource.partnerId must equal user.partner_id
   If check fails → deny (OUT_OF_SCOPE)

6. grant: true, scope: effective_scope, scopePredicate: <built from step 5>
```

**Deny-by-default:** any path not reaching step 6 returns `{ granted: false }`.

**Multi-attribute users** (e.g. a user with both `branch_id` and `team_id`): the `role_permission.max_scope` acts as the ceiling; the effective scope is the narrower of what the role allows and what the user's attributes can supply. If a resource spans two branches, the **owning branch** (`leads.branch_id`) governs.

### Validation Logic

`EntitlementService` itself performs no Zod validation (it receives typed inputs from guards). Validation applies at the `AbacGuard` boundary:

| Input | Rule | Error |
|---|---|---|
| `capability` | Must be a member of the `capability` enum (`@shared/enums`) | `VALIDATION_ERROR` (400) — unknown action → deny |
| `resource.resourceType` | Must be a known resource type from `auth-matrix.json` | `VALIDATION_ERROR` (400) — unknown → deny |
| `user.status` | `active` only; else deny before capability lookup | `FORBIDDEN` (403) |

---

### Error Cases

| Scenario | HTTP | Code | `error.detail.reason` | Audit? |
|---|---|---|---|---|
| Missing/expired JWT | 401 | `AUTH_REQUIRED` | — | No |
| User suspended (`status != 'active'`) | 403 | `FORBIDDEN` | `SUSPENDED_USER` | Yes — `audit_logs` via AuditAppender |
| Role has no permission for capability | 403 | `FORBIDDEN` | `NO_CAPABILITY` | Yes |
| Resource outside caller's scope | 403 | `FORBIDDEN` | `OUT_OF_SCOPE` | Yes |
| ADMIN/DPO lead-content access without active break-glass | 403 | `FORBIDDEN` | `ADMIN_LEAD_BLOCKED` | Yes |
| PARTNER accessing another partner's lead | 403 | `FORBIDDEN` | `PARTNER_CROSS_ACCESS` | Yes |
| Resource existence hidden per §8.4 policy (cross-partner) | 404 | `NOT_FOUND` | — | Yes (hidden deny) |
| Unknown capability (invalid enum value) | 400 | `VALIDATION_ERROR` | — | No |
| Unhandled exception in EntitlementService | 500 | `INTERNAL_ERROR` | — | Yes (pino error) |

**Existence-hiding policy (from `auth-matrix.json` `http_status_rules`):**
- Out-of-scope resource where the user could know it exists → **403 `FORBIDDEN`**.
- Cross-partner resource (where BRD §8.4 hides existence) → **404 `NOT_FOUND`**.
- This decision is made by the controller, not `AbacGuard`; `AbacGuard` returns `{ granted: false, reason }` and the controller applies the policy.

All 403 deny events are emitted via `AuditAppender` (action `break_glass_access` for unmask events; for general ABAC denies the audit action is `lead_update` or the relevant BRD action with a deny flag). The `AuditChainConsumer` (single-writer) appends to `audit_logs`.

### Backend Flow (Step-by-Step)

The following is the request processing pipeline for **any protected endpoint** as implemented by FR-002's middleware:

```
1. HTTP request arrives
   → CorrelationMiddleware attaches X-Correlation-Id to req + logger context

2. JwtAuthGuard (global)
   a. Extract JWT from Authorization header or httpOnly cookie
   b. Verify signature + expiry (throw AUTH_REQUIRED 401 on failure)
   c. Load user entitlement from Redis cache (key abac:{orgId}:{userId})
      Cache miss → execute Read 1 (role_permissions JOIN) → populate cache (TTL 60 s)
   d. If user.status != 'active' → throw FORBIDDEN 403 (SUSPENDED_USER)
   e. Attach AuthenticatedUser to req.user

3. AbacGuard (on endpoints decorated with @Requires)
   a. Read capability + scopeResolver from @Requires metadata
   b. Invoke scopeResolver(req) → AbacResource
   c. Call EntitlementService.can(req.user, capability, resource)
      i.  Resolve scope — if T, fetch team_members (cache or Read 2)
          If R, fetch region_branches (cache or Read 3)
          If ADMIN/DPO + lead-content capability, run Read 4 (break_glass_grants)
      ii. Run evaluation algorithm (steps 1–6 above)
   d. On deny:
      → Emit audit intent via AuditAppender (non-blocking; appended by AuditChainConsumer)
      → If reason = PARTNER_CROSS_ACCESS → throw NotFoundException (404)
      → Otherwise → throw ForbiddenException (403)
   e. On grant:
      → Attach req.scopePredicate + req.effectiveScope to request context
      → If effectiveScope = 'M' or user has break_glass_grant, attach req.maskingLevel

4. ValidationPipe (Zod)
   → Validate DTO shape; VALIDATION_ERROR (400) on failure

5. Controller handler executes
   → Passes req.scopePredicate to the repository / service
   → Repository applies scope predicate to every Kysely query (see Data Operations)
   → Service applies business logic

6. ResponseEnvelopeInterceptor
   → Wraps result in { data, meta: { correlation_id, pagination? }, error: null }

7. MaskingInterceptor
   → Reads req.effectiveScope (from step 3e)
   → If scope = 'M' (DPO) or export context: apply strictest masking
   → Otherwise: apply role-based masking per masking rules below
   → Emits audit intent via AuditAppender if unmasked PII is accessed under break-glass

8. AllExceptionsFilter (global)
   → Catches unhandled errors → INTERNAL_ERROR (500) envelope; stack logged server-side only
```

### Masking Rules

**Server-side (MaskingInterceptor)** — applied on serialisation to every outbound response:

| Field | Default (most roles) | DPO (scope M) | Export (strictest) |
|---|---|---|---|
| `mobile` | `98xxxxxx10` (first 2 + last 2) | `98xxxxxx10` (masked) | masked unless break-glass |
| `pan_masked` | `ABCxxxx1F` (first 3 + last 2) | `ABCxxxx1F` | masked unless break-glass |
| `aadhaar_ref_token` | last-4 only | last-4 only | last-4 only (never full) |
| `email` | `ab****@domain.com` | masked | masked |
| `full_name` | unmasked (internal roles) | first-name only | first-name only |
| Break-glass unmasked | full value | full value | full value + audit |

Raw Aadhaar number is never stored or returned (security.md).

**Client-side (`MaskedField` component)** — for cases where the raw token is passed to the client (e.g. RM viewing own leads):

- Receives `maskedValue` (already server-masked) and an optional `canUnmask` boolean (derived from `effectiveScope`).
- If `canUnmask=true` (break-glass grant active): renders a "Reveal" button that calls `POST /audit/unmask` (FR-003) and then displays the raw value inline.
- Otherwise: renders the masked string with a tooltip explaining the masking.

---

### Transaction Boundaries via UnitOfWork

FR-002 itself does **not** open a `UnitOfWork` transaction (it performs only reads for the ABAC decision). However:

- All Redis cache reads are performed outside any transaction (Kysely transaction does not span Redis).
- The `AuditAppender` emit (on deny or unmask) is **fire-and-forget** to the `AuditChainConsumer` queue; it does not need to be in a transaction because the guard's deny is already complete before the audit write.
- If a caller's `UnitOfWork` transaction rolls back, the audit intent already emitted is not rolled back — this is intentional (audit of deny/unmask is unconditional).

---

### State Machine Reference

FR-002 does **not** transition any `lead_stage`. It only appends to `audit_logs` (via `AuditChainConsumer`, the single writer). There is no entity state machine for ABAC decisions.

---

### External Service Calls

None. FR-002 does not call `IntegrationGateway` or any external port. All data is read from Postgres (with Redis caching).

## Ambiguities

The following items are not fully resolved by the contracts and are flagged for human decision before implementation:

1. **Unmask endpoint ownership.** The `MaskedField` "Reveal" button calls `POST /audit/unmask`. This endpoint is not listed in `api-contract.yaml` with an `x-frs` tag. FR-003 (break-glass) is the closest owner, but FR-002 renders the button. Decision needed: does FR-003 define this endpoint, or does FR-002? Best-effort: assign the endpoint to FR-003's LLD; FR-002's `MaskedField` is the only caller.

2. **`AuditAppender` action code for ABAC deny events.** `audit_logs.action` is typed as the `audit_action` enum in `schema.sql`. The enum does not include a dedicated `abac_deny` value. The closest existing value for a generic deny is unclear — `lead_update` covers too wide a range. Best-effort: use the most specific action available (e.g. `lead_view` for a `view_lead` deny) and add a `detail.denied=true` flag. If a dedicated `abac_deny` enum value is added to `schema.sql`, update accordingly.

3. **Cache invalidation trigger for `role_permissions`.** The LLD specifies TTL-based expiry (60 s).

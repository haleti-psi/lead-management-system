## Summary

Allows a customer to view a **customer-safe, stage-mapped status** of their lead (no internal RM detail) and to **request a callback slot** — which creates a `Task` of type `callback` assigned to the lead's current owner. The callback task raises a hot-signal (`ScoringService.setHotFlag`) because a callback request is a high-intent event (BRD FR-031 rule 2.g). Both endpoints are token-scoped to a single lead; the token is validated by `CustomerLinkGuard` (opaque, never a JWT). All customer views and actions are audited.

No external-service calls are made. No `leads` stage transition occurs. The sole write is a `tasks` row insertion plus hot-flag update on `leads` (via `LeadService.setHotFlag`).

## Endpoints

### 1. GET /api/v1/c/{token}/status

**Request**

| Element | Value |
|---|---|
| Method | GET |
| Path | `/api/v1/c/{token}/status` |
| Path param | `token` — opaque string (resolved to `customer_links.token_hash` by `CustomerLinkGuard`) |
| Query params | none |
| Request body | none |
| Rate limit tier | public/customer-link: 10 req/min per IP (`RATE_LIMITED` 429) |

**Response 200**

```json
{
  "data": {
    "lead_code": "LD-2026-000123",
    "customer_name": "Rajesh K.",
    "stage_label": "Documents Required",
    "stage_description": "We need a few documents from you before we can proceed.",
    "pending_actions": ["Upload PAN copy", "Upload address proof"],
    "is_handed_off": false,
    "los_status_label": null
  },
  "meta": { "correlation_id": "corr_abc123" },
  "error": null
}
```

`stage_label` and `stage_description` are derived from the **customer-safe stage map** (see Data Operations §1). When `leads.stage = 'handed_off'`, `is_handed_off` is `true` and the response shows only "With lending team" messaging; internal `los_application_id` and RM details are never surfaced.

**Field masking:** `customer_name` is taken from `customer_profiles.display_name` — no PAN, mobile, or Aadhaar is included in this response.

### 2. POST /api/v1/c/{token}/callback

**Request**

| Element | Value |
|---|---|
| Method | POST |
| Path | `/api/v1/c/{token}/callback` |
| Path param | `token` — same as above |
| Headers | `Idempotency-Key: <client-generated UUID>` (required) |
| Content-Type | `application/json` |
| Rate limit tier | public/customer-link: 10 req/min per IP (`RATE_LIMITED` 429) |

**Request body**

```json
{
  "preferred_slot": "2026-06-12T10:00:00+05:30",
  "note": "Please call after 10 AM"
}
```

| Field | Type | Required | Constraints |
|---|---|---|---|
| `preferred_slot` | ISO-8601 datetime (with TZ) | Yes | Must be within the allowed callback window (see Validation) |
| `note` | string | No | Max 500 chars |

**Response 201**

```json
{
  "data": {
    "task_id": "uuid",
    "message": "Your callback request has been received. We'll call you at your preferred time."
  },
  "meta": { "correlation_id": "corr_xyz789" },
  "error": null
}
```

**Idempotency:** a replayed `Idempotency-Key` within 24 h returns the original 201 response with the same `task_id`; no duplicate Task is created. The key is stored in Redis with a 24 h TTL. Response has `detail.reason = IDEMPOTENT_REPLAY` in the meta when replayed (HTTP 200 with original body, per `IDEMPOTENT_REPLAY` sub-reason).

## Auth Check

Both endpoints are listed in `auth-matrix.json → public_endpoints`; they are therefore decorated with `@Public()` and exempted from `JwtAuthGuard`.

**`CustomerLinkGuard`** (located at `apps/api/src/modules/self-service/`) applies to all `/c/{token}/*` routes:

1. Resolve `token` → `sha256(token)` → query `customer_links` by `token_hash`.
2. If no row found: respond `NOT_FOUND` (404) — existence is hidden.
3. If `link_status` is `'expired'`, `'revoked'`, or `'used'`: respond `NOT_FOUND` (404).
4. If `otp_verified_at IS NULL` (OTP step-up not completed on this session): respond `AUTH_REQUIRED` (401).
5. If all checks pass: attach `{ leadId, customerLinkId, orgId }` to the request context (`cls.set`).

There is no `AbacGuard` or `@Requires` on these endpoints; the `CustomerLinkGuard` is the sole authz boundary, scoping the session to one `lead_id`.

**Important:** the response for an invalid/expired token is always `NOT_FOUND` (404) — the endpoint auth note says existence is hidden for `/c/{token}/*` (auth-matrix `endpoint_auth_notes`).

### 1. GET /c/{token}/status — read path

All queries use the Kysely typed instance (`Db`) with parameterised bindings. No raw SQL strings.

const lead = await db
  .selectFrom('leads')

  .where('lead_id', '=', ctx.leadId)
  .where('deleted_at', 'is', null)

// Step 2: resolve customer display name

### 2. POST /c/{token}/callback — write path

The write consists of a single `tasks` INSERT plus `LeadService.setHotFlag` for the hot-signal, plus an audit-log emit. These three writes are wrapped in a single `UnitOfWork` transaction.

```typescript
await uow.run(async (tx) => {
  // Step 1: re-validate lead is in a state where callback makes sense
  const lead = await db
    .selectFrom('leads')
    .select(['lead_id', 'stage', 'owner_id', 'is_hot', 'version'])
    .where('lead_id', '=', ctx.leadId)
    .where('deleted_at', 'is', null)
    .forUpdate()           // row-lock to prevent concurrent double-insert
    .executeTakeFirstOrThrow();

  // Step 2: idempotency check (Redis) — if key seen, return cached response
  // (handled in controller before uow.run — see Backend Flow)

  // Step 3: insert Task of type 'callback'
  const task = await db
    .insertInto('tasks')
    .values({
      task_id: generateUuid(),
      org_id: ctx.orgId,
      lead_id: ctx.leadId,
      type: 'callback',
      owner_id: lead.owner_id,   // assigned to current lead owner
      due_at: dto.preferred_slot,
      priority: 'high',          // callback = high-intent → high priority (FR-031)
      status: 'open',
      result_note: dto.note ?? null,
      created_by: SYSTEM_USER_ID,
      updated_by: SYSTEM_USER_ID,
    })
    .returning(['task_id'])
    .executeTakeFirstOrThrow();

  // Step 4: set hot flag via LeadService (owner-writes rule; only LeadService writes leads)
  if (!lead.is_hot) {
    await leadService.setHotFlag(
      ctx.leadId,
      true,
      ['callback_requested'],    // score_reasons field
      tx,
    );
  }

  // Step 5: audit log emit (append-only sink)
  await auditAppender.emit({
    action: 'comm_send',           // closest audit_action for customer interaction
    entityType: 'task',
    entityId: task.task_id,
    leadId: ctx.leadId,
    actorId: SYSTEM_USER_ID,
    detail: { task_type: 'callback', preferred_slot: dto.preferred_slot },
    ipDevice: extractIpDevice(request),
  }, tx);

  return task;
});
```

**Note on owner assignment:** if `lead.owner_id IS NULL` (lead not yet assigned), the Task `owner_id` is set to the org's default assignment queue owner (a system/ops user configured in `UNASSIGNED_LEAD_OWNER_ID` env var). This edge case is logged at `warn` level.

## Validation Logic

All validation uses Zod schemas evaluated by the `ValidationPipe` before the handler is called. `VALIDATION_ERROR` (400) is returned with `fields[]` on failure.

### GET /c/{token}/status — no body validation; path param validated by guard.

### POST /c/{token}/callback

```typescript
const CallbackRequestDto = z.object({
  preferred_slot: z
    .string({ required_error: "preferred_slot is required" })
    .datetime({ offset: true, message: "preferred_slot must be a valid ISO-8601 datetime with timezone" })
    .refine(
      (val) => {
        const dt = new Date(val);
        const now = new Date();
        const maxFutureDays = 7; // configurable via CALLBACK_SLOT_MAX_FUTURE_DAYS env
        const minFutureMinutes = 30;
        const minDt = new Date(now.getTime() + minFutureMinutes * 60 * 1000);
        const maxDt = new Date(now.getTime() + maxFutureDays * 24 * 60 * 60 * 1000);
        return dt >= minDt && dt <= maxDt;
      },
      { message: "preferred_slot must be at least 30 minutes in the future and within 7 days" }
    ),
  note: z
    .string()
    .max(500, "note must be at most 500 characters")
    .optional(),
});
```

| Field | Rule | Error field | Error message |
|---|---|---|---|
| `preferred_slot` | Required | `preferred_slot` | "preferred_slot is required" |
| `preferred_slot` | ISO-8601 with TZ | `preferred_slot` | "preferred_slot must be a valid ISO-8601 datetime with timezone" |
| `preferred_slot` | ≥ 30 min from now | `preferred_slot` | "preferred_slot must be at least 30 minutes in the future and within 7 days" |
| `preferred_slot` | ≤ 7 days from now | `preferred_slot` | "preferred_slot must be at least 30 minutes in the future and within 7 days" |
| `note` | Max 500 chars | `note` | "note must be at most 500 characters" |

**Lead state guard for callback:** if the lead is in stage `'handed_off'` or `'rejected'`, the POST returns `VALIDATION_ERROR` (400) with message "Callback requests cannot be made for applications in this status" (field: `preferred_slot`). A handed-off lead is owned by the LOS; the RM task would be meaningless.

|---|---|---|---|---|
| Token not found, expired, revoked, or used | 404 | `NOT_FOUND` | — | CustomerLinkGuard; existence hidden |
| OTP step-up not completed | 401 | `AUTH_REQUIRED` | — | CustomerLinkGuard |
| Rate limit exceeded | 429 | `RATE_LIMITED` | — | `Retry-After` header set; Redis ThrottlerGuard |

| `note` > 500 chars | 400 | `VALIDATION_ERROR` | — | `fields[]` populated |

| Unhandled server error | 500 | `INTERNAL_ERROR` | — | No stack/SQL in response; alert raised |

## Backend Flow

### GET /c/{token}/status (step-by-step)

1. **CorrelationMiddleware** stamps `X-Correlation-Id` on request.
2. **JwtAuthGuard** skipped (`@Public()`).
3. **CustomerLinkGuard** resolves token → `customer_links` lookup → validates `link_status = 'active'` and OTP step-up → attaches `{ leadId, customerLinkId, orgId }` to CLS context. Returns 404/401 on failure.
4. **ThrottlerGuard** checks Redis rate bucket (10/min per IP). Returns 429 on breach.
5. **Controller** calls `SelfServiceService.getLeadStatus(ctx)`.
6. **Service** — `selectFrom('leads')` by `lead_id`; throws `NotFoundException` if not found or soft-deleted.
7. **Service** — `selectFrom('customer_profiles')` by `customer_profile_id`.
8. **Service** — maps `lead.stage` via `CUSTOMER_STAGE_MAP` constant.
9. **Service** — if stage is `documents_pending`: queries `documents` WHERE `doc_status IN ('pending','mismatch')` LIMIT 10 to produce `pending_actions[]`.
10. **Service** — calls `AuditAppender.emit({ action: 'link_open', ... })` (fire-and-forget via outbox; does not affect response).
11. **ResponseEnvelopeInterceptor** wraps result in `{ data, meta, error: null }`.
12. Return 200.

### POST /c/{token}/callback (step-by-step)

1. **CorrelationMiddleware** stamps `X-Correlation-Id`.
2. **JwtAuthGuard** skipped (`@Public()`).
3. **CustomerLinkGuard** — same as above; attaches context. Returns 404/401 on failure.
4. **ThrottlerGuard** — 429 if rate limit breached.
5. **ValidationPipe (Zod)** parses and validates `CallbackRequestDto`. Returns 400 with `fields[]` on failure.
6. **Controller** extracts `Idempotency-Key` header; calls `SelfServiceService.requestCallback(ctx, dto, idempotencyKey)`.
7. **Service** — checks Redis idempotency cache: `GET idempotency:callback:{idempotencyKey}`. If hit, deserialize cached response and return with `meta.detail.reason = IDEMPOTENT_REPLAY` (HTTP 200).
8. **Service** — opens `UnitOfWork.run(async (tx) => { ... })`:
   a. SELECT lead (with `FOR UPDATE` lock) by `lead_id`; throws 404 if deleted.
   b. Validates lead stage is not `'handed_off'` or `'rejected'`; throws 400 if so.
   c. INSERTs `tasks` row (`type='callback'`, `status='open'`, `priority='high'`, `owner_id=lead.owner_id`, `due_at=dto.preferred_slot`).
   d. If `!lead.is_hot`: calls `LeadService.setHotFlag(leadId, true, ['callback_requested'], tx)` — which UPDATEs `leads.is_hot = true` and `leads.score_reasons` with the callback reason (optimistic lock uses a dedicated volatile mutator path — no `expectedVersion` conflict with RM edits per §11.2).
   e. Calls `AuditAppender.emit({ action: 'comm_send', entityType: 'task', ... }, tx)`.
9. **Service** — on successful commit: stores `{ task_id }` in Redis idempotency cache with 24 h TTL: `SET idempotency:callback:{idempotencyKey} <json> EX 86400`.
10. **ResponseEnvelopeInterceptor** wraps result in `{ data: { task_id, message }, meta, error: null }`.
11. Return 201.

**Transaction boundary:** the `UnitOfWork` in step 8 covers `tasks INSERT` + `leads UPDATE` (if hot-flag) + `audit_logs emit`. If any step fails, the entire transaction rolls back. No partial state persists.

## State Machine

### CustomerLink.link_status (CustomerLinkGuard validation)

`CustomerLinkGuard` validates `link_status = 'active'`. Per state-machines.md:

- `active` → `used` / `expired` / `revoked` (all terminal for customer access)
- Only `active` links allow status view and callback requests.
- This FR does **not** transition the `CustomerLink` status (it remains `active` for re-use within its 7-day TTL).

### Task.task_status (new Task created)

A new `tasks` row is inserted with `status = 'open'`. Per state-machines.md:

```
open → in_progress → done
open/in_progress → cancelled
open → overdue (SLA sweep)
```

This FR only creates the task in `'open'` state. Subsequent transitions (RM picks up → `in_progress` → `done`) are handled by FR-100 (Task Management) outside this FR.

### leads.is_hot (side effect)

`is_hot` is a derived volatile field updated by `LeadService.setHotFlag`. The callback event is a high-intent signal (BRD FR-031 rule 2.g). This is not a `lead_stage` transition — no `stage_history` row is written, no `event_outbox(LEAD_STAGE_CHANGED)` is emitted for this side effect. A separate `HOT_LEAD` outbox event is emitted by `LeadService.setHotFlag` if the flag flips from false to true (FR-031 implementation responsibility).

- **Mutation error (NOT_FOUND):** redirect to token-expired page.

- All inputs have visible `<label>` — no placeholder-only.

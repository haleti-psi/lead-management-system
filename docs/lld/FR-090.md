## Summary

FR-090 provides CRUD management of the **Partner master** — the authoritative registry of external channel partners (DSA, Dealer, Connector, OEM, Aggregator, Referral) used by the NBFC. It exposes three endpoints: list partners, create a partner, and update/change the status of an existing partner. Status changes are governed by a simple state machine (`active → suspended / expired`) and audited. A suspended or expired partner blocks new lead submissions (enforced by FR-091 at submission time — FR-090 owns only the master record). Configuration changes are versioned via `ConfigurationVersion` governance where applicable.

All endpoints sit under `/api/v1/partners` and share the uniform `{ data, meta, error }` envelope.

## Auth Check

### Authentication

Global `JwtAuthGuard` is applied. No endpoint carries `@Public()`. Unauthenticated requests receive `AUTH_REQUIRED` (401).

### Authorisation

`AbacGuard` + `@Requires('configuration', scopeResolver)` applied to all three endpoints. `EntitlementService.can(user, 'configuration', partnerResource)` is called.

**Capability matrix for `configuration`:**

| Role | Scope | Allowed |
|---|---|---|
| ADMIN | A (all org) | Yes — full CRUD, any partner |
| HEAD | A (all org) | Yes — full CRUD, any partner |
| BM | B (branch) | Yes — partners mapped to their branch |
| SM | — | Not listed in capability_matrix for `configuration`; denied (see Ambiguities) |
| RM / KYC / DPO / PARTNER / CUSTOMER | — | No `configuration` capability; FORBIDDEN (403) |

**Scope enforcement for BM:**
- On list (`GET`): `WHERE branch_id = :userBranchId OR branch_id IS NULL` (org-wide partners visible to BM; see Ambiguities).
- On get-by-id / PATCH: if `partners.branch_id` is not null and does not match the caller's `branch_id`, return `NOT_FOUND` (404) to hide existence (per §8.4).

**Status-change restriction:** changing `status` to `suspended` or `expired` requires an additional check — only ADMIN/HEAD may suspend/expire any partner; BM may only update metadata for partners in their branch scope. The service enforces this; the guard grants entry by capability, not by status-change action.

## Data Operations

All queries use the **Kysely** typed query builder. Table name: `partners`. Column names are exact from `docs/data-model/schema.sql`.

### List partners

```typescript
// PartnerRepository.list(orgId, filters, scopeFilter, pagination)
const query = db
  .selectFrom('partners')
  .select([
    'partner_id', 'partner_code', 'type', 'legal_name',
    'branch_id', 'products', 'contact_person', 'contact_mobile',
    'status', 'agreement_ref', 'commission_flag', 'mapped_rm_id',
    'risk_category', 'quality_score', 'valid_until',
    'created_at', 'updated_at'
  ])
  .where('org_id', '=', orgId)
  .$if(filters.status !== undefined, (qb) => qb.where('status', '=', filters.status!))
  .$if(filters.type !== undefined, (qb) => qb.where('type', '=', filters.type!))
  .$if(scopeFilter.branchId !== undefined, (qb) =>
    qb.where((eb) =>
      eb.or([
        eb('branch_id', '=', scopeFilter.branchId!),
        eb('branch_id', 'is', null)
      ])
    )
  )
  .orderBy(resolvedSortColumn, resolvedSortDirection)
  .limit(pagination.limit)      // max 100, enforced before call
  .offset((pagination.page - 1) * pagination.limit);
```

A separate `COUNT(*)` query (same `WHERE` clause, no `LIMIT`) provides the total for pagination meta.

### Create partner

```typescript
// PartnerRepository.create(payload, createdBy, tx)
await db
  .insertInto('partners')
  .values({
    partner_id:      crypto.randomUUID(),
    org_id:          orgId,
    partner_code:    payload.partnerCode,
    type:            payload.type,
    legal_name:      payload.legalName,
    branch_id:       payload.branchId ?? null,
    products:        JSON.stringify(payload.products ?? []),
    contact_person:  payload.contactPerson ?? null,
    contact_mobile:  payload.contactMobile ?? null,
    status:          'active',          // always starts active
    agreement_ref:   payload.agreementRef ?? null,
    commission_flag: payload.commissionFlag ?? false,
    mapped_rm_id:    payload.mappedRmId ?? null,
    risk_category:   payload.riskCategory ?? null,
    valid_until:     payload.validUntil ?? null,
    created_at:      new Date(),
    updated_at:      new Date(),
    created_by:      createdBy,
    updated_by:      createdBy,
  })
  .executeTakeFirstOrThrow();
```

PostgreSQL unique constraint `uq_partners_code` on `(org_id, partner_code)` raises a uniqueness violation that is caught and mapped to `CONFLICT` (409).

### Update partner

```typescript
// PartnerRepository.update(partnerId, orgId, changes, updatedBy, tx)
const result = await db
  .updateTable('partners')
  .set({
    ...changesMap,               // only provided non-immutable fields
    updated_at: new Date(),
    updated_by: updatedBy,
  })
  .where('partner_id', '=', partnerId)
  .where('org_id', '=', orgId)
  .returningAll()
  .executeTakeFirst();

if (!result) throw new NotFoundException();
```

Zod schema validated at the controller boundary (Nest `ValidationPipe` backed by Zod).

### CreatePartnerDto

| Field | Rule | Error message (field-level) |
|---|---|---|
| `partnerCode` | string, 1–20 chars, `/^[A-Z0-9_-]+$/i` | "Partner code must be 1–20 alphanumeric characters." |

| `contactMobile` | optional, matches `^[6-9][0-9]{9}$` | "Mobile must be a valid 10-digit Indian mobile number." |
| `riskCategory` | optional, enum `risk_band`: `low`, `medium`, `high` | "Risk category must be one of: low, medium, high." |
| `validUntil` | optional, ISO date string, must be a future date | "Valid until must be a future date." |
| `commissionFlag` | optional boolean (default false) | — |
| `mappedRmId` | optional UUID | "Mapped RM ID must be a valid UUID." |
| `agreementRef` | optional string, max 80 chars | "Agreement reference must be under 80 characters." |

### UpdatePartnerDto

Same field rules as above for each field, plus:

| Field | Rule | Error message |
|---|---|---|
| `partnerCode` | **forbidden** (immutable) | "Partner code cannot be changed after creation." |
| `type` | **forbidden** (immutable) | "Partner type cannot be changed after creation." |
| `status` | enum `partner_status` if provided; transition must be valid | See state machine below |
| `statusReason` | required if `status` is being changed to `suspended` or `expired` | "A reason is required when suspending or expiring a partner." |

### State machine validation (PATCH status field)

Valid transitions (from `state-machines.md` compact table):

| From | To | Allowed |
|---|---|---|
| `active` | `suspended` | Yes |
| `active` | `expired` | Yes |
| `suspended` | `active` | Yes (reactivation) |
| `suspended` | `expired` | Yes |
| `expired` | `active` | No — returns `VALIDATION_ERROR` with field `status` |
| `expired` | `suspended` | No — returns `VALIDATION_ERROR` with field `status` |

Attempting an invalid transition → `VALIDATION_ERROR` (400) with `fields: [{ field: "status", issue: "Transition from '<current>' to '<requested>' is not permitted." }]`.

## Backend Flow

### POST /api/v1/partners — step by step

1. **Correlation middleware** attaches `X-Correlation-Id` to the request context.
2. **JwtAuthGuard** validates the access token; loads `user` (including `role_id`, `branch_id`, `org_id`) onto the request.
3. **AbacGuard** calls `EntitlementService.can(user, 'configuration', { orgId })`. ADMIN/HEAD pass with scope A; BM passes with scope B. All others receive `FORBIDDEN` (403).
4. **ValidationPipe (Zod)** validates the `CreatePartnerDto`. Failures return `VALIDATION_ERROR` (400) with `fields[]`.
5. **PartnerController** calls `PartnerService.create(dto, user)`.
6. **PartnerService** opens a `UnitOfWork` transaction:
   a. Calls `PartnerRepository.create(dto, user.userId, tx)` — inserts into `partners`. If the DB unique constraint fires, the service catches the Postgres `23505` error code and throws `ConflictException` → `CONFLICT` (409).
   b. Calls `AuditAppender.append({ action: 'partner_created', entityType: 'partner', entityId: newPartnerId, actorId: user.userId, detail: { after: sanitisedPartner } }, tx)` — sanitised means `contact_mobile` is masked before being written to `audit_logs.detail`.
   c. Calls `OutboxService.emit({ event: 'PARTNER_CREATED', payload: { partnerId: newPartnerId } }, tx)`.
7. Transaction commits; all three writes (insert + audit + outbox) are atomic.
8. **ResponseEnvelopeInterceptor** wraps the returned partner object in `{ data, meta, error: null }`.
9. HTTP 201 returned.

### GET /api/v1/partners — step by step

1. **CorrelationMiddleware** → **JwtAuthGuard** → **AbacGuard** (same pattern as above).
2. **Scope resolution:** if `user.role === 'BM'`, `scopeFilter.branchId = user.branchId`; otherwise no branch filter (ADMIN/HEAD see all).
3. **PartnerController** calls `PartnerService.list(queryParams, scopeFilter, user.orgId)`.
4. **PartnerRepository.list(...)** executes the paginated Kysely query with filters applied.
5. **MaskingService** masks `contact_mobile` on all returned rows before serialisation.
6. Returns `{ data: partnerArray, meta: { pagination: { page, limit, total } } }`.

### PATCH /api/v1/partners/{id} — step by step

1. **CorrelationMiddleware** → **JwtAuthGuard** → **AbacGuard** (`configuration` capability).
2. **ValidationPipe (Zod)** validates `UpdatePartnerDto`. Immutable fields (`partnerCode`, `type`) rejected if present.
3. **PartnerController** calls `PartnerService.update(id, dto, user)`.
4. **PartnerService.update:**
   a. Fetches current partner row: `PartnerRepository.findById(id, user.orgId)`. If not found or out of BM branch scope → `NotFoundException`.
   b. If `dto.status` is provided, validates the transition against the state machine. Invalid → `VALIDATION_ERROR`.
   c. If transitioning to `suspended` or `expired`, asserts `dto.statusReason` is present. Missing → `VALIDATION_ERROR`.
   d. Opens `UnitOfWork` transaction:
      - `PartnerRepository.update(id, user.orgId, changesMap, user.userId, tx)`.
      - `AuditAppender.append({ action: statusChanged ? 'partner_status_changed' : 'partner_updated', entityType: 'partner', entityId: id, actorId: user.userId, detail: { before: sanitisedBefore, after: sanitisedAfter } }, tx)`.
5. Commits; returns updated partner.

## State Machine & Transaction Boundaries

FR-090 touches the `Partner.partner_status` state machine (from `state-machines.md`):

```
active → suspended   (PATCH status=suspended; statusReason required; audited)
active → expired     (PATCH status=expired;  statusReason required; audited; also set by validity job)
suspended → active   (reactivation; audited)
suspended → expired  (PATCH; audited)
expired → active     INVALID → VALIDATION_ERROR
expired → suspended  INVALID → VALIDATION_ERROR
```

**Transaction boundary** for status-changing PATCH:

```
BEGIN
  UPDATE partners SET status=..., updated_at=..., updated_by=... WHERE partner_id=... AND org_id=...
  INSERT INTO audit_logs (action='partner_status_changed', detail={before, after, reason}, ...)
COMMIT
```

`event_outbox` is emitted on create only (PARTNER_CREATED). Status changes do not emit an outbox event in this FR (the downstream effect — blocking FR-091 submissions — is enforced at submission time by checking `partners.status` directly).

No `LeadService` writes are involved; `partners` is owned by `PartnerService`.

## Ambiguities

1. **SM and `configuration` capability:** `auth-matrix.json` does not list `configuration` in SM's `capability_matrix`. The BRD FR-090 text lists SM as an allowed role. Best-effort resolution: the SM role does **not** have `configuration` capability as specified in the contracts; SM users will receive `FORBIDDEN` (403) when accessing partner management endpoints. If the intent is to grant SM access, the `capability_matrix` entry for SM must be updated in `auth-matrix.json` before implementation.

2. **BM branch scope on list — org-wide partners:** Some partners may have `branch_id = NULL` (org-wide). The LLD above makes these visible to all BMs. If the intent is that BMs only see partners explicitly mapped to their branch (i.e. `branch_id = userBranchId`), this query must be adjusted. The current resolution (org-wide + own-branch) is the safer default.

## Summary

FR-020 implements real-time duplicate and near-duplicate detection for NBFC leads. Detection runs at lead creation (FR-010), lead edit, bulk import, API intake, and explicitly via `POST /leads/{id}/duplicate-check`. The engine (`DuplicateService.match()`) queries `lead_identities` using indexed exact and trigram-fuzzy keys, computes a confidence score (Strong/Medium/Weak) per the BRD default-match table, and takes the highest-confidence action. Results are persisted as `duplicate_matches` rows; `leads.duplicate_status` is recomputed from those rows by `LeadService`. Strong-block actions raise `CONFLICT` (409) with sub-reason `DUPLICATE_BLOCKED`; weaker actions return 200 with the match list and the configured action. All actions (including overrides) are audited. The service is internal-only for intake hooks; the REST endpoint is for on-demand/manual checks.

## Endpoint

### POST /api/v1/leads/{id}/duplicate-check

**Purpose:** Run or re-run duplicate detection on a specific lead and return the match result + action taken.

#### Path Parameter

| Parameter | Type | Constraint |
|---|---|---|
| `id` | UUID | must reference an existing, non-deleted lead |

#### Request Body (optional)

```json
{
  "override_reason": "string (max 500, required only when action=override)",
  "requested_action": "block | warn | queue | link | override"
}
```

When called internally (from FR-010 / import / handoff guard) no body is provided; the service uses the configured default action from the match table.

#### Response — 200 OK (warn / queue / link / idempotent-re-check)

```json
{
  "data": {
    "lead_id": "uuid",
    "duplicate_status": "none | flagged | linked | merged",
    "action_taken": "warned | queued | linked | overridden",
    "matches": [
      {
        "duplicate_match_id": "uuid",
        "matched_lead_id": "uuid",
        "matched_lead_code": "LD-2026-000123",
        "confidence": "strong | medium | weak",
        "matched_on": ["pan_token", "mobile"],
        "action": "warned",
        "status": "open"
      }
    ]
  },
  "meta": { "correlation_id": "uuid" },
  "error": null
}
```

#### Response — 409 CONFLICT (strong block on create / explicit re-check with block action)

```json
{
  "data": null,
  "meta": { "correlation_id": "uuid" },
  "error": {
    "code": "CONFLICT",
    "message": "This action conflicts with the current state. Refresh and retry.",
    "retryable": false,
    "detail": {
      "reason": "DUPLICATE_BLOCKED",
      "matches": [
        {
          "duplicate_match_id": "uuid",
          "matched_lead_id": "uuid",
          "matched_lead_code": "LD-2026-000050",
          "confidence": "strong",
          "matched_on": ["pan_token", "mobile"]
        }
      ],
      "override_allowed_by": ["BM", "SM"]
    }
  }
}
```

## Auth Check

### Global authentication
`JwtAuthGuard` is applied globally; no `@Public()` on this endpoint. Unauthenticated → `AUTH_REQUIRED` (401).

### Authorisation — read/check phase
```
@Requires('edit_lead', scopeResolver)
```
`EntitlementService.can(user, 'edit_lead', lead)` must return true.

Scope enforcement:
- RM → scope O: `lead.owner_id === user.user_id`
- SM → scope T: `lead.team_id IN user.managedTeams`
- BM → scope B: `lead.branch_id === user.branch_id`
- HEAD → scope A: `lead.org_id === user.org_id`
- KYC → scope B: `lead.branch_id === user.branch_id`

Out-of-scope lead → `FORBIDDEN` (403).

### Authorisation — override action (block-override)
When `requested_action === 'override'` the service additionally checks:

```
EntitlementService.can(user, 'edit_lead', lead)
  AND user.role IN ['BM', 'SM']
```

RM / KYC attempting override → `FORBIDDEN` (403), audited.

### Authorisation — queue action
`requested_action === 'queue'` is allowed for BM, SM, KYC with branch/team scope.

## Data Operations

All queries use the Kysely `Db` instance with parameterised bindings. No raw SQL strings.

### Step 1 — Fetch the lead and its identity

```typescript
// Fetch lead with scope filter
const lead = await db
  .selectFrom('leads as l')
  .innerJoin('lead_identities as li', 'li.lead_identity_id', 'l.lead_identity_id')
  .leftJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .select([
    'l.lead_id', 'l.org_id', 'l.lead_code', 'l.stage', 'l.branch_id',
    'l.owner_id', 'l.team_id', 'l.duplicate_status', 'l.version',
    'l.product_code', 'l.pin_code', 'l.master_lead_id',
    'li.lead_identity_id', 'li.mobile', 'li.pan_token', 'li.email',
    'li.ckyc_id', 'li.gstin', 'li.name', 'li.dob',
    'sa.source', 'sa.partner_id'
  ])
  .where('l.lead_id', '=', leadId)
  .where('l.org_id', '=', user.org_id)
  .where('l.deleted_at', 'is', null)
  .executeTakeFirst();
```

Returns `undefined` → `NOT_FOUND` (404).

### Step 2 — Execute match queries (parallel, parameterised)

Each key-match query is capped at `LIMIT 10` to prevent unbounded scans. Results are scored individually and then de-duped by `matched_lead_id` keeping highest confidence.

**Exact-key lookups against `lead_identities`** (using indexed columns: `ix_lead_identities_mobile`, `ix_lead_identities_pan`, `ix_lead_identities_ckyc`, `ix_lead_identities_gstin`):

```typescript
// PAN match
const panMatches = pan_token
  ? await db
      .selectFrom('lead_identities as li')
      .innerJoin('leads as l', 'l.lead_identity_id', 'li.lead_identity_id')
      .select(['l.lead_id', 'l.lead_code', 'l.stage', 'l.master_lead_id',
               'l.branch_id', 'l.pin_code', 'l.product_code',
               'li.mobile', 'li.pan_token'])
      .where('li.pan_token', '=', pan_token)
      .where('l.lead_id', '!=', leadId)
      .where('l.org_id', '=', user.org_id)
      .where('l.deleted_at', 'is', null)
      .limit(10)
      .execute()
  : [];

// Mobile match
const mobileMatches = await db
  .selectFrom('lead_identities as li')
  .innerJoin('leads as l', 'l.lead_identity_id', 'li.lead_identity_id')
  .select(['l.lead_id', 'l.lead_code', 'l.stage', 'l.master_lead_id',
           'l.branch_id', 'l.pin_code', 'l.product_code',
           'li.mobile', 'li.pan_token'])
  .where('li.mobile', '=', mobile)
  .where('l.lead_id', '!=', leadId)
  .where('l.org_id', '=', user.org_id)
  .where('l.deleted_at', 'is', null)
  .limit(10)
  .execute();

// CKYC ID match
const ckycMatches = ckyc_id
  ? await db
      .selectFrom('lead_identities as li')
      .innerJoin('leads as l', 'l.lead_identity_id', 'li.lead_identity_id')
      .select(['l.lead_id', 'l.lead_code', 'l.stage', 'l.master_lead_id',
               'l.branch_id', 'l.pin_code', 'l.product_code'])
      .where('li.ckyc_id', '=', ckyc_id)
      .where('l.lead_id', '!=', leadId)
      .where('l.org_id', '=', user.org_id)
      .where('l.deleted_at', 'is', null)
      .limit(10)
      .execute()
  : [];

// GSTIN + product match
const gstinMatches = gstin
  ? await db
      .selectFrom('lead_identities as li')
      .innerJoin('leads as l', 'l.lead_identity_id', 'li.lead_identity_id')
      .select(['l.lead_id', 'l.lead_code', 'l.stage', 'l.master_lead_id',
               'l.branch_id', 'l.pin_code', 'l.product_code'])
      .where('li.gstin', '=', gstin)
      .where('l.product_code', '=', product_code)
      .where('l.lead_id', '!=', leadId)
      .where('l.org_id', '=', user.org_id)
      .where('l.deleted_at', 'is', null)
      .limit(10)
      .execute()
  : [];

// Fuzzy name + pin + source (weak signal; uses GIN trigram index ix_lead_identities_name_trgm)
const fuzzyMatches = await db
  .selectFrom('lead_identities as li')
  .innerJoin('leads as l', 'l.lead_identity_id', 'li.lead_identity_id')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .select(['l.lead_id', 'l.lead_code', 'l.stage', 'l.master_lead_id',
           'l.branch_id', 'l.pin_code', 'l.product_code'])
  .where(sql`li.name % ${name}`)           // trigram similarity operator
  .where('l.pin_code', '=', pin_code)
  .where('sa.source', '=', source)
  .where('l.lead_id', '!=', leadId)
  .where('l.org_id', '=', user.org_id)
  .where('l.deleted_at', 'is', null)
  .limit(10)
  .execute();
```

Note: `sql` is imported from `kysely` for the raw similarity operator fragment only; all values remain parameterised.

### Step 3 — Upsert `duplicate_matches` rows (within UnitOfWork transaction)

```typescript
// Insert or update the match record; unique constraint uq_dup_pair prevents duplicates
await db
  .insertInto('duplicate_matches')
  .values({
    org_id: user.org_id,
    lead_id: leadId,
    matched_lead_id: candidateLeadId,
    confidence: computedConfidence,      // 'strong' | 'medium' | 'weak'
    matched_on: JSON.stringify(matchedKeys),  // JSONB array of key names
    action: resolvedAction,              // 'blocked' | 'warned' | 'queued' | 'linked' | 'overridden'
    action_by: overrideActorId ?? null,
    action_reason: overrideReason ?? null,
    status: 'open',
    created_by: user.user_id,
    updated_by: user.user_id,
  })
  .onConflict(oc =>
    oc.constraint('uq_dup_pair').doUpdateSet({
      confidence: eb => eb.ref('excluded.confidence'),
      matched_on: eb => eb.ref('excluded.matched_on'),
      action: eb => eb.ref('excluded.action'),
      action_by: eb => eb.ref('excluded.action_by'),
      action_reason: eb => eb.ref('excluded.action_reason'),
      status: 'open',
      updated_by: user.user_id,
      updated_at: new Date(),
    })
  )
  .execute();
```

### Step 4 — Recompute `leads.duplicate_status` via LeadService

`LeadService` is the sole writer of `leads`. The service derives `duplicate_status` from the highest-severity open `duplicate_matches` row:

```typescript
// Inside LeadService.recomputeDuplicateStatus(leadId, tx)
const openMatches = await db
  .selectFrom('duplicate_matches')
  .select(['action', 'status'])
  .where('lead_id', '=', leadId)
  .where('status', '=', 'open')
  .where('org_id', '=', orgId)
  .orderBy(
    sql`CASE action
          WHEN 'merged'    THEN 1
          WHEN 'linked'    THEN 2
          WHEN 'blocked'   THEN 3
          WHEN 'queued'    THEN 4
          WHEN 'warned'    THEN 5
          WHEN 'overridden' THEN 6
        END`
  )
  .limit(1)
  .executeTakeFirst();

// Map to dup_status enum
const dupStatus: DupStatus =
  action === 'merged'                          ? 'merged'
  : action === 'linked'                        ? 'linked'
  : action === 'blocked' || action === 'queued'
    || action === 'warned'                     ? 'flagged'
  : /* overridden or no open matches */          'none';

await db
  .updateTable('leads')
  .set({ duplicate_status: dupStatus, updated_by: actorId, updated_at: new Date() })
  .where('lead_id', '=', leadId)
  .where('version', '=', expectedVersion)   // optimistic lock
  .execute()
  .then(r => { if (r.numUpdatedRows === 0n) throw new ConflictException('CONFLICT'); });
```

### Step 5 — Emit audit log (via AuditAppender, within the same tx)

Audit action: `lead_override` (for override action) or `lead_update` (for all other actions).

```typescript
await auditAppender.append({

```

### Step 6 — Emit outbox event (within the same tx, via OutboxService)

```typescript
await outboxService.emit(
  'LEAD_STAGE_CHANGED',           // closest available event_code for status derivation change
```

## Backend Flow — Step by Step

1. **Request arrives** at `DedupController.check(id, dto, user)`.
2. **Zod validation** on path param and optional body (controller boundary).
3. **ABAC check**: `EntitlementService.can(user, 'edit_lead', {lead_id:id})` — FORBIDDEN if denied.
4. **Fetch lead + identity** with scope filter (see Data Operations Step 1). NOT_FOUND if absent.
5. **Terminal-stage guard**: reject if `stage IN ('handed_off')` (no point checking on a closed lead).
6. **Run match queries in parallel** (Promise.all across the 5 key-type groups). Each query is independently bounded at LIMIT 10.
7. **Score and de-duplicate results** using the confidence-scoring rules below.
8. **Determine action** from the BRD default-match table (or from `requested_action` if provided). If any match is `strong` AND action is not `override`, the service will raise `CONFLICT` with `DUPLICATE_BLOCKED`.
9. **Open `UnitOfWork` transaction**:
   a. Upsert `duplicate_matches` rows (one per unique `matched_lead_id`, keeping highest confidence).
   b. Call `LeadService.recomputeDuplicateStatus(leadId, expectedVersion, tx)` — sole writer of `leads`.
   c. Append `audit_logs` via `AuditAppender`.
   d. Emit outbox event via `OutboxService.emit(...)`.
10. **Commit transaction**. Any failure rolls back entirely.
11. **Build and return response**. 409 for strong-block; 200 with match list for all other outcomes.

### Confidence Scoring Rules (BRD default-match table)

| Match condition | Confidence | Default action |
|---|---|---|
| Same `pan_token` AND same `mobile` | `strong` | block (unless BM/SM override with reason) |
| Same `pan_token`, different `mobile` | `strong` | warn → identity-review queue |
| Same `mobile`, no `pan_token` on either | `medium` | warn; allow with `duplicate_status=flagged` |
| Same `ckyc_id` | `strong` | block → identity-review queue |
| Same asset/vehicle ID (via `gstin` product proxy) | `strong` | block → asset-review queue |
| Same `gstin` AND same `product_code` | `medium` | warn; link to business profile |
| Fuzzy name + same `pin_code` + same `source` | `weak` | warning only |
| Multiple matches | highest-confidence match wins for action |  |
| Matched lead has `master_lead_id IS NOT NULL` (merged-master) | inherit master's confidence | point to master |

When no match is found: `duplicate_status=none`, action=none, 200 with empty `matches` array.

### Internal invocation (FR-010 / import / handoff)

`DuplicateService.match(leadId, tx)` is called internally by the lead capture flow and handoff guard. It uses the same logic but skips the HTTP layer. On strong-block during capture it throws `DuplicateBlockedException` (mapped to CONFLICT 409 at the intake controller).

## State Machine & Transaction Boundaries

### `DuplicateMatch.dup_record_status`

From `state-machines.md` compact section:

```
open → resolved
```

- `open`: match detected; action applied (blocked/warned/queued/linked/overridden).
- `resolved`: set by FR-021 (merge) or a BM/SM explicit resolution action.

FR-020 only creates/updates matches in `open` status. FR-021 sets `status=resolved`.

### `leads.duplicate_status` (derived field)

```
none → flagged  (any open warn/block/queue)
none → linked   (open link action)
flagged → none  (all open matches resolved / overridden)
linked → merged (FR-021 merge)
```

`duplicate_status` is a derived field: it is recomputed by `LeadService.recomputeDuplicateStatus` every time `duplicate_matches` changes. It is never set directly.

### Transaction boundary (UnitOfWork)

All writes in Step 9 (duplicate_matches upsert + leads update + audit_logs insert + event_outbox insert) execute within a single `UnitOfWork.run()` transaction. Partial failure in any step rolls back all writes. The outbox event is published post-commit by the `OutboxService` relay (not within the tx).

## Assumptions

1. No dedicated `DUPLICATE_FLAGGED` or `DUPLICATE_RESOLVED` event code exists in the `event_code` enum. FR-020 emits `LEAD_STAGE_CHANGED` as the closest available code. A future schema migration should add `DUPLICATE_FLAGGED`; this is tracked as an ambiguity.
2. Vehicle/asset/engine/chassis identifiers are not separate columns in `lead_identities`; the BRD references them as match keys. The schema stores GSTIN as a proxy for business asset identity. True vehicle ID matching (VAHAN) is Phase 1.5 (`AssetVerificationPort`). FR-020 MVP covers GSTIN+product as the business-asset proxy.
3. The BRD mentions "branch/product/pin proximity" as a match key. This is implemented as: exact `pin_code` match + exact `product_code` match (on `leads`) combined with fuzzy-name similarity. Geographic proximity radius matching is deferred to Phase 1.5.
4. The `requested_action` field in the request body is optional. If omitted the service derives action from the BRD default-match table configuration. The match table thresholds are currently hardcoded in `DuplicateService`; a future Phase 1.5 `DuplicateConfig` table may make them configurable per NBFC.
5. `email` is an optional match key (present in `lead_identities`) but absent from the BRD's explicit match table. FR-020 uses email as a supplementary weak signal only (same email → weak confidence, warn). This is noted as an ambiguity if the NBFC wants email to be a stronger signal.

# FR-010: Omnichannel Lead Capture — Low-Level Design

**Tier: 2** (Moderate — multiple data entities written in a single transaction, stateful outbox/audit emission, idempotency, duplicate-gate integration, no external provider call on the primary path; bulk import path elevates complexity to the upper bound of Tier 2.)

## Summary

FR-010 is the entry point for every lead in the LMS regardless of channel. It handles three surfaces:

1. **Manual / API** — `POST /leads` (authenticated staff or partner)
2. **Bulk import** — `POST /leads/import` (CSV/Excel, async job)
3. **Public / QR / website** — `POST /public/leads` (unauthenticated, captcha + rate-limit)

Every path must: validate the payload against `ProductConfig.field_schema`, run the duplicate check (FR-020 service call, non-blocking except on strong-block), persist `Lead + LeadIdentity + SourceAttribution + (CustomerProfile) + LeadProductDetail(stub)` atomically via `UnitOfWork`, emit `LEAD_CREATED` into `event_outbox`, append to `audit_logs` (action `lead_create`) and `stage_history` (from=null, to=`captured`) — all in a single database transaction. A replayed `Idempotency-Key` returns the original result without a duplicate row.

```json
{
  "data": {
    "lead_id": "uuid",
    "lead_code": "LD-2026-000123",

    "product_code": "CV",
    "consent_status": "partial",
```

**Response 200 (idempotent replay)**
```json
{
  "data": { "lead_id": "original-uuid", "lead_code": "LD-2026-000123", ... },
  "meta": { "correlation_id": "uuid" },
  "error": null
}
```
Note: HTTP 200 is returned (not 201) for a replayed `Idempotency-Key`. The `error.detail.reason = "IDEMPOTENT_REPLAY"` is NOT set in the envelope `error` field (which is `null`); the replay is transparent — the client receives its original response.

**Response 400 (validation)**
```json
{
  "data": null,
  "meta": { "correlation_id": "uuid" },
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Please correct the highlighted fields.",
    "retryable": false,
    "fields": [
      { "field": "identity.mobile", "issue": "Must be a valid 10-digit Indian mobile number." },
      { "field": "source.partner_code", "issue": "partner_code is required when source is DSA or Dealer." }
    ]
  }
}
```

**Response 409 (strong duplicate block)**
```json
{
  "data": null,
  "meta": { "correlation_id": "uuid" },
  "error": {
    "code": "CONFLICT",
    "message": "This action conflicts with the current state. Refresh and retry.",
    "retryable": false,
    "detail": {
      "reason": "DUPLICATE_BLOCKED",
      "matches": [
        { "lead_id": "uuid", "lead_code": "LD-2026-000100", "confidence": "strong", "matched_on": ["mobile", "pan"] }
      ]
    }
  }
}
```

## Auth Check

### POST /api/v1/leads
- **Authentication:** `JwtAuthGuard` (global, applied by default).
- **Authorisation:** `AbacGuard` + `@Requires('create_lead', scopeResolver)`.
  - `EntitlementService.can(user, 'create_lead', { org_id })` evaluates against the `capability_matrix`.
  - Scope enforcement: RM can only create leads that will be owned by themselves (scope `O`); BM can create for any RM in their branch (scope `B`); SM for their team (scope `T`); PARTNER for their own `partner_id` only (scope `P`). HEAD creates at `A` scope.
  - PARTNER check: `user.partner_id` must match `source.partner_id` resolved from `partner_code`; mismatch → `FORBIDDEN`.

### POST /api/v1/leads/import
- **Authentication:** `JwtAuthGuard`.
- **Authorisation:** `AbacGuard` + `@Requires('bulk_action', scopeResolver)`. Only BM, SM, HEAD have `bulk_action` capability per `capability_matrix`.

### POST /api/v1/public/leads
- **Authentication:** `@Public()` — `JwtAuthGuard` is bypassed.
- **Captcha:** `X-Captcha-Token` validated via `CaptchaService` before processing; failure → `FORBIDDEN`.
- **Rate limit:** `ThrottlerGuard` configured at 10/min per IP (same tier as auth endpoints).
- `channel_created_by` is forced to `website` or `qr` (from query param `?channel=qr`); caller cannot override.

## Data Operations

All writes use **Kysely parameterised queries** inside `UnitOfWork.run(async (tx) => { ... })`. No raw string SQL.

### Step A — Idempotency check (before transaction)
```typescript
// Redis key: `idempotency:create_lead:<idempotency_key>`
const cached = await redis.get(`idempotency:create_lead:${idempotencyKey}`);
if (cached) return JSON.parse(cached);  // return original response
```

### Step B — Read ProductConfig (active version)
```typescript
const config = await db
  .selectFrom('product_configs')
  .where('product_code', '=', dto.product_code)
  .where('status', '=', 'active')
  .where('org_id', '=', orgId)
  .select(['product_config_id', 'field_schema', 'pan_required_at', 'document_checklist'])
  .limit(1)
  .executeTakeFirstOrThrow();
```
No active config → `VALIDATION_ERROR` with `fields: [{ field: 'product_code', issue: 'No active configuration for this product.' }]`.

### Step C — Resolve branch_id (if branch_code provided)
```typescript
const branch = await db
  .selectFrom('branches')
  .where('code', '=', dto.branch_code)
  .where('org_id', '=', orgId)
  .where('is_active', '=', true)
  .select(['branch_id'])
  .limit(1)
  .executeTakeFirst();
```

### Step D — Resolve partner_id (if source is DSA or Dealer)
```typescript
const partner = await db
  .selectFrom('partners')
  .where('partner_code', '=', dto.source.partner_code)
  .where('org_id', '=', orgId)
  .where('status', '=', 'active')
  .select(['partner_id', 'type'])
  .limit(1)
  .executeTakeFirstOrThrow();  // throws NOT_FOUND if absent
```
Partner suspended/expired → `VALIDATION_ERROR`: `{ field: 'source.partner_code', issue: 'Partner is not active.' }`.

### Step E — UnitOfWork transaction (all inserts atomic)

```typescript
await uow.run(async (tx) => {

  // E1. Insert lead_identities
  const identity = await tx
    .insertInto('lead_identities')
    .values({
      org_id: orgId,
      name: dto.identity.name,
      mobile: dto.identity.mobile,
      email: dto.identity.email ?? null,
      pan_token: dto.identity.pan_token ?? null,
      pan_masked: dto.identity.pan_masked ?? null,
      preferred_language: dto.identity.preferred_language ?? null,
      created_by: actorId,
      updated_by: actorId,
    })
    .returning('lead_identity_id')
    .executeTakeFirstOrThrow();

  // E2. Upsert customer_profiles (match on org_id + primary_mobile)
  //     ON CONFLICT DO NOTHING — existing profile is linked, not updated
  await tx
    .insertInto('customer_profiles')
    .values({
      org_id: orgId,
      primary_mobile: dto.identity.mobile,
      display_name: dto.identity.name,
      customer_type: dto.customer_type ?? 'individual',
      created_by: actorId,
      updated_by: actorId,
    })
    .onConflict((oc) => oc.constraint('uq_customer_profiles_mobile').doNothing())
    .execute();

  const customerProfile = await tx
    .selectFrom('customer_profiles')
    .where('org_id', '=', orgId)
    .where('primary_mobile', '=', dto.identity.mobile)
    .select('customer_profile_id')
    .limit(1)
    .executeTakeFirst();

  // E3. Insert source_attributions
  const attribution = await tx
    .insertInto('source_attributions')
    .values({
      org_id: orgId,
      source: dto.source.source,
      sub_source: dto.source.sub_source ?? null,
      partner_id: resolvedPartnerId ?? null,
      campaign_code: dto.source.campaign_code ?? null,
      utm: dto.source.utm ? JSON.stringify(dto.source.utm) : null,
      creator_channel: channelCreatedBy,
      attribution_status: 'original',
      created_by: actorId,
      updated_by: actorId,
    })
    .returning('source_attribution_id')
    .executeTakeFirstOrThrow();

  // E4. Generate lead_code via CodeGenerator
  const leadCode = await codeGenerator.nextLeadCode(tx, orgId);  // LD-{YYYY}-{seq6}

  // E5. Insert leads (sole writer: LeadService.create)
  const lead = await tx
    .insertInto('leads')
    .values({
      org_id: orgId,
      lead_code: leadCode,
      stage: 'captured',
      product_code: dto.product_code,
      product_config_id: config.product_config_id,
      branch_id: resolvedBranchId ?? null,
      pin_code: dto.pin_code ?? null,
      owner_id: resolvedOwnerId ?? null,
      source_attribution_id: attribution.source_attribution_id,
      customer_profile_id: customerProfile?.customer_profile_id ?? null,
      lead_identity_id: identity.lead_identity_id,
      channel_created_by: channelCreatedBy,
      consent_status: derivedConsentStatus,   // see Validation Logic §consent_status
      requested_amount: dto.requested_amount ?? null,
      priority: 'normal',
      duplicate_status: 'none',
      kyc_status: 'not_started',
      import_job_id: importJobId ?? null,
      version: 1,
      created_by: actorId,
      updated_by: actorId,
    })
    .returning('lead_id')
    .executeTakeFirstOrThrow();

  // E6. Insert lead_product_details (stub)
  await tx
    .insertInto('lead_product_details')
    .values({
      org_id: orgId,
      lead_id: lead.lead_id,
      product_config_id: config.product_config_id,
      attributes: dto.product_detail ? JSON.stringify(dto.product_detail) : '{}',
      validation_status: 'incomplete',
      created_by: actorId,
      updated_by: actorId,
    })
    .execute();

  // E7. Insert stage_history (append-only; from_stage=null for initial capture)
  await tx
    .insertInto('stage_history')
    .values({
      org_id: orgId,
      lead_id: lead.lead_id,
      from_stage: null,
      to_stage: 'captured',
      actor_id: actorId,
      reason: 'Initial capture',
      occurred_at: new Date(),
    })
    .execute();

  // E8. Emit audit log (append-only; single-writer AuditAppender enqueues intent)
  await auditAppender.append(tx, {
    actor_id: actorId,
    action: 'lead_create',
    entity_type: 'leads',
    entity_id: lead.lead_id,
    lead_id: lead.lead_id,
    detail: { lead_code: leadCode, channel: channelCreatedBy, source: dto.source.source },
    ip_device: requestMetadata.ipDevice,
  });

  // E9. Emit outbox event (append-only)
  await outboxService.emit(tx, {
    event_code: 'LEAD_CREATED',
    aggregate_type: 'leads',
    aggregate_id: lead.lead_id,
    payload: { lead_id: lead.lead_id, lead_code: leadCode, product_code: dto.product_code, stage: 'captured' },
  });

  // E10. Consent records (if consents[] provided — append-only)
  for (const consent of dto.consents ?? []) {
    await tx
      .insertInto('consent_records')
      .values({
        org_id: orgId,
        lead_id: lead.lead_id,
        customer_profile_id: customerProfile?.customer_profile_id ?? null,
        purpose: consent.purpose,
        state: consent.state,
        channel: channelCreatedBy,
        language: consent.language ?? null,
        notice_version: consent.notice_version,
        consent_text_version: consent.consent_text_version,
        actor: consent.actor,
        ip_device: requestMetadata.ipDevice ? JSON.stringify(requestMetadata.ipDevice) : null,
      })
      .execute();
  }

  return lead.lead_id;
});
```

### Step F — Post-commit: duplicate check (non-blocking for weak/medium)
Call `DuplicateService.matchAsync(leadId)` after the transaction commits. The service creates `duplicate_matches` rows and updates `leads.duplicate_status` in a separate transaction. A strong-block result that slips through (timing edge) is logged but the lead already exists — the BM/SM override path in FR-020 handles remediation.

For **strong duplicates detected before commit** (in the sync validation path): `DuplicateService.matchSync(dto.identity, tx)` runs inside the transaction; if confidence=`strong` and configured action=`blocked`, throw `CONFLICT` with `detail.reason=DUPLICATE_BLOCKED` and the matched leads — the whole transaction rolls back.

### Step G — Post-commit: idempotency cache
```typescript
await redis.set(
  `idempotency:create_lead:${idempotencyKey}`,
  JSON.stringify(responseEnvelope),
  'EX', 86400  // 24 h TTL
);
```

### Step H — Post-commit: scoring (non-blocking)
Dispatch `ScoringService.evaluateAsync(leadId)` via Cloud Tasks (does not block the response). Updates `leads.score` and `leads.score_reasons` in a separate write through `LeadService.setScore(leadId, score, reasons, expectedVersion)`.

## Validation Logic

All validation runs via Zod at the controller boundary before the service layer processes the request.

| Field | Rule | Error message |
|---|---|---|
| `identity.name` | Required; 1–150 chars; not blank | "Name is required." |
| `identity.mobile` | Required; regex `^[6-9]\d{9}$` | "Must be a valid 10-digit Indian mobile number." |
| `identity.email` | Optional; valid email format; max 255 | "Invalid email address." |
| `identity.pan_token` | Optional unless `ProductConfig.pan_required_at = at_capture`; opaque token string | "PAN is required at capture for this product." |
| `product_code` | Required; member of `product_code` enum (`CV`, `CAR`, `TRACTOR`, `CE`, `TW`, `SBL`, `HRM`) | "Invalid product code." |
| `source.source` | Required; member of `lead_source` enum (`DSA`, `Dealer`, `Branch`, `Website`, `Referral`, `Telecalling`, `Field`) | "Source is required and must be from the configured master." |
| `source.partner_code` | Required when `source.source` ∈ `{DSA, Dealer}` | "partner_code is required when source is DSA or Dealer." |
| `pin_code` | Optional; regex `^[0-9]{6}$` | "Invalid pin code format." |
| `requested_amount` | Optional; >= 0; max 15-digit NUMERIC | "Requested amount must be a non-negative number." |
| `consents[*].purpose` | Member of `consent_purpose` enum | "Invalid consent purpose." |
| `consents[*].state` | Member of `consent_state` enum | "Invalid consent state." |
| `consents[*].notice_version` | Required if consent provided; non-empty string | "notice_version is required." |
| `consents[*].consent_text_version` | Required if consent provided; non-empty string | "consent_text_version is required." |
| `consents[*].actor` | Member of `consent_actor` enum | "Invalid consent actor." |
| Bulk file | Content-type multipart/form-data; MIME by inspection (CSV=`text/csv`, XLSX=`application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`); max 10 MB | "File is too large." / "Unsupported file type." |
| Public endpoint `X-Captcha-Token` | Present and valid (CaptchaService verification) | FORBIDDEN |

**`consent_status` derivation** (computed from `consents[]` provided at intake):
- No consents provided → `pending`
- At least one purpose=`lead_contact` granted → `partial` (minimum)
- All required purposes for the product granted → `captured`
- Any purpose `withdrawn` or `denied` → `withdrawn` (if that is the only consent and it covers lead_contact)

The lead is always created; `consent_status` is a derived summary field, not a gate at capture.

**PAN timing rule**: if `ProductConfig.pan_required_at = 'at_capture'` and `identity.pan_token` is absent → `VALIDATION_ERROR` with `fields: [{ field: 'identity.pan_token', issue: 'PAN is required at capture for this product.' }]`. For `before_kyc` or `before_handoff` timing, PAN is optional at capture.

**Bulk row validation**: each row of the CSV/Excel is validated against the same Zod schema (minus file-level fields). Invalid rows are collected into the error report with `(row, column, VALIDATION_ERROR code, human message)`. Valid rows proceed to create leads. The `ImportJob` tracks `total_rows`, `success_rows`, `failed_rows`, and `error_file_ref` (GCS path to the row-error CSV).

## Error Cases

| Code | HTTP | Condition | `detail.reason` |
|---|---|---|---|
| `VALIDATION_ERROR` | 400 | Missing/invalid field, invalid product code, missing partner for DSA/Dealer, PAN required at capture but absent, invalid enum value, bulk file wrong MIME | — |
| `AUTH_REQUIRED` | 401 | Missing or expired JWT on `/leads` or `/leads/import` | — |
| `FORBIDDEN` | 403 | Role lacks `create_lead` capability; PARTNER submitting for another partner's `partner_id`; captcha invalid on `/public/leads` | — |
| `CONFLICT` | 409 | Strong duplicate blocked by configured action | `DUPLICATE_BLOCKED` |
| `CONFLICT` | 409 | Replayed `Idempotency-Key` — this is surfaced as HTTP 200 with original data, NOT a 409; documented here for completeness | `IDEMPOTENT_REPLAY` (transparent) |
| `PAYLOAD_TOO_LARGE` | 413 | Bulk import file > 10 MB | — |
| `UNSUPPORTED_MEDIA` | 415 | Bulk import file not CSV or XLSX (by content inspection) | — |
| `RATE_LIMITED` | 429 | `/public/leads` exceeds 10/min per IP; `/leads` exceeds 60/min per user | — |
| `INTERNAL_ERROR` | 500 | Unhandled exception (transaction failure, code-gen error, etc.) | — |

## Backend Flow — Step by Step

```
POST /api/v1/leads
│
├─ 1. CorrelationMiddleware attaches X-Correlation-Id
│
├─ 2. JwtAuthGuard validates access token → loads user + ABAC attributes
│    (for /public/leads: @Public() skips this; ThrottlerGuard + CaptchaService run instead)
│
├─ 3. AbacGuard + @Requires('create_lead', scopeResolver)
│    → EntitlementService.can(user, 'create_lead', { org_id })
│    → FORBIDDEN (403) on deny
│
├─ 4. Zod DTO validation (CaptureController → CreateLeadDto)
│    → VALIDATION_ERROR (400) with field[] on any schema violation
│
├─ 5. CaptureService.createLead(dto, requestMeta)
│
│   ├─ 5a. Idempotency check (Redis GET)
│   │       → if hit: return cached response (HTTP 200)
│   │
│   ├─ 5b. Read ProductConfig (active, for product_code)
│   │       → 400 VALIDATION_ERROR if none active
│   │
│   ├─ 5c. Validate PAN timing vs ProductConfig.pan_required_at
│   │       → 400 VALIDATION_ERROR if pan required at_capture but absent
│   │
│   ├─ 5d. Resolve branch_id from branch_code (if provided)
│   │       → 400 VALIDATION_ERROR if branch code not found or inactive
│   │
│   ├─ 5e. Resolve partner_id from partner_code (if source DSA/Dealer)
│   │       → 400 VALIDATION_ERROR if partner inactive
│   │       → 403 FORBIDDEN if PARTNER user and partner_id mismatch
│   │
│   ├─ 5f. Sync duplicate pre-check via DuplicateService.matchSync(identity)
│   │       → if strong-block configured: 409 CONFLICT + DUPLICATE_BLOCKED
│   │
│   ├─ 5g. UnitOfWork.run(tx) — ATOMIC:
│   │       • INSERT lead_identities
│   │       • UPSERT customer_profiles (ON CONFLICT DO NOTHING)
│   │       • SELECT customer_profiles for link
│   │       • INSERT source_attributions
│   │       • CodeGenerator.nextLeadCode(tx)
│   │       • LeadService.insertLead(tx, ...) → INSERT leads (stage=captured, version=1)
│   │       • INSERT lead_product_details (stub, validation_status=incomplete)
│   │       • INSERT stage_history (from_stage=null, to_stage=captured)
│   │       • AuditAppender.append(tx, lead_create)
│   │       • OutboxService.emit(tx, LEAD_CREATED)
│   │       • INSERT consent_records[] (if consents provided)
│   │       → rollback on any failure → INTERNAL_ERROR (500)
│   │
│   ├─ 5h. Idempotency cache write (Redis SET, 24h TTL)
│   │
│   ├─ 5i. Dispatch ScoringService.evaluateAsync(leadId) → Cloud Tasks (non-blocking)
│   │
│   └─ 5j. Dispatch DuplicateService.matchAsync(leadId) → Cloud Tasks (non-blocking)
│           for medium/weak matches (strong handled in 5f)
│
├─ 6. ResponseEnvelopeInterceptor wraps response → { data, meta, error }
│     MaskingInterceptor applies role-based masking on name/mobile/PAN
│
└─ 7. Return 201 Created
```

**Bulk import additional flow (POST /leads/import)**:
```
POST /api/v1/leads/import
│
├─ 1–4. Same auth/validation as above (ThrottlerGuard mutation tier)
│
├─ 5. CaptureService.acceptBulkImport(file, importJobId, requestMeta)
│   ├─ 5a. Validate MIME (content inspection) → 415 UNSUPPORTED_MEDIA
│   ├─ 5b. Validate file size ≤ 10 MB → 413 PAYLOAD_TOO_LARGE
│   ├─ 5c. Upload file to GCS → import_jobs.file_ref
│   ├─ 5d. INSERT import_jobs (status=queued)
│   ├─ 5e. Dispatch ImportProcessorJob via Cloud Tasks
│   └─ 5f. Return 202 Accepted with import_job_id
│
└─ [Async] ImportProcessorJob:
   ├─ Parse CSV/XLSX row by row
   ├─ For each row: validate → createLead (same 5a–5j flow) with import_job_id
   │   Success → success_rows++
   │   Failure → collect (row, col, code, message) → failed_rows++
   ├─ Write error CSV to GCS → import_jobs.error_file_ref
   └─ UPDATE import_jobs (status=completed, total/success/failed counts)
```

## State Machine & Transaction Boundaries

FR-010 sets the **initial** state `captured` for every lead. The state machine entry (from §state-machines.md) is:

```
[NONE] → captured     trigger: lead creation (FR-010)
                      who: RM / BM / SM / PARTNER / CUSTOMER / system
                      side effects (all in ONE transaction):
                        • INSERT leads (stage='captured')
                        • INSERT stage_history (from_stage=null, to_stage='captured')
                        • INSERT audit_logs (action='lead_create')
                        • INSERT event_outbox (LEAD_CREATED)
```

The transaction uses `UnitOfWork.run()` (nestjs-cls ambient transaction). All five writes succeed or all roll back. No partial state is ever committed.

Subsequent state transitions (e.g. `captured → assigned`) are handled by FR-030 (allocation), not FR-010.

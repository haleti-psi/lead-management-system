## Summary

FR-051 exposes a single `GET /api/v1/leads/{id}` endpoint that returns a fully-aggregated "Lead 360" response: lead core fields, linked identity (masked per role), product/source card, stage history, consent coverage, latest eligibility snapshot, latest LOS application mirror status, document checklist counts, KYC status summary, open tasks, notes/activity timeline, related/duplicate leads summary, and partner details.

The response is read-only. DPO access is audited via `AuditAppender`. No data is mutated. All PII fields are masked on serialisation by the `MaskingService` / masking interceptor according to the caller's role.

**Notes on masking:**
- DPO (scope M): `identity.mobile` → fully masked (`98xxxxxx10`); `identity.panMasked` returned as-is (already masked); `identity.email` partially masked; `identity.name` returned; `identity.dob` → omitted; `identity.address` → omitted. `notes[].body` → omitted for DPO unless break-glass.
- PARTNER (scope P): receives only their own leads; `identity.mobile` masked; `identity.panMasked` returned; no access to `notes` marked `is_internal=true`.
- RM/BM/KYC: standard masking applied by `MaskingService` / `ResponseEnvelopeInterceptor`.

| 500 | `INTERNAL_ERROR` | Unhandled server error |

## Auth Check

1. **Authentication:** Global `JwtAuthGuard` validates the JWT and loads `req.user` (user_id, role, scope attributes: branch_id, team_id, region_id, partner_id).
2. **Authorisation:** `@Requires('view_lead', scopeResolver)` → `AbacGuard` → `EntitlementService.can(user, 'view_lead', { leadId })`.
   - The `scopeResolver` resolves the lead's `owner_id`, `branch_id`, `team_id`, `org_id`, `source_attribution.partner_id` against the caller's scope:
     - RM (O): `leads.owner_id = req.user.userId`
     - BM (B): `leads.branch_id = req.user.branchId`
     - SM (T): `leads.team_id = req.user.teamId`
     - HEAD (A): any lead in `leads.org_id = req.user.orgId`
     - KYC (B): `leads.branch_id = req.user.branchId`
     - DPO (M): any lead in org (masked view); every access is audited
     - PARTNER (P): `source_attributions.partner_id = req.user.partnerId`
3. Out-of-scope or non-existent lead → `NOT_FOUND` (404) — existence is hidden from out-of-scope callers per BRD §8.4.
4. **DPO audit:** if `req.user.role === 'DPO'`, after the response is assembled, `AuditAppender.emit({ action: 'view_sensitive', entityType: 'lead', entityId: leadId, actorId, detail: { role: 'DPO' } })` is called (fire-and-forget; does not block response).

## Data Operations

All queries use the Kysely `Db` instance with parameterised bindings. No raw string interpolation.

### Step 1 — Fetch lead core + scope check

```typescript
// apps/api/src/modules/workspace/lead360.repository.ts
const lead = await db
  .selectFrom('leads as l')
  .innerJoin('lead_identities as li', 'li.lead_identity_id', 'l.lead_identity_id')
  .leftJoin('customer_profiles as cp', 'cp.customer_profile_id', 'l.customer_profile_id')
  .leftJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .leftJoin('lead_product_details as lpd', 'lpd.lead_id', 'l.lead_id')
  .leftJoin('branches as b', 'b.branch_id', 'l.branch_id')
  .leftJoin('users as owner', 'owner.user_id', 'l.owner_id')
  .leftJoin('teams as t', 't.team_id', 'l.team_id')
  .leftJoin('partners as p', 'p.partner_id', 'sa.partner_id')
  .select([
    'l.lead_id', 'l.lead_code', 'l.stage', 'l.priority', 'l.is_hot',
    'l.score', 'l.score_reasons', 'l.requested_amount', 'l.channel_created_by',
    'l.consent_status', 'l.kyc_status', 'l.duplicate_status',
    'l.los_application_id', 'l.sla_first_contact_due_at', 'l.reopened_count',
    'l.nurture_next_at', 'l.created_at', 'l.updated_at', 'l.version',
    'l.branch_id', 'l.owner_id', 'l.team_id',
    'li.lead_identity_id', 'li.name', 'li.mobile', 'li.email',
    'li.pan_masked', 'li.gstin', 'li.dob', 'li.preferred_language',
    'cp.customer_profile_id', 'cp.display_name', 'cp.customer_type', 'cp.is_existing_customer',
    'sa.source_attribution_id', 'sa.source', 'sa.sub_source', 'sa.partner_id',
    'sa.campaign_code', 'sa.utm',
    'lpd.lead_product_detail_id', 'lpd.product_config_id', 'lpd.attributes', 'lpd.validation_status',
    'l.product_code',
    'b.name as branch_name',
    'owner.user_id as owner_user_id', 'owner.display_name as owner_display_name',
    't.name as team_name',
    'p.partner_id as partner_id_val', 'p.partner_code', 'p.legal_name as partner_legal_name',
    'p.type as partner_type', 'p.status as partner_status',
  ])
  .where('l.lead_id', '=', leadId)
  .where('l.deleted_at', 'is', null)
  .executeTakeFirst();

if (!lead) throw new NotFoundException();  // mapped to NOT_FOUND (404)
```

Scope assertion is performed by `EntitlementService.can(user, 'view_lead', lead)` **after** the fetch — existence hidden from out-of-scope callers via 404 (per §8.4).

### Step 10 — Duplicate matches summary (open only, max 5)

```typescript
const duplicateMatches = await db
  .selectFrom('duplicate_matches as dm')
  .innerJoin('leads as ml', 'ml.lead_id', 'dm.matched_lead_id')
  .select([
    'dm.duplicate_match_id', 'dm.matched_lead_id', 'ml.lead_code as matched_lead_code',
    'dm.confidence', 'dm.status', 'dm.action',
  ])
  .where('dm.lead_id', '=', leadId)
  .where('dm.status', '=', 'open')
  .limit(5)
  .execute();
```

All ten queries run sequentially within the request handler. Given this is a read-only aggregate with no lock contention, no UnitOfWork transaction is required (no writes occur).

Validation via Zod at the controller boundary before the service is called.

| Field | Rule | Error message |
|---|---|---|
| `id` (path param) | Valid UUID v4 (`z.string().uuid()`) | "id must be a valid UUID" |

No request body. No query parameters. Zod schema: `UUIDParam` from `packages/shared/src/...`.

## Error Cases

| Scenario | Code | HTTP | Notes |
|---|---|---|---|
| Path `id` is not a UUID | `VALIDATION_ERROR` | 400 | `fields: [{ field: "id", issue: "id must be a valid UUID" }]` |
| Missing/invalid JWT | `AUTH_REQUIRED` | 401 | Generic message; no user enumeration |
| Lead exists but caller is out of scope | `NOT_FOUND` | 404 | Existence hidden per §8.4 — same as not found |
| Lead does not exist or is soft-deleted | `NOT_FOUND` | 404 | |
| Unhandled internal error | `INTERNAL_ERROR` | 500 | Full error logged server-side; no stack in response |

`FORBIDDEN` (403) is not returned for this endpoint — existence is hidden via 404 (BRD §8.4 policy: out-of-scope lead → hide existence).

## Backend Flow (Step by Step)

```
1. CorrelationMiddleware — assigns/propagates X-Correlation-Id
2. JwtAuthGuard — validates JWT; loads user onto request context
3. AbacGuard (@Requires('view_lead', scopeResolver)) — calls EntitlementService.can(...)
   ↳ EntitlementService resolves scope; if unresolvable at this point, defers to post-fetch
4. ValidationPipe (Zod) — validates path param id is a UUID; 400 VALIDATION_ERROR if not
5. Lead360Controller.getOne(id, req.user)
   ↳ calls Lead360Service.getAggregate(id, user)
6. Lead360Service
   a. Lead360Repository.fetchCore(leadId) — Step 1 query (lead + joins)
   b. If no row returned: throw NotFoundException() → 404 NOT_FOUND
   c. EntitlementService.can(user, 'view_lead', lead) — scope check against loaded lead
      If denied: throw NotFoundException() → 404 NOT_FOUND (existence hidden)
   d. Execute Steps 2–10 queries in sequence (all read-only)
   e. Reduce doc counts, KYC counts, consent de-dup in application layer
   f. If user.role === 'DPO':
      AuditAppender.emit(audit intent for 'view_sensitive') — fire-and-forget
   g. Return assembled Lead360Dto
7. MaskingService / ResponseEnvelopeInterceptor — applies role-based masking
   (mobile, email, dob/address for DPO, internal notes for PARTNER)
8. ResponseEnvelopeInterceptor — wraps result in { data, meta, error: null }
9. Response 200
```

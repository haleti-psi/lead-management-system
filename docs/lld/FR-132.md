## Summary

FR-132 provides the maker-checker governance layer that wraps every configuration write in the system. When any authorised user (maker) submits a high-impact configuration change, the system creates a `configuration_versions` row in `pending` status capturing the diff. A second authorised user (checker) who is different from the maker either approves or rejects the change. On approval the version transitions to `approved` and then `active` (immediately, or at `effective_at`). At any point an `active` version may be rolled back; rollback re-activates the version referenced in `rollback_ref` and marks the current version `rolled_back`.

This FR is complex (Tier 3): it involves a multi-step state machine, two separate UnitOfWork transactions (one per action), self-approval guard, `CONFIG_CHANGED` outbox events, append-only `audit_logs`, and a UI approval queue with destructive confirm dialogs. No external provider is called.

The two contract endpoints (`POST /admin/config/{id}/approve` and `POST /admin/config/{id}/rollback`) govern the checker and rollback actions respectively. The creation of `ConfigurationVersion` rows (pending) is performed by FR-131 master-config write paths; FR-132 owns the approval and rollback lifecycle only.

## Auth Check

Both endpoints are protected by the global `JwtAuthGuard` (no `@Public()` decorator). Authorization uses `AbacGuard` + `@Requires('configuration', scopeResolver)` which calls `EntitlementService.can(user, 'configuration', resource)`.

The `configuration_versions` resource is governed by the `configuration` capability (auth-matrix.json `resource_governance`). Roles with this capability:

| Role | Scope |
|---|---|
| ADMIN | A (all org) |
| BM | B (branch) |
| HEAD | A (all org) |
| KYC | B (branch) |
| DPO | B (branch) |

Configuration resources are not scoped to individual leads, so scope resolution is `org_id` only (no `owner_id` / `partner_id` filter).

**Self-approval guard (approve endpoint only):** After the `AbacGuard` passes, the service fetches the `configuration_versions` row and checks `checker_id_candidate !== maker_id`. If equal, throw `FORBIDDEN` (403) — this is enforced both in service logic and backed by the DB constraint `ck_config_maker_checker`. The DB constraint is a safety net; the service must check first to surface a correct error message.

## Data Operations

All Kysely queries use parameterised bindings. Table and column names are exact from `docs/data-model/schema.sql`.

### Approve / Reject a ConfigurationVersion

```typescript
// 1. Fetch the pending version (inside UnitOfWork tx)
const cv = await db
  .selectFrom('configuration_versions')
  .selectAll()
  .where('configuration_version_id', '=', versionId)
  .where('org_id', '=', user.orgId)
  .executeTakeFirst();

// 2. Guard: must exist and be pending
if (!cv) throw new NotFoundException('NOT_FOUND');
if (cv.status !== 'pending') throw new ConflictException('CONFLICT');

// 3. Self-approval guard
if (cv.maker_id === user.userId) throw new ForbiddenException('FORBIDDEN');

// 4a. On approval — set status to 'active' (immediate) or 'approved' (future effective_at)
const newStatus: ConfigChangeStatus =
  action === 'approved'
    ? (cv.effective_at && cv.effective_at > new Date() ? 'approved' : 'active')
    : 'rejected';

await db
  .updateTable('configuration_versions')
  .set({
    status: newStatus,
    checker_id: user.userId,
    updated_at: new Date(),
    updated_by: user.userId,
  })
  .where('configuration_version_id', '=', versionId)
  .where('status', '=', 'pending')
  .execute();
```

Note: The `WHERE status = 'pending'` in the UPDATE acts as an optimistic guard — if a concurrent checker already acted, zero rows update and the service raises `CONFLICT`.

### Rollback an Active ConfigurationVersion

```typescript
// 1. Fetch the active version
const cv = await db
  .selectFrom('configuration_versions')
  .selectAll()
  .where('configuration_version_id', '=', versionId)
  .where('org_id', '=', user.orgId)
  .executeTakeFirst();

if (!cv) throw new NotFoundException('NOT_FOUND');
if (cv.status !== 'active') throw new ConflictException('CONFLICT');

// 2. Mark current version rolled_back
await db
  .updateTable('configuration_versions')
  .set({
    status: 'rolled_back',
    updated_at: new Date(),
    updated_by: user.userId,
  })
  .where('configuration_version_id', '=', versionId)
  .where('status', '=', 'active')
  .execute();

// 3. Re-activate rollback_ref (if present)
if (cv.rollback_ref) {
  await db
    .updateTable('configuration_versions')
    .set({
      status: 'active',
      updated_at: new Date(),
      updated_by: user.userId,
    })
    .where('configuration_version_id', '=', cv.rollback_ref)
    .where('org_id', '=', user.orgId)
    .execute();
}
```

## Backend Flow (step-by-step)

### Approve / Reject flow

1. `CorrelationMiddleware` attaches `X-Correlation-Id` to the request context.
2. `JwtAuthGuard` validates the access token; loads `user` (userId, orgId, roles, capabilityMap) onto the request.
3. `AbacGuard` + `@Requires('configuration', orgScopeResolver)` calls `EntitlementService.can(user, 'configuration', {org_id})`. Deny → `FORBIDDEN` (403).
4. `ValidationPipe` runs `ApproveConfigDto` Zod schema on the request body. Failure → `VALIDATION_ERROR` (400) with `fields[]`.
5. `ConfigGovernanceService.approve(versionId, dto, user)` is called.
6. `UnitOfWork.run(async (tx) => { … })` opens a single ambient transaction.
7. Inside tx: fetch `configuration_versions` row by `configuration_version_id` and `org_id`. Not found → `NOT_FOUND` (404).
8. Guard: `status !== 'pending'` → `CONFLICT` (409).
9. Guard: `cv.maker_id === user.userId` → `FORBIDDEN` (403). (Self-approval block.)
10. Determine `newStatus`: `action === 'approved'` → `active` (or `approved` if `effective_at` is future); `action === 'rejected'` → `rejected`.
11. `UPDATE configuration_versions SET status = newStatus, checker_id = user.userId, updated_at = now(), updated_by = user.userId WHERE configuration_version_id = versionId AND status = 'pending'`. Zero rows affected → `CONFLICT` (concurrent checker).
12. Emit audit intent via `AuditAppender` (action `config_change`) within the same `tx`.
13. Emit `CONFIG_CHANGED` event via `OutboxService.emit(…, tx)` within the same `tx`.
14. `UnitOfWork` commits. If any step throws, the whole transaction rolls back (no partial state).
15. `ResponseEnvelopeInterceptor` wraps the return value in `{ data, meta, error: null }`.
16. `AuditChainConsumer` (async, single-writer) picks up the audit intent and appends to `audit_logs` with hash chain.

### Rollback flow

1–5. Same as above (authentication, authorization, validation with `RollbackConfigDto`).
6. `ConfigGovernanceService.rollback(versionId, dto, user)` is called.
7. `UnitOfWork.run(async (tx) => { … })` opens a transaction.
8. Inside tx: fetch `configuration_versions` row. Not found → `NOT_FOUND` (404).
9. Guard: `status !== 'active'` → `CONFLICT` (409).
10. `UPDATE configuration_versions SET status = 'rolled_back', updated_at, updated_by WHERE configuration_version_id = versionId AND status = 'active'`. Zero rows → `CONFLICT`.
11. If `cv.rollback_ref` is set: `UPDATE configuration_versions SET status = 'active' WHERE configuration_version_id = cv.rollback_ref AND org_id = user.orgId`.
12. Emit audit intent via `AuditAppender` (action `config_change`, detail includes `reason` and `rollback_ref`).
13. Emit `CONFIG_CHANGED` event via `OutboxService.emit(…, tx)`.
14. `UnitOfWork` commits.
15. Response wrapped in envelope.

## State Machine & Transaction Boundaries

Reference: `docs/contracts/state-machines.md` — ConfigurationVersion section.

```
pending ──[checker approve; checker ≠ maker]──► approved ──[effective_at reached or immediate]──► active
pending ──[checker reject;  checker ≠ maker]──► rejected
active  ──[authorised rollback + reason]───────► rolled_back
         (rollback_ref version is simultaneously set to active)

Invalid:
  checker == maker           → FORBIDDEN (403)
  approve when not pending   → CONFLICT (409)
  rollback when not active   → CONFLICT (409)
```

**Transaction boundaries:**

Each action (approve/reject, rollback) runs in its own `UnitOfWork` transaction. Both transactions include:

- UPDATE on `configuration_versions` (with status guard in WHERE clause)
- Optional UPDATE on `rollback_ref` version (rollback only)
- `AuditAppender.emit(…, tx)` — audit intent enqueued within tx
- `OutboxService.emit('CONFIG_CHANGED', …, tx)` — `event_outbox` INSERT within tx

If any operation within the transaction throws, the entire transaction is rolled back. Post-commit side effects (audit hash chain computation by `AuditChainConsumer`, Pub/Sub relay by outbox publisher) are asynchronous and retryable; they do not roll back the governance action.

No `LeadService` mutators are involved — `configuration_versions` is not a lead entity.

## External Service Calls

None. FR-132 makes no calls via `IntegrationGateway` or any port. The `CONFIG_CHANGED` outbox event is relayed to Pub/Sub by the outbox publisher worker (FR-141) post-commit; failure in that relay does not affect the governance action outcome.

## UI Component Tree

The approval queue and rollback action are surfaced in the Admin Settings area (`/admin/config`).

```
AppShell (role-filtered nav: Admin Settings visible for ADMIN/BM/HEAD/KYC/DPO)
└── /admin/config — Config Governance Page
    ├── PageHeader (title="Configuration Approvals", scoped actions for filtered role)
    ├── DataTable (server-paginated, sortable)
    │   ├── columns: Config Type | Version | Maker | Status | Effective At | Created At | Actions
    │   ├── StatusChip (pending=yellow, approved=blue, active=green, rejected=red, rolled_back=grey)
    │   ├── filter: status (pending | active | all), config_type
    │   └── Row actions:
    │       ├── [pending rows, checker ≠ current user] → "Review" button → ApproveDrawer
    │       └── [active rows] → "Rollback" button → RollbackConfirmDialog
    ├── ApproveDrawer (shadcn Drawer)
    │   ├── DiffViewer (JSONB diff rendered as key-value change list)
    │   ├── EntityForm (RHF+Zod, ApproveConfigSchema)
    │   │   ├── RadioGroup: Approve / Reject
    │   │   └── Textarea: comment (optional)
    │   ├── ConfirmDialog (action=approve: "Approve this configuration change?"; action=reject: "Reject?")
    │   ├── LoadingSkeleton (while fetching version detail)
    │   └── Toast (success: "Configuration approved" / "Configuration rejected"; error: FORBIDDEN/CONFLICT mapped)
    └── RollbackConfirmDialog (shadcn ConfirmDialog — destructive action)
        ├── Textarea: reason (required, 1–500 chars)
        ├── Confirm button (disabled while isSubmitting)
        └── Toast (success: "Configuration rolled back"; error message per taxonomy code)
```

**State handling (every view):**
- Loading: `LoadingSkeleton` in table and drawer.
- Empty: `EmptyState` ("No pending configuration changes") when queue is empty.
- Error: inline `ErrorState` for query failures; `Toast` for mutation failures (maps `CONFLICT` → "Action conflicts with current state. Refresh and retry.", `FORBIDDEN` → "You don't have access to this.", `VALIDATION_ERROR` → field-level inline errors via `EntityForm`).
- Destructive action (rollback): always uses `ConfirmDialog` with mandatory reason input.

**WCAG 2.1 AA:** All buttons keyboard-reachable; `aria-label` on icon-only actions; `ConfirmDialog` uses Radix `Dialog` with focus trap and `aria-modal`; `StatusChip` uses `role="status"` and colour-independent indicators; visible `:focus-visible` ring.

**i18n:** All user-visible strings via `t('admin.config.*')` keys.

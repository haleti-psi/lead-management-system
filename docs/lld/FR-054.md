## Summary

FR-054 provides a global, scope-filtered, PII-masked search across three entity types — **Leads**, **Partners**, and **Tasks** — from a single `GET /search?q=` endpoint. The endpoint returns top-N results per entity type in a grouped envelope. Every result is masked per the caller's role and filtered to the caller's data scope. No writes occur; no state transitions are triggered. This is a **Tier 2** FR because it involves multi-entity query logic, scope filtering, masking, and a dedicated service — but no external service calls, no state machine transitions, and no optimistic-lock concerns.

The UI entry point is the `AppShell` global search palette (`cmd-k`) rendered in the top bar.

## Auth Check

**Step 1 — Authentication:** `JwtAuthGuard` (global, no `@Public()`) verifies the JWT access token and loads `user` (including `role`, `branch_id`, `team_id`, `partner_id`, `org_id`) onto the request context. Returns `AUTH_REQUIRED` (401) if absent/expired.

**Step 2 — Authorization:** `AbacGuard` with `@Requires('view_lead', scopeResolver)`. The scope resolver returns the caller's data scope from `capability_matrix`:

| Role | Scope | Lead filter condition |
|---|---|---|
| RM | O | `leads.owner_id = user.user_id` |
| SM | T | `leads.team_id = user.team_id` |
| BM | B | `leads.branch_id = user.branch_id` |
| HEAD | A | no additional scope filter |
| KYC | B | `leads.branch_id = user.branch_id` |
| DPO | M | no scope filter; all results masked |
| PARTNER | P | `leads` joined via `source_attributions.partner_id = user.partner_id` |

If `EntitlementService.can(user, 'view_lead', leads)` returns false (e.g., ADMIN without break-glass), returns `FORBIDDEN` (403).

Partners and tasks follow the same scope:
- **Tasks**: `tasks.owner_id` scoped identically to leads for RM/SM/BM; HEAD sees all; DPO has no task access (no `view_lead` on tasks beyond compliance view — see Ambiguities).
- **Partners**: partner search is only served for roles that have any lead-view capability; results are unscoped by partner_id (any internal user can look up a partner by name/code for lead context). PARTNER role sees only their own partner record.

**Rate limit:** `ThrottlerGuard` (Redis) — reads tier: 300 requests/min per user.

## Data Operations

All queries use the **Kysely** typed query builder with parameterised bindings. Raw string interpolation is forbidden. Every query includes `org_id` in the WHERE clause and a `LIMIT 5`.

### Lead search

Searches `leads` joined to `lead_identities` for scope-filtered results matching the query string on:
- `leads.lead_code` (exact prefix `ILIKE`)
- `lead_identities.name` (trigram similarity via `pg_trgm`)
- `lead_identities.mobile` (equality on unmasked mobile)
- `lead_identities.pan_token` (equality lookup; display masked)
- `lead_identities.gstin` (equality)
- `leads.los_application_id` (exact match)

```
-- Conceptual shape (implemented via Kysely builder, not raw SQL):
SELECT
  l.lead_id,
  l.lead_code,
  l.stage,
  l.product_code,
  l.owner_id,
  l.branch_id,
  l.created_at,
  li.name    AS applicant_name,
  li.mobile,
  li.pan_masked
FROM leads l
JOIN lead_identities li ON li.lead_identity_id = l.lead_identity_id
WHERE l.org_id = :orgId
  AND l.deleted_at IS NULL
  AND (
    l.lead_code ILIKE :prefix
    OR similarity(li.name, :q) > 0.3
    OR li.mobile = :q
    OR li.pan_token = :panToken
    OR li.gstin = :q
    OR l.los_application_id = :q
  )
  AND <scope_filter>   -- owner_id / team_id / branch_id / partner join
ORDER BY similarity(li.name, :q) DESC, l.created_at DESC
LIMIT 5;
```

- `similarity()` uses the `pg_trgm` GIN index `ix_lead_identities_name_trgm` on `lead_identities.name`.
- `lead_identities.mobile` uses `ix_lead_identities_mobile`.
- `leads.lead_code` uses `uq_leads_code` (unique index).
- `lead_identities.pan_token` uses `ix_lead_identities_pan`.

### Partner search

Searches `partners` on:
- `partners.partner_code` (exact prefix `ILIKE`)
- `partners.legal_name` (trigram similarity; no separate trgm index — uses `similarity()` with the `ix_partners_type_status` b-tree; tolerated given top-N = 5 and partners table is small)

```
SELECT partner_id, partner_code, legal_name, type, status
FROM partners
WHERE org_id = :orgId
  AND (
    partner_code ILIKE :prefix
    OR similarity(legal_name, :q) > 0.3
  )
  AND (
    <if PARTNER role>: partner_id = :userPartnerId
    <else>: TRUE
  )
ORDER BY similarity(legal_name, :q) DESC
LIMIT 5;
```

### Task search

Searches `tasks` on:
- Joined `leads.lead_code` (exact prefix)
- The task type or disposition does not expose PII directly

```
SELECT
  t.task_id,
  t.type,
  t.lead_id,
  l.lead_code,
  t.due_at,
  t.status,
  t.priority
FROM tasks t
LEFT JOIN leads l ON l.lead_id = t.lead_id
WHERE t.org_id = :orgId
  AND (
    l.lead_code ILIKE :prefix
    OR l.lead_identity_id IN (
         SELECT lead_identity_id FROM lead_identities
         WHERE mobile = :q OR similarity(name, :q) > 0.3
       )
  )
  AND <scope_filter>   -- tasks.owner_id / leads.team_id / leads.branch_id
LIMIT 5;
```

- Tasks are scoped identically to leads (the task's associated lead determines branch/team).
- Tasks with `lead_id IS NULL` (system tasks) are excluded from search results.

### Execution strategy

The three queries run in **parallel** via `Promise.all` inside `SearchService.search()`. If any individual sub-query throws, the error is caught, logged, and the result bucket returns an empty array — the overall response still succeeds with the other buckets populated. (INTERNAL_ERROR is only raised if all three fail simultaneously.)

## Error Cases

| Scenario | Code | HTTP | detail.reason | Notes |
|---|---|---|---|---|
| `q` missing or < 2 chars | `VALIDATION_ERROR` | 400 | — | `fields: [{field:"q", issue:"must be at least 2 characters"}]` |
| `q` > 100 chars | `VALIDATION_ERROR` | 400 | — | `fields: [{field:"q", issue:"must not exceed 100 characters"}]` |
| No JWT / expired token | `AUTH_REQUIRED` | 401 | — | — |
| Role has no `view_lead` (ADMIN w/o break-glass) | `FORBIDDEN` | 403 | — | — |
| >300 reads/min per user | `RATE_LIMITED` | 429 | — | Sets `Retry-After` header |
| Unhandled DB/infra error | `INTERNAL_ERROR` | 500 | — | Stack logged server-side; never in response |

No `CONFLICT`, `NOT_FOUND`, or `UPSTREAM_UNAVAILABLE` codes are raised by this endpoint (no writes, no external calls).

## Backend Flow

```
1. CorrelationMiddleware — assign / propagate X-Correlation-Id
2. JwtAuthGuard — validate JWT; load user (user_id, role, branch_id, team_id, partner_id, org_id)
3. ThrottlerGuard — reads tier 300/min per user (Redis); RATE_LIMITED (429) if exceeded
4. AbacGuard + @Requires('view_lead', scopeResolver)
   → EntitlementService.can(user, 'view_lead', leads) → FORBIDDEN (403) if denied
5. ValidationPipe (Zod SearchQueryDto) → VALIDATION_ERROR (400) on failure
6. SearchController.search(query, user)
   → SearchService.search(q, user, orgId)
       a. Build scope predicate from user.role + scope (see auth check table)
       b. Sanitise q for ILIKE: escape '%' and '_'; the trigram operator uses parameterised binding
       c. Prepare partial PAN token lookup only if q matches PAN regex ([A-Z]{5}[0-9]{4}[A-Z])
       d. Execute three queries in parallel (Promise.all):
          i.  LeadSearchRepository.search(q, scopePredicate, orgId, limit=5)
          ii. PartnerSearchRepository.search(q, partnerScopePredicate, orgId, limit=5)
          iii.TaskSearchRepository.search(q, scopePredicate, orgId, limit=5)
       e. If any sub-query rejects: catch, log at error level with correlation_id, return [] for that bucket
       f. Apply MaskingService to lead results based on user.role:
          - pan_masked is already masked in DB; name masked if DPO (M scope)
          - mobile: always masked in search results for all roles (MaskedField contract)
       g. Build response: { leads, partners, tasks }, top_n=5, counts
7. ResponseEnvelopeInterceptor — wrap in { data, meta, error }
8. AllExceptionsFilter — catch any unhandled error → INTERNAL_ERROR (500)
```

**No writes are issued. No AuditAppender, UnitOfWork, or OutboxService is invoked.**

## Assumptions

1. The `pg_trgm` extension is already installed (schema.sql line 13). No additional migration is required for the name trigram index (`ix_lead_identities_name_trgm`) — it exists.
2. A trigram GIN index on `partners.legal_name` is **not** present in `schema.sql`. Partner search falls back to a sequential similarity scan. Given the partners table is small (tens to hundreds of rows), this is acceptable at top-N=5. If partner volume grows, a follow-up migration should add `CREATE INDEX ix_partners_legal_name_trgm ON partners USING GIN (legal_name gin_trgm_ops)`.
3. Task search only finds tasks linked to a lead (`lead_id IS NOT NULL`). Standalone system tasks have no searchable name field.
4. The DPO masked view (`M` scope) for tasks is not explicitly defined in the auth-matrix. See Ambiguities section.
5. `top_n` is hardcoded to 5 per entity type and is not a configurable parameter in this FR.

---

## Ambiguities

1. **DPO task visibility in search:** The `auth-matrix.json` gives DPO `view_lead: M` (masked compliance view). It does not explicitly list task visibility for DPO. Best-effort: DPO can see tasks linked to leads that fall within their masked view scope (all leads, masked). If compliance determines DPO should not see tasks at all, the task bucket should return `[]` for DPO — this must be resolved in the auth-matrix before implementation.

2. **Trigram threshold (0.3):** The similarity threshold for name fuzzy match is set to 0.3 (consistent with FR-020 duplicate detection usage). If business testing shows too many false positives (e.g., short names), it should be tuned. This is a runtime constant, not a configurable parameter.

3. **PAN search — token lookup:** `lead_identities.pan_token` is a tokenised value (not the raw PAN). If the user enters a raw PAN string, the service must tokenise it before lookup. The tokenisation utility is not specified in `shared-utilities.md`. Best-effort: if a `PanTokenService` or equivalent exists in M2/M8, reuse it; otherwise detect PAN-regex input and skip PAN lookup with a log warning until the utility is registered in `shared-utilities.md`.

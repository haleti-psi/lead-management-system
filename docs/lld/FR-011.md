**Tier: 2** (Moderate ‚Äî pure compute/data enrichment invoked as a side-effect of lead create/update; reads multiple tables; writes two fields on `leads` via `LeadService.setScore`; no external provider call on the primary path; scoring rule evaluation is non-trivial business logic requiring deterministic test coverage.)

## Summary

FR-011 computes a transparent, rules-based quality score (0‚Äì100) and an array of human-readable reason codes on a `Lead` whenever scoring-relevant fields are present or change. Scoring is invoked by M2 (CaptureController/UpdateLeadUseCase) and delegates to `ScoringService` (M4 allocation module). The result is written back onto the lead via `LeadService.setScore`. Scoring errors never block the caller; the capture/update succeeds regardless and the score is left `null` with a structured log entry.

**Not in scope for FR-011:** hot-flag determination (`is_hot`) is owned by FR-031; `ScoringService.evaluate` may be shared, but the `setHotFlag` write is driven by FR-031 logic.

## Auth Check

### On `POST /api/v1/leads` (create path)
1. **`JwtAuthGuard`** (global) ‚Äî validates JWT; loads user + ABAC attributes onto request. Returns `AUTH_REQUIRED` (401) if invalid.
2. **`AbacGuard` + `@Requires('create_lead', scopeResolver)`** ‚Äî calls `EntitlementService.can(user, 'create_lead', { branch_id, team_id, partner_id })`. Returns `FORBIDDEN` (403) on deny.
3. Score is a system side-effect; no additional capability check for the scoring step.

### On `PATCH /api/v1/leads/{id}` (update path)
1. **`JwtAuthGuard`** (global).
2. **`AbacGuard` + `@Requires('edit_lead', scopeResolver)`** ‚Äî `EntitlementService.can(user, 'edit_lead', lead)`. Lead is loaded first; if not found, return `NOT_FOUND` (404). Scope:
   - `RM`: `O` ‚Äî own assigned leads only.
   - `BM`: `B` ‚Äî any lead in their branch.
   - `SM`: `T` ‚Äî team leads.
   - `KYC`: `B` ‚Äî branch leads (KYC fields only per capability_conditions).
   - `HEAD`: `A` ‚Äî all org.
3. `PARTNER` does **not** have `edit_lead` capability for FR-011-related fields; partner scope is limited to own submissions view (score is stripped at serialization).
4. `CUSTOMER` role never reaches this endpoint (customer uses `/c/{token}/*`).

### Score visibility (masking)
- `MaskingService` / `ResponseEnvelopeInterceptor` strips `score` and `score_reasons` from the serialized response when the actor's role is `PARTNER` or `CUSTOMER`.
- Internal roles (`RM`, `BM`, `SM`, `KYC`, `HEAD`, `DPO`) see the score subject to their scoped access.

## Data Operations

All queries use Kysely parameterised builders. No string interpolation.

### Read operations (inside `ScoringService.evaluate`)

**1. Load lead + lead_identity + source_attribution (single join)**
```typescript
// ScoringService calls its own repository or receives a hydrated ScoringContext
const row = await db
  .selectFrom('leads as l')
  .innerJoin('lead_identities as li', 'li.lead_identity_id', 'l.lead_identity_id')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .select([
    'l.lead_id', 'l.org_id', 'l.product_code', 'l.product_config_id',
    'l.pin_code', 'l.requested_amount', 'l.priority', 'l.version',
    'li.pan_token', 'li.mobile', 'li.preferred_language',
    'sa.source', 'sa.partner_id'
  ])
  .where('l.lead_id', '=', leadId)
  .where('l.deleted_at', 'is', null)
  .executeTakeFirstOrThrow();
```

**2. Load partner quality (if partner_id present)**
```typescript
const partner = row.partner_id
  ? await db
      .selectFrom('partners')
      .select(['quality_score', 'risk_category', 'status'])

      .executeTakeFirst()
  : null;
```

**3. Load product_config for thresholds (hot-amount threshold; pan_required_at)**
```typescript
const productConfig = await db
  .selectFrom('product_configs')
```

**4. Load lead_product_details attributes (employment_type, asset details, customer_type)**

### Write operation (via `LeadService.setScore`)

`ScoringService` returns a `ScoringResult` to the use case; the use case calls `LeadService.setScore` ‚Äî the only writer of `leads`:

```typescript
// LeadService.setScore signature (from ¬ß11.2)
async setScore(
  leadId: string,
  score: number | null,
  reasons: string[] | null,
  tx: Transaction<DB>
): Promise<void>

// Implementation (inside UnitOfWork.run)
await db
  .updateTable('leads')
  .set({
    score: score,
    score_reasons: reasons ? JSON.stringify(reasons) : null,
    updated_at: new Date(),
    updated_by: systemUserId,
  })
  .where('lead_id', '=', leadId)
  // Note: setScore does NOT use expectedVersion ‚Äî it is a volatile-field mutator
  // (¬ß11.2: "volatile fields like score are updated by dedicated mutators so background
  //  scoring writes don't raise false 409s against RM edits")
  .execute();
```

**Audit log** ‚Äî `AuditAppender` emits a `lead_update` action within the same transaction:
```typescript
await auditAppender.emit({
  action: 'lead_update',
  entity_type: 'Lead',
  entity_id: leadId,
  lead_id: leadId,
  detail: { field: 'score', new_score: score, reason_codes: reasons },
  actor_id: systemUserId,
}, tx);
```

No `event_outbox` entry is written by FR-011 alone. The `LEAD_STAGE_CHANGED` outbox event (if any) is managed by the outer FR-010/FR-050 transaction, not by the scoring side-effect.

## Validation Logic

FR-011 has no inbound request schema of its own; it reads from the already-validated `Lead` object. However, the `ScoringService` enforces output constraints:

| Field | Rule | Zod / Guard |
|---|---|---|
| `score` (output) | Must be `null` OR integer `[0, 100]` | `z.number().int().min(0).max(100).nullable()` ‚Äî enforced against DB constraint `ck_leads_score` |
| `score_reasons` (output) | If `score !== null`, must be a non-empty string array | `z.array(z.string().min(1)).min(1).nullable()` |
| `score_reasons` individual codes | Must be one of the known `ScoreReasonCode` enum values from `@shared/enums` | Zod `z.array(ScoreReasonCodeSchema)` |

These are **output** validations ‚Äî if the scoring rules produce an invalid output, the service logs the error and returns `{ score: null, reasons: null }` rather than persisting an invalid value.

## Scoring Rules (Configurable ‚Äî loaded from `product_configs.sla_config` / `configuration_versions`)

> **‚úî Arbiter decisions 2026-06-12 (see `docs/decisions/seed-config-decisions.md`):**
> (1) **FR-011 owns `leads.score`** ‚Äî this 13-factor model is THE score model; FR-031 owns `is_hot` only and never writes `score`. (2) **D1 seed signed off:** ship the factor table below as `V3__seed_default_scoring_rules.sql`, one `configuration_versions` row at `status='active'` (FR-041 precedent), `diff` JSON per the decision brief; thereafter governed via FR-132 approve/rollback. (3) **D3:** `penalised_sources: []` + `source_rejection_rate_threshold: null` seeded PENDING-BUSINESS (factor inert until business supplies values). (4) Hot-amount threshold home = `sla_config.hot_amount_threshold` (D2; FR-031 amended accordingly).

The scoring engine applies additive weighted rules. Factors and weights are configurable via `ConfigurationVersion` (config_type = `scoring_rules`). The **baseline default** (seeded at bootstrap) is:

| Factor | Condition | Points (default) | Reason Code |
|---|---|---|---|
| Mobile verified | `lead_identities.mobile` present + valid format | +10 | `mobile_verified` |
| PIN present | `leads.pin_code` is not null | +8 | `pin_present` |
| Requested amount present | `leads.requested_amount` is not null | +7 | `requested_amount_present` |
| High amount | `requested_amount` >= product threshold (from `product_configs.sla_config.hot_amount_threshold`) | +10 | `high_amount` |
| Language preference set | `lead_identities.preferred_language` not null | +5 | `language_preference_set` |
| PAN present at capture | `lead_identities.pan_token` is not null AND `product_configs.pan_required_at = 'at_capture'` OR pan present regardless | +15 | `pan_present` |
| PAN missing (penalty) | `pan_token` is null AND `product_configs.pan_required_at = 'at_capture'` | -15 | `pan_missing_penalty` |
| Partner quality good | `partners.quality_score >= 70` AND `partners.status = 'active'` | +10 | `partner_quality_good` |
| Partner quality poor (penalty) | `partners.quality_score < 40` OR `partners.risk_category = 'high'` | -10 | `partner_high_risk` |
| Source high-rejection (penalty) | Source attribution has historically high rejection rate (configurable threshold; `source_attributions.source` in penalised list per `configuration_versions`) | -10 | `source_high_rejection` |
| Customer type business | `lead_product_details.attributes.customer_type = 'business'` | +5 | `customer_type_business` |
| Employment/business type present | `lead_product_details.attributes.employment_type` not null | +5 | `employment_type_present` |
| Asset details present | `lead_product_details.attributes.asset_type` not null | +5 | `asset_details_present` |

**Score clamping:** raw weighted sum is clamped to `[0, 100]`. A score of `0` with non-empty reasons is valid and indicates minimum enrichment.

**Configurability:** the above table is the seed default. At runtime, `ScoringService` loads the active `ConfigurationVersion` (config_type = `scoring_rules`, status = `active`) and merges its `diff` JSONB over the defaults. If no active version exists, fall back to the built-in defaults.

**Reason codes** are defined as `ScoreReasonCode` enum in `@shared/enums`; never construct reason strings ad hoc.

## Error Cases

| Scenario | Code | HTTP | `detail.reason` | Blocking? |
|---|---|---|---|---|
| JWT missing / expired | `AUTH_REQUIRED` | 401 | ‚Äî | Yes (request rejected) |
| Actor out-of-scope for lead | `FORBIDDEN` | 403 | ‚Äî | Yes (request rejected) |
| Lead not found (updateLead path) | `NOT_FOUND` | 404 | ‚Äî | Yes (request rejected) |
| Stale `expected_version` (updateLead) | `CONFLICT` | 409 | ‚Äî | Yes (request rejected) |
| Zod validation on updateLead body | `VALIDATION_ERROR` | 400 | ‚Äî | Yes (request rejected) |
| `ScoringService.evaluate` throws (any reason: DB error reading partner/config, rule engine exception) | logged as `error` internally; no API error returned | ‚Äî | ‚Äî | **No** ‚Äî capture/update succeeds; `score = null`, `score_reasons = null` written |
| `LeadService.setScore` DB write fails within the main UnitOfWork tx | triggers tx rollback; outer use-case returns `INTERNAL_ERROR` (500) if unhandled | `INTERNAL_ERROR` | ‚Äî | Yes (full request fails) |
| Score output violates `[0,100]` constraint | caught by output-validation guard in `ScoringService`; returns null result with log | ‚Äî | ‚Äî | **No** |
| Rate limit exceeded on mutation | `RATE_LIMITED` | 429 | ‚Äî | Yes |

**Key design rule:** scoring is a best-effort enrichment. If `ScoringService.evaluate` throws, the calling use-case catches the error, logs it with `correlation_id`, and proceeds with `score = null`. It does NOT re-throw. The capture/update is not rolled back for a scoring failure.

## Backend Flow (Step-by-Step)

### Trigger: POST /api/v1/leads (via FR-010 CaptureUseCase)

```
1.  CorrelationMiddleware adds X-Correlation-Id to request context.
2.  JwtAuthGuard validates JWT; loads user + ABAC attributes.
3.  AbacGuard: EntitlementService.can(user, 'create_lead', context) ‚Üí 403 if denied.
4.  ThrottlerGuard: 60 mutations/min per user; 429 if exceeded.
5.  ValidationPipe (Zod): validates LeadCreate DTO ‚Üí 400 if invalid.
6.  CaptureController ‚Üí CaptureUseCase.execute(dto, user).
7.  UnitOfWork.run(async (tx) => {
      a. FR-010: LeadService.create(..., tx) ‚Äî creates lead_identity, source_attribution,
         lead, lead_product_details stub, stage_history(captured), audit_logs(lead_create),
         event_outbox(LEAD_CREATED).
      b. FR-011 (this FR): ScoringService.evaluate(leadId, tx) is called AFTER the lead exists.
         i.   Load ScoringContext: lead + lead_identity + source_attribution + partner + product_config
              + lead_product_details (all reads within tx for consistency).
         ii.  Load active scoring ConfigurationVersion (config_type='scoring_rules'); fall back
              to built-in defaults if none.
         iii. Apply factor rules; accumulate weighted score; collect reason codes.
         iv.  Clamp score to [0, 100]; validate reason codes against ScoreReasonCode enum.
         v.   Return ScoringResult { score: number | null, reasons: string[] | null }.
      c. If ScoringService throws:
         - Catch error; log { level: 'error', module: 'scoring', lead_id, correlation_id, err }.
         - Set scoringResult = { score: null, reasons: null }.
         - Do NOT rethrow; do NOT rollback the outer tx.
      d. LeadService.setScore(leadId, scoringResult.score, scoringResult.reasons, tx).
         - UPDATE leads SET score=?, score_reasons=?, updated_at=? WHERE lead_id=?
         - AuditAppender.emit('lead_update', ..., tx).
      e. [FR-031 may also run here if hot-flag rules are evaluated in the same cycle]
    });
8.  ResponseEnvelopeInterceptor wraps result; MaskingService strips score/score_reasons for
    PARTNER/CUSTOMER roles.
9.  Return 201 LeadEnvelope with score present (or null if scoring failed).
```

### Trigger: PATCH /api/v1/leads/{id} (via UpdateLeadUseCase)

```
1.  CorrelationMiddleware, JwtAuthGuard, AbacGuard (edit_lead), ThrottlerGuard, Zod validation.
2.  UpdateLeadUseCase.execute(leadId, dto, user):
    a. Load existing lead (check deleted_at; 404 if not found).
    b. EntitlementService scope check on loaded lead ‚Üí 403 if denied.
    c. Determine if any scoring-relevant fields are in the patch payload:
       SCORING_RELEVANT_FIELDS = [
         'pin_code', 'requested_amount',
         'product_detail.employment_type', 'product_detail.asset_type',
         'product_detail.customer_type'
       ]
       (Note: identity changes like pan_token / preferred_language are not patchable here;
        they go through identity update flow. Source changes are not patchable post-create.)
    d. UnitOfWork.run(async (tx) => {
         i.   Apply field updates to leads and/or lead_product_details.
         ii.  If any SCORING_RELEVANT_FIELD changed:
              - Call ScoringService.evaluate(leadId, tx) (same flow as create path, steps b‚Äìd above).
         iii. LeadService writes via appropriate mutators (expected_version check on leads update).
         iv.  AuditAppender.emit('lead_update', ..., tx).
       });
3.  Return 200 LeadEnvelope with updated score.
```

**Scoring re-evaluation is skipped on PATCH if no scoring-relevant field is in the diff**, to avoid unnecessary DB reads and score thrash on unrelated updates (e.g. adding a note or changing priority).

## Transaction Boundaries

FR-011 does not own a transaction; it enlists in the caller's `UnitOfWork`:

```
UnitOfWork.run(tx)
  ‚îú‚îÄ‚îÄ [FR-010] LeadService.create(...)        ‚Äî lead + identity + attribution + stage_history + outbox
  ‚îú‚îÄ‚îÄ [FR-011] ScoringService.evaluate(...)   ‚Äî reads (within tx for read consistency)
  ‚îî‚îÄ‚îÄ [FR-011] LeadService.setScore(...)      ‚Äî UPDATE leads SET score, score_reasons
                                               ‚Äî AuditAppender.emit(lead_update)
```

If `LeadService.setScore` fails (e.g. DB connection lost), the entire UnitOfWork rolls back ‚Äî the lead is not persisted with a partial score. If `ScoringService.evaluate` itself throws before writing, the exception is caught; the UnitOfWork continues with `score = null`.

**Optimistic locking:** `LeadService.setScore` does NOT use `expectedVersion` (it is a volatile-field mutator per ¬ß11.2). This prevents false 409 conflicts when a concurrent RM edit and a background re-score happen simultaneously.

---

## State Machine

FR-011 does NOT transition `lead_stage`. The `leads.stage` is unchanged by scoring. `score` and `score_reasons` are volatile fields updated in-place without a stage transition.

No `stage_history` row is written by FR-011. No `event_outbox(LEAD_STAGE_CHANGED)` event is emitted by FR-011.

**`is_hot` flag** is not set by FR-011. `ScoringService.evaluate` computes the score and reasons; the hot-flag determination belongs to FR-031 (`LeadService.setHotFlag`), which may be called in the same UnitOfWork cycle.

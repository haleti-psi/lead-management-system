## Summary

FR-031 defines a rules-based, explainable hot-lead prioritisation system. After every relevant mutation of a lead (create, profile update, doc upload, callback logged, LOS indicative received), `ScoringService.evaluate(lead, ctx)` is called to:

1. Apply a weighted scoring model (0–100) producing `score_reasons` reason codes.
2. Apply the hot-rule set (eight named rules) to derive `leads.is_hot`.
3. Persist both fields via `LeadService.setScore()` and `LeadService.setHotFlag()` (the sole writer of `leads`).
4. Emit a `HOT_LEAD` outbox event (and notify via `NotificationDispatchService`) whenever `is_hot` transitions `false → true`.
5. Clear `is_hot` with an updated reason code when a lead "cools" (was hot but no rule now fires).

Scoring errors must never block the triggering operation — the lead write succeeds; scoring failure is logged and the prior score is retained.

Score is used only for prioritisation routing. It is never used for credit approval or rejection decisions.

### Internal trigger: rescore after callback (FR-062) and doc upload (FR-070)

`ScoringService.evaluate()` is also called internally (not via an HTTP path in this FR) by:
- `TaskService` when a `callback` task is created/completed — triggers the `HIGH_INTENT_EVENT` rule.
- `DocumentService` when a document row transitions to `uploaded` — triggers the `CUSTOMER_SUBMITTED_DOCS` rule.
- `EligibilityService` when a snapshot `status` becomes `received` and `indicative_amount IS NOT NULL` — triggers the `POSITIVE_LOS_INDICATIVE` rule.

In all three cases, rescoring calls `LeadService.setScore()` and `LeadService.setHotFlag()` with a fresh `UnitOfWork` transaction.

## Data Operations

All queries use **Kysely** parameterised builders. Column names are exact `snake_case` matches from `schema.sql`.

### Read — fetch lead context for scoring

```typescript
// ScoringService reads the full lead row plus joined context
const lead = await db
  .selectFrom('leads as l')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .leftJoin('customer_profiles as cp', 'cp.customer_profile_id', 'l.customer_profile_id')
  .leftJoin('partners as p', 'p.partner_id', 'sa.partner_id')
  .leftJoin('product_configs as pc', 'pc.product_config_id', 'l.product_config_id')
  .select([
    'l.lead_id',
    'l.priority',
    'l.requested_amount',
    'l.is_hot',
    'l.score',
    'l.score_reasons',
    'l.version',
    'sa.source',
    'sa.partner_id',
    'cp.is_existing_customer',
    'p.quality_score as partner_quality_score',
    'pc.sla_config',          // contains hot_amount_threshold per product (D2 decision 2026-06-12)
  ])
  .where('l.lead_id', '=', leadId)
  .where('l.org_id', '=', orgId)
  .executeTakeFirst();
```

### Read — check for customer-submitted docs (CUSTOMER_SUBMITTED_DOCS rule)

```typescript
const docCount = await db

  .where('uploaded_via', '=', 'customer_link')
  .where('status', 'in', ['uploaded', 'under_review', 'verified'])

  .executeTakeFirst();
```

### Read — check for positive LOS indicative (POSITIVE_LOS_INDICATIVE rule)

```typescript
const snap = await db
  .selectFrom('eligibility_snapshots')
  .select(['indicative_amount', 'status'])
  .where('lead_id', '=', leadId)
  .where('org_id', '=', orgId)
  .where('status', '=', 'received')
  .where('indicative_amount', 'is not', null)

```

### Write — update score + hot flag (via LeadService mutators)

These are the only writes FR-031 makes to `leads`. Both go through `LeadService` (the sole writer of `leads`).

```typescript
// LeadService.setScore — updates score and score_reasons; does not bump optimistic version
// (volatile field, per architecture §11.2)
await db
  .updateTable('leads')
  .set({
    score:          score,
    score_reasons:  JSON.stringify(reasons),
    updated_at:     new Date(),
    updated_by:     systemUserId,
  })
  .where('lead_id', '=', leadId)
  .where('org_id', '=', orgId)
  .execute();

// LeadService.setHotFlag — updates is_hot; does not bump optimistic version
await db
  .updateTable('leads')
  .set({
    is_hot:     isHot,
    updated_at: new Date(),
    updated_by: systemUserId,
  })
  .where('lead_id', '=', leadId)
  .where('org_id', '=', orgId)
  .execute();
```

**Note on optimistic locking:** Per architecture §11.2, volatile fields (`is_hot`, `score`, `sla_*_due_at`) are updated by dedicated mutators that do **not** increment `version` and do **not** require `expectedVersion`. This prevents background scoring writes from raising false 409s against concurrent RM edits on human fields.

## Validation Logic (Zod)

Scoring itself has no separate HTTP endpoint, so there is no independent Zod schema for this FR. The triggering endpoints' DTOs are owned by FR-010 (POST /leads) and FR-050 (PATCH /leads/:id). FR-031 adds the following internal validation rules inside `ScoringService`:

| Field | Rule | Error / handling |
|---|---|---|
| `score` result | Must be integer 0–100 (enforced by `ck_leads_score` CHECK and pre-write assertion) | If ScoringService computes out-of-range, clamp to 0–100 and log `warn` — never raise to caller |
| `score_reasons` | Must be a non-empty array of known reason codes when `is_hot = true` | If reasons empty and hot=true, log `warn` + set hot=false (self-healing) |
| Product config threshold | `hot_amount_threshold` read from `product_configs.sla_config.hot_amount_threshold` *(amended per D2 decision 2026-06-12 — was `field_schema.hot_threshold`, which the built M5 Zod strips)*; if absent, default 500000 INR | Absence is not an error — default applied with reason `AMOUNT_ABOVE_DEFAULT_THRESHOLD` |
| Input lead row | Must be found and not soft-deleted | If not found, log `warn` + skip — never raise (scoring is a side-effect) |

## Hot-Rule Set (BRD §3 AC-2)

`ScoringService.evaluateHotRules(ctx)` applies the following rules in order. A lead is hot if **any** rule fires. Reason codes are included in `score_reasons`. A lead cools when **no** rule fires and `is_hot` was previously `true`.

| # | Rule name | Reason code | Condition |
|---|---|---|---|
| H1 | Priority high | `PRIORITY_HIGH` | `leads.priority = 'high'` |
| H2 | Amount above product threshold | `AMOUNT_ABOVE_THRESHOLD` | `leads.requested_amount > product_configs.sla_config.hot_amount_threshold` (default 500 000 INR; D2 decision 2026-06-12) |
| H3 | Returning customer | `RETURNING_CUSTOMER` | `customer_profiles.is_existing_customer = true` |
| H4 | Partner verified | `PARTNER_VERIFIED` | `source_attributions.partner_id IS NOT NULL AND partners.quality_score >= 70 AND partners.status = 'active'` |
| H5 | Customer submitted docs | `CUSTOMER_SUBMITTED_DOCS` | one or more documents uploaded via `customer_link` channel with `status IN ('uploaded','under_review','verified')` |
| H6 | Positive LOS indicative | `POSITIVE_LOS_INDICATIVE` | latest `eligibility_snapshots.status = 'received' AND indicative_amount IS NOT NULL` |
| H7 | High-intent callback event | `HIGH_INTENT_EVENT` | a `tasks` row of `task_type = 'callback'` exists for this lead with `status IN ('open','in_progress','done')` |
| H8 | Amount above default threshold | `AMOUNT_ABOVE_DEFAULT_THRESHOLD` | H2 fallback when `hot_amount_threshold` is absent in product config: `requested_amount > 500000` |

**Cool-down:** when `ScoringService` re-evaluates a previously hot lead and no rule fires, it calls `LeadService.setHotFlag(leadId, false, ['COOLED'])`. The `HOT_LEAD` outbox event is NOT emitted for a cool-down — only for hot transitions.

## Scoring Model (0–100)

> **⚠ SUPERSEDED — arbiter decision 2026-06-12 (see `docs/decisions/seed-config-decisions.md`):**
> `leads.score` is owned by **FR-011's 13-factor model** (written only via `LeadService.setScore` on the FR-011 path). **FR-031 owns the `is_hot` flag only** — it evaluates the Hot-Rule Set above and writes via `LeadService.setHotFlag`; it MUST NOT write `score`. The table below is retained for reference and is NOT implemented.

The numeric score is independent of the binary hot flag. It is a weighted sum of rule signals plus bonus factors:

| Signal | Max points | Condition |
|---|---|---|
| Priority = high | 25 | `priority = 'high'` |
| Priority = normal | 10 | `priority = 'normal'` |
| Amount ≥ product threshold | 20 | amount rule H2/H8 |
| Amount 50%–100% of threshold | 10 | partial amount match |
| Returning customer | 15 | rule H3 |
| Partner quality ≥ 70 | 15 | rule H4 |
| Customer submitted docs | 10 | rule H5 |
| Positive LOS indicative | 20 | rule H6 |
| High-intent callback | 15 | rule H7 |
| **Cap** | **100** | clamp after sum |

All active reason codes for the applied signals are included in `score_reasons`. The weights above are **initial defaults**; they are intentionally hard-coded in `ScoringService` for MVP. Making them configurable via `product_configs.field_schema` is a post-MVP enhancement (not in scope for this FR).

## Error Cases

| Scenario | Code | HTTP | `error.detail.reason` | Notes |
|---|---|---|---|---|
| Missing/invalid JWT on PATCH | `AUTH_REQUIRED` | 401 | — | Global `JwtAuthGuard` |
| PATCH lead outside caller's scope | `FORBIDDEN` | 403 | — | `AbacGuard`; out-of-scope read/write |
| Lead not found (PATCH /:id) | `NOT_FOUND` | 404 | — | Existence hidden if out of scope |
| PATCH with stale `version` (human fields) | `CONFLICT` | 409 | — | Returned by FR-050's edit flow; not from scoring |
| Scoring compute error (internal) | Score retained; `INTERNAL_ERROR` logged | — | — | Scoring error never surfaces to caller; logged with `correlation_id`; prior score/is_hot kept |
| Product config missing hot_threshold | No error | — | — | Default 500 000 INR applied silently |
| Rate limit on PATCH | `RATE_LIMITED` | 429 | — | 60 mutations/min per user |

**Key constraint:** a scoring failure must never block or roll back the lead create/update that triggered it. The scoring write is attempted after the primary lead write commits. If scoring fails, the lead is still created/updated successfully.

## Backend Flow (Step-by-Step)

The flow below describes `POST /leads` (create). `PATCH /leads/:id` (update) follows the same scoring steps from step 7 onward.

**Step 1 — Request ingress**
`CorrelationMiddleware` stamps `X-Correlation-Id`. `JwtAuthGuard` validates JWT and loads `req.user`.

**Step 2 — Authorization**
`AbacGuard` calls `EntitlementService.can(user, 'create_lead', scope)`. FORBIDDEN (403) if denied.

**Step 3 — DTO validation**
`ValidationPipe` runs FR-010's `CreateLeadDto` (Zod schema). VALIDATION_ERROR (400) if invalid.

**Step 4 — Idempotency check**
If `Idempotency-Key` header is present, check Redis + `integration_logs` (FR-140 pattern). If replay detected, return cached 201 response with `IDEMPOTENT_REPLAY` sub-reason — skip all writes.

**Step 5 — Primary lead create (FR-010)**
`UnitOfWork.run(async (tx) => { ... })` opens a transaction. `LeadService.create(input, tx)` inserts the `leads` row with `is_hot = false`, `score = null`, `score_reasons = null` as initial values.

**Step 6 — Duplicate check (FR-020)**
`DuplicateService.match()` runs in the same tx. If DUPLICATE_BLOCKED → rollback → CONFLICT (409).

**Step 7 — Scoring context load**
After tx commits (or within the same tx for updates), `ScoringService.evaluate(leadId, orgId, triggerSource)` reads the lead + joined context (source_attributions, customer_profiles, partners, product_configs) via the Kysely read queries above.

**Step 8 — Hot-rule evaluation**
`ScoringService.evaluateHotRules(ctx)` applies rules H1–H8. Returns `{ isHot: boolean, hotReasons: string[] }`.

**Step 9 — Score computation**
`ScoringService.computeScore(ctx)` applies the weighted model. Returns `{ score: number, allReasons: string[] }` where `allReasons` is the union of all fired rule codes. Score clamped to [0, 100].

**Step 10 — Hot flag / score persistence**
If scoring succeeds:
- `LeadService.setScore(leadId, score, allReasons, orgId)` — updates `leads.score`, `leads.score_reasons` (volatile mutator, no version bump).
- `LeadService.setHotFlag(leadId, isHot, allReasons, orgId)` — updates `leads.is_hot` (volatile mutator, no version bump).

If scoring throws, the error is caught, logged at `warn` level with `correlation_id`, and the prior values are retained. The scoring failure is never re-thrown to the HTTP handler.

**Step 11 — HOT_LEAD outbox event**
If `isHot = true` AND prior `is_hot` was `false` (transition): `OutboxService.emit('HOT_LEAD', aggregate)` inserts into `event_outbox` (inside a short dedicated tx or the same tx if triggered from within one). No event emitted on cool-down or on re-confirmed-hot (idempotent).

**Step 12 — HOT_LEAD notification (post-commit)**
After the transaction commits, `NotificationDispatchService.send(lead, 'HOT_LEAD_ALERT', ['in_app', 'sms'])` is called via the `engagement` module for the assigned RM and BM. This is a post-commit async call — failure here does not roll back the lead write or scoring.

**Step 13 — Audit log**
`AuditAppender.emit(...)` records `lead_update` with `fields_changed: ['is_hot', 'score', 'score_reasons']`. The single-writer `AuditChainConsumer` appends to `audit_logs` out-of-band.

**Step 14 — Response**
`ResponseEnvelopeInterceptor` wraps the serialized lead (with masking applied). Scoring fields `is_hot`, `score`, `score_reasons` are included for internal roles and excluded for PARTNER/CUSTOMER roles by the masking interceptor.

## Transaction Boundaries

FR-031's scoring writes are **volatile** (architecture §11.2) and intentionally run outside the primary lead create/update transaction to prevent a scoring failure from rolling back the lead write.

```
┌─── Primary transaction (FR-010/FR-050 owns this) ──────────────┐
│  LeadService.create() or LeadService field-update mutators      │
│  DuplicateService.match() (FR-020)                              │
│  stage_history + audit_logs + event_outbox (LEAD_STAGE_CHANGED) │
│  COMMIT                                                         │
└─────────────────────────────────────────────────────────────────┘
         ↓ (after primary commit)
┌─── Scoring mini-transaction ────────────────────────────────────┐
│  ScoringService.evaluate() reads context                        │
│  LeadService.setScore()   — UPDATE leads (volatile)             │
│  LeadService.setHotFlag() — UPDATE leads (volatile)             │
│  OutboxService.emit(HOT_LEAD) — INSERT event_outbox             │
│  COMMIT (or rollback if scoring error — lead row unchanged)     │
└─────────────────────────────────────────────────────────────────┘
         ↓ (post-commit, async)
NotificationDispatchService.send() — in_app + sms notification
AuditAppender.emit() — queued for AuditChainConsumer
```

When scoring is triggered from within an existing transaction (e.g., from the callback handler or doc-upload handler), the scoring mini-transaction runs as a nested/enlist in the caller's UnitOfWork if one is active. The `setScore`/`setHotFlag` mutators accept an optional `tx` parameter.

### Usage in Lead List (FR-050 DataTable)

The `is_hot` column in `DataTable` renders `<LeadHotBadge>`. It is sortable (filter `filter[is_hot]=true`). The column is included for RM/BM/SM/HEAD roles, excluded for PARTNER.

### Usage in Lead 360 (FR-051)

In the lead header section of Lead 360, both `<LeadHotBadge>` and `<LeadScoreChip>` appear adjacent to the lead stage `<StatusChip>`.

### Loading / Error states

- `<LoadingSkeleton>` shown while lead data is fetching.
- If score is `null` (scoring pending or errored), `<LeadScoreChip>` renders "—" with tooltip `t('score.unavailable')`.

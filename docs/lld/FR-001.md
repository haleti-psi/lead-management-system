## Summary

FR-001 implements the secure authentication and session lifecycle for all internal users. It covers password-based login, an optional TOTP/OTP MFA challenge step, short-lived JWT access token issuance (15 min), rotating refresh token (httpOnly cookie), account lockout (5 failed attempts, 15-min window), and audit logging for every auth event. There is no UI for SSO/OIDC in the MVP (deferred to post-MVP configuration); the flow is password-only with a configurable TOTP MFA challenge.

All four `/auth/*` endpoints are explicitly `@Public()` — they are the only routes exempt from `JwtAuthGuard`. The `users.status` field follows the `user_status` state machine: `active → locked` on threshold failure, returning to `active` only via an admin unlock (FR-130). No other FR writes `users.status` or `users.last_login_at`.

## Endpoints

### POST /api/v1/auth/login

**Auth:** `@Public()` — no JWT required.

**Rate limit:** 10 requests / min per IP (Redis ThrottlerGuard).

#### Request

```json
{
  "username": "rm.delhi.001",
  "password": "S3cur3P@ssword!"
}
```

| Field | Type | Rules |
|---|---|---|
| `username` | `string` | required; 1–150 chars |
| `password` | `string` | required; 1–255 chars |

#### Responses

**200 — MFA not required (token issued directly)**

```json
{
  "data": {
    "access_token": "<jwt>",
    "token_type": "Bearer",
    "expires_in": 900,
    "mfa_required": false
  },
  "meta": { "correlation_id": "corr_…" },
  "error": null
}
```

**200 — MFA required (challenge issued)**

```json
{
  "data": {
    "mfa_required": true,
    "mfa_challenge_token": "<opaque short-lived token>",
    "mfa_method": "totp"
  },
  "meta": { "correlation_id": "corr_…" },
  "error": null
}
```

The `mfa_challenge_token` is a signed JWT (separate secret, 5-min TTL) that carries only `{ sub: user_id, type: "mfa_challenge" }`. It is stored as a key in Redis (`mfa:challenge:<user_id>`) to support single-use invalidation.

**400** — missing/invalid fields (`VALIDATION_ERROR`).

**401** — wrong credentials or account does not exist (`AUTH_REQUIRED`, generic — no user enumeration).

**423 / mapped to 403** — account locked (`FORBIDDEN`, `detail.reason: "ACCOUNT_LOCKED"`, includes `retry_after_seconds`).

**429** — rate limit exceeded (`RATE_LIMITED`).

### POST /api/v1/auth/mfa

**Auth:** `@Public()` — caller presents the `mfa_challenge_token` in the request body.

**Rate limit:** 10 requests / min per IP.

#### Request

```json
{
  "mfa_challenge_token": "<opaque token from /auth/login>",
  "otp": "482910"
}
```

| Field | Type | Rules |
|---|---|---|
| `mfa_challenge_token` | `string` | required |
| `otp` | `string` | required; exactly 6 digits |

#### Responses

**200 — tokens issued**

Same shape as the non-MFA login 200 (`mfa_required: false`, access token + refresh cookie set).

**401** — invalid/expired `mfa_challenge_token`, wrong OTP, or OTP replay (`AUTH_REQUIRED`).

**429** — rate limit exceeded (`RATE_LIMITED`).

---

### POST /api/v1/auth/refresh

**Auth:** `@Public()` — reads refresh token from `httpOnly` cookie `lms_refresh`.

**Rate limit:** inherited from default; no extra throttle.

#### Request

No request body. The refresh token is read from the `lms_refresh` `httpOnly` cookie only.

#### Responses

**200** — new access token + rotated refresh cookie (previous refresh token invalidated in Redis).

```json
{
  "data": {
    "access_token": "<new-jwt>",
    "token_type": "Bearer",
    "expires_in": 900,
    "mfa_required": false
  },
  "meta": { "correlation_id": "corr_…" },
  "error": null
}
```

**401** — refresh token absent, expired, or already rotated (`AUTH_REQUIRED`).

---

### POST /api/v1/auth/reset

**Auth:** `@Public()`.

**Rate limit:** 10 requests / min per IP.

#### Request

```json
{
  "email": "user@nbfc.in"
}
```

| Field | Type | Rules |
|---|---|---|
| `email` | `string` | required; valid email format |

#### Response

**200** — always returns success (no user enumeration). If the email matches an active user, a password-reset link is dispatched via the `NotificationChannelPort` (email channel). The link token is a signed JWT (3-hr TTL) stored in Redis (`pw_reset:<user_id>`) for single-use invalidation.

```json
{
  "data": null,
  "meta": { "correlation_id": "corr_…" },
  "error": null
}
```

**429** — rate limit exceeded.

## Auth Check

All four endpoints use `@Public()` — they bypass `JwtAuthGuard` and `AbacGuard` entirely. This is the only legitimate path to bypass global auth. No capability or scope check applies pre-login. Post-login endpoints (any non-`@Public()` route) use `JwtAuthGuard` and optionally `AbacGuard + @Requires(...)` per FR.

## Data Operations

All queries use the `Db` Kysely instance with parameterised bindings. No string interpolation.

### Lookup user on login

```typescript
const user = await db
  .selectFrom('users as u')
  .innerJoin('roles as r', 'r.role_id', 'u.role_id')
  .select([
    'u.user_id', 'u.username', 'u.password_hash', 'u.status',
    'u.mfa_enabled', 'u.org_id', 'u.role_id',
    'r.code as role_code', 'r.default_scope',
    'u.branch_id', 'u.team_id', 'u.region_id', 'u.partner_id',
  ])
  .where('u.org_id', '=', ORG_ID)
  .where('u.username', '=', dto.username)
  .executeTakeFirst();
```

### Update last_login_at on successful authentication

```typescript
await db
  .updateTable('users')
  .set({ last_login_at: new Date(), updated_by: user.user_id })
  .where('user_id', '=', user.user_id)
  .execute();
```

### Lock account on threshold

```typescript
await db
  .updateTable('users')
  .set({ status: 'locked', updated_by: SYSTEM_USER_ID })
  .where('user_id', '=', user.user_id)
  .execute();
```

### Unlock account (admin action, belongs to FR-130 but referenced here)

The identity module exposes `AuthService.unlockUser(userId)` which sets `status = 'active'` and clears the Redis lockout key.

### Insert audit log (every auth event)

```typescript
await db
  .insertInto('audit_logs')
  .values({
    org_id: user.org_id,
    actor_id: user.user_id,          // or SYSTEM_USER_ID for failed lookups where user exists
    action: 'login' | 'login_failed' | 'mfa_failed' | 'logout',
    entity_type: 'users',
    entity_id: user.user_id,
    lead_id: null,
    detail: { /* masked — no passwords/tokens */ ip_device: maskedIpDevice },
    ip_device: { ip: req.ip, user_agent: req.headers['user-agent'] },
    created_at: new Date(),
  })
  .execute();
```

The `AuditAppender` is called; the `AuditChainConsumer` (single-writer) computes and writes `prev_audit_hash`. The auth service must not write the hash chain directly.

## Validation Logic

Implemented with Zod at the controller boundary (NestJS `ValidationPipe`). All validation failures return `VALIDATION_ERROR` (400) with `error.fields[]`.

| Endpoint | Field | Rule | Error message |
|---|---|---|---|
| `/auth/login` | `username` | required, string, 1–150 chars | "Username is required." |
| `/auth/login` | `password` | required, string, 1–255 chars | "Password is required." |
| `/auth/mfa` | `mfa_challenge_token` | required, string | "MFA challenge token is required." |
| `/auth/mfa` | `otp` | required, exactly 6 digits (`/^\d{6}$/`) | "OTP must be 6 digits." |
| `/auth/reset` | `email` | required, valid email (Zod `z.string().email()`) | "A valid email address is required." |

Password complexity is enforced only on the **password-reset completion** endpoint (not on login — the stored hash already encodes the policy). That endpoint is outside this FR's MVP scope but referenced in BRD §4.6: minimum 10 chars, upper + lower + digit + symbol.

## Error Cases

| Scenario | HTTP | Code | `detail.reason` |
|---|---|---|---|
| Missing `username` or `password` | 400 | `VALIDATION_ERROR` | — |
| OTP not 6 digits | 400 | `VALIDATION_ERROR` | — |
| Email not valid format (reset) | 400 | `VALIDATION_ERROR` | — |
| Wrong credentials (user exists or not) | 401 | `AUTH_REQUIRED` | — (generic; no enumeration) |
| Account locked (`status = 'locked'`) | 403 | `FORBIDDEN` | `"ACCOUNT_LOCKED"` |
| MFA challenge token invalid/expired | 401 | `AUTH_REQUIRED` | — |
| Wrong OTP | 401 | `AUTH_REQUIRED` | — |
| OTP replayed (Redis key already consumed) | 401 | `AUTH_REQUIRED` | — |
| Refresh token absent/expired/rotated | 401 | `AUTH_REQUIRED` | — |
| Rate limit exceeded (any auth endpoint) | 429 | `RATE_LIMITED` | — (sets `Retry-After` header) |
| Unhandled server error | 500 | `INTERNAL_ERROR` | — |

Notes on generic messaging: wrong password and non-existent username both return `AUTH_REQUIRED` with the same message ("Please sign in to continue."). The response body must never include which field was wrong or whether the account exists.

## Backend Flow (Step-by-Step)

### POST /auth/login

1. **Rate-limit check** — `ThrottlerGuard` checks Redis `login:ip:<ip>`. If over 10/min, return `RATE_LIMITED` (429) immediately.
2. **Zod validation** — `LoginDto` validated; missing/invalid fields → `VALIDATION_ERROR` (400).
3. **User lookup** — Kysely query on `users JOIN roles` by `username` + `org_id`. If no row returned, insert `audit_logs(action='login_failed', actor_id=SYSTEM_USER_ID, detail={reason:'user_not_found'})` and return `AUTH_REQUIRED` (401) — same response as bad password.
4. **Lockout check** — if `users.status = 'locked'`, check Redis `lockout:<user_id>`. If still active, return `FORBIDDEN` (403) + `detail.reason='ACCOUNT_LOCKED'` + `retry_after_seconds`. Append `audit_logs(login_failed)`.
5. **Password verify** — `argon2.verify(user.password_hash, dto.password)`. On mismatch:
   a. Increment Redis counter `fail:<user_id>` (TTL 15 min; expiry resets on each increment).
   b. If counter reaches 5: set `users.status = 'locked'`, set Redis `lockout:<user_id>` with 15-min TTL, insert `audit_logs(login_failed, detail={reason:'lockout_triggered'})`.
   c. Otherwise insert `audit_logs(login_failed, detail={reason:'bad_password', attempt: count})`.
   d. Return `AUTH_REQUIRED` (401).
6. **Lockout counter reset** — on successful password verify, delete Redis key `fail:<user_id>`.
7. **MFA decision** — check `users.mfa_enabled` **and** whether the user's `role_code` is in `['ADMIN','DPO','HEAD','PARTNER']` (MFA mandatory for these regardless of `mfa_enabled` flag). If MFA is required:
   a. Generate `mfa_challenge_token` (JWT, `type:'mfa_challenge'`, 5-min TTL, signed with `MFA_CHALLENGE_SECRET`).
   b. Store `mfa:challenge:<user_id>` in Redis (TTL 300 s) for single-use invalidation.
   c. Return 200 with `{ mfa_required: true, mfa_challenge_token, mfa_method: 'totp' }`.
8. **Token issuance (no MFA)** — call `AuthService.issueTokens(user)`:
   a. Sign JWT access token (15 min, payload: `{ sub: user_id, org_id, role: role_code, scope: default_scope, jti: uuid }`).
   b. Generate opaque refresh token (UUID v4), store in Redis `refresh:<token>` → `user_id` with 30-day TTL.
   c. Set `httpOnly; Secure; SameSite=Strict` cookie `lms_refresh=<refresh_token>; Path=/api/v1/auth; Max-Age=2592000`.
9. **Audit** — insert `audit_logs(action='login', actor_id=user.user_id, entity_type='users', entity_id=user.user_id)` and update `users.last_login_at`.
10. **Response** — return 200 with access token.

### POST /auth/mfa

1. **Rate-limit check** — same Redis throttle (10/min per IP).
2. **Zod validation** — `MfaDto`; bad OTP format → `VALIDATION_ERROR`.
3. **Verify challenge token** — `@nestjs/jwt` verify with `MFA_CHALLENGE_SECRET`. Invalid/expired → `AUTH_REQUIRED` (401).
4. **Check Redis single-use** — `EXISTS mfa:challenge:<user_id>`. If key absent (already used), return `AUTH_REQUIRED`.
5. **TOTP verify** — `otplib.authenticator.verify({ token: dto.otp, secret: user.totp_secret })`. Wrong OTP:
   a. Increment `mfa_fail:<user_id>` counter. On threshold (reuse the 5-attempt rule), lock and return `FORBIDDEN`.
   b. Insert `audit_logs(action='mfa_failed')`.
   c. Return `AUTH_REQUIRED` (401).
6. **Invalidate challenge** — `DEL mfa:challenge:<user_id>` (prevents replay).
7. **Token issuance** — same as login step 8 above.
8. **Audit** — insert `audit_logs(action='login', ...)`, update `last_login_at`.
9. **Response** — 200 with tokens.

### POST /auth/refresh

1. **Read cookie** — extract `lms_refresh` cookie. Absent → `AUTH_REQUIRED`.
2. **Lookup in Redis** — `GET refresh:<token>` → `user_id`. Not found (expired/rotated/never issued) → `AUTH_REQUIRED`.
3. **Idle timeout check** — if `last_login_at` + 30 min < now (configurable via env `SESSION_IDLE_TIMEOUT_MINUTES`), invalidate refresh token (`DEL refresh:<token>`), return `AUTH_REQUIRED`.
4. **Rotate refresh** — `DEL refresh:<old_token>`, generate new UUID refresh token, store `refresh:<new_token>` → `user_id` (30-day TTL).
5. **Issue new access token** — re-read user row (to pick up any role/status changes), sign new JWT.
6. **Set new cookie** — overwrite `lms_refresh` cookie.
7. **Response** — 200 with new access token.

### POST /auth/reset

1. **Rate-limit check** — 10/min per IP.
2. **Zod validation** — email format.
3. **User lookup by email** — Kysely query on `users` by `email` + `org_id`. Whether the user exists or not, the response is always 200 (no enumeration).
4. **If user found and status=active** — generate password-reset JWT (3-hr TTL, `{ sub: user_id, type: 'pw_reset', jti: uuid }`), store `pw_reset:<user_id>` in Redis (single-use), dispatch via `NotificationChannelPort` email channel using the `pw_reset_link` template.
5. **Audit** — if user found: `audit_logs(action='login', detail={reason:'password_reset_requested'})`. Note: `password_reset_requested` is not a distinct `audit_action` enum value; it is recorded under the closest applicable action with a typed detail. If no matching enum value covers password reset, record under `user_change` with `detail.sub_action='password_reset_requested'`.
6. **Response** — 200 with `data: null`.

## State Machine (User Status)

Per `docs/contracts/state-machines.md` compact machines section.

```
active ──(5 failed logins)──▶ locked ──(admin unlock / TTL expiry)──▶ active
active ◀──(admin reactivate)── inactive ──(admin deactivate)── active
```

This FR touches only the `active → locked` transition (via login failure counter). The `locked → active` admin-unlock is executed by FR-130 (`AuthService.unlockUser`). The `active ↔ inactive` transitions are admin operations (FR-130).

There is no multi-entity write requiring a `UnitOfWork` for the auth flow: the `users.status` update and the `audit_logs` INSERT are the only writes, both on separate tables. They are executed sequentially within the same request. If the audit insert fails after the status update, the exception filter logs the failure; the lock still stands (the conservative direction is safer). This is an acceptable trade-off documented here as an ambiguity note — see the Ambiguities section.

## External Service Calls

### NotificationChannelPort (password reset email)

- **Port:** `NotificationChannelPort` → `EmailAdapter` (prod) / `MockChannelAdapter` (test).
- **When:** only during `POST /auth/reset` for a confirmed active user.
- **Call:** `notificationChannelPort.send({ channel: 'email', templateCode: 'pw_reset_link', recipient: user.email, variables: { reset_url, expires_in: '3 hours' } })`.
- **Failure handling:** provider failure → log at `warn` level; do **not** throw — the response is always 200. The reset email failure does not surface to the caller (no user enumeration). An operational alert fires on repeated send failures (monitored via `communication_logs.status = 'failed'`).
- **No `IntegrationGateway` wrapping required** here: this is a fire-and-forget notification with no idempotency key needed (the reset link itself carries a one-use JWT). The `NotificationDispatchService.send()` shared utility handles consent/opt-out checks; for password reset this is a system-initiated transactional message exempt from marketing consent gating.

## Ambiguities

1. **TOTP secret storage** — the schema has no `totp_secret` column on `users`. The MFA flow requires a per-user TOTP secret for `otplib` verification. This column must be added via a Flyway migration (`ALTER TABLE users ADD COLUMN totp_secret_enc VARCHAR(255);` — encrypted at rest via application-level key from Secret Manager). This is a genuine schema gap for FR-001 and must be resolved before implementation. Best-effort: add `totp_secret_enc` to the `users` table, encrypted with a KMS-managed key; value is `NULL` until MFA is enrolled.

2. **MFA enrollment flow** — FR-001's acceptance criteria mandate MFA for certain roles but do not specify how the TOTP secret is initially set up (QR code enrollment). The BRD LLD table references the "MFA challenge" step but not the enrollment screen. Best-effort: implement a separate `POST /auth/mfa/enroll` endpoint (same module, `@Public()` gated by a short-lived enrollment token) that returns the TOTP URI for QR rendering. This is tracked as an open scope item.

3. **Audit action for password-reset-requested** — the `audit_action` enum does not include `password_reset_requested`. Best-effort: record under `user_change` with `detail.sub_action = 'password_reset_requested'`.

4. **Idle timeout enforcement** — the 30-min idle timeout is checked on token refresh but not on every API call (which would require server-side session state). The JWT access token TTL (15 min) already limits the effective idle window. An additional server-side check is performed on refresh: if `now - last_login_at > SESSION_IDLE_TIMEOUT_MINUTES`, the refresh is denied. This is the agreed implementation approach.

## Summary

FR-003 implements emergency (break-glass) privileged access to lead content for ADMIN and DPO roles who have no standing lead-data access (BRD §3.4). A grant is created in `pending` status by the requester, then approved by a second authorised user (four-eyes / approver ≠ grantee). Once approved the grant becomes `active`; `EntitlementService` consults active grants at every lead-content access for these roles. Grants auto-expire at `valid_until` (via a Cloud Scheduler sweep) and may be revoked early. Every lead access that relies on a break-glass grant is written to `audit_logs` with action `break_glass_access` and the `grant_id` reference. No external provider is involved.

The feature is complex (Tier 3): it touches multi-entity writes inside a UnitOfWork, a state machine with scheduled expiry, time-bound authorisation logic in `EntitlementService`, and the audit hash-chain.


```json
{
  "granteeId": "uuid",
  "scopeType": "lead | branch | all",
  "scopeRef": "uuid | null",
  "reason": "string (1–500 chars)",
  "validFrom": "ISO-8601 TIMESTAMPTZ",
  "validUntil": "ISO-8601 TIMESTAMPTZ"
}
```

**Response (201 Created)**

```json
{
  "data": {
    "grantId": "uuid",
    "granteeId": "uuid",
    "approverId": null,
    "scopeType": "lead",
    "scopeRef": "uuid | null",
    "reason": "Incident #4471 — data review",
    "status": "active",
    "validFrom": "2026-06-09T09:00:00Z",
    "validUntil": "2026-06-09T11:00:00Z",
    "createdAt": "2026-06-09T08:55:00Z"
  },
  "meta": { "correlation_id": "corr_…" },
  "error": null
}
```

**Design note:** the BRD flow ("pending → approver approves → active") is the normative description; the schema's `grant_status` enum contains `active | expired | revoked` (no `pending` variant). The schema also bears a DB-level `ck_break_glass_four_eyes` constraint that enforces `approver_id <> grantee_id`. To honour both: the `POST /admin/break-glass` endpoint requires the requester to supply the `approverId` in the request body at creation time; the row is inserted with `status = 'active'` only after the approve step (`POST /admin/break-glass/{id}/approve`). Because the schema has no `pending` state, the record is created with a sentinel: insert is done inside a transaction that also validates four-eyes and window; the grant is **not yet active** (implementation-internal hold) until approve fires. See Ambiguities section — the missing `pending` enum state is noted. Best-effort resolution: treat the freshly inserted grant as logically pending until `approver_id` is set; `EntitlementService` checks `approver_id IS NOT NULL AND status = 'active'`.

**Simpler safe resolution used here:** the `POST /admin/break-glass` call **requires an `approverId`** from the requester (the requester nominates an approver) and inserts the row. The row's `status` starts as `active` only after the separate `POST /{id}/approve` is called by a *different* user identified as `approverId`. Until that call, the grant is stored but `EntitlementService` ignores it (it checks `approver_id IS NOT NULL`). This satisfies both the four-eyes constraint and the `grant_status` enum.

## Auth Check

### POST /admin/break-glass

1. **JwtAuthGuard** (global) validates the JWT; loads `req.user` (includes `userId`, `roleCode`, `orgId`).
2. **AbacGuard** + `@Requires('break_glass', scopeResolver)`:
   - calls `EntitlementService.can(user, 'break_glass', null)`.
   - Allowed roles: `ADMIN` (scope `A`) and `DPO` (scope `M`) per `auth-matrix.json`.
   - Any other role → `FORBIDDEN` (403).
3. The requester **may not be the same user as the nominated `granteeId`** (four-eyes on the request side: the requester nominates another user as grantee OR nominates themselves, but the approver must be different from the grantee). The DB constraint `ck_break_glass_four_eyes` enforces `approver_id <> grantee_id` at INSERT; service-layer pre-check raises `FORBIDDEN` before the DB insert.
4. Requester (`req.user.userId`) sets `created_by` and `updated_by`.

### POST /admin/break-glass/{id}/approve

1. **JwtAuthGuard** validates JWT.
2. **AbacGuard** + `@Requires('break_glass', scopeResolver)` — only `ADMIN` or `DPO` may approve.
3. Service checks: `approver_id` on the fetched grant must equal `req.user.userId` (only the nominated approver may approve their own nominated grant).
4. Service checks: `req.user.userId !== grant.granteeId` (four-eyes: approver ≠ grantee). If they are the same → `FORBIDDEN` (403).
5. Grant must currently have `approver_id = null` (not yet approved). Already-approved → `CONFLICT` (409).

## Data Operations

All queries use **Kysely** with parameterised bindings. Table and column names are exact per `schema.sql`.

### Insert grant (POST /admin/break-glass)

```typescript
// Inside UnitOfWork transaction
const grant = await db
  .insertInto('break_glass_grants')
  .values({
    grant_id: randomUUID(),
    org_id: user.orgId,
    grantee_id: dto.granteeId,
    approver_id: dto.approverId,   // nominated; not yet confirmed — approver_id IS set but status check deferred
    scope_type: dto.scopeType,
    scope_ref: dto.scopeRef ?? null,
    reason: dto.reason,
    status: 'active',              // see Ambiguities; logically pending until approve step
    valid_from: dto.validFrom,
    valid_until: dto.validUntil,
    created_by: user.userId,
    updated_by: user.userId,
  })
  .returningAll()
  .executeTakeFirstOrThrow();
```

After insert, `AuditAppender` emits `break_glass_access` intent (action = `break_glass_access`, entity_type = `break_glass_grants`, entity_id = grant.grant_id, detail = `{ event: 'grant_requested', scopeType, reason }` — no raw PII).

### Update grant on approval (POST /admin/break-glass/{id}/approve)

```typescript
// Inside UnitOfWork transaction
const updated = await db
  .updateTable('break_glass_grants')
  .set({
    status: 'active',
    updated_by: approver.userId,
    updated_at: new Date(),
  })
  .where('grant_id', '=', grantId)
  .where('org_id', '=', approver.orgId)
  .where('approver_id', '=', approver.userId)   // must match nominated approver
  .where('grantee_id', '!=', approver.userId)   // four-eyes: DB-level guard matches service check
  .where('status', '=', 'active')               // idempotency guard (already resolved above)
  .returningAll()
  .executeTakeFirstOrThrow();
```

The `ck_break_glass_four_eyes` DB constraint (`approver_id <> grantee_id`) is a backstop; the service layer pre-validates and returns `FORBIDDEN` before hitting the DB.

### Read active grants for EntitlementService

```typescript
// Called by EntitlementService on every ADMIN/DPO lead-data access
const grant = await db
  .selectFrom('break_glass_grants')
  .selectAll()
  .where('grantee_id', '=', userId)

  .where('valid_from', '<=', new Date())
  .where('valid_until', '>', new Date())
```

## Validation Logic (Zod)

```typescript
const BreakGlassRequestSchema = z.object({
  granteeId:  z.string().uuid({ message: 'granteeId must be a valid UUID' }),
  approverId: z.string().uuid({ message: 'approverId must be a valid UUID' }),
  scopeType:  z.enum(['lead', 'branch', 'all'], { message: 'scopeType must be lead, branch, or all' }),
  scopeRef:   z.string().uuid().nullish(),
  reason:     z.string().min(1, 'reason is required').max(500, 'reason must be 500 characters or fewer'),
  validFrom:  z.string().datetime({ message: 'validFrom must be a valid ISO-8601 datetime' }),
  validUntil: z.string().datetime({ message: 'validUntil must be a valid ISO-8601 datetime' }),
}).superRefine((val, ctx) => {
  const from = new Date(val.validFrom);
  const until = new Date(val.validUntil);
  if (until <= from) {
    ctx.addIssue({ code: z.ZodIssueCode.custom, path: ['validUntil'], message: 'validUntil must be after validFrom' });
  }
  const maxHours = config.get<number>('BREAK_GLASS_MAX_WINDOW_HOURS');   // see env-contract Ambiguity
  const windowHours = (until.getTime() - from.getTime()) / (1000 * 60 * 60);
  if (windowHours > maxHours) {
    ctx.addIssue({ code: z.ZodIssueCode.custom, path: ['validUntil'],
      message: `Access window must not exceed ${maxHours} hours` });
  }
  if (val.approverId === val.granteeId) {
    ctx.addIssue({ code: z.ZodIssueCode.custom, path: ['approverId'],
      message: 'Approver must be different from grantee (four-eyes required)' });
  }
  if (val.scopeType !== 'all' && !val.scopeRef) {
    ctx.addIssue({ code: z.ZodIssueCode.custom, path: ['scopeRef'],
      message: 'scopeRef is required when scopeType is lead or branch' });
  }
});
```

Validation at the controller boundary via `ZodValidationPipe`; failures emit `VALIDATION_ERROR` (400) with `fields[]`.

## Backend Flow (Step-by-Step)

### POST /admin/break-glass — Create grant request

1. **CorrelationMiddleware** stamps `X-Correlation-Id` on request.
2. **JwtAuthGuard** validates JWT; loads `req.user` (`userId`, `roleCode`, `orgId`).
3. **AbacGuard** + `@Requires('break_glass', …)` calls `EntitlementService.can(user, 'break_glass', null)`. Non-ADMIN/DPO → `FORBIDDEN` (403).
4. **ZodValidationPipe** runs `BreakGlassRequestSchema` on the body. Failure → `VALIDATION_ERROR` (400) with `fields[]`.
5. **Service layer (`BreakGlassService.request`)** pre-validates:
   a. Confirm `granteeId` exists in `users` table (same `org_id`). Not found → `NOT_FOUND` (404).
   b. Confirm `approverId` exists in `users` table (same `org_id`). Not found → `NOT_FOUND` (404).
   c. Confirm `approverId !== granteeId`. If equal → `FORBIDDEN` (403).
   d. Confirm the nominated approver has `break_glass` capability (ADMIN or DPO role). If not → `FORBIDDEN` (403).
   e. If `scopeType` ∈ {`lead`, `branch`} confirm `scopeRef` references an existing row in the appropriate table. Not found → `NOT_FOUND` (404).
6. **`UnitOfWork.run(async (tx) => { … })`** opens a single transaction:
   a. `BreakGlassRepository.insert(dto, tx)` — Kysely `insertInto('break_glass_grants')`.
   b. `AuditAppender.emit({ action: 'break_glass_access', event: 'grant_requested', entity_type: 'break_glass_grants', entity_id: grant.grant_id, detail: { scopeType, reason } }, tx)` — audit intent queued via outbox; no raw PII.
   c. Transaction commits.
7. **ResponseEnvelopeInterceptor** wraps result in `{ data, meta, error: null }`.
8. HTTP 201 returned.

### POST /admin/break-glass/{id}/approve — Four-eyes approval

1. **CorrelationMiddleware** → **JwtAuthGuard** → **AbacGuard** (`break_glass` capability required). Non-ADMIN/DPO → `FORBIDDEN` (403).
2. **Service layer (`BreakGlassService.approve`)**:
   a. Fetch grant by `grant_id` + `org_id` from `break_glass_grants`. Not found → `NOT_FOUND` (404).
   b. Confirm `grant.approver_id === req.user.userId`. Wrong approver → `FORBIDDEN` (403).
   c. Confirm `req.user.userId !== grant.grantee_id` (four-eyes). Self-approve → `FORBIDDEN` (403).
   d. Confirm grant is not already active with a non-null approver (re-approve guard). Duplicate → `CONFLICT` (409).
3. **`UnitOfWork.run(async (tx) => { … })`**:
   a. `BreakGlassRepository.setActive(grantId, approverId, tx)` — Kysely `updateTable` with `WHERE approver_id = :approverId AND grantee_id != :approverId`.
   b. `AuditAppender.emit({ action: 'break_glass_access', event: 'grant_approved', entity_type: 'break_glass_grants', entity_id: grantId }, tx)`.
   c. Transaction commits.
4. HTTP 200 with updated grant returned.

### EntitlementService integration (every ADMIN/DPO lead-content access)

```
EntitlementService.can(user, 'view_lead', resource)
  if user.roleCode ∈ ['ADMIN', 'DPO']:
    grant = BreakGlassRepository.findActiveForUser(user.userId, user.orgId, scopeRef)
    if !grant → FORBIDDEN (403)
    AuditAppender.emit({ action: 'break_glass_access', entity_type: 'leads', entity_id: resource.leadId,
                          detail: { grant_id: grant.grant_id, scope_type: grant.scope_type } })
    return true (access granted)
```

This check runs on every protected lead-content read/write for these roles, ensuring every access is audited with the grant reference (BRD AC-3).

### Scheduled expiry sweep (Cloud Scheduler)

Cloud Scheduler fires `BreakGlassExpiryJob` on a configured cron (e.g., every 5 minutes). The job:
1. Kysely `updateTable('break_glass_grants') SET status='expired' WHERE status='active' AND valid_until <= now() LIMIT 100`.
2. For each expired grant, `AuditAppender.emit({ action: 'break_glass_access', event: 'grant_expired', entity_id: grantId })`.
3. Job is idempotent (re-run safe: already-expired rows are not matched by `WHERE status='active'`).

## State Machine

Entity: **BreakGlassGrant** (`grant_status` enum: `active | expired | revoked`).

```
[created, logically pending] ──(approve by nominated approver)──► active
                                                                      │
                                                             ┌────────┴────────┐
                                                             ▼                 ▼
                                                           expired           revoked
                                                       (valid_until ≤ now) (manual early)
```

| Transition | Trigger | Actor | Guard | Side effects |
|---|---|---|---|---|
| created → active | `POST /approve` | Nominated approver (ADMIN/DPO; ≠ grantee) | `approver_id === req.user.userId`; `approver_id ≠ grantee_id` | `AuditAppender(break_glass_access, grant_approved)` |
| active → expired | Cloud Scheduler sweep | System | `valid_until <= now()` | `AuditAppender(break_glass_access, grant_expired)`; per-grant, LIMIT 100 |
| active → revoked | Revoke action (ADMIN/DPO) | ADMIN or DPO | Grant is active | `AuditAppender(break_glass_access, grant_revoked)` |

Invalid transitions:
- `expired → active` — terminal; not possible.
- `revoked → active` — terminal; not possible.
- Self-approval → `FORBIDDEN` (403).

### Transaction Boundaries (UnitOfWork)

Each state-changing operation runs inside one `UnitOfWork.run(async (tx) => { … })`:
- Create: `INSERT break_glass_grants` + `AuditAppender.emit(grant_requested)` — single commit.
- Approve: `UPDATE break_glass_grants SET status='active'` + `AuditAppender.emit(grant_approved)` — single commit.
- Revoke: `UPDATE break_glass_grants SET status='revoked'` + `AuditAppender.emit(grant_revoked)` — single commit.
- Expiry sweep: bulk `UPDATE` + audit events — single commit per batch.

If any step fails, the whole transaction rolls back; no partial state persists.

## Ambiguities

1. **Missing `pending` state in `grant_status` enum.** The BRD flow ("pending → approve → active") describes a two-step process, but `schema.sql` defines `grant_status AS ENUM ('active','expired','revoked')` — no `pending` variant. Best-effort resolution: the grant is inserted but `EntitlementService` requires `approver_id IS NOT NULL` to consider it active. The `status` column is set to `'active'` only after the approve step; however, the DB enum does not model `pending`. This is a schema/BRD gap. Resolution needed: either add `pending` to the enum (Flyway migration) or confirm the two-step is not needed and a single call creates an active grant (single-authoriser variant). **LLD uses the two-call flow with status set at approve time.**

2. **`BREAK_GLASS_MAX_WINDOW_HOURS` not in `docs/contracts/environment-contract.md`.** The BRD says "bounded window (≤ configurable max)" but the env contract has no such variable. Best-effort: added as an optional env var with default `48` hours, read via `ConfigService`. Must be formally added to `environment-contract.md` before implementation.

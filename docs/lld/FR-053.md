# FR-053 â€” Role-Based Dashboard & Home (Low-Level Design)

**Tier: 2 (Moderate)** â€” multi-table scope-filtered aggregate reads across `leads`, `tasks`, and `integration_logs` with Redis short-TTL caching, role-differentiated widget sets, drill-through to the lead list, and isolated per-widget error handling. No state-machine transition (reads-only, no writes to `leads`), no external service call via `IntegrationGateway`, and no UnitOfWork transaction â€” excludes Tier 3. Richer than a simple CRUD endpoint (role-branching logic, four query aggregations, cache layer, five error code paths, UI composition) â€” excludes Tier 1. ## Summary

FR-053 delivers a **role-scoped home dashboard** to RM, BM, SM, and HEAD. On load the frontend calls `GET /dashboard`, which returns a structured payload of widgets containing KPI counts, SLA-alert lists, hot leads, pending tasks, source summaries, and hand-off failure counts â€” each scoped to the calling user's ABAC scope and cached in Redis for a short TTL (60 s default).

The backend assembles the widget payload from four Kysely aggregate queries executed concurrently via `Promise.allSettled` so that one failing query degrades only its own widget without collapsing the response. Every KPI count is derived using the same filter predicates as the corresponding BRD Â§12 metric (reconciliation rule Â§12.5: dashboard and reports cannot diverge).

This FR is **entirely read-only**. It issues no writes to any table and calls no external service via `IntegrationGateway`. The endpoint is protected by `JwtAuthGuard` (global) + `AbacGuard(@Requires('reports', scopeResolver))`. PARTNER, ADMIN, CUSTOMER, and KYC roles are redirected: KYC gets a subset widget set (documents + kyc queue counts only â€” see Ambiguities section); PARTNER and CUSTOMER are forbidden (403); ADMIN has no `reports` capability for lead content.

Drill-through is handled entirely on the frontend: each widget card links to `GET /leads` with a pre-built filter object derived from the widget context (e.g. hot-lead widget links to `?filter[is_hot]=true&filter[stage]=first_contact_pending`).

## Tier 2 Specification

### Endpoint â€” `GET /dashboard` (operationId `getDashboard`)

Returns all widgets for the authenticated user's role in a single response.

**Request (query string):**

| Param | Type | Default | Rule |
|---|---|---|---|

### Auth Check

| Layer | Detail |
|---|---|
| Authn | Global `JwtAuthGuard` â€” validates Bearer JWT; rejects missing/expired â†’ `AUTH_REQUIRED` (401) |
| Authz | `@Requires('reports', dashboardScopeResolver)` via `AbacGuard` â†’ `EntitlementService.can(user, 'reports', resource)` |
| Scope resolution | `dashboardScopeResolver` reads `user.role` + optional query params (`branch_id`, `team_id`) and resolves the effective scope: RMâ†’O, BMâ†’B, SMâ†’T, HEADâ†’A, KYCâ†’B (subset widgets), DPOâ†’M (aggregate counts only, PII masked). If `branch_id`/`team_id` override supplied, verify the resolved branch/team is within the caller's permitted scope; if not â†’ `FORBIDDEN` (403). |
| PII masking | `MaskingService` applied before serialisation: `name_masked` (partial mask), `mobile_masked` (regex mask). `MaskedField` never exposes raw values for RM, BM, SM scopes. DPO scope: all name/mobile fields masked. |
| Rate limit | Standard read tier: 300 requests / minute per user (Redis `ThrottlerGuard`). Exceeding â†’ `RATE_LIMITED` (429). |

### Data Operations

All queries use the project's **Kysely** typed query builder against exact column names from `docs/data-model/schema.sql`. Queries run concurrently via `Promise.allSettled`.

#### Scope predicate helper

```typescript
// dashboard.repository.ts â€” applies once, reused by all widget queries
function applyScopeFilter(
  qb: SelectQueryBuilder<DB, 'leads', unknown>,
  ctx: DashboardScopeContext,   // { role, userId, teamIds, branchIds }
): SelectQueryBuilder<DB, 'leads', unknown> {
  if (ctx.role === 'RM') return qb.where('leads.owner_id', '=', ctx.userId);
  if (ctx.role === 'SM') return qb.where('leads.team_id', 'in', ctx.teamIds);
  if (ctx.role === 'BM' || ctx.role === 'KYC') return qb.where('leads.branch_id', 'in', ctx.branchIds);
  // HEAD and DPO: no additional predicate (org-wide)
  return qb;
}
```

#### Query 1 â€” KPI counts

```typescript
// Single aggregate query; uses FILTER clause to avoid N queries
db.selectFrom('leads')
  .select([
    db.fn.count<string>('lead_id').filterWhere('deleted_at', 'is', null)
      .filterWhere(sql`stage NOT IN ('handed_off','rejected')`).as('active_pipeline'),
    db.fn.count<string>('lead_id').filterWhere('deleted_at', 'is', null)
      .filterWhere(sql`created_at >= date_trunc('day', now() AT TIME ZONE 'Asia/Kolkata') AT TIME ZONE 'Asia/Kolkata'`).as('captured_today'),
    db.fn.count<string>('lead_id').filterWhere('is_hot', '=', true)
      .filterWhere('deleted_at', 'is', null)
      .filterWhere(sql`stage NOT IN ('handed_off','rejected')`).as('hot_leads'),
    db.fn.count<string>('lead_id').filterWhere('deleted_at', 'is', null)
      .filterWhere(sql`sla_first_contact_due_at < now()`)
      .filterWhere(sql`stage = 'first_contact_pending'`).as('sla_breached'),
    db.fn.count<string>('lead_id').filterWhere('deleted_at', 'is', null)
      .filterWhere(sql`stage NOT IN ('handed_off','rejected')`)
      .filterWhere(sql`handed_off_this_month`).as('handed_off_this_month'),
  ])
  .$call(qb => applyScopeFilter(qb, ctx))
  .executeTakeFirstOrThrow()
```

`handed_off_this_month`: derived via `FILTER WHERE stage = 'handed_off' AND updated_at >= date_trunc('month', now() AT TIME ZONE 'Asia/Kolkata') AT TIME ZONE 'Asia/Kolkata'`.

`consent_coverage_pct`: computed in application code as `Math.round(active_with_required_consent / active_pipeline * 100)` where `active_with_required_consent` is a separate count with `WHERE consent_status IN ('partial','captured') AND deleted_at IS NULL AND stage NOT IN ('handed_off','rejected')`.

#### Query 2 â€” SLA alerts (top 10 most overdue)

```typescript
db.selectFrom('leads')
  .innerJoin('users as owner', 'owner.user_id', 'leads.owner_id')
  .select([
    'leads.lead_id', 'leads.lead_code', 'leads.stage',
    'leads.owner_id', 'owner.full_name as owner_name',
    'leads.sla_first_contact_due_at',
    sql<number>`EXTRACT(EPOCH FROM (now() - leads.sla_first_contact_due_at)) / 60`.as('minutes_overdue'),
  ])
  .where('leads.stage', '=', 'first_contact_pending')
  .where('leads.sla_first_contact_due_at', '<', asOf)
  .where('leads.deleted_at', 'is', null)
  .$call(qb => applyScopeFilter(qb, ctx))
  .orderBy('leads.sla_first_contact_due_at', 'asc')
  .limit(10)
  .execute()
```

#### Query 3 â€” Hot leads (top 10 by score desc)

```typescript
db.selectFrom('leads')
  .innerJoin('lead_identities as li', 'li.lead_identity_id', 'leads.lead_identity_id')
  .innerJoin('users as owner', 'owner.user_id', 'leads.owner_id')
  .select([
    'leads.lead_id', 'leads.lead_code', 'leads.stage', 'leads.score',
    'li.name_masked', 'li.mobile_masked',
    'owner.full_name as owner_name',
  ])
  .where('leads.is_hot', '=', true)
  .where('leads.deleted_at', 'is', null)
  .where(sql`leads.stage NOT IN ('handed_off','rejected')`)
  .$call(qb => applyScopeFilter(qb, ctx))
  .orderBy('leads.score', 'desc')
  .limit(10)
  .execute()
```

`name_masked` and `mobile_masked` are stored-masked columns from `lead_identities` (populated by M2 `MaskingService` on write); no runtime masking needed for these columns.

#### Query 4 â€” Tasks (open/overdue for the calling user; top 20 by due_at asc)

```typescript
db.selectFrom('tasks')
  .innerJoin('leads', 'leads.lead_id', 'tasks.lead_id')
  .select([
    'tasks.task_id', 'tasks.type', 'tasks.due_at', 'tasks.priority', 'tasks.status',
    'leads.lead_code',
  ])
  .where('tasks.owner_id', '=', ctx.userId)              // tasks always own-user for this widget
  .where('tasks.status', 'in', ['open', 'in_progress', 'overdue'])
  .where('tasks.org_id', '=', ctx.orgId)
  .orderBy('tasks.due_at', 'asc')
  .limit(20)
  .execute()
```

#### Query 5 â€” Source summary (top 5 sources by captured desc, last 30 days)

```typescript
db.selectFrom('leads')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'leads.source_attribution_id')
  .select([
    'sa.source_name',
    db.fn.count<string>('leads.lead_id').as('captured'),
    db.fn.count<string>('leads.lead_id')
      .filterWhere('leads.stage', '=', 'handed_off').as('handed_off'),
  ])
  .where('leads.deleted_at', 'is', null)
  .where(sql`leads.created_at >= now() - INTERVAL '30 days'`)
  .$call(qb => applyScopeFilter(qb, ctx))
  .groupBy('sa.source_name')
  .orderBy('captured', 'desc')
  .limit(5)
  .execute()
```

#### Query 6 â€” Hand-off failures (integration_logs)

```typescript
db.selectFrom('integration_logs as il')
  .innerJoin('leads', 'leads.lead_id', 'il.lead_id')
  .select([
    'il.lead_id', 'leads.lead_code',
    db.fn.max<string>('il.created_at').as('last_attempt_at'),
  ])
  .where('il.integration', '=', 'los')
  .where('il.direction', '=', 'outbound')
  .where('il.status', 'in', ['failed', 'retrying'])
  .where('leads.deleted_at', 'is', null)
  .$call(qb => {
    // scope on leads, not integration_logs (integration_logs has no owner_id)
    if (ctx.role === 'RM') return qb.where('leads.owner_id', '=', ctx.userId);
    if (ctx.role === 'SM') return qb.where('leads.team_id', 'in', ctx.teamIds);
    if (ctx.role === 'BM') return qb.where('leads.branch_id', 'in', ctx.branchIds);
    return qb;
  })
  .groupBy(['il.lead_id', 'leads.lead_code'])
  .limit(10)
  .execute()
```

### Caching

- **Cache key:** `dashboard:{orgId}:{userId}:{role}:{resolvedScopeHash}:{asOf_date}` (SHA-256 truncated to 16 hex chars for `resolvedScopeHash`).
- **TTL:** 60 seconds (configurable via env `DASHBOARD_CACHE_TTL_SECONDS`, default 60).
- **Store:** Redis (`ioredis` â€” already in dependency-register for idempotency/throttler).
- **Invalidation:** passive expiry only (TTL). Dashboard is intentionally slightly stale; fresh data is available via drill-through to `GET /leads`.
- **Cache miss / Redis unavailable:** `DashboardService` catches the Redis error, logs at `warn` level, and proceeds with a live DB query. The response includes `"cache_hit": false`. Redis failure does **not** cause a 503 (it is not a critical path for this read-only endpoint).
- `cache_hit: true` is included in the response when data was served from cache.

| Field | Rule | Error message (VALIDATION_ERROR fields[]) |

### Error Cases

| Scenario | Code | HTTP | Notes |
|---|---|---|---|
| Missing or expired JWT | `AUTH_REQUIRED` | 401 | Raised by `JwtAuthGuard` before the controller is reached. |
| Role has no `reports` capability for lead content (PARTNER, CUSTOMER, ADMIN) | `FORBIDDEN` | 403 | `AbacGuard` â†’ `EntitlementService.can` returns false. |
| `branch_id`/`team_id` out of caller's scope | `FORBIDDEN` | 403 | Scope override check in `DashboardService`. |
| `as_of` is in the future; `branch_id` + `team_id` both supplied | `VALIDATION_ERROR` | 400 | Zod schema with `superRefine`. |
| One widget DB query throws unexpectedly | Per-widget degradation; `widget_errors[]` populated; overall response still 200 | 200 | `Promise.allSettled` â€” partial failure does not collapse the response. |
| All widget queries fail (e.g. DB unavailable) | `INTERNAL_ERROR` | 500 | `AllExceptionsFilter` maps the unhandled throw; full error logged with `correlation_id`. |
| Read throttle exceeded | `RATE_LIMITED` | 429 | Redis `ThrottlerGuard`; sets `Retry-After` header. |

Note: `UPSTREAM_UNAVAILABLE` (503) does not apply to FR-053 because no external service is called via `IntegrationGateway`. The `integration_logs` table is a DB read, not a live integration call.

### Backend Flow (Step-by-Step)

1. Request arrives at `GET /api/v1/dashboard`. `CorrelationMiddleware` stamps `X-Correlation-Id`.
2. `JwtAuthGuard` validates the Bearer JWT. Missing/expired â†’ `AUTH_REQUIRED` (401).
3. `AbacGuard` evaluates `@Requires('reports', dashboardScopeResolver)` via `EntitlementService.can`. Role without `reports` capability â†’ `FORBIDDEN` (403).
4. `ValidationPipe` (Zod) parses and validates `GetDashboardQueryDto`. Invalid â†’ `VALIDATION_ERROR` (400) with `fields[]`.
5. `DashboardController.getDashboard(user, query)` calls `DashboardService.getWidgets(user, query)`.
6. `DashboardService` resolves effective scope context (`DashboardScopeContext`) from `user.role`, `user.branchIds`, `user.teamIds`, and optional query overrides. Scope override validated against `EntitlementService`; out-of-scope override â†’ `FORBIDDEN` (403).
7. Cache lookup: `RedisService.get(cacheKey)`. On hit â†’ return cached payload immediately with `cache_hit: true`.
8. Cache miss: build 6 Kysely queries (KPI, SLA alerts, hot leads, tasks, source summary, hand-off failures), execute with `Promise.allSettled`.
9. For each settled result: if `fulfilled`, include widget data; if `rejected`, set widget to `null` and append to `widget_errors[]` with `INTERNAL_ERROR` code. Log the rejection at `error` level with `correlation_id` and widget name.
10. Assemble `DashboardPayload` DTO. Apply `MaskingService` to PII fields (`name_masked`, `mobile_masked`) based on caller's role (DPO â†’ full mask).
11. Store assembled payload in Redis with TTL. Redis write failure is caught and swallowed (log `warn`).
12. `ResponseEnvelopeInterceptor` wraps the payload in `{ data, meta, error: null }`.
13. Response returned. Total p95 target: â‰¤ 500 ms (NFR read threshold). Redis cache hit path: â‰¤ 100 ms.

FR-053 performs **no state transitions** and **no writes**. No UnitOfWork transaction is needed. All operations are read-only Kysely `selectFrom` queries using a standard (non-transactional) DB connection from the pool.

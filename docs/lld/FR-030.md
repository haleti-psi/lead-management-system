## Summary

FR-030 delivers transparent, rules-based lead allocation for NBFC branch sales. When a lead is created (or triggered for re-allocation), `AllocationService.allocate(lead)` evaluates active `allocation_rules` ordered by `priority_order`, applies capacity checks, and sets `owner_id` + `team_id` on the lead. Manual reassignment (`POST /leads/{id}/reassign`) requires an auditable reason and is restricted to BM/SM/HEAD within their respective scope (B/T/A). Both paths write `leads + stage_history + audit_logs + event_outbox` in one `UnitOfWork` transaction. No-match falls back to the branch unassigned pool and fires an alert.

## Auth Check

### POST /leads/{id}/reassign
- **Authentication:** Global `JwtAuthGuard` (JWT must be valid; 401 otherwise).
- **Authorisation:** `@Requires('allocate', scopeResolver)` → `AbacGuard` calls `EntitlementService.can(user, 'allocate', lead)`.
  - BM: scope B — lead's `branch_id` must equal caller's `branch_id`.
  - SM: scope T — lead's `team_id` must be in caller's teams.
  - HEAD: scope A — all branches.
  - RM has no `allocate` capability → 403 `FORBIDDEN`.
- **Ownership check:** `new_owner_id` must be a user whose branch/team is within the caller's scope.

### GET + POST /admin/allocation-rules
- **Authentication:** Global `JwtAuthGuard`.
- **Authorisation:** `@Requires('allocate', scopeResolver)` → BM/SM/HEAD only. ADMIN has `configuration` capability but not `allocate`; admin config changes (rule CRUD) go through M14's maker-checker (FR-131/FR-132). In MVP, BM/SM/HEAD may create rules within their scope.
- No `@Public()`.

## Validation Logic

### POST /leads/{id}/reassign — ReassignLeadDto (Zod)

| Field | Rule | Error message |
|-------|------|---------------|
| `new_owner_id` | Required, UUID v4 | "new_owner_id must be a valid UUID" |
| `reason` | Required, string, min 5 chars, max 500 chars | "Reason is required (min 5, max 500 characters)" |
| `override_capacity` | Optional boolean, default false | "override_capacity must be a boolean" |

**Service-layer checks:**
- Lead must exist and not be soft-deleted (`deleted_at IS NULL`).
- Lead must not be in `handed_off` stage (terminal) — returns `CONFLICT`.
- `new_owner_id` must resolve to an active user (`status = 'active'`) in the caller's scope.
- Capacity check: if `override_capacity` is false and the target RM's active-lead count equals `allocation_rules.capacity_limit` for the matching rule, return `CONFLICT` (409) with message "Target RM is at capacity; set override_capacity=true if authorised."
- `override_capacity: true` is only honoured by BM (scope B) or HEAD (scope A); SM gets `FORBIDDEN` if they attempt it.

### POST /admin/allocation-rules — CreateAllocationRuleDto (Zod)

| Field | Rule | Error message |
|-------|------|---------------|
| `name` | Required, string, min 1, max 120 | "name is required, max 120 characters" |
| `priority_order` | Required, integer ≥ 1 | "priority_order must be a positive integer" |
| `method` | Required, one of `allocation_method` enum | "method must be one of: round_robin, capacity, specialist, branch, partner, escalation" |
| `criteria` | Required, non-empty object | "criteria must be a non-empty object" |
| `target` | Required, non-empty object (must contain `team_ids` array or `partner_id`) | "target must specify team_ids array or partner_id" |
| `capacity_limit` | Optional, integer ≥ 1, max 500 | "capacity_limit must be a positive integer (max 500)" |
| `is_active` | Optional boolean, default true | "is_active must be a boolean" |

## Backend Flow (step-by-step)

### Path A — Automatic allocation (internal, called post-lead-create or SLA escalation)

1. **Caller** — `CaptureService` (post lead-create) or `SlaEngine` (escalation trigger) calls `AllocationService.allocate(leadId, { actorId: SYSTEM_USER_ID, expectedVersion })` within the creating `UnitOfWork` transaction.

2. **Load rules** — `AllocationRuleRepository.findActive(orgId)` fetches all `is_active=true` rules ordered by `priority_order ASC` (LIMIT 100).

3. **Match criteria** — For each rule, evaluate `rule.criteria` JSONB against the lead's attributes:
   - `branch_id` match, `product_code` match, `source` (from `source_attributions.source`), `partner_id` (from `source_attributions.partner_id`), `priority` (`leads.priority`), `language` (from `customer_profiles.preferred_language`), `is_hot` (`leads.is_hot`).
   - A rule matches if all criteria keys are satisfied. First match wins (priority_order ascending).

4. **Resolve RM pool** — From `rule.target` JSONB extract `team_ids` (or `partner_id` for `partner` method). Load active RMs in those teams from `users` where `status='active'` and `team_id IN (target.team_ids)`.

5. **Capacity check** — Count each RM's active leads (stages not in `handed_off`, `rejected`, `dormant`). If `method='capacity'`, filter out RMs whose count ≥ `rule.capacity_limit`. If all are at capacity, continue to next rule.

6. **Apply method logic:**
   - `round_robin` — pick the RM with the lowest `active_count` among tied candidates (if all equal, pick by `users.created_at ASC` as deterministic tie-break per BRD §5).
   - `capacity` — pick the RM with the lowest current active-lead count (that is below `capacity_limit`).
   - `specialist` — filter pool by `users.product_skills @> [rule.criteria.product_code]` then apply round-robin among specialists.
   - `branch` — all RMs in `rule.target.team_ids` under the lead's `branch_id`.
   - `partner` — all RMs assigned to `rule.target.partner_id` (users whose `partner_id` matches).
   - `escalation` — pick the BM (reporting_manager of the team) as the escalated RM.

7. **No match** — If all rules fail or all pools are at capacity:
   - `LeadService.assignOwner` is called with `owner_id = null`, `team_id = lead.branch_id`'s default team, `reason = 'no_rule_match'`.
   - `OutboxService.emit(LEAD_ASSIGNED, { owner_id: null, reason: 'unassigned_pool' })`.
   - External alert fires (pino `warn` + Cloud Monitoring custom metric `allocation.no_match`).

8. **Write in UnitOfWork transaction** — `LeadService.assignOwner(leadId, { ownerId, teamId, allocationReason, method, expectedVersion }, tx)`:
   - UPDATE `leads` SET `owner_id`, `team_id`, `stage='assigned'`, `sla_first_contact_due_at`, `version+1` WHERE `version=expectedVersion`.
   - If 0 rows updated → throw `CONFLICT`.
   - INSERT `stage_history` (captured → assigned).
   - `AuditAppender.append(action='allocate', ...)`.
   - `OutboxService.emit('LEAD_ASSIGNED', ...)`.

9. **Post-commit** — `NotificationDispatchService` sends `LEAD_ASSIGNED` in-app + email to `owner_id` (consent-gated; non-blocking; retryable; does not roll back the transaction on failure).

---

### Path B — Manual reassign (POST /leads/{id}/reassign)

1. **Request arrives** — `AllocationController.reassignLead(id, dto, req)`.

2. **Auth** — `JwtAuthGuard` validates JWT → loads user. `AbacGuard` checks `EntitlementService.can(user, 'allocate', lead)` with scope resolution. 403 on fail.

3. **Load lead** — `AllocationRepository.findLeadForReassign(id)` — SELECT `lead_id, owner_id, team_id, stage, branch_id, version, deleted_at` WHERE `lead_id=id AND deleted_at IS NULL`. 404 if not found.

4. **Stage guard** — If `lead.stage === 'handed_off'` → throw `CONFLICT` (terminal stage, cannot reassign).

5. **Validate new owner** — Load `users` WHERE `user_id=dto.new_owner_id AND status='active'`. Verify in caller's scope. 403 if out of scope.

6. **Capacity check** — Count `new_owner_id`'s active leads. If at capacity and `dto.override_capacity=false` → `CONFLICT`. If `dto.override_capacity=true` and caller is SM → `FORBIDDEN`.

7. **UnitOfWork transaction:**
   - `LeadService.assignOwner(id, { ownerId: dto.new_owner_id, teamId: resolvedTeamId, reason: dto.reason, method: 'manual', expectedVersion: lead.version }, tx)`.
   - Writes: `leads` UPDATE, `stage_history` INSERT (from current stage → `assigned` if not already assigned, else stays `assigned` but owner changes), `audit_logs` INSERT (`action='reassign'`), `event_outbox` INSERT (`LEAD_ASSIGNED`).

8. **Post-commit** — Notification to new `owner_id`.

9. **Response** — Return updated lead data in the `{ data, meta, error }` envelope.

## State Machine + Transaction Boundaries

Per `docs/contracts/state-machines.md`:

```
captured ──(allocation trigger)──→ assigned
assigned ──(manual reassign)──→ assigned   [owner changes; stage stays assigned]
```

**Valid triggering stages for automatic allocation:** `captured` (primary), `dormant` (reactivation), `rejected` (reopen, previous stage was assigned).

**Invalid transitions that return `CONFLICT`:**
- Attempting to reassign a lead in `handed_off` stage.

**Transaction boundary (UnitOfWork):** Every allocation or reassignment writes the following in ONE transaction:
1. `leads` UPDATE (owner_id, team_id, stage, sla_first_contact_due_at, version)
2. `stage_history` INSERT
3. `audit_logs` INSERT (via AuditAppender)
4. `event_outbox` INSERT (via OutboxService)

If any step fails, the entire transaction rolls back — no partial allocation state can persist.

**Optimistic lock:** `leads.version` must equal `expectedVersion` at update time; 0 rows updated → `CONFLICT` (409).

**SLA:** `sla_first_contact_due_at` is computed by `SlaEngine.computeDue('first_contact', branchId)` using `BusinessCalendarService`; set only on the `captured → assigned` transition; not reset on reassignment.

## External Service Calls

FR-030 does **not** call any external port via `IntegrationGateway`. All operations are internal database reads and writes. The `NotificationDispatchService` (post-commit, non-blocking) uses `NotificationChannelPort` (SMS/email) but notification failure does NOT roll back the allocation.

## Shared Utilities to Use

| Utility | Usage |
|---------|-------|
| `LeadService.assignOwner(...)` | Sole writer of `leads.owner_id / team_id / stage`; takes `expectedVersion` |
| `UnitOfWork.run(fn(tx))` | Wraps all writes (leads + stage_history + audit_logs + event_outbox) in one transaction |
| `AuditAppender.append(...)` | Emits `allocate` / `reassign` audit action into `audit_logs` within the tx |
| `OutboxService.emit('LEAD_ASSIGNED', ..., tx)` | Transactional outbox write within the same tx |
| `SlaEngine.computeDue('first_contact', branchId)` | Computes `sla_first_contact_due_at` using `BusinessCalendarService` |
| `EntitlementService.can(user, 'allocate', resource)` | ABAC decision; called by `AbacGuard` |
| `NotificationDispatchService.send(...)` | Post-commit LEAD_ASSIGNED notification (non-blocking) |
| `@shared/enums` — `AllocationMethod`, `LeadStage`, `Priority`, `AuditAction` | Never redefine locally |
| `UUIDParam` (Zod) | Path `id` validation |
| `PaginationParams` (Zod) | List endpoints |
| `ApiEnvelope<T>` | All response shapes |
| `DataTable`, `EntityForm`, `Modal`/`Drawer`, `ConfirmDialog`, `Toast`, `PageHeader`, `EmptyState`, `LoadingSkeleton`, `ErrorState` | UI primitives — do not recreate |

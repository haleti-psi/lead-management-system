## Summary

FR-100 implements the Task Management feature for Module M11 (Engagement). It allows RM, BM, SM, and KYC users to create, list, and update tasks associated with leads. Tasks track follow-up activity (calls, visits, document requests, KYC appointments, callbacks, etc.) with due dates, priorities, SLA policies, dispositions, and result notes. An overdue sweep job transitions `open` and `in_progress` tasks whose `due_at` has passed to `overdue` and triggers escalation notifications.

The `nurture` task type has a side effect: completing it sets `leads.nurture_next_at` to the value supplied in `next_action_at`. All task writes are audited. No external service calls (SLA is served by the internal `SlaEngine`/`BusinessCalendarService`). No `leads` mutation is performed by this FR except the single `nurture_next_at` field update, which goes through `LeadService.setNurtureNextAt()` (see Ambiguities).

## Auth Check

### Authentication
Global `JwtAuthGuard` applies to all three endpoints. No `@Public()` override.

### Authorisation

```
@Requires('edit_lead', scopeResolver)
AbacGuard -> EntitlementService.can(user, 'edit_lead', { lead_id, owner_id })
```

**Scope resolution:**

| Role | Max scope | Effective filter |
|---|---|---|
| RM | O | Tasks where `tasks.owner_id = user.user_id` OR `tasks.lead_id` has `leads.owner_id = user.user_id` |
| BM | B | Tasks where `tasks.lead_id` has `leads.branch_id = user.branch_id` |
| SM | T | Tasks where `tasks.lead_id` has `leads.team_id IN (user's teams)` |
| KYC | B | Same as BM |

**Ownership for PATCH:** the caller must be the task owner OR have BM/SM scope over the lead. An RM attempting to update another RM's task on a shared lead returns `FORBIDDEN`.

**Owner reassignment (`owner_id` field on PATCH):** only BM/SM may change `owner_id`; RM attempting this gets `FORBIDDEN`.

## Data Operations

### List tasks (GET)

```typescript
// TaskRepository.list(filters, caller)
db.selectFrom('tasks as t')
  .innerJoin('leads as l', 'l.lead_id', 't.lead_id')
  .selectAll('t')
  .where('t.org_id', '=', orgId)
  // scope filter applied per role (one of):
  .where('t.owner_id', '=', caller.userId)                   // RM
  // or .where('l.branch_id', '=', caller.branchId)          // BM/KYC
  // or .where('l.team_id', 'in', caller.teamIds)             // SM
  .$if(filters.lead_id != null, q => q.where('t.lead_id', '=', filters.lead_id!))
  .$if(filters.status != null, q => q.where('t.status', '=', filters.status!))
  .$if(filters.owner_id != null, q => q.where('t.owner_id', '=', filters.owner_id!))
  .$if(filters.type != null, q => q.where('t.type', '=', filters.type!))
  .$if(filters.due_before != null, q => q.where('t.due_at', '<=', filters.due_before!))
  .orderBy('t.due_at', 'asc')
  .limit(Math.min(filters.limit ?? 25, 100))
  .offset(((filters.page ?? 1) - 1) * (filters.limit ?? 25))
  .execute()
```

Count query uses the same `WHERE` conditions.

### Create task (POST)

```typescript
// TaskRepository.insert(data, tx)
db.insertInto('tasks')
  .values({
    task_id: newUuid(),
    org_id: caller.orgId,
    lead_id: input.lead_id,
    type: input.type,
    owner_id: input.owner_id,
    due_at: input.due_at,
    priority: input.priority ?? 'normal',
    sla_policy_id: input.sla_policy_id ?? null,
    status: 'open',
    disposition: null,
    result_note: null,
    geo: null,
    next_action_at: null,
    created_at: now(),
    updated_at: now(),
    created_by: caller.userId,
    updated_by: caller.userId,
  })
  .returningAll()
  .executeTakeFirstOrThrow()
```

Followed by `AuditAppender.append({ action: 'task_created', lead_id, actor_id, ... }, tx)`.

### Update task (PATCH)

```typescript
// TaskRepository.update(taskId, patch, tx)
db.updateTable('tasks')
  .set({ ...patch, updated_at: now(), updated_by: caller.userId })
  .where('task_id', '=', taskId)
  .where('org_id', '=', orgId)
  .returningAll()
  .executeTakeFirstOrThrow()
```

When `status` transitions to `done` and `type === 'nurture'` and `next_action_at` is provided, `LeadService.setNurtureNextAt(lead_id, next_action_at, tx)` is called within the same transaction (see Ambiguities — this mutator is not in the current `LeadService` interface; the implementation must add it or use the closest equivalent).

`AuditAppender.append({ action: 'task_updated', lead_id, actor_id, detail: { old_status, new_status, disposition } }, tx)` is written in the same transaction.

### Overdue sweep job

```typescript
// TaskOverdueSweepJob — scheduled via Cloud Tasks (cron-like, every ~5 min)
db.updateTable('tasks')
  .set({ status: 'overdue', updated_at: now(), updated_by: SYSTEM_USER_ID })
  .where('status', 'in', ['open', 'in_progress'])
  .where('due_at', '<', now())
  .where('org_id', '=', orgId)
  .execute()
```

After the batch update, the job reads the updated task IDs and emits `TASK_OVERDUE` events via `OutboxService` for each, which the notification dispatcher processes to trigger escalation per the linked `sla_policy_id.escalation_chain`.

| `FORBIDDEN` | 403 | Lead or task is outside the caller's ABAC scope; RM attempting to reassign `owner_id` | — |

## Backend Flow

### POST /api/v1/tasks (Create)

1. `JwtAuthGuard` validates JWT; injects `AuthUser` into request context.
2. `AbacGuard` calls `EntitlementService.can(user, 'edit_lead', { lead_id })`. Fails → `FORBIDDEN` (403).
3. `Zod` validates `CreateTaskDto`. Fails → `VALIDATION_ERROR` (400) with `fields[]`.
4. `TaskService.create(dto, caller)`:
   a. Verify `lead_id` exists and is within caller scope (Kysely select with scope filter).
   b. Verify `owner_id` references an active user in the org.
   c. If `sla_policy_id` provided, verify it references an `is_active = true` sla_policy.
   d. Open `UnitOfWork.run(async (tx) => { ... })`:
      - `TaskRepository.insert(data, tx)` → inserts the `tasks` row.
      - `AuditAppender.append({ action: 'task_created', lead_id, actor_id: caller.userId, detail: { type, owner_id, due_at } }, tx)`.
   e. Return the created task.
5. `ResponseEnvelopeInterceptor` wraps result → 201.

### PATCH /api/v1/tasks/{id} (Update/Complete)

1. `JwtAuthGuard` validates JWT.
2. `TaskService.update(taskId, dto, caller)`:
   a. Fetch task by `task_id` (scoped to `org_id`). Not found → `NOT_FOUND` (404).
   b. `AbacGuard` / ownership check: caller must be task owner OR BM/SM in scope over lead. Fails → `FORBIDDEN` (403).
   c. `Zod` validates `UpdateTaskDto`. Fails → `VALIDATION_ERROR` (400).
   d. Validate status transition via `TaskStateMachine.canTransition(currentStatus, newStatus)`. Invalid → `CONFLICT` (409).
   e. If `status === 'done'` and `disposition` is absent → `VALIDATION_ERROR` (400).
   f. If caller is RM and `owner_id` in patch → `FORBIDDEN` (403).
   g. Open `UnitOfWork.run(async (tx) => { ... })`:
      - `TaskRepository.update(taskId, patch, tx)`.
      - If `status === 'done'` AND `type === 'nurture'` AND `next_action_at` provided:
        `LeadService.setNurtureNextAt(lead_id, next_action_at, tx)` (updates `leads.nurture_next_at`).
      - `AuditAppender.append({ action: 'task_updated', lead_id, actor_id, detail: { from_status, to_status, disposition } }, tx)`.
   h. Return updated task.
3. `ResponseEnvelopeInterceptor` wraps result → 200.

### GET /api/v1/tasks (List)

1. `JwtAuthGuard` validates JWT.
2. `AbacGuard` checks `edit_lead` capability at the role level (no specific resource yet).
3. `Zod` validates query params (PaginationParams + filter params).
4. `TaskService.list(filters, caller)`:
   a. Build scoped Kysely query (scope filter applied per role).
   b. Run paginated select with `LIMIT` enforced (max 100).
   c. Run count query.
5. Return `{ data: tasks[], meta: { page, limit, total } }` → 200.

### Overdue Sweep Job (TaskOverdueSweepJob)

Runs every 5 minutes via Cloud Tasks:

1. Batch update: set `status = 'overdue'` where `status IN ('open','in_progress') AND due_at < NOW()`.
2. Fetch IDs of updated tasks.
3. For each overdue task, emit `TASK_OVERDUE` event via `OutboxService.emit('TASK_OVERDUE', { task_id, lead_id, owner_id, sla_policy_id }, tx)` (inside a transaction per batch to ensure atomicity).
4. The outbox relay delivers the event to Pub/Sub; the notification dispatcher reads `sla_policy_id.escalation_chain` and dispatches escalation notifications.

## State Machine: Task Status

Reference: `docs/contracts/state-machines.md` — Task section.

```
open ──────────────────→ in_progress
open ──────────────────→ done          (direct complete, disposition required)
open ──────────────────→ cancelled
open ──────────────────→ overdue       (sweep only; not user-settable)
in_progress ───────────→ done          (disposition required)
in_progress ───────────→ cancelled
overdue ───────────────→ in_progress   (user picks it up)
overdue ───────────────→ done          (disposition required)
overdue ───────────────→ cancelled
done ──────────────────→ (terminal; no transitions out)
cancelled ─────────────→ (terminal; no transitions out)
```

**Invalid transitions (→ 409 CONFLICT):** any write to `done` or `cancelled` from those terminal states; any user-initiated write to `overdue` (sweep-only).

### Transaction Boundaries

Task create and task update each run inside a `UnitOfWork` transaction:

```
UnitOfWork.run(async (tx) => {
  TaskRepository.insert/update(... , tx)      // tasks row
  LeadService.setNurtureNextAt(... , tx)       // leads.nurture_next_at — only on nurture done
  AuditAppender.append(... , tx)              // audit_logs row
  // OutboxService.emit for TASK_OVERDUE is in the sweep job's own tx
})
```

Post-commit (not in the tx): notification dispatch (retryable, does not roll back the task state).

## Ambiguities

1. **`LeadService.setNurtureNextAt` does not exist in the shared-utilities `LeadService` mutator interface.** The state machines doc says `nurture task sets leads.nurture_next_at`; the `leads` table has the column. Best-effort resolution: the implementation agent should add `setNurtureNextAt(leadId: string, nextAt: Date, tx: Transaction): Promise<void>` to `LeadService` in `apps/api/src/modules/capture/lead.service.ts`, or use the nearest available mutator if added by another FR. This must be confirmed and written back to the LeadService interface in `shared-utilities.md` before coding.

2. **Idempotency-Key behaviour on POST /tasks.** The api-contract.yaml does not include `Idempotency-Key` as a parameter on `POST /tasks` (unlike `POST /leads`). Best-effort: accept the header, but do not enforce Redis-backed idempotency for tasks in MVP. Document this in the controller JSDoc.

3. **Owner scope validation on create.** The BRD says "Owner in scope"; the auth-matrix does not define a separate `manage_task` capability — tasks are governed by `edit_lead`. Best-effort: validate that the proposed `owner_id` user is accessible within the caller's ABAC scope (same branch for BM, same team for SM, same user for RM). Confirm with product if RM can assign to another RM in their team.

4. **`TASK_OVERDUE` event shape.** The event outbox event name is not defined in the contracts package. Best-effort: use `TASK_OVERDUE` as event type with `{ task_id, lead_id, owner_id, sla_policy_id, due_at }` payload. This should be registered in `integration-map.md`/`state-machines.md`.

## Summary

FR-070 provides the document checklist and upload workflow for a lead. The checklist is **derived at runtime** from `product_configs.document_checklist` JSONB filtered by `applicant_scope`. Each checklist item corresponds to a `documents` row; items that do not yet have a row exist only in the derived checklist response (not_required / pending). An authorised actor uploads a file, which is streamed to GCS via a server-generated signed URL and then a scan is triggered. A document item may also be **waived** with a reason by KYC or BM roles. Customer uploads arrive through the customer self-service path (`POST /c/{token}/documents`) guarded by `CustomerLinkGuard`.

The `leads.kyc_status` derived field is recomputed (via `LeadService.setKycStatus`) after every document status change; no direct `UPDATE leads` is issued by M8 code. Stage transition from `documents_pending` to `kyc_in_progress` is owned by the stage-transition FR (FR-052) and is a **side effect** of checklist completion — it is not triggered directly from this FR, but this FR fires the `DOC_UPLOADED` outbox event that may trigger the transition.

### 2. POST /api/v1/leads/{id}/documents

**Purpose:** Upload a document file for the given lead. The server generates a GCS signed PUT URL, creates a `documents` row in `pending` / `uploaded` status, and schedules a virus scan via IntegrationGateway.

**Request** (multipart/form-data OR JSON pre-flight to get signed URL — see flow below)

The upload follows a **two-phase protocol**:
- **Phase A (initiate):** Client POSTs metadata JSON. Server validates, creates the `documents` row, and returns a GCS signed PUT URL.
- **Phase B (confirm):** After the client uploads the binary to GCS, it calls the same endpoint with `action: "confirm"` and `document_id`. Server sets `status = "uploaded"`, enqueues scan.

For simplicity in this FR the two phases share the same `POST /leads/{id}/documents` path differentiated by body shape.

**Phase A — initiate (JSON body)**
```json
{
  "doc_type": "pan",
  "applicant_scope": "applicant",
  "file_name": "pan_card.pdf",
  "file_type": "application/pdf",
  "file_size_kb": 320
}
```

**Phase A — Response 201**
```json
{
  "data": {
    "document_id": "uuid",
    "upload_url": "https://storage.googleapis.com/lms-prod-docs/leads/...?X-Goog-Signature=...",
    "upload_url_expires_at": "2026-06-01T09:25:00Z",
    "status": "pending"
  },
  "meta": { "correlation_id": "corr_..." },
  "error": null
}
```

**Phase B — confirm (JSON body)**
```json
{
  "action": "confirm",
  "document_id": "uuid"
}
```

**Phase B — Response 200**
```json
{
  "data": {
    "document_id": "uuid",
    "status": "uploaded",
    "virus_scan_status": "pending"
  },
  "meta": { "correlation_id": "corr_..." },
  "error": null
}
```

**Error responses:** 400 (VALIDATION_ERROR), 403 (FORBIDDEN), 404 (NOT_FOUND), 409 (CONFLICT — infected scan), 413 (PAYLOAD_TOO_LARGE), 415 (UNSUPPORTED_MEDIA), 429 (RATE_LIMITED), 503 (UPSTREAM_UNAVAILABLE — GCS failure).

### 3. POST /api/v1/leads/{id}/documents/{did}/waive

**Note:** The `api-contract.yaml` only lists `GET/POST /leads/{id}/documents` at the M8 section boundary. No waiver endpoint appears in the contract's explicit endpoint map. However:
- BRD §FR-070 AC-3 explicitly requires waiver with authorised role, reason, expiry/review date, and audit.
- The state machine defines `any -> waived` as a Document transition.
- The auth-matrix `verify_doc` capability for KYC/BM covers waiver operations.

**Best-effort resolution (AMBIGUITY noted below):** This LLD defines the waiver as a PATCH on the document sub-resource, which is consistent with the api-contract's PATCH patterns elsewhere (e.g., `PATCH /leads/{id}/kyc/{kid}/resolve`). The path used is:

```
PATCH /api/v1/leads/{id}/documents/{did}/waive
```

**Request**

```json
  "expires_at": "2026-09-01",
  "review_note": "Resubmit within 90 days"
}
```

**Response 200**
```json
{
  "data": {
    "document_id": "uuid",
    "status": "waived",
    "waiver_reason": "...",
    "updated_at": "2026-06-01T10:00:00Z"
```

**Error responses:** 400 (VALIDATION_ERROR — missing reason or invalid expires_at), 401, 403 (FORBIDDEN — insufficient role), 404, 409 (CONFLICT — already waived or in a terminal state that cannot be waived).

### 4. POST /api/v1/c/{token}/documents  (Customer self-service upload)

**Purpose:** Customer uploads a document via a tokenised link (no JWT). Uses `CustomerLinkGuard` (opaque token + OTP step-up).

**Request** — same Phase A/B protocol as endpoint 2. Body adds no extra fields; `uploaded_via` is automatically set to `'customer_link'`.

**Response:** same shapes as endpoint 2.

**Guards:** `@Public()` (in auth-matrix `public_endpoints`); `CustomerLinkGuard` validates the token, resolves the `lead_id`, and scopes the write to `upload_doc:C`.

## Auth Check

### Staff endpoints (GET, POST, PATCH on /leads/{id}/documents[/{did}/waive])

| Guard | Detail |
|---|---|
| `JwtAuthGuard` | Global — validates Bearer JWT; rejects with 401 if absent/expired |
| `AbacGuard` + `@Requires('upload_doc', scopeResolver)` | For upload (POST) and list (GET) |
| `AbacGuard` + `@Requires('verify_doc', scopeResolver)` | For waiver (PATCH /waive) — only KYC and BM hold `verify_doc` |
| `EntitlementService.can(user, 'upload_doc', { lead_id, org_id, owner_id, branch_id, partner_id })` | Returns boolean; 403 on false |

**Scope enforcement in `DocumentRepository`:**

Every query against `documents` is joined through `leads` to enforce the caller's data scope:

| Role | Scope | Filter applied |
|---|---|---|
| RM | O | `leads.owner_id = $userId AND leads.org_id = $orgId` |
| BM | B | `leads.branch_id = $branchId AND leads.org_id = $orgId` |
| KYC | B | `leads.branch_id = $branchId AND leads.org_id = $orgId` |
| SM | T | `leads.team_id = $teamId AND leads.org_id = $orgId` |
| PARTNER | P | `leads.source_attribution_id IN (SELECT ... WHERE partner_id = $partnerId)` |

**Waiver role check (additional explicit check in `DocumentService`):**

`verify_doc` capability is not held by RM or SM (per capability_matrix). The service also checks the caller's role code is in `{KYC, BM}` for waiver operations; otherwise returns `FORBIDDEN` (403).

### Customer endpoint (POST /c/{token}/documents)

| Guard | Detail |
|---|---|
| `@Public()` | Opt-out of JwtAuthGuard |
| `CustomerLinkGuard` | Validates opaque token; resolves `lead_id`; checks `link_status = 'active'`; enforces OTP step-up per `GCS_SIGNED_URL_TTL` window |
| Scope | Implicitly scoped to the single `lead_id` bound to the token; `upload_doc:C` |

### 3. Insert document row (Phase A)

```typescript
// DocumentRepository.insert(input, tx)
const doc = await $tx
  .insertInto('documents')
  .values({
    lead_id:           input.leadId,
    org_id:            input.orgId,
    doc_type:          input.docType,           // validated enum from @shared/enums
    applicant_scope:   input.applicantScope,
    status:            'pending',
    file_type:         input.fileType,          // MIME, after content inspection
    file_size_kb:      input.fileSizeKb,
    version:           existingVersionCount + 1,
    uploaded_via:      input.uploadedVia,
    classification:    'pii',
    virus_scan_status: 'pending',
    storage_ref:       null,                    // set on Phase B confirm
    created_by:        input.actorId,
    updated_by:        input.actorId,

  .returningAll()
  .executeTakeFirstOrThrow();
```

**Re-upload version increment:** before insert, count existing non-deleted rows for `(lead_id, doc_type, applicant_scope)` to derive `version`. The state machine says `verified -> pending` is invalid — re-uploads create a NEW version row, not an update to the verified row.

### 8. Recompute leads.kyc_status (via LeadService — owner-writes)

After any document status change, `DocumentService` calls `LeadService.setKycStatus(leadId, derivedStatus, tx)`:

```typescript
// Derivation logic (DocumentService private method)
function deriveKycStatus(checklist: ChecklistItem[], docs: Document[]): KycStatus {
  const mandatoryItems = checklist.filter(i => i.mandatory);
  const allMandatoryDone = mandatoryItems.every(item => {
    const latest = docs.find(d => d.doc_type === item.doc_type &&
                                  d.applicant_scope === item.applicant_scope);
    return latest && ['verified', 'waived', 'not_required'].includes(latest.status);
  });
  const anyInProgress = docs.some(d => ['uploaded', 'under_review'].includes(d.status));
  if (allMandatoryDone) return 'verified';
  if (anyInProgress)    return 'in_progress';
  return 'not_started';
}
```

`LeadService.setKycStatus` issues `UPDATE leads SET kyc_status = $v WHERE lead_id = $id AND org_id = $orgId` (no optimistic lock bump — this is a volatile derived field per §11.2).

## Validation Logic

All validated via Zod at the controller boundary before the service layer.

### Phase A upload initiation (UploadInitiateDto)

| Field | Rule | Error message |
|---|---|---|
| `doc_type` | `z.enum([...DocTypeEnum])` — values from `@shared/enums DocType` | "Invalid document type." |
| `applicant_scope` | `z.enum([...ApplicantScopeEnum])` — `applicant/co_applicant/guarantor/business` | "Invalid applicant scope." |
| `file_name` | Non-empty string, 1–255 chars | "File name is required." |
| `file_type` | MIME must be one of `application/pdf`, `image/jpeg`, `image/png`, `image/heic` | "Unsupported file type." → `UNSUPPORTED_MEDIA` (415) |
| `file_size_kb` | integer, 1 ≤ x ≤ `MAX_UPLOAD_MB * 1024` (default 10240) | "File is too large." → `PAYLOAD_TOO_LARGE` (413) |
| `doc_type` membership in checklist | `doc_type` + `applicant_scope` must exist in `product_configs.document_checklist` for this lead | "This document type is not in the product checklist." → `VALIDATION_ERROR` 400 |

**MIME by content inspection (security rule):** The server does not trust the client-supplied `file_type`. After the client uploads to GCS, a content-inspect step reads the GCS object magic bytes (via the GCS port) before confirming status. If content MIME does not match declared MIME → `VALIDATION_ERROR` 400 + delete GCS object.

### Phase B confirm (UploadConfirmDto)

| Field | Rule | Error message |
|---|---|---|
| `action` | must equal `"confirm"` | "Invalid action." |
| `document_id` | valid UUID; must exist in `documents` with `status='pending'` and belong to the same `lead_id` and caller scope | "Document not found." → `NOT_FOUND` 404 |

### Waiver (WaiverDto)

| Field | Rule | Error message |
|---|---|---|
| `reason` | non-empty string, 10–500 chars | "Waiver reason is required (10–500 characters)." |
| `expires_at` | optional ISO-8601 date; if provided must be ≥ today + 1 day | "Expiry date must be in the future." |
| `review_note` | optional string ≤ 500 chars | — |
| Caller role | must be KYC or BM (verified via `EntitlementService.can(user, 'verify_doc', ...)`) | "You don't have access to this." → `FORBIDDEN` 403 |

## Backend Flow

### Upload flow (Phase A + Phase B)

```
1. Request arrives at DocumentController.upload(leadId, dto, user)
   - CorrelationMiddleware assigns / propagates X-Correlation-Id
   - JwtAuthGuard validates JWT -> resolves User
   - AbacGuard.@Requires('upload_doc', ...) -> EntitlementService.can -> 403 if denied

2. Zod validation of UploadInitiateDto (controller pipe)
   - file_type: 415 UNSUPPORTED_MEDIA if invalid MIME
   - file_size_kb: 413 PAYLOAD_TOO_LARGE if > MAX_UPLOAD_MB*1024

3. DocumentService.initiateUpload(leadId, dto, user) enters UnitOfWork.run(async tx => {

   3a. DocumentRepository.getProductChecklist(leadId, orgId, tx)
       -> NOT_FOUND 404 if lead absent / out of scope
       -> parse product_configs.document_checklist JSONB
       -> VALIDATION_ERROR 400 if doc_type+applicant_scope not in checklist

   3b. Compute version: count existing non-deleted documents for
       (lead_id, doc_type, applicant_scope) + 1

   3c. GcsPort.generateSignedPutUrl(bucketPath, contentType, ttl=GCS_SIGNED_URL_TTL)
       via IntegrationGateway.call('GcsPort', ...)
       -> UPSTREAM_UNAVAILABLE 503 on GCS failure (IntegrationLog(status=failed))

   3d. DocumentRepository.insert(doc row with status='pending', storage_ref=null, tx)

   3e. AuditAppender.emit(action='doc_upload', status='pending', tx)

   3f. UnitOfWork commits (doc row + audit_log in one transaction)

   3g. Return { document_id, upload_url, upload_url_expires_at, status:'pending' }
})

4. Client PUTs binary to GCS signed URL (outside LMS — direct to GCS)

5. Client calls POST /leads/{id}/documents with { action:'confirm', document_id }

6. DocumentService.confirmUpload(leadId, documentId, user) enters UnitOfWork.run(async tx => {

   6a. Validate document_id exists in documents with status='pending',
       lead_id matches, scope check -> 404 / 403

   6b. GcsPort.verifyObjectExists(storageRef)
       -> UPSTREAM_UNAVAILABLE 503 on GCS failure

   6c. GcsPort.inspectMimeType(storageRef)
       -> if mismatch vs. declared file_type:
            GcsPort.deleteObject(storageRef)
            throw VALIDATION_ERROR 400 "File type mismatch"

   6d. DocumentRepository.confirmUpload(documentId, storageRef, orgId, actorId, tx)
       sets status='uploaded'

   6e. Schedule async virus scan:
       IntegrationGateway.call('VirusScanPort', { document_id, storage_ref },
         { idempotencyKey: `virusScan:${documentId}` })
       -- Cloud Tasks enqueues; does NOT block response

   6f. LeadService.setKycStatus(leadId, deriveKycStatus(...), tx)
       -- re-derives from all current doc statuses

   6g. OutboxService.emit({ eventCode:'DOC_UPLOADED', ... }, tx)

   6h. AuditAppender.emit(action='doc_upload', status='uploaded', tx)

   6i. UnitOfWork commits

   6j. Return { document_id, status:'uploaded', virus_scan_status:'pending' }
})
```

### Virus scan async callback (Cloud Tasks webhook)

```
POST /api/v1/internal/documents/{did}/scan-result  (service-to-service, HMAC-verified)

1. Validate HMAC signature -> 403 if invalid
2. UnitOfWork.run(tx => {
   2a. DocumentRepository.setScanStatus(did, result.status, orgId, tx)
   2b. If result.status = 'infected':
         GcsPort.deleteObject(storageRef)
         DocumentRepository.setScanStatus(...) sets status back to 'pending' (rejected)
         OutboxService.emit({ eventCode:'DOC_UPLOADED', status:'rejected' }, tx)
   2c. If result.status = 'clean':
         DocumentRepository.markUnderReview(did, orgId, tx)
         LeadService.setKycStatus(leadId, re-derive, tx)
         OutboxService.emit({ eventCode:'DOC_UPLOADED', status:'under_review' }, tx)
   2d. AuditAppender.emit(action='doc_upload', detail.scan_result, tx)
   2e. UnitOfWork commits
})
```

## State Machine & Transaction Boundaries

Per `docs/contracts/state-machines.md` — Document `doc_status`:

```
not_required ──> pending ──> uploaded ──> under_review ──> verified
                                                        └──> mismatch (FR-071)
any ──> waived (KYC/BM only, with reason + audit)
any ──> expired (validity sweep job)
scan: pending -> clean -> under_review  |  pending -> infected -> rejected (back to pending)
```

**Invalid transitions (-> 409 CONFLICT):**
- `verified -> pending` (re-upload creates a NEW version row, not a status revert of the verified row)
- `waived -> waived` (already waived)
- Any status change to `expired` by user (system-only)

**Transaction scope (every document write):**
Each state-changing operation runs inside `UnitOfWork.run(async tx => { ... })`:

| Write | Table | Note |
|---|---|---|
| Document row insert or update | `documents` | owner: M8 DocumentRepository |
| Audit log | `audit_logs` | append-only sink |
| Outbox event | `event_outbox` | append-only sink |
| `leads.kyc_status` update | `leads` | via `LeadService.setKycStatus` (owner-writes) |

All four writes commit atomically. If any fails, the entire unit of work rolls back — no partial state.

## Ambiguities

1. **Waiver endpoint path not in api-contract.yaml.** The contract maps FR-070 only to `GET/POST /leads/{id}/documents`. The waiver requirement is in BRD AC-3 and the state machine, but no PATCH endpoint is defined. This LLD uses `PATCH /leads/{id}/documents/{did}/waive` by analogy with the `PATCH /leads/{id}/kyc/{kid}/resolve` pattern. The api-contract.yaml should be updated to add this endpoint.

2. **Virus scan port not in integration-map.md.** The integration map covers LOS/KYC/comms/telephony. BRD AC-4 requires virus scanning but names no vendor or port. This LLD introduces `VirusScanPort` with a mock adapter following the same hexagonal pattern. The integration map should be updated with the vendor choice and env vars.

3. **GCS signed URL protocol (Phase A/B vs. server-proxied upload).** The BRD says "GCS signed URL" but does not specify whether the client uploads directly to GCS (requiring a two-phase protocol) or the server proxies the stream. This LLD chose the direct-to-GCS signed-URL approach (lower server bandwidth cost, standard for GCS), which requires Phase A (initiate) + Phase B (confirm) on the same endpoint distinguished by body shape. If the NBFC's GCS IAM policy does not permit direct client uploads, the api-contract.yaml would need a server-proxy endpoint. Confirm with infrastructure.

4. **`not_required` items in the checklist.** The state machine and BRD mention `not_required` as a starting state for optional items. This LLD treats `not_required` as a `product_configs.document_checklist` attribute (field `mandatory: false`), not a `doc_status` value stored in `documents`. A row is only created in `documents` when an upload is initiated. `not_required` items whose `mandatory=false` and for which no row exists appear in the checklist response with `status: 'not_required'` derived from the checklist definition, not from the DB. This is consistent with the doc_status enum which does include `not_required` as a valid value — but it is set explicitly only by the checklist derivation logic.

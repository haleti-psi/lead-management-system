# FR-121 — NBFC Differentiator Reports (LLD)
**Tier: 2 (Moderate)** — multi-table parameterised aggregate reads across up to nine source tables, ten report codes with distinct formulas (including the §12.4 DSA/dealer quality sub-formula), scope-filtered by role, zero-denominator guard, async fallback for heavy payloads, and a report viewer UI with heatmap/SLA visualisations. No state transitions and no external service calls.

## Summary

`GET /reports/{code}` with a differentiator `code` value returns scope-filtered aggregate data for one of the ten NBFC-specific management reports defined in BRD §12 / FR-121:

| `code` | Report | Priority |
|---|---|---|
| `first_contact_sla` | First Contact SLA | Must |
| `kyc_doc_ageing` | KYC & Document Ageing | Must |
| `dsa_dealer_quality` | DSA/Dealer Quality | Must |
| `duplicate_leakage` | Duplicate Leakage | Must |
| `handoff_failure` | Handoff Failure | Must |
| `source_roi` | Source ROI | Should |
| `contactability` | Contactability | Should |
| `consent_privacy_ops` | Consent & Privacy Ops | Should |
| `product_branch_heatmap` | Product/Branch Heatmap | Should |
| `rm_capacity_load` | RM Capacity & Load | Should |

Each report is a stateless parameterised read: the service resolves the caller's data scope, builds one or more Kysely aggregate queries bounded to that scope and the requested filter window, evaluates zero-denominator guard rules (§12.5 — display `null` not 0%), and returns the `data` payload inside the uniform envelope. No writes occur.

Heavy reports (> 5 000 rows of raw data or query time > 2 000 ms, configurable) are rejected synchronously with a `202 Accepted` response that points the caller to `POST /exports` (FR-122 async path). This protects OLTP latency (NFR-02 ≤ 500 ms read p95). Read-replica routing is advised for all queries (architecture §12.6).

The `dsa_dealer_quality` code re-uses the §12.4 formula logic also used by FR-092 (`GET /partners/{id}/quality`); the implementation must call `PartnerQualityService.computeScore(partnerId, window)` rather than duplicating the formula inline.

## Endpoint

### `GET /api/v1/reports/{code}` — `getReport`

**Path parameter:**

| Parameter | Type | Constraint |
|---|---|---|
| `code` | string | Must be one of the ten differentiator codes or the core-pack codes (FR-120). For FR-121 the valid codes are listed in the table above. Unknown code → `VALIDATION_ERROR`. |

**Query parameters (all optional):**

| Parameter | Type | Default | Constraint |
|---|---|---|---|
| `from` | `YYYY-MM-DD` | today − 29 days (IST) | ISO date; must be ≤ `to` |
| `to` | `YYYY-MM-DD` | today (IST) | ISO date; must be ≥ `from` |
| `branch_id` | UUID | — | Scope sub-filter; access denied if outside caller scope |
| `team_id` | UUID | — | Scope sub-filter |
| `source` | enum `lead_source` | — | Filter by source channel |
| `partner_id` | UUID | — | Filter by specific partner; PARTNER role auto-binds to own `partner_id` |
| `product_code` | enum `product_code` | — | Filter by product |
| `page` | integer ≥ 1 | 1 | Pagination (tabular reports only) |
| `limit` | integer 1–100 | 25 | Pagination |

## Auth Check

Global `JwtAuthGuard` is applied. Authorization via `AbacGuard` + `@Requires('reports', reportScopeResolver)` → `EntitlementService.can(user, 'reports', resource)`.

The `reportScopeResolver` is a function injected at the decorator level that receives the incoming request and returns the resource descriptor `{ reportCode, filters }`. `EntitlementService.can` evaluates:

1. Checks `capability_matrix[user.role].reports` is present; absent → `FORBIDDEN`.
2. Resolves the caller's **data scope ceiling** from the auth matrix:

| Role | Scope | Data restriction applied to all queries |
|---|---|---|
| `HEAD` | `A` | `org_id = user.org_id` only |
| `BM` | `B` | additionally filter by `branch_id IN user.branch_ids` |
| `SM` | `T` | additionally filter by `team_id IN user.team_ids` |
| `KYC` | `B` | additionally filter by `branch_id IN user.branch_ids` |
| `DPO` | `M` | `org_id` scope, masked PII fields; access limited to compliance-relevant reports (`consent_privacy_ops`); other codes return `FORBIDDEN` |
| `PARTNER` | `P` | `source_attributions.partner_id = user.partner_id`; any `partner_id` query param that does not match → `FORBIDDEN` |
| `RM` | `O` | `leads.owner_id = user.user_id` |
| `ADMIN` | — | No `reports` capability on lead data (auth-matrix `ADMIN.*` note); returns `FORBIDDEN` |

3. Sub-filter validation: if `branch_id` query param is supplied but is not within the caller's scope → `FORBIDDEN`. Same for `team_id`.

4. `PARTNER` role: `partner_id` query param is auto-set to `user.partner_id`; any attempt to supply a different value → `FORBIDDEN`.

## Validation Logic

All request parameters are validated by a Zod schema in the DTO before the service layer is reached. Failures return `VALIDATION_ERROR` (400) with `fields[]`.

| Field | Rule | Error message |
|---|---|---|
| `code` (path) | Must be one of the ten FR-121 differentiator codes listed above (or FR-120 core codes) | "Unknown report code." |
| `from` | If present: `YYYY-MM-DD` format, parseable as a valid date in IST | "Invalid from date. Use YYYY-MM-DD." |
| `to` | If present: `YYYY-MM-DD` format, parseable as a valid date in IST | "Invalid to date. Use YYYY-MM-DD." |
| `from`, `to` pair | `from` ≤ `to` (after parsing) | "from must not be after to." |
| `branch_id`, `team_id`, `partner_id` | UUID format when supplied | "Must be a valid UUID." |
| `source` | Must be a value from `@shared/enums` `lead_source` enum | "Invalid source value." |
| `product_code` | Must be a value from `@shared/enums` `product_code` enum | "Invalid product_code value." |
| `limit` | Integer 1–100; default 25 | "limit must be between 1 and 100." |
| `page` | Integer ≥ 1; default 1 | "page must be a positive integer." |

## Data Operations

All queries use **Kysely** typed parameterised builders. Every query includes `org_id` as a mandatory filter. Every list query has a `LIMIT` clause (max 100). No raw SQL strings. Read-replica routing is recommended (architecture §12.6) but is infrastructure-level; the service uses the standard `Db` instance injected from `core/db/`.

The service builds a `ScopeFilter` object from the resolved auth scope + query params, then dispatches to the appropriate per-report aggregate function. The ten report functions are:

### Common scope filter applied to all reports

```ts
// Constructed once per request; all report queries receive this
interface ReportScopeFilter {
  orgId: string;
  fromTs: Date;          // IST midnight of `from`
  toTs: Date;            // IST midnight + 23:59:59.999 of `to`
  branchIds?: string[];  // BM/KYC scope
  teamIds?: string[];    // SM scope
  ownerId?: string;      // RM scope
  partnerId?: string;    // PARTNER scope or partner_id filter
  source?: string;
  productCode?: string;
  page: number;
  limit: number;
}
```

### 1. `first_contact_sla` — First Contact SLA

Answers: are hot/new leads contacted in time by branch/team/RM/source?

```ts
// Summary row
db.selectFrom('leads as l')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .select([
    db.fn.count('l.lead_id').as('total_leads'),
    db.fn.sum(
      sql`CASE WHEN l.stage NOT IN ('captured','assigned','first_contact_pending') THEN 1 ELSE 0 END`
    ).as('contacted_count'),
    db.fn.sum(
      sql`CASE WHEN l.sla_first_contact_due_at IS NOT NULL AND l.sla_first_contact_due_at < now() AND l.stage IN ('captured','assigned','first_contact_pending') THEN 1 ELSE 0 END`
    ).as('sla_breached'),
    db.fn.sum(
      sql`CASE WHEN l.stage IN ('captured','assigned','first_contact_pending') AND (l.sla_first_contact_due_at IS NULL OR l.sla_first_contact_due_at >= now()) THEN 1 ELSE 0 END`
    ).as('pending_first_contact'),
  ])
  .where('l.org_id', '=', filter.orgId)
  .where('l.created_at', '>=', filter.fromTs)
  .where('l.created_at', '<=', filter.toTs)
  .$if(filter.branchIds != null, q => q.where('l.branch_id', 'in', filter.branchIds!))
  .$if(filter.teamIds != null, q => q.where('l.team_id', 'in', filter.teamIds!))
  .$if(filter.ownerId != null, q => q.where('l.owner_id', '=', filter.ownerId!))
  .$if(filter.partnerId != null, q => q.where('sa.partner_id', '=', filter.partnerId!))
  .$if(filter.productCode != null, q => q.where('l.product_code', '=', filter.productCode!))
  .executeTakeFirstOrThrow()

// Breakdown rows (by branch or team depending on scope)
db.selectFrom('leads as l')
  .innerJoin('branches as b', 'b.branch_id', 'l.branch_id')
  .select([
    'l.branch_id', 'b.name as branch_name',
    db.fn.count('l.lead_id').as('total'),
    db.fn.sum(sql`CASE WHEN l.stage NOT IN ('captured','assigned','first_contact_pending') THEN 1 ELSE 0 END`).as('contacted'),
    db.fn.sum(sql`CASE WHEN l.sla_first_contact_due_at < now() AND l.stage IN ('captured','assigned','first_contact_pending') THEN 1 ELSE 0 END`).as('breached'),
  ])
  .where('l.org_id', '=', filter.orgId)
  // ... scope + window filters ...
  .groupBy(['l.branch_id', 'b.name'])
  .orderBy('breached', 'desc')
  .limit(filter.limit)
  .offset((filter.page - 1) * filter.limit)
```

SLA compliance % = contacted_count / (total_leads − pending_first_contact) × 100; if denominator = 0 → `null`.

### 2. `kyc_doc_ageing` — KYC & Document Ageing

Answers: which docs/products/branches delay hand-off?

Uses `stage_history` to compute the dwell time at each KYC-related stage (`documents_pending`, `kyc_in_progress`):

```ts
db.selectFrom('stage_history as sh')
  .innerJoin('leads as l', 'l.lead_id', 'sh.lead_id')
  .select([
    'l.product_code',
    'l.branch_id',
    db.fn.avg(
      sql`EXTRACT(EPOCH FROM (sh.occurred_at - lag(sh.occurred_at) OVER (PARTITION BY sh.lead_id ORDER BY sh.occurred_at)))/3600`
    ).as('avg_stage_dwell_hrs'),
    db.fn.count('l.lead_id').as('lead_count'),
  ])
  .where('sh.to_stage', 'in', ['documents_pending', 'kyc_in_progress', 'eligibility_requested'])
  .where('sh.org_id', '=', filter.orgId)
  .where('sh.occurred_at', '>=', filter.fromTs)
  .where('sh.occurred_at', '<=', filter.toTs)
  // ... scope filters ...
  .groupBy(['l.product_code', 'l.branch_id'])
  .limit(filter.limit)
  .offset((filter.page - 1) * filter.limit)

// Document-level ageing (time from upload to verification or today if still pending)
db.selectFrom('documents as d')
  .innerJoin('leads as l', 'l.lead_id', 'd.lead_id')
  .select([
    'd.doc_type',
    'l.product_code',
    db.fn.avg(
      sql`EXTRACT(EPOCH FROM (COALESCE(d.updated_at, now()) - d.created_at))/86400`
    ).as('avg_age_days'),
    db.fn.count('d.document_id').as('doc_count'),
    db.fn.sum(sql`CASE WHEN d.status = 'verified' THEN 1 ELSE 0 END`).as('verified_count'),
    db.fn.sum(sql`CASE WHEN d.status IN ('uploaded','under_review','mismatch') THEN 1 ELSE 0 END`).as('pending_count'),
  ])
  .where('d.org_id', '=', filter.orgId)
  .where('d.created_at', '>=', filter.fromTs)
  .where('d.created_at', '<=', filter.toTs)
  .where('d.deleted_at', 'is', null)
  // ... scope filters via leads join ...
  .groupBy(['d.doc_type', 'l.product_code'])
  .limit(filter.limit)
  .offset((filter.page - 1) * filter.limit)
```

### 3. `dsa_dealer_quality` — DSA/Dealer Quality

Delegates to `PartnerQualityService.computeScoreBatch(partnerIds[], window)` (reuses FR-092 §12.4 formula — do **not** duplicate the formula). Returns a ranked list of all partners in scope:

```ts
// Step 1: resolve partner ids in scope
const partnerIds = await db
  .selectFrom('partners as p')
  .select('p.partner_id')
  .where('p.org_id', '=', filter.orgId)
  .where('p.type', 'in', ['DSA', 'Dealer'])
  .$if(filter.partnerId != null, q => q.where('p.partner_id', '=', filter.partnerId!))
  .execute()
  .then(rows => rows.map(r => r.partner_id))

// Step 2: batch compute scores (reuses PartnerQualityService)
const scores = await partnerQualityService.computeScoreBatch(partnerIds, {
  from: filter.fromTs,
  to: filter.toTs,
  orgId: filter.orgId,
})

// Step 3: paginate
return scores
  .sort((a, b) => (b.quality_score ?? -1) - (a.quality_score ?? -1))
  .slice((filter.page - 1) * filter.limit, filter.page * filter.limit)
```

Each row in the response mirrors the FR-092 metrics/factors structure with `partner_id`, `legal_name`, `type`, `quality_score`, `insufficient_data` flag, and `metrics`.

### 4. `duplicate_leakage` — Duplicate Leakage

Answers: which sources create duplicates and where are they caught?

```ts
db.selectFrom('duplicate_matches as dm')
  .innerJoin('leads as l', 'l.lead_id', 'dm.lead_id')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .select([
    'sa.source',
    'sa.partner_id',
    'dm.confidence',
    'dm.action',
    'dm.status',
    db.fn.count('dm.duplicate_match_id').as('count'),
  ])
  .where('dm.org_id', '=', filter.orgId)
  .where('dm.created_at', '>=', filter.fromTs)
  .where('dm.created_at', '<=', filter.toTs)
  .$if(filter.branchIds != null, q => q.where('l.branch_id', 'in', filter.branchIds!))
  .$if(filter.source != null, q => q.where('sa.source', '=', filter.source!))
  .$if(filter.partnerId != null, q => q.where('sa.partner_id', '=', filter.partnerId!))
  .groupBy(['sa.source', 'sa.partner_id', 'dm.confidence', 'dm.action', 'dm.status'])
  .orderBy('count', 'desc')
  .limit(filter.limit)
  .offset((filter.page - 1) * filter.limit)
```

Summary: total duplicates, caught at `captured` vs later stages, blocked vs warned vs merged.

### 5. `handoff_failure` — Handoff Failure

Answers: why are hand-offs failing; which integration errors recur?

```ts
db.selectFrom('integration_logs as il')
  .leftJoin('leads as l', 'l.lead_id', 'il.lead_id')
  .select([
    'il.integration',
    'il.error_code',
    'il.http_status',
    db.fn.count('il.integration_log_id').as('failure_count'),
    db.fn.avg('il.retry_count').as('avg_retries'),
    db.fn.max('il.created_at').as('last_seen_at'),
  ])
  .where('il.org_id', '=', filter.orgId)
  .where('il.status', 'in', ['failed'])
  .where('il.integration', '=', 'LOS')
  .where('il.created_at', '>=', filter.fromTs)
  .where('il.created_at', '<=', filter.toTs)
  .$if(filter.branchIds != null, q => q.where('l.branch_id', 'in', filter.branchIds!))
  .groupBy(['il.integration', 'il.error_code', 'il.http_status'])
  .orderBy('failure_count', 'desc')
  .limit(filter.limit)
  .offset((filter.page - 1) * filter.limit)
```

Summary: total hand-off attempts, failure rate, most common error codes, avg retries.

### 6. `source_roi` — Source ROI

Answers: which source/campaign converts best after cost?

```ts
db.selectFrom('leads as l')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .select([
    'sa.source',
    'sa.campaign_code',
    'sa.partner_id',
    db.fn.count('l.lead_id').as('total_leads'),
    db.fn.sum(sql`CASE WHEN l.stage = 'handed_off' THEN 1 ELSE 0 END`).as('converted'),
    db.fn.sum(sql`CASE WHEN l.stage = 'rejected' THEN 1 ELSE 0 END`).as('rejected'),
  ])
  .where('l.org_id', '=', filter.orgId)
  .where('l.created_at', '>=', filter.fromTs)
  .where('l.created_at', '<=', filter.toTs)
  .$if(filter.branchIds != null, q => q.where('l.branch_id', 'in', filter.branchIds!))
  .$if(filter.source != null, q => q.where('sa.source', '=', filter.source!))
  .groupBy(['sa.source', 'sa.campaign_code', 'sa.partner_id'])
  .orderBy('converted', 'desc')
  .limit(filter.limit)
  .offset((filter.page - 1) * filter.limit)
```

Conversion rate = converted / total_leads × 100; null if total = 0. Cost-per-conversion is not stored in the LMS (no cost field in schema); the report exposes conversion rates only and adds a `cost_data_available: false` flag.

### 7. `contactability` — Contactability

Answers: which leads fail (wrong number/no response) and from where?

```ts
db.selectFrom('communication_logs as cl')
  .innerJoin('leads as l', 'l.lead_id', 'cl.lead_id')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')

    'sa.source',
    'sa.partner_id',
    'cl.channel',
    'cl.failure_reason',
    db.fn.count('cl.communication_log_id').as('total_attempts'),
    db.fn.sum(sql`CASE WHEN cl.status IN ('delivered') THEN 1 ELSE 0 END`).as('delivered'),
    db.fn.sum(sql`CASE WHEN cl.status IN ('failed') THEN 1 ELSE 0 END`).as('failed'),
  ])
  .where('cl.org_id', '=', filter.orgId)
  .where('cl.created_at', '>=', filter.fromTs)
  .where('cl.created_at', '<=', filter.toTs)
  .$if(filter.branchIds != null, q => q.where('l.branch_id', 'in', filter.branchIds!))
  .$if(filter.source != null, q => q.where('sa.source', '=', filter.source!))
  .groupBy(['sa.source', 'sa.partner_id', 'cl.channel', 'cl.failure_reason'])
  .orderBy('failed', 'desc')
  .limit(filter.limit)
  .offset((filter.page - 1) * filter.limit)
```

Contactability rate = delivered / total_attempts × 100; null if total = 0.

### 8. `consent_privacy_ops` — Consent & Privacy Ops

Answers: which leads lack consent / have withdrawals / open requests?

```ts
// Consent status aggregation
db.selectFrom('leads as l')
  .select([
    'l.consent_status',
    'l.product_code',
    db.fn.count('l.lead_id').as('count'),
  ])
  .where('l.org_id', '=', filter.orgId)
  .where('l.created_at', '>=', filter.fromTs)
  .where('l.created_at', '<=', filter.toTs)
  .$if(filter.branchIds != null, q => q.where('l.branch_id', 'in', filter.branchIds!))
  .groupBy(['l.consent_status', 'l.product_code'])
  .execute()

// Open data rights requests
db.selectFrom('data_rights_requests as drr')
  .select([
    'drr.request_type',
    db.fn.count('drr.data_rights_request_id').as('open_count'),
  ])
  .where('drr.org_id', '=', filter.orgId)
  .where('drr.status', '=', 'open')
  .where('drr.created_at', '>=', filter.fromTs)
  .where('drr.created_at', '<=', filter.toTs)
  .groupBy('drr.request_type')
  .execute()

// Open grievances related to privacy/consent
db.selectFrom('grievances as g')
  .select([
    'g.category',
    db.fn.count('g.grievance_id').as('open_count'),
  ])
  .where('g.org_id', '=', filter.orgId)
  .where('g.status', 'in', ['open', 'in_progress'])
  .where('g.category', '=', 'privacy')
  .where('g.created_at', '>=', filter.fromTs)
  .where('g.created_at', '<=', filter.toTs)
  .groupBy('g.category')
  .execute()
```

DPO role may access this report at `M` scope (org-wide, PII masked). Other roles see only within their data scope.

### 9. `product_branch_heatmap` — Product/Branch Heatmap

Answers: which products/branches have volume/conversion/TAT issues?

```ts
db.selectFrom('leads as l')
  .innerJoin('branches as b', 'b.branch_id', 'l.branch_id')
  .select([
    'l.product_code',
    'l.branch_id',
    'b.name as branch_name',
    db.fn.count('l.lead_id').as('volume'),
    db.fn.sum(sql`CASE WHEN l.stage = 'handed_off' THEN 1 ELSE 0 END`).as('converted'),
    db.fn.sum(sql`CASE WHEN l.stage = 'rejected' THEN 1 ELSE 0 END`).as('rejected'),
  ])
  .where('l.org_id', '=', filter.orgId)
  .where('l.created_at', '>=', filter.fromTs)
  .where('l.created_at', '<=', filter.toTs)
  .$if(filter.branchIds != null, q => q.where('l.branch_id', 'in', filter.branchIds!))
  .$if(filter.productCode != null, q => q.where('l.product_code', '=', filter.productCode!))
  .groupBy(['l.product_code', 'l.branch_id', 'b.name'])
  .orderBy(['l.product_code', 'volume desc'])
  .limit(filter.limit)
  .offset((filter.page - 1) * filter.limit)
```

TAT is derived from `stage_history` in a second query joining on `lead_id`, computing average dwell from `assigned` → `handed_off` in hours. Zero denominator → `null`.

### 10. `rm_capacity_load` — RM Capacity & Load

Answers: which RMs are overloaded/under-utilised?

```ts
db.selectFrom('users as u')
  .innerJoin('leads as l', 'l.owner_id', 'u.user_id')
  .leftJoin('tasks as t', eb =>
    eb.on('t.lead_id', '=', 'l.lead_id').on('t.status', 'in', ['open', 'in_progress'])
  )
  .select([
    'u.user_id', 'u.full_name', 'u.branch_id', 'u.team_id',
    db.fn.count(sql`DISTINCT l.lead_id`).as('active_leads'),
    db.fn.sum(sql`CASE WHEN l.stage IN ('assigned','first_contact_pending','contacted') THEN 1 ELSE 0 END`).as('early_stage_leads'),
    db.fn.count(sql`DISTINCT t.task_id`).as('open_tasks'),
    db.fn.sum(sql`CASE WHEN t.status = 'open' AND t.due_at < now() THEN 1 ELSE 0 END`).as('overdue_tasks'),
  ])
  .where('u.org_id', '=', filter.orgId)
  .where('u.status', '=', 'active')
  .where('l.created_at', '>=', filter.fromTs)
  .where('l.created_at', '<=', filter.toTs)
  .where('l.stage', 'not in', ['handed_off', 'rejected', 'dormant'])
  .$if(filter.branchIds != null, q => q.where('u.branch_id', 'in', filter.branchIds!))
  .$if(filter.teamIds != null, q => q.where('u.team_id', 'in', filter.teamIds!))
  .groupBy(['u.user_id', 'u.full_name', 'u.branch_id', 'u.team_id'])
  .orderBy('active_leads', 'desc')
  .limit(filter.limit)
  .offset((filter.page - 1) * filter.limit)
```

## Backend Flow (Step-by-Step)

1. **Request enters** `ReportingController.getReport(code, query, req)`. `CorrelationMiddleware` sets `X-Correlation-Id`.
2. **JwtAuthGuard** validates the access token; loads `user` (role, org_id, branch_ids, team_ids, partner_id) onto the request. Invalid token → `AUTH_REQUIRED` (401).
3. **AbacGuard** calls `EntitlementService.can(user, 'reports', { code, filters: query })`. No `reports` capability → `FORBIDDEN` (403).
4. **GetReportDto** (Zod) validates path `code` and query params. Invalid input → `VALIDATION_ERROR` (400) with `fields[]`.
5. **Scope resolution**: `ReportingService.buildScopeFilter(user, dto)` constructs the `ReportScopeFilter`. Checks sub-filter validity (`branch_id`, `team_id` inside scope). Out-of-scope sub-filter → `FORBIDDEN` (403). PARTNER role overrides `partner_id` to `user.partner_id`.
6. **Async threshold check**: `ReportingService.estimateComplexity(code, filter)` executes a fast `COUNT(*)` query. If the estimated result set exceeds the configured threshold (default 5 000 rows), returns `202` with export guidance — no further queries.
7. **Dispatch** to the appropriate per-report function (e.g. `this.firstContactSlaReport(filter)`, `this.dsaDealerQualityReport(filter)`). For `dsa_dealer_quality`, delegates to `PartnerQualityService.computeScoreBatch`.
8. **Zero-denominator guard** (§12.5): each rate field is computed as `numerator / denominator`; if `denominator === 0` the rate is set to `null`, not `0`.
9. **Reconciliation block** is populated with `numerator` and `denominator` for every rate, satisfying §12.5.
10. **ResponseEnvelopeInterceptor** wraps the result in `{ data, meta, error: null }` with pagination metadata.
11. **MaskingInterceptor** is applied post-serialization. Reports do not return raw PII fields (no mobile/PAN in aggregate rows). The `consent_privacy_ops` report for DPO (`M` scope) masks any lead-level identifiers.
12. Response returned to caller.

## State Machine

FR-121 is read-only. No `lead_stage`, `doc_status`, `kyc_check_status`, `consent_state`, or any other state machine is touched. No `UnitOfWork` transaction. No `AuditAppender` call. No `OutboxService` emit.

The only write side effect is that the `dsa_dealer_quality` report, when it delegates to `PartnerQualityService.computeScoreBatch`, **may** trigger a write to `partners.quality_score` as a cache update — that write is owned by `PartnerQualityService` (M10, FR-092) and is out of scope for this FR. `ReportingService` treats the result as read-only.

None. FR-121 is a pure database aggregate read. No `IntegrationGateway` call is made.

## Assumptions

1. **`PartnerQualityService.computeScoreBatch`** is a method exposed by M10's `PartnerModule` for batch computation. If it does not yet exist (FR-092 not yet implemented), the `ReportingModule` must declare a dependency on `PartnerModule` and this method must be added as part of FR-092's implementation. Until then, `dsa_dealer_quality` returns a stub with `insufficient_data: true`.
2. **Cost fields** for `source_roi` do not exist in the schema. The report returns conversion rates only and sets `cost_data_available: false` in the response. Actual cost-per-conversion requires a future schema extension.
3. **Async threshold** values (row count and/or query time) are configurable via environment variables `REPORT_ASYNC_ROW_THRESHOLD` (default 5000) and `REPORT_ASYNC_TIME_MS` (default 2000). These must be added to `docs/contracts/environment-contract.md` before implementation.
4. **Charts library**: `ui.md` states "a lightweight lib with a low-bandwidth fallback". No chart library is currently registered in `dependency-register.md`. The chart components must use whichever library is added there before this FR is implemented. The low-bandwidth fallback (`DataTable`) is implemented regardless.
5. **Read replica routing** is an infrastructure-level concern. The service uses the standard `Db` instance; read-replica configuration is handled at the connection pool / Cloud SQL Proxy level.
6. The `tasks.due_at` column is referenced in `rm_capacity_load`; this column must exist in the `tasks` table. It was not visible in the schema excerpt read; this is flagged as an ambiguity below.

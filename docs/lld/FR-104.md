## Summary

FR-104 provides two capabilities:

1. **SLA Policy Administration** (human): ADMIN / Sales Ops configure named `sla_policies` rows — threshold in business minutes, `applies_to` enum (`sla_target`), and a JSON escalation chain — subject to maker-checker (`configuration_versions`). Active policies are activated via `POST /admin/config/{id}/approve` (FR-132); this FR covers the `GET /admin/sla-policies` list and `POST /admin/sla-policies` create/version operations only.

2. **SLA Engine** (system): `SlaEngine` (shared core service at `core/sla/`) computes business-time-aware due timestamps whenever relevant state transitions occur, and a Cloud Scheduler job (`SLA_SWEEP_JOB`) scans for approaching/breached records every minute, emits `FIRST_CONTACT_DUE` / `FIRST_CONTACT_BREACH` event outbox entries, dispatches in-app notifications, and — on breach — triggers reassignment via `AllocationService` (FR-030). No direct HTTP endpoint drives the sweep; it is a background job.

`BusinessCalendarService.resolve(branchId?, regionId?)` is the single business-time clock (ADR-6). `UnitOfWork` is used wherever SLA timestamp writes accompany a state transition. `OutboxService.emit` queues events transactionally. `NotificationDispatchService.send` fires post-commit.

**Auth guard:** `JwtAuthGuard` (global) + `AbacGuard` + `@Requires('configuration', scopeResolver)`.

## Auth Check

### Endpoints
- Guard stack: `JwtAuthGuard` (global) → `AbacGuard` → `@Requires('configuration', scopeResolver)`.
- `EntitlementService.can(user, 'configuration', { org_id })` is called inside `AbacGuard`.
- Roles with `configuration` capability: `ADMIN` (scope A), `HEAD` (scope A), `BM` (scope B), `KYC` (scope B), `DPO` (scope B).
- `POST /admin/sla-policies` additionally enforces `scope === 'A'` in the service layer; a BM/KYC/DPO call returns `FORBIDDEN`.
- Maker-checker: the actor who creates the policy (`maker_id`) may NOT be the same as the approver (`checker_id`); enforced by `ck_config_maker_checker` DB constraint and validated in FR-132.

### SLA Engine (background job)
- Runs as a Cloud Scheduler → Cloud Tasks HTTP target calling `POST /api/v1/internal/sla/sweep` (service-to-service, not a user-facing endpoint; guarded by `InternalTaskGuard` verifying the `X-CloudTasks-QueueName` header and a service-account-bound OIDC token, not a user JWT).

```typescript

const approaching = await tx
  .selectFrom('leads')
  .select(['lead_id', 'owner_id', 'branch_id', 'sla_first_contact_due_at', 'version'])

## Validation Logic (Zod)

### CreateSlaPolicyDto
| Field | Rule | Error message |
|-------|------|---------------|
| `name` | `z.string().min(1).max(120)` | "Name is required and must be at most 120 characters." |
| `applies_to` | `z.enum(['first_contact','document','kyc_exception','grievance','handoff_retry'])` | "applies_to must be a valid SLA target." |
| `condition` | `z.record(z.unknown()).optional().nullable()` | — |
| `threshold_minutes` | `z.number().int().positive()` (enforces DB check `> 0`) | "Threshold must be a positive integer (business minutes)." |
| `escalation_chain` | `z.array(EscalationStepSchema).min(1)` | "Escalation chain must have at least one step." |
| `escalation_chain[].at_minutes` | `z.number().int().positive()` | "at_minutes must be a positive integer." |
| `escalation_chain[].notify_roles` | `z.array(z.enum([role_code values])).min(1)` | "notify_roles must be a non-empty array of valid roles." |
| `escalation_chain[].action` | `z.enum(['notify','reassign'])` | "action must be 'notify' or 'reassign'." |

**Cross-field rule:** All `escalation_chain[].at_minutes` values must be distinct (no duplicate warning thresholds). Violation → `VALIDATION_ERROR` with `fields: [{ field: 'escalation_chain', issue: 'Duplicate at_minutes values are not allowed.' }]`.

**Service-layer rule:** A `reassign` action must appear no more than once per chain and, if present, must be the step with the highest `at_minutes`. Violation → `VALIDATION_ERROR`.

### ListSlaPoliciesQueryDto
| Field | Rule |
|-------|------|
| `page` | `z.coerce.number().int().positive().default(1)` |
| `limit` | `z.coerce.number().int().min(1).max(100).default(25)` |
| `applies_to` | `z.enum([...sla_target values]).optional()` |
| `is_active` | `z.coerce.boolean().optional()` |

## Backend Flow (Step-by-Step)

### POST /admin/sla-policies

1. `CorrelationMiddleware` attaches `X-Correlation-Id` to the request context.
2. `JwtAuthGuard` validates the JWT; `AbacGuard` calls `EntitlementService.can(user, 'configuration', { org_id })`.
3. For `POST`, the service additionally checks `user.scope === 'A'`; if not, throws `FORBIDDEN`.
4. `CreateSlaPolicyDto` is Zod-parsed; field errors collected and thrown as `VALIDATION_ERROR` with `fields[]`.
5. Cross-field and service-layer rules validated (duplicate `at_minutes`, reassign position).
6. `UnitOfWork.run(async (tx) => { ... })` opens one ambient transaction:
   a. Duplicate check: query `sla_policies` for existing active row with same `name` + `applies_to` + `org_id`; if found, throw `CONFLICT`.
   b. `SLAPolicyRepository.create(dto, actorId, tx)` — inserts `sla_policies` row with `is_active = false`.
   c. `ConfigurationVersionRepository.create({ config_type: 'sla_policy', config_ref: policyId, version: 1, maker_id: actorId }, tx)` — inserts `configuration_versions` row with `status = 'pending'`.
   d. `AuditAppender.append({ action: 'config_change', entity_type: 'sla_policy', entity_id: policyId, actor_id: actorId, detail: { op: 'create', threshold_minutes: dto.threshold_minutes, applies_to: dto.applies_to } }, tx)`.
7. Transaction commits. `ResponseEnvelopeInterceptor` wraps response in `{ data, meta, error: null }`.
8. Response 201 returned.

### GET /admin/sla-policies

1. `JwtAuthGuard` + `AbacGuard` check `configuration` capability.
2. `ListSlaPoliciesQueryDto` Zod-parsed.
3. `SLAPolicyRepository.list(orgId, filters, pagination)` — two Kysely queries (rows + count); LIMIT ≤ 100.
4. Response 200 with pagination metadata.

### SLA Due-At Setting (triggered by AllocationService on `captured → assigned`)

This runs inside the same `UnitOfWork` transaction as the lead stage transition (see FR-030 and state-machines.md):

1. `AllocationService.allocate(lead, tx)` calls `SlaEngine.setFirstContactDue(lead, tx)`.
2. `SlaEngine.setFirstContactDue`:
   a. Queries `sla_policies` for the most specific active policy: `applies_to = 'first_contact'` + matches `condition.product_code` (if set) + `is_active = true`. Falls back to policy with no condition. If no policy found → skips (no due timestamp set, logs warning).
   b. Calls `BusinessCalendarService.resolve(lead.branch_id, regionId)` to get the correct calendar.
   c. Computes `dueAt = addBusinessMinutes(now(), policy.threshold_minutes, calendar)`.
   d. Calls `LeadService.setSlaDueAt({ lead_id, sla_first_contact_due_at: dueAt, expectedVersion }, tx)` — the only writer of `leads`.
3. On `failed → exception` in a KYC verification: `SlaEngine.setKycExceptionDue(kycVerification, tx)` follows the same pattern, writing `kyc_verifications.exception_sla_due_at`.
4. On grievance intake: `SlaEngine.setGrievanceDue(grievance, tx)` writes `grievances.sla_due_at`.

### SLA Sweep (Cloud Scheduler → Cloud Tasks → internal endpoint, every minute)

1. Cloud Scheduler fires `POST /api/v1/internal/sla/sweep` with OIDC token.
2. `InternalTaskGuard` verifies OIDC token; no user JWT involved.
3. `SlaEngine.sweep(orgId)` is called inside `UnitOfWork.run(...)`.
4. **Approaching scan** (within 30 min of `sla_first_contact_due_at`, lead not yet in terminal/contacted stage):
   a. Retrieves at most 100 leads (LIMIT enforced).
   b. For each lead, evaluates the escalation chain: finds the step whose `at_minutes` <= `(dueAt - now())` and has not been notified (idempotency: check if `FIRST_CONTACT_DUE` outbox entry already exists for this lead + this sweep window).
   c. Emits `FIRST_CONTACT_DUE` outbox event via `OutboxService.emit(...)` within the same transaction.
5. **Breached scan** (`sla_first_contact_due_at <= now()`, not terminal):
   a. Retrieves at most 100 leads (LIMIT enforced).
   b. Emits `FIRST_CONTACT_BREACH` outbox event within the same transaction.
   c. For escalation chain steps with `action: 'reassign'`: calls `AllocationService.escalate(lead, escalationChain, tx)` which calls `LeadService.assignOwner(...)` — reassigning the lead to the BM/SM per chain, within the same transaction, with `stage_history` + `audit_logs` + `event_outbox(LEAD_ASSIGNED)` written atomically.
   d. `AuditAppender.append({ action: 'reassign', entity_type: 'lead', entity_id: leadId, detail: { reason: 'sla_breach', policy_id } }, tx)`.
6. Transaction commits. Post-commit: `NotificationDispatchService.send(...)` dispatches in-app notifications (consent-gated, does not roll back the transaction on failure).
7. Sweep job is **idempotent**: a second run within the same minute will find the same leads but the outbox deduplication (same `event_code` + `aggregate_id` within a sweep window) prevents duplicate events. Breach reassignment also checks if `owner_id` already matches the target before issuing `LeadService.assignOwner`.

## State Machine & Transaction Boundaries

### ConfigurationVersion (maker-checker)
```
pending → approved → active
       └→ rejected
active → rolled_back
```
FR-104 creates the `sla_policy` + `configuration_versions(status=pending)` in one transaction.
`POST /admin/config/{id}/approve` (FR-132) updates `status → approved → active` and sets `sla_policies.is_active = true` in one transaction.

### Lead SLA State (side-effect, not a lead_stage transition)
- `sla_first_contact_due_at` is set in the same transaction as the `captured → assigned` stage transition (via `LeadService.setSlaDueAt`).
- No separate `lead_stage` transition is triggered by the SLA engine; escalation reassignment does trigger a `LeadService.assignOwner` mutator (which writes `stage_history` only if the stage also changes; reassignment within `assigned` stage does not produce a new `stage_history` row — only an `audit_logs(reassign)` + `event_outbox(LEAD_ASSIGNED)` entry).

### Transaction boundaries summary
| Write path | Entities written atomically |
|---|---|
| POST /admin/sla-policies | `sla_policies` + `configuration_versions` + `audit_logs` |
| Lead assignment (FR-030) | `leads(sla_first_contact_due_at)` + `stage_history` + `audit_logs` + `event_outbox` |
| KYC exception (FR-072) | `kyc_verifications(exception_sla_due_at)` + `audit_logs` |
| Grievance intake (FR-114) | `grievances(sla_due_at)` + `audit_logs` |
| Sweep breach + reassign | `leads(owner_id, version)` + `audit_logs(reassign)` + `event_outbox(LEAD_ASSIGNED, FIRST_CONTACT_BREACH)` |

All inside `UnitOfWork.run(async (tx) => { ... })`. Post-commit side effects (notifications) do not participate in the transaction.

## External Service Calls

FR-104 does not call external providers (LOS/KYC/Comms) directly. However:

- **`BusinessCalendarService.resolve(branchId?, regionId?)`** — internal `core/sla/` service; reads from `business_calendars` table via Kysely. No external call. Resolves: branch-specific calendar → region-specific calendar → org default (`is_default = true`). Falls back to a hardcoded Mon–Sat 09:30–18:30 IST calendar if no DB record found (logged as a warning).

- **`NotificationDispatchService.send(recipientUserId, eventCode, leadId, templateData)`** — internal `modules/engagement/` service. Calls `NotificationChannelPort` (SMS/email/WhatsApp) via `IntegrationGateway`. This is invoked **post-commit** (not inside the `UnitOfWork` transaction). If the notification send fails: the breach event is already committed and published via outbox; the notification failure is logged and the `communication_logs` row is written with `status = 'failed'`; `UPSTREAM_UNAVAILABLE` is **not** surfaced to the sweep caller (it is a background job).

└── StatusChip (pending | approved | active | rejected)
```

**Accessibility:** All form fields have `aria-label`; validation errors linked via `aria-describedby`. Drag-drop list uses keyboard-accessible pattern (arrow keys to reorder). `StatusChip` uses `role="status"`. WCAG 2.1 AA required.

**Mobile/PWA:** Drawer replaces full-page modal on < 768 px. `DataTable` collapses to card view on 360 px min viewport. Escalation chain editor stacks vertically.

**i18n:** All labels use `i18next` keys. Threshold rendered as formatted number; no currency.

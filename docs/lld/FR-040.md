# FR-040: Product Configuration Without Credit BRE — Low-Level Design

**Tier: 3** (Complex — multi-table transactional writes across `product_configs` + `configuration_versions` + `audit_logs` + `event_outbox`; maker-checker state machine with four status transitions; versioning invariant that must never mutate a pinned config row; ABAC `configuration/A`; builder UI with three tabbed editors plus version history.)

## Summary

FR-040 lets ADMIN and Product Ops staff create and maintain product configurations that drive capture-form field groups, mandatory document checklists, SLA thresholds, eligibility-payload mappings, rejection reasons, and the `pan_required_at` timing — all without code changes. It is explicitly **not** a credit BRE: pricing, LTV, FOIR, and sanction logic remain in LOS/BRE.

Every configuration change goes through a **maker-checker workflow** backed by `configuration_versions`. Activating a config sets the `product_configs.status` to `active`; editing an already-active config does **not** mutate the live row — it creates a new `draft` row plus a `configuration_versions` row in state `pending`, leaving existing leads safely pinned to the old `product_config_id`. Retiring a config does not touch in-flight leads.

The feature spans:
1. **List** — GET all product configs (paged, filterable by status/product_code).
2. **Create** — POST a new draft product config (maker step).
3. **Edit active** — PATCH an existing active config; produces a new draft version row.
4. **Approve/reject** — POST to approve or reject a pending configuration version (checker step).
5. **Activate** — automatic after approval; product_config.status flips to `active` in the same transaction.
6. **Retire** — PATCH status=retired on an active config (guarded so in-flight leads are unaffected).

## Auth Check

| Guard | Detail |
|---|---|
| `JwtAuthGuard` (global) | Validates JWT access token on every request; returns `AUTH_REQUIRED` (401) if absent/invalid/expired. |
| `AbacGuard` + `@Requires('configuration', scopeResolver)` | Calls `EntitlementService.can(user, 'configuration', { org_id })`. Only roles with `configuration` capability proceed. Returns `FORBIDDEN` (403) on deny. |
| Eligibility-mapping checker gate | At `POST /admin/config/{id}/approve`: if the pending `configuration_version.diff` includes an `eligibility_mapping` change, `EntitlementService.can(user, 'configuration', { org_id })` must return scope `A` (ADMIN or HEAD only). BM/KYC/DPO (`B` scope) may not approve eligibility-mapping changes → `FORBIDDEN` (403). |
| Self-approval guard | At approve: `checker_id` (current user) must not equal `maker_id` on the `configuration_versions` row. DB constraint `ck_config_maker_checker` backs this up; service layer enforces it first and returns `FORBIDDEN` (403) with a clear message. |

**Roles with `configuration` capability and their scopes (from auth-matrix.json):**

| Role | Scope | Can approve eligibility-mapping? |
|---|---|---|
| ADMIN | A | Yes |
| HEAD | A | Yes |
| BM | B | No |
| KYC | B | No |
| DPO | B | No |
| RM | — | No (no `configuration` capability) |
| PARTNER | — | No |

## Data Operations

All queries use the **Kysely** query builder with the generated DB types. No raw string SQL. Every list query has a `LIMIT`.

### List product configs

```typescript
// ProductConfigRepository.list(params, orgId)
db.selectFrom('product_configs')
  .where('org_id', '=', orgId)
  .$if(params.status !== undefined, (qb) => qb.where('status', '=', params.status!))
  .$if(params.product_code !== undefined, (qb) => qb.where('product_code', '=', params.product_code!))
  .select([
    'product_config_id', 'product_code', 'name', 'version', 'status',
    'pan_required_at', 'created_at', 'updated_at', 'created_by'
  ])
  .orderBy(allowListedSortColumn, 'desc')
  .limit(params.limit)
  .offset((params.page - 1) * params.limit)
  .execute();
```

### Get single (full detail)

```typescript
db.selectFrom('product_configs')
  .where('product_config_id', '=', id)
  .where('org_id', '=', orgId)
  .selectAll()
  .executeTakeFirst();
```

### Create new draft (POST — new product config)

Compute `version` as `MAX(version) + 1` for the same `(org_id, product_code)` within the transaction:

```typescript
// inside UnitOfWork.run(async (tx) => { ... })
const maxVersion = await tx
  .selectFrom('product_configs')
  .where('org_id', '=', orgId)
  .where('product_code', '=', input.product_code)
  .select(db.fn.max('version').as('max_v'))
  .executeTakeFirst();

const newVersion = (maxVersion?.max_v ?? 0) + 1;

const inserted = await tx
  .insertInto('product_configs')
  .values({
    org_id: orgId,
    product_code: input.product_code,
    name: input.name,
    version: newVersion,
    status: 'draft',
    field_schema: JSON.stringify(input.field_schema),
    document_checklist: JSON.stringify(input.document_checklist),
    sla_config: input.sla_config ? JSON.stringify(input.sla_config) : null,
    eligibility_mapping: input.eligibility_mapping ? JSON.stringify(input.eligibility_mapping) : null,
    pan_required_at: input.pan_required_at,
    created_by: userId,
    updated_by: userId,
  })
  .returning('product_config_id')
  .executeTakeFirstOrThrow();
```

Then insert `configuration_versions` row (still inside the same `tx`):

```typescript
await tx
  .insertInto('configuration_versions')
  .values({
    org_id: orgId,
    config_type: 'product_config',
    config_ref: inserted.product_config_id,
    version: newVersion,
    maker_id: userId,
    status: 'pending',
    diff: JSON.stringify({ action: 'create', product_code: input.product_code }),
    created_by: userId,
    updated_by: userId,
  })
  .executeTakeFirstOrThrow();
```

### Edit active config (PATCH — create new draft version)

The active config row is **not mutated**. Read the current active row, create a new draft row with `version = maxVersion + 1`, compute a diff for `configuration_versions.diff`:

```typescript
// inside UnitOfWork.run(async (tx) => { ... })
const existing = await tx
  .selectFrom('product_configs')
  .where('product_config_id', '=', targetId)
  .where('org_id', '=', orgId)
  .selectAll()
  .executeTakeFirstOrThrow();

// Guard: only active configs can be edited via PATCH to produce a new version
if (existing.status !== 'active') throw new ConflictException(/* CONFLICT */);

const newVersion = existing.version + 1;
const mergedConfig = mergeConfigFields(existing, input); // pure function; applies partial update

const newDraft = await tx
  .insertInto('product_configs')
  .values({
    ...mergedConfig,
    product_config_id: undefined, // let DB generate
    version: newVersion,
    status: 'draft',
    created_by: userId,
    updated_by: userId,
    created_at: undefined,
    updated_at: undefined,
  })
  .returning('product_config_id')
  .executeTakeFirstOrThrow();

const diffPayload = computeDiff(existing, mergedConfig); // tracks changed keys

await tx.insertInto('configuration_versions').values({
  org_id: orgId,
  config_type: 'product_config',
  config_ref: newDraft.product_config_id,
  version: newVersion,
  maker_id: userId,
  status: 'pending',
  diff: JSON.stringify(diffPayload),
  created_by: userId,
  updated_by: userId,
}).executeTakeFirstOrThrow();
```

### Approve config version (checker)

```typescript
// inside UnitOfWork.run(async (tx) => { ... })
const cv = await tx
  .selectFrom('configuration_versions')
  .where('configuration_version_id', '=', cvId)
  .where('org_id', '=', orgId)
  .selectAll()
  .executeTakeFirstOrThrow();

// Self-approval guard
if (cv.maker_id === userId) throw new ForbiddenException(/* FORBIDDEN */);
// State guard
if (cv.status !== 'pending') throw new ConflictException(/* CONFLICT */);
// Eligibility-mapping scope guard (if diff includes eligibility_mapping)
if (hasEligibilityMappingChange(cv.diff) && userScope !== 'A') throw new ForbiddenException(/* FORBIDDEN */);

const now = new Date();

// Transition: pending → approved
await tx.updateTable('configuration_versions')
  .set({ status: 'approved', checker_id: userId, effective_at: now, updated_by: userId })
  .where('configuration_version_id', '=', cvId)
  .execute();

// Activate: set product_config.status = active, previous active → retired
await tx.updateTable('product_configs')
  .set({ status: 'active', updated_by: userId })
  .where('product_config_id', '=', cv.config_ref!)
  .execute();

// Retire the previous active config for same product_code (if any, and different row)
const draftRow = await tx.selectFrom('product_configs')
  .where('product_config_id', '=', cv.config_ref!).selectAll().executeTakeFirstOrThrow();

await tx.updateTable('product_configs')
  .set({ status: 'retired', updated_by: userId })
  .where('org_id', '=', orgId)
  .where('product_code', '=', draftRow.product_code)
  .where('status', '=', 'active')
  .where('product_config_id', '!=', cv.config_ref!)
  .execute();

// Also transition ConfigurationVersion to active
await tx.updateTable('configuration_versions')
  .set({ status: 'active', updated_by: userId })
  .where('configuration_version_id', '=', cvId)
  .execute();
```

### Retire product config

```typescript
// Guard: confirm no draft/pending CV references this config (would orphan it)
// Note: in-flight leads are NOT affected; they keep their pinned product_config_id
await tx.updateTable('product_configs')
  .set({ status: 'retired', updated_by: userId })
  .where('product_config_id', '=', id)
  .where('org_id', '=', orgId)
  .where('status', '=', 'active')
  .execute();
```

## Backend Flow — Step by Step

### POST /admin/products (create new draft)

1. **JwtAuthGuard** validates token; loads user + ABAC attributes into request context.
2. **AbacGuard** calls `EntitlementService.can(user, 'configuration', { org_id })`. Fail → `FORBIDDEN` (403).
3. **ValidationPipe** runs `CreateProductConfigDto` (Zod) on request body. Fail → `VALIDATION_ERROR` (400) with `fields[]`.
4. **ProductConfigService.createDraft(dto, userId, orgId)** opens `UnitOfWork.run(async (tx) => { ... })`:
   a. Query `MAX(version)` for `(org_id, product_code)` — `newVersion = max + 1` (or 1 if none).
   b. Validate `eligibility_mapping.fields[*].lms_field` cross-references against `field_schema`. Fail → throw `VALIDATION_ERROR`.
   c. `INSERT INTO product_configs` (status=draft, version=newVersion).
   d. `INSERT INTO configuration_versions` (config_type='product_config', config_ref=new_id, status='pending', maker_id=userId, diff=create_action_json).
   e. `AuditAppender.emit(action='config_change', ...)` within `tx`.
   f. `OutboxService.emit(CONFIG_CHANGED, ...)` within `tx`.
   g. Commit.
5. Return 201 with `{ product_config_id, version, status:'draft', configuration_version_id, config_version_status:'pending' }`.

### PATCH /admin/products/{id} (edit active config → new draft)

1. Auth checks (same as POST).
2. **ValidationPipe** runs `UpdateProductConfigDto`.
3. **ProductConfigService.editActive(id, dto, userId, orgId)**:
   a. Fetch `product_configs` row for `id`; verify `org_id`. Not found → `NOT_FOUND` (404).
   b. If `dto.status === 'retired'` → jump to retire flow (step 6 below).
   c. Guard: `status` must be `active`. Fail → `CONFLICT` (409).
   d. Open `UnitOfWork.run(async (tx) => { ... })`:
      - Compute `mergedConfig` by overlaying `dto` fields onto existing row values.
      - Cross-validate `eligibility_mapping` against `field_schema` (merged). Fail → `VALIDATION_ERROR`.
      - Compute `diff` JSON of changed keys.
      - `INSERT INTO product_configs` (new row, version = existing.version + 1, status = 'draft').
      - `INSERT INTO configuration_versions` (config_ref = new_id, status = 'pending', maker_id = userId, diff).
      - `AuditAppender.emit` + `OutboxService.emit` within tx.
      - Commit.
4. Return 200 with new draft's `product_config_id`, `version`, `configuration_version_id`.

### POST /admin/config/{id}/approve (checker approves or rejects)

1. Auth checks.
2. **ValidationPipe** runs `ApproveConfigDto`.
3. **ProductConfigService.processCheckerDecision(cvId, dto, userId, orgId)**:
   a. Fetch `configuration_versions` row for `cvId`; verify `org_id`. Not found → `NOT_FOUND` (404).
   b. Status guard: must be `pending`. Fail → `CONFLICT` (409).
   c. Self-approval guard: `cv.maker_id === userId` → `FORBIDDEN` (403).
   d. If `action === 'rejected'`:
      - Open `UnitOfWork.run(async (tx) => { ... })`:
        - Update `configuration_versions.status = 'rejected'`, set `checker_id`, `updated_by`.
        - Update the linked `product_configs` row `status = 'retired'` (it was draft; no leads pinned to it).
        - `AuditAppender.emit` + `OutboxService.emit(CONFIG_CHANGED)` within tx.
      - Return 200 with `{ configuration_version_id, status:'rejected' }`.
   e. If `action === 'approved'`:
      - Eligibility-mapping scope guard: if `hasEligibilityMappingChange(cv.diff)` and `userScope !== 'A'` → `FORBIDDEN` (403).
      - Open `UnitOfWork.run(async (tx) => { ... })`:
        - `UPDATE configuration_versions SET status='approved', checker_id=userId, effective_at=now`.
        - Fetch the linked `product_configs` row (the new draft). Get its `product_code`.
        - Retire any current `active` config for the same `(org_id, product_code)` that is not the new row.
        - `UPDATE product_configs SET status='active'` for the new draft row.
        - `UPDATE configuration_versions SET status='active'` for the CV row (final settled state).
        - `AuditAppender.emit(action='config_change', ...)` within tx.
        - `OutboxService.emit(CONFIG_CHANGED, { product_code, version, status:'active' })` within tx.
      - Return 200 with `{ configuration_version_id, status:'active', product_config_id, product_config_status:'active', effective_at }`.

### PATCH /admin/products/{id} with status=retired (retire active config)

1. Auth checks.
2. Fetch and verify `org_id`; must be `status=active`. Fail → `CONFLICT` (409) or `NOT_FOUND` (404).
3. Open `UnitOfWork.run(async (tx) => { ... })`:
   a. `UPDATE product_configs SET status='retired'` for the target row.
   b. `AuditAppender.emit(action='config_change', operation:'retire')` within tx.
   c. `OutboxService.emit(CONFIG_CHANGED, { status:'retired' })` within tx.
4. Return 200 with `{ product_config_id, status:'retired' }`.

## State Machine — ConfigurationVersion + ProductConfig

### ConfigurationVersion (`config_change_status`)

```
pending ──(checker approves, self ≠ maker, scope-ok)──► approved ──(auto in same tx)──► active
   │
   └──(checker rejects)──► rejected
active ──(checker/ADMIN rollback)──► rolled_back
```

Invalid transitions → `CONFLICT` (409). Self-approval → `FORBIDDEN` (403). Scope insufficient for eligibility-mapping → `FORBIDDEN` (403).

### ProductConfig (`config_status`)

```
draft ──(CV moves to active)──► active ──(explicit retire)──► retired
```

**Invariant (never violated):** A `product_config` row that is referenced by any `leads.product_config_id` row must never have its own fields mutated. The PATCH handler only reads the active row and creates a new draft. The activate step writes the new draft row's status to `active`; it does not change any field on the previously active row (it only sets the old active row's `status` to `retired` — a status-only change that does not affect the data already pinned by leads).

### Transaction Boundaries

All mutating operations use one `UnitOfWork.run(async (tx) => { ... })` call. Every write (`product_configs`, `configuration_versions`, `audit_logs` via `AuditAppender`, `event_outbox` via `OutboxService`) uses the same `tx`. If any write fails, the entire transaction rolls back — no partial state persists.

## Assumptions

1. **Eligibility-mapping IT approval scope:** BRD §FR-040 AC-4 states "IT approval" for eligibility-mapping changes. The auth-matrix.json does not define a separate `IT` role. This LLD maps "IT checker" to roles with `configuration` capability at scope `A` (ADMIN and HEAD), which is the closest match. If a distinct `IT_APPROVER` role is introduced in the auth-matrix later, the guard condition must be updated.

2. **Version number strategy:** The BRD does not specify whether `version` is global (per org) or per product-code. This LLD uses `MAX(version) + 1` scoped to `(org_id, product_code)`. The unique constraint `uq_product_configs_version (org_id, product_code, version)` backs this.

3. **`diff` field content:** The schema does not specify the shape of `configuration_versions.diff`. This LLD stores a JSON object `{ action: 'create'|'update'|'retire', changed_keys: string[], before: { ... }, after: { ... } }` stripped of PII. A standard shape is assumed; if a different format is required, update this LLD.

4. **GET single product config endpoint:** The api-contract.yaml defines a generic `GET /admin/{masterResource}` for list only. This LLD assumes a `GET /admin/products/{id}` endpoint is implied for detail fetches (consistent with REST conventions and the UI requirement for the config detail page). This is listed as an ambiguity below.

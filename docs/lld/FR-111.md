**Tier: 2**

> **Note on endpoints:** The `api-contract.yaml` marks FR-111 as `"cross-cutting: data minimisation in services + data_sharing_logs (consents, eligibility, handoff)"`. There is no dedicated CRUD endpoint for creating sharing-log entries (they are inserted by the owning service — `DataSharingService` — within the `UnitOfWork` transaction of the triggering FR). The only user-facing endpoint is a read-only DPO oversight view. The endpoint path `GET /leads/{id}/sharing-logs` is inferred from the resource name and DPO oversight AC-5; see Ambiguities section.

## Summary

FR-111 implements data minimisation and resource-access controls for M12 Compliance. It has two complementary concerns:

**1. Schema-level data minimisation enforcement (passive, always-on):**
The data model enforces data minimisation by construction — no `raw_aadhaar` or biometrics column exists anywhere in the schema (enforced by the column-level comments in `schema.sql §5.6.9` and the `aadhaar_ref_token` tokenisation-only pattern). FR-111 codifies a `DataMinimisationService` that validates incoming field writes against the `field_schema` defined on `product_configs.field_schema` (JSONB), rejecting persistence of any `data_category` not declared as allowed for the configured purposes. This guard runs inside the `LeadService` and `DocumentService` write paths, not as a standalone endpoint.

**2. Third-party data-sharing audit (`data_sharing_logs`):**
Whenever a service shares lead data with a third party (LOS hand-off via FR-081, eligibility call via FR-080, KYC via FR-071, Account Aggregator via future FR), it must first verify that a `granted` `consent_record` exists for the relevant `consent_purpose` + `data_category`. On success, it inserts a `data_sharing_logs` row inside the caller's `UnitOfWork` transaction. Unconsented sharing attempts are rejected with `FORBIDDEN` (`CONSENT_MISSING`).

**3. DPO oversight view:**
DPO can query the `data_sharing_logs` for a given lead to audit all third-party sharing events — recipient, purpose, data category, consent reference, status, timestamp.

No device permissions (camera/mic/location) are managed server-side; those are governed by the PWA frontend permission prompts (one-time, purpose-bound, logged to `audit_logs`). See UI Component Tree section.

## Auth Check

### GET /leads/{id}/sharing-logs

1. `JwtAuthGuard` (global) validates the JWT; loads `user.role`, `user.scope`, and `user.abac_attributes` onto the request context. Returns `AUTH_REQUIRED` (401) if missing/expired.
2. `AbacGuard` + `@Requires('consent_ledger', scopeResolver)` calls `EntitlementService.can(user, 'consent_ledger', { leadId })`.
   - Only `DPO` has `consent_ledger` at scope `A` (all org, masked view) in `auth-matrix.json`.
   - All other roles have `consent_ledger` at narrower scopes (O/T/B/P) — they may query `consent_records` via FR-110 but do NOT have access to `data_sharing_logs`, which is DPO/compliance-only.
   - Callers without `consent_ledger` at scope `A` receive `FORBIDDEN` (403).
3. Existence check: if `lead_id` does not resolve to a non-deleted lead, return `NOT_FOUND` (404).

### Internal DataSharingService.logShare (no HTTP endpoint — called within UnitOfWork)

Before writing a `data_sharing_logs` row, `DataSharingService.logShare` verifies:
- A `consent_records` row for `(lead_id, purpose, data_category)` exists with `state = 'granted'` and `expires_at IS NULL OR expires_at > now()`.
- If no such record exists: throw `FORBIDDEN` with `detail.reason = 'CONSENT_MISSING'` — the outer service's `UnitOfWork` rolls back the entire operation.

## Data Operations

### 1. List data-sharing logs for a lead (repository read)

```typescript
// apps/api/src/modules/compliance/data-sharing-logs.repository.ts

async findByLeadId(
  leadId: string,
  pagination: { page: number; limit: number },
  db: Kysely<DB>,  // or transaction handle
): Promise<{ rows: DataSharingLog[]; total: number }> {
  const offset = (pagination.page - 1) * pagination.limit;

  const [rows, countResult] = await Promise.all([
    db
      .selectFrom('data_sharing_logs')
      .selectAll()
      .where('lead_id', '=', leadId)
      .orderBy('shared_at', 'desc')
      .limit(pagination.limit)
      .offset(offset)
      .execute(),
    db
      .selectFrom('data_sharing_logs')
      .select(db.fn.countAll<number>().as('count'))
      .where('lead_id', '=', leadId)
      .executeTakeFirstOrThrow(),
  ]);

  return { rows, total: Number(countResult.count) };
}
```

Column names used (exact, from `schema.sql §3.30`):
`data_sharing_log_id`, `org_id`, `lead_id`, `recipient`, `purpose`, `data_category`, `consent_id`, `status`, `shared_at`, `created_at`, `updated_at`, `created_by`, `updated_by`.

### 2. Insert a data-sharing log (service-internal, within UnitOfWork tx)

```typescript
// apps/api/src/modules/compliance/data-sharing.service.ts

async logShare(
  input: {
    leadId: string;
    recipient: string;
    purpose: ConsentPurpose;
    dataCategory: DataCategory;
    consentId: string | null;
    actorId: string;
  },
  tx: Transaction<DB>,
): Promise<void> {
  // Step 1: verify granted consent exists (if consentId provided, validate it; else search by purpose+category)
  const consent = await tx
    .selectFrom('consent_records')
    .select(['consent_id', 'state', 'expires_at'])
    .where('lead_id', '=', input.leadId)
    .where('purpose', '=', input.purpose)
    .where('data_category', '=', input.dataCategory)
    .where('state', '=', 'granted')
    .where((eb) =>
      eb.or([
        eb('expires_at', 'is', null),
        eb('expires_at', '>', new Date()),
      ])
    )
    .orderBy('created_at', 'desc')
    .limit(1)
    .executeTakeFirst();

  if (!consent) {
    throw new ForbiddenException({
      code: 'FORBIDDEN',
      message: "You don't have access to this.",
      retryable: false,
      detail: { reason: 'CONSENT_MISSING' },
    });
  }

  // Step 2: insert the sharing log row
  await tx
    .insertInto('data_sharing_logs')
    .values({
      lead_id: input.leadId,
      recipient: input.recipient,
      purpose: input.purpose,
      data_category: input.dataCategory,
      consent_id: input.consentId ?? consent.consent_id,
      status: 'shared',
      shared_at: new Date(),
      created_by: input.actorId,
      updated_by: input.actorId,
    })
    .execute();

  // Step 3: emit audit intent
  // NOTE: 'data_share_logged' is not in the audit_action enum. Use 'lead_update' as the closest
  // available value until the enum is extended. See Ambiguity #4.
  await this.auditAppender.emit(
    {
      action: 'lead_update',   // best-fit existing enum value; see Ambiguity #4
      entity_type: 'data_sharing_logs',
      entity_id: input.leadId,
      actor_id: input.actorId,
      detail: { recipient: input.recipient, purpose: input.purpose, dataCategory: input.dataCategory },
    },
    tx,
  );
}
```

### 3. Data minimisation guard (field-schema enforcement)

`DataMinimisationGuard` is a NestJS service called by `LeadService` and document-capture services before persisting custom field values. It reads the `field_schema` JSONB from `product_configs` and rejects any field whose `data_category` is not declared in the product's `allowed_categories` array.

```typescript
// apps/api/src/modules/compliance/data-minimisation.service.ts

async assertAllowed(
  productConfigId: string,
  incomingFields: Record<string, unknown>,
  db: Kysely<DB>,
): Promise<void> {
  const config = await db
    .selectFrom('product_configs')
    .select(['field_schema'])
    .where('product_config_id', '=', productConfigId)
    .executeTakeFirstOrThrow();

  const fieldSchema = config.field_schema as ProductFieldSchema;
  const disallowedFields = Object.keys(incomingFields).filter(
    (key) => !fieldSchema.allowedFields.includes(key),
  );

  if (disallowedFields.length > 0) {
    throw new BadRequestException({
      code: 'VALIDATION_ERROR',
      message: 'Please correct the highlighted fields.',
      retryable: false,
      fields: disallowedFields.map((f) => ({
        field: f,
        issue: 'Field not permitted for this product configuration.',
      })),
    });
  }
}
```

## Validation Logic

### GET /leads/{id}/sharing-logs

| Field | Rule | Error message |
|---|---|---|
| `id` (path) | UUID v4 format | "We couldn't find that item." (404, not a field error) |
| `page` | integer ≥ 1, default 1 | "Please correct the highlighted fields." |
| `limit` | integer 1–100, default 25 | "Please correct the highlighted fields." |

### DataSharingService.logShare (internal — validated before UnitOfWork insert)

| Field | Rule | Error |
|---|---|---|
| `recipient` | non-empty string, max 120 chars | `VALIDATION_ERROR` 400 |
| `purpose` | value in `consent_purpose` enum (`@shared/enums`) | `VALIDATION_ERROR` 400 |
| `dataCategory` | value in `data_category` enum (`@shared/enums`) | `VALIDATION_ERROR` 400 |
| Active granted consent | `consent_records` row with `state='granted'` and not expired | `FORBIDDEN` 403 + `detail.reason='CONSENT_MISSING'` |

### DataMinimisationService.assertAllowed (internal)

| Field | Rule | Error |
|---|---|---|
| Incoming field keys | Each key must appear in `product_configs.field_schema.allowedFields` | `VALIDATION_ERROR` 400 + `fields[]` listing disallowed fields |
| Raw Aadhaar | No field named `aadhaar_number`, `aadhaar_raw`, `biometric_*` permitted in any product schema | `VALIDATION_ERROR` 400 (enforced at schema definition time, not runtime — schema validation in M14 config) |

## Backend Flow

### GET /leads/{id}/sharing-logs

1. `CorrelationMiddleware` assigns `X-Correlation-Id` if absent.
2. `JwtAuthGuard` validates token; loads `user` onto request context.
3. `AbacGuard` + `@Requires('consent_ledger', scopeAResolver)` calls `EntitlementService.can(user, 'consent_ledger', { leadId: id })`. Denies if scope is not `A`.
4. Controller calls `DataSharingLogsService.listForLead(leadId, pagination)`.
5. Service calls `LeadsRepository.findById(leadId, db)` — if not found or `deleted_at IS NOT NULL`, throws `NotFoundException`.
6. Service calls `DataSharingLogsRepository.findByLeadId(leadId, pagination, db)`.
7. `ResponseEnvelopeInterceptor` wraps result in `{ data, meta, error: null }`.
8. Response returned.

### DataSharingService.logShare (called within UnitOfWork by FR-080/FR-081/FR-071)

1. Caller (e.g. `HandoffService`) opens `uow.run(async (tx) => { ... })`.
2. Inside the transaction, caller invokes `DataSharingService.logShare(input, tx)`.
3. `logShare` queries `consent_records` for a valid granted consent (scoped to `lead_id + purpose + data_category`).
4. If not found: throws `ForbiddenException` with `CONSENT_MISSING`. The `UnitOfWork` transaction rolls back entirely — the triggering operation (hand-off, eligibility, KYC) does not proceed.
5. If found: inserts `data_sharing_logs` row within the same `tx`.
6. Calls `AuditAppender.emit(...)` within same `tx` (audit intent queued to single-writer consumer).
7. Returns; caller continues its own UnitOfWork steps.

### DataMinimisationService.assertAllowed (called in LeadService write path)

1. `LeadService.create` or field-update mutator receives incoming `attributes` payload.
2. Calls `DataMinimisationService.assertAllowed(productConfigId, incomingFields, db)`.
3. Service reads `product_configs.field_schema` for the lead's `product_config_id`.
4. Compares incoming field keys against `field_schema.allowedFields`.
5. If any disallowed field present: throws `BadRequestException` with `VALIDATION_ERROR` + `fields[]`.
6. If all allowed: returns; caller proceeds with write.

## Transaction Boundaries

FR-111 does not own a standalone multi-entity write transaction. Instead:

- **`DataSharingService.logShare`** enlists in the **caller's** `UnitOfWork` transaction (e.g., the FR-081 hand-off UnitOfWork). The `data_sharing_logs` insert and `AuditAppender.emit` call both use the same `tx` handle. This is the "stated exception" from architecture §11.1: any owning service may `INSERT` into `data_sharing_logs` (an append-only sink) within the caller's transaction.
- If the `logShare` call throws (consent missing, insert fails), the caller's entire `UnitOfWork` rolls back — no partial state persists.
- The `GET /leads/{id}/sharing-logs` read endpoint runs outside a transaction (read-only Kysely query on the `db` instance).

**Device permission prompts (PWA, client-side only):**
The BRD AC-3 ("camera/mic/location one-time, purpose-bound, logged") is enforced entirely on the frontend. The PWA requests permissions only when the specific feature requiring them is activated (e.g. document photo capture, visit geo-tag). No permission is requested at startup. Each permission request is logged to `audit_logs` via the API with `action='device_permission_requested'`. This is not a server-enforced data minimisation control — it is a UX policy. No additional server endpoint is needed for this; it piggybacks on existing `AuditAppender` calls in the document/task write paths.

## Ambiguities

1. **`GET /leads/{id}/sharing-logs` not explicitly listed in `api-contract.yaml` paths.** The contract marks FR-111 as `"cross-cutting"` with no dedicated path entry. The path `GET /leads/{id}/sharing-logs` is the best-effort inference from the resource name and AC-5 (DPO oversight). If the contract is extended, the path/operationId must match exactly. A coding agent must confirm this with the DPO/product owner before finalising the endpoint registration.

2. **`ProductFieldSchema` type for `product_configs.field_schema`.** The `field_schema` column is `JSONB` — its internal shape (`allowedFields`, `allowedCategories`) is not defined in `schema.sql`. `DataMinimisationService.assertAllowed` uses `fieldSchema.allowedFields` as a `string[]`. The exact JSONB shape is the responsibility of FR-040 (M5 product config). The coding agent must confirm with FR-040's LLD that this property exists and is named `allowedFields`.

3. **Who calls `DataSharingService.logShare` for eligibility (FR-080) vs hand-off (FR-081) vs KYC (FR-071)?** The integration map says `data_sharing_logs` are written at eligibility call and hand-off. The calling service (e.g. `HandoffService`, `EligibilityService`, `KycService`) must inject and invoke `DataSharingService.logShare` within its own `UnitOfWork`. This LLD specifies the `logShare` contract; the caller's LLD is responsible for the invocation.

4. **`audit_action` enum does not contain a `data_share_logged` value.** The `audit_action` type in `schema.sql` does not include a value specific to data-sharing events. The best-effort value `'lead_update'` is used in this LLD pending a Flyway migration that adds `'data_share'` (or similar) to the enum. Before implementing, confirm with the data-model owner whether the enum should be extended or whether `'lead_update'` is accepted as a proxy for this audit event.

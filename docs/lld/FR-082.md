## Summary

FR-082 implements a **read-only LOS application status mirror**. The LMS receives status push events from the LOS via a signed webhook (`POST /los/webhooks/status`) and also reconciles missed events via a scheduled poll. On each inbound event the service:

1. Verifies the HMAC-SHA256 signature using `LOS_WEBHOOK_HMAC_SECRET`.
2. Looks up the lead using `leads.los_application_id`.
3. Upserts `los_application_mirrors` — keeping the row with the latest `status_date` (out-of-order protection).
4. Writes an `integration_logs` row (kind `los_status`, direction `inbound`).
5. Emits an audit intent via `AuditAppender`.

The lead stage is **not** changed by this FR; the mirror is display-only on the Lead 360 screen. A separate Cloud Scheduler job calls an internal reconciliation handler that polls the LOS for any mirror records whose `status_date` is stale.

## Endpoint

### Inbound webhook

```
POST /api/v1/los/webhooks/status
```

| Property | Value |
|---|---|
| Auth | `@Public()` — no JWT; HMAC signature in request header |
| Guard | `HmacGuard` (see Auth Check) |
| operationId | `losStatusWebhook` |
| Tags | `LOS` |
| Idempotency | payload field `event_id` deduplicated via `uq_integration_idempotency` unique index on `integration_logs.idempotency_key` |

**Request headers:**

| Header | Required | Description |
|---|---|---|
| `X-LOS-Signature` | Yes | `sha256=<hex-hmac>` of the raw request body using `LOS_WEBHOOK_HMAC_SECRET` |
| `X-Correlation-Id` | No | Forwarded or generated |
| `Content-Type` | Yes | `application/json` |

**Request body (Zod-validated):**

```json
{
  "event_id":         "evt_abc123",
  "los_application_id": "LOS-2026-00123",
  "status":           "CREDIT_APPRAISAL",
  "status_date":      "2026-06-09T10:30:00Z",
  "correlation_id":   "corr_xyz",
  "remarks":          "Moved to credit team"
}
```

| Field | Type | Rule |
|---|---|---|
| `event_id` | `string, min 1, max 120` | Idempotency key; deduplicated against `integration_logs.idempotency_key` |
| `los_application_id` | `string, min 1, max 64` | Must match a `leads.los_application_id` in the same org |
| `status` | `string, min 1, max 40` | LOS-owned string; stored verbatim (no enum constraint — LOS defines these) |
| `status_date` | `ISO-8601 datetime` | Used as the currency date for out-of-order comparison |
| `correlation_id` | `string, max 120, optional` | Propagated to `integration_logs.correlation_id` and log lines |
| `remarks` | `string, max 500, optional` | Informational; stored in `integration_logs.request_ref` |

**Success response (HTTP 200):**

```json
{ "data": null, "meta": { "correlation_id": "corr_…" }, "error": null }
```

The contract (`api-contract.yaml`) specifies `Empty` (HTTP 200) on success. No resource is returned — the LOS only needs an acknowledgement.

### Internal reconciliation (scheduled poll)

```
POST /api/v1/internal/los/reconcile   [Cloud Scheduler → service account; not exposed publicly]
```

This endpoint is **not** exposed in the public API contract. It is called by Cloud Scheduler (cron, configurable interval, default every 15 minutes) using a service-account-signed request. The handler queries `LosPort.pollApplicationStatuses()` for leads in stage `handed_off` whose mirror `status_date` is older than a configurable threshold (default 60 minutes), then processes each result as if it were a webhook delivery (same upsert logic, `received_via = 'poll'`).

## Auth Check

### Webhook endpoint

The endpoint is decorated `@Public()` to exempt it from `JwtAuthGuard`. Instead, a dedicated **`HmacGuard`** executes before the handler:

1. Reads the raw request body (Buffer, before JSON parse — preserve byte-for-byte exact body for HMAC computation).
2. Computes `HMAC-SHA256(rawBody, LOS_WEBHOOK_HMAC_SECRET)` using Node `crypto.timingSafeEqual`.
3. Compares the computed digest against the `X-LOS-Signature` header value (strips the `sha256=` prefix).
4. If mismatch or header absent: throws `ForbiddenException` → HTTP 403 `FORBIDDEN`. Logs `{ event: 'hmac_mismatch', correlation_id }` at `warn` level (no body content logged — no PII risk from webhook body is assumed but is not guaranteed; log at warn, not debug).
5. If match: passes through to the controller.

The auth-matrix classifies this endpoint as `service_to_service_only` with note "HMAC-signature verified (no user session)".

### Reconciliation endpoint

Protected by `@Requires()` with a service-account identity check (implementation detail: validate a GCP OIDC token from Cloud Scheduler, not a user JWT). Out of scope for user-facing ABAC. The endpoint must not be reachable from the public internet (Cloud Run ingress rule: internal-only for the `/internal/*` path prefix).

## Data Operations

All queries use the Kysely typed builder with exact column names from `docs/data-model/schema.sql`. No raw string SQL.

### 1. Idempotency check (before transaction)

```typescript
// Check for replay using the unique partial index uq_integration_idempotency
const existing = await db
  .selectFrom('integration_logs')
  .select(['integration_log_id', 'status'])
  .where('idempotency_key', '=', dto.event_id)
  .executeTakeFirst();

if (existing) {
  // IDEMPOTENT_REPLAY — return 200 immediately, no further writes
  return;
}
```

### 2. Lead lookup

```typescript
const lead = await db
  .selectFrom('leads')
  .select(['lead_id', 'org_id', 'stage', 'version', 'los_application_id'])
  .where('los_application_id', '=', dto.los_application_id)
  .where('deleted_at', 'is', null)
  .executeTakeFirst();
```

If not found: write `integration_logs` with `status = 'failed'` and `error_code = 'UNKNOWN_APP_ID'`, log at `warn`, return HTTP 200 (acknowledge to LOS; BRD says "unknown app id → log + ignore").

### 3. Upsert los_application_mirrors (inside UnitOfWork transaction)

```typescript
// Upsert: on conflict (los_application_id) update only if incoming status_date > existing status_date
await tx
  .insertInto('los_application_mirrors')
  .values({
    los_mirror_id:     generateUuid(),
    org_id:            lead.org_id,
    lead_id:           lead.lead_id,
    los_application_id: dto.los_application_id,
    status:            dto.status,
    status_date:       dto.status_date,
    correlation_id:    dto.correlation_id ?? correlationId,
    received_via:      receivedVia,           // 'webhook' | 'poll'
    created_by:        SYSTEM_USER_ID,
    updated_by:        SYSTEM_USER_ID,
  })
  .onConflict((oc) =>
    oc.column('los_application_id').doUpdateSet((eb) => ({
      status:       eb.ref('excluded.status'),
      status_date:  eb.ref('excluded.status_date'),
      correlation_id: eb.ref('excluded.correlation_id'),
      received_via: eb.ref('excluded.received_via'),
      updated_by:   SYSTEM_USER_ID,
      updated_at:   new Date(),
    }))
    // Conditional: only update if incoming status_date > current status_date
    .where(eb =>
      eb('excluded.status_date', '>', eb.ref('los_application_mirrors.status_date'))
    )
  )
  .execute();
```

Note: the `WHERE` clause on the `ON CONFLICT DO UPDATE` implements out-of-order protection — if the incoming `status_date` is older than the stored one, the update is a no-op. The row already existing with `status_date >= incoming` is not an error; HTTP 200 is returned.

### 4. Insert integration_logs (same transaction)

```typescript
await tx
  .insertInto('integration_logs')
  .values({
    integration_log_id: generateUuid(),
    org_id:             lead.org_id,
    integration:        'los_status',
    direction:          'inbound',
    lead_id:            lead.lead_id,
    correlation_id:     dto.correlation_id ?? correlationId,
    idempotency_key:    dto.event_id,
    request_ref:        dto.remarks ?? null,
    status:             'success',
    http_status:        200,
    completed_at:       new Date(),
    created_by:         SYSTEM_USER_ID,
    updated_by:         SYSTEM_USER_ID,
  })
  .execute();
```

### 5. Audit intent (same transaction)

```typescript
await auditAppender.append(
  {
    action:    'handoff_success',   // closest audit_action enum value for an inbound LOS event
    actor_id:  SYSTEM_USER_ID,
    lead_id:   lead.lead_id,
    detail:    {
      los_application_id: dto.los_application_id,
      new_status:         dto.status,
      status_date:        dto.status_date,
      received_via:       receivedVia,
    },
  },
  tx,
);
```

The `AuditChainConsumer` (single-writer, `max-instances=1`) computes the hash chain asynchronously — `AuditAppender.append()` writes the intent to `event_outbox` / a dedicated queue, not directly to `audit_logs`.

## Validation Logic

Implemented as a Zod schema in `los-status-webhook.dto.ts`. The controller's `ValidationPipe` runs it before the handler is reached.

| Field | Rule | Error message |
|---|---|---|
| `event_id` | `z.string().min(1).max(120)` | "event_id is required and must be at most 120 characters" |
| `los_application_id` | `z.string().min(1).max(64)` | "los_application_id is required and must be at most 64 characters" |
| `status` | `z.string().min(1).max(40)` | "status is required and must be at most 40 characters" |
| `status_date` | `z.string().datetime()` | "status_date must be a valid ISO-8601 datetime" |
| `correlation_id` | `z.string().max(120).optional()` | "correlation_id must be at most 120 characters" |
| `remarks` | `z.string().max(500).optional()` | "remarks must be at most 500 characters" |

Validation failures produce `VALIDATION_ERROR` (HTTP 400) with `fields[]` in the envelope. The HMAC check runs **before** the Zod parse — an invalid signature never reaches the parse step.

---

## Error Cases

| Scenario | Code | HTTP | detail.reason | Notes |
|---|---|---|---|---|
| Missing or bad `X-LOS-Signature` | `FORBIDDEN` | 403 | — | HmacGuard rejects before parse |
| Zod validation fails | `VALIDATION_ERROR` | 400 | — | fields[] carries field-level errors |
| `los_application_id` not found in `leads` | (none — 200 ack) | 200 | — | Log + ignore per BRD; integration_log written with `error_code='UNKNOWN_APP_ID'` |
| Duplicate `event_id` (idempotent replay) | (none — 200 ack) | 200 | `IDEMPOTENT_REPLAY` | No writes; original 200 returned |
| Out-of-order `status_date` (older than stored) | (none — 200 ack) | 200 | — | Upsert WHERE clause skips the update; treated as success |
| DB write fails (transaction rollback) | `INTERNAL_ERROR` | 500 | — | Full error logged with correlation_id; no stack in response; alert raised |
| LOS polling failure (reconcile path) | `UPSTREAM_UNAVAILABLE` | 503 | — | Cloud Tasks retry; circuit-breaker on repeated failure |

## Backend Flow (Step-by-step)

```
POST /api/v1/los/webhooks/status
  │
  ├─ 1. CorrelationMiddleware  — assigns/forwards X-Correlation-Id
  ├─ 2. HmacGuard              — HMAC-SHA256 verify (constant-time compare)
  │       FAIL → 403 FORBIDDEN (log warn, no body in log)
  ├─ 3. ValidationPipe (Zod)   — parse LosStatusWebhookDto
  │       FAIL → 400 VALIDATION_ERROR with fields[]
  │
  ├─ 4. LosWebhookController.receiveStatus(dto, correlationId)
  │       └─ calls LosService.processStatusUpdate(dto, 'webhook', correlationId)
  │
  ├─ 5. LosService.processStatusUpdate
  │       a. Idempotency check — SELECT from integration_logs WHERE idempotency_key = dto.event_id
  │          HIT  → return immediately (200, IDEMPOTENT_REPLAY log entry)
  │          MISS → continue
  │       b. Lead lookup — SELECT from leads WHERE los_application_id = dto.los_application_id
  │          NOT FOUND → write integration_log(failed, UNKNOWN_APP_ID) + return 200
  │          FOUND     → continue
  │
  ├─ 6. UnitOfWork.run(async (tx) => {
  │         i.  INSERT/UPDATE los_application_mirrors (upsert with status_date guard)
  │        ii.  INSERT integration_logs (idempotency_key = dto.event_id, status='success')
  │       iii.  AuditAppender.append(handoff_success intent, tx)
  │       })
  │       ERROR → transaction rollback → 500 INTERNAL_ERROR (alert raised)
  │
  └─ 7. Return { data: null, meta: { correlation_id }, error: null } HTTP 200
```

### Reconciliation (scheduled poll) flow

```
Cloud Scheduler → POST /api/v1/internal/los/reconcile
  │
  ├─ 1. Service-account OIDC verification (not user JWT)
  ├─ 2. LosService.reconcile()
  │       a. SELECT leads WHERE stage = 'handed_off'
  │          AND los_application_id IS NOT NULL
  │          AND (los_application_mirrors does not exist
  │               OR los_application_mirrors.status_date < now() - RECONCILE_STALE_THRESHOLD)
  │          LIMIT 100  (NFR-17 — no unbounded query)
  │       b. For each lead: call LosPort.getApplicationStatus(los_application_id)
  │          → on success: call LosService.processStatusUpdate(result, 'poll', correlationId)
  │          → on failure: write integration_log(status='failed') + schedule Cloud Tasks retry
  └─ 3. Return summary count; errors do not abort the batch
```

## State Machine & Transaction Boundaries

Per `docs/contracts/state-machines.md`:

> `LOSApplicationMirror` — mirror of LOS status; keep latest by `status_date`; never edited by LMS.

FR-082 does **not** transition the `lead_stage` state machine. The lead remains in `handed_off` (terminal in LMS). Only the `los_application_mirrors` row is updated.

**Transaction boundary (single UnitOfWork):**

```
UnitOfWork.run(tx)
  ├── INSERT/UPDATE los_application_mirrors
  ├── INSERT integration_logs
  └── AuditAppender.append (intent → event_outbox / audit queue)
```

All three writes succeed or all roll back. The `IntegrationLog` record uses the `event_id` as `idempotency_key`, protected by the `uq_integration_idempotency` unique partial index — a concurrent duplicate delivery will hit a DB constraint and be caught as an idempotent replay.

## External Service Calls via IntegrationGateway

### Polling path (LosPort.getApplicationStatus)

```typescript
// Called only during reconciliation, not during webhook processing
const result = await integrationGateway.call(
  losPort,
  { operation: 'getApplicationStatus', los_application_id },
  { idempotencyKey: `los-poll-${lead_id}-${runId}` },
);
```

| Failure | Behaviour |
|---|---|
| LOS 5xx / timeout | `IntegrationLog(status=failed)` + Cloud Tasks retry (exp backoff, max 3) |
| Circuit-breaker open | Reconcile job skips + logs; alert raised |
| Mapping missing | `VALIDATION_ERROR` (config) — block with structured message |

Webhook processing itself does **not** call the LOS outbound — it only processes the inbound push. The `IntegrationGateway` is not used for webhook receipt (no outbound call needed).

## UI Component Tree (Lead 360 — LOS Status Panel)

FR-082 has a **read-only** UI component on the Lead 360 screen (module M6 workspace). No write actions from the UI.

```
Lead360Page (apps/web/src/components/workspace/Lead360Page.tsx)
  └── LosStatusPanel (apps/web/src/components/los/LosStatusPanel.tsx)
        ├── PageHeader (shared) — title "LOS Application Status"
        │     badge: "LOS-owned · Read-only"
        ├── LoadingSkeleton (shared) — while query is loading
        ├── EmptyState (shared) — if no mirror record exists ("No LOS application linked")
        ├── ErrorState (shared) — on query error
        └── LosStatusTimeline (apps/web/src/components/los/LosStatusTimeline.tsx)
              ├── StatusChip (shared) — current status value
              ├── <dl> metadata grid
              │     ├── LOS Application ID: {los_application_id}
              │     ├── Last Updated: {status_date} (IST, dd-MM-yyyy HH:mm)
              │     ├── Source: {received_via} ("Webhook" | "Reconciliation poll")
              │     └── Correlation ID: {correlation_id}
              └── Timeline list — all mirror history rows, newest first
                    Each entry: StatusChip + status_date + received_via
```

**Data fetch:** `GET /api/v1/leads/{id}/los-status` (read endpoint served by M6 workspace — reads `los_application_mirrors WHERE lead_id = ?` ordered by `status_date DESC LIMIT 25`). This read endpoint is governed by `view_lead` capability via `AbacGuard`. The webhook endpoint (`POST /los/webhooks/status`) is **not** called from the UI.

**Roles with access:** RM (scope O), BM (scope B), SM (scope T), HEAD (scope A), KYC (scope B). PARTNER and CUSTOMER cannot see the LOS panel.

**Accessibility:** `StatusChip` uses `role="status"` + `aria-label`. Timeline list uses `<ul>` with `<li>` elements. All interactive states (loading/empty/error) use the shared primitives.

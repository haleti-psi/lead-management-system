**Tier: 3**


## Summary

FR-072 provides the managed exception resolution flow for KYC verifications that are in `exception` status — rows created by FR-071 when a provider returns a mismatch, expired/unreadable document, CKYC unavailability, V-CIP failure, or provider downtime.

The sole mutating endpoint is `PATCH /leads/{id}/kyc/{kid}/resolve`. It:

1. Validates the caller (KYC or BM, branch-scoped).
2. Confirms the target `KYCVerification` is in `exception` status.
3. Validates the `resolution_code` and conditional evidence rules (evidence required for waiver and provider-downtime fallback; provider-downtime fallback additionally requires the compliance flag).
4. Updates `kyc_verifications` (status, resolution_code, updated_by/at).
5. Recomputes `leads.kyc_status` via `LeadService.setKycStatus`.
6. Emits `AuditAppender` intent (action `kyc_exception`, detail masked).
7. Emits `KYC_EXCEPTION` event into `event_outbox`.

All five writes execute inside a single `UnitOfWork` transaction. On resolution, if no other open exceptions remain for this lead, `leads.kyc_status` transitions to `verified` (or `waived` if the resolution was a waiver), which unblocks the FR-081 hand-off guard.

The FR also drives the read path: the KYC Workbench exception queue is served by the existing `GET /leads/{id}/kyc/{type}` list (FR-071 scope) filtered to `status = exception` — no new GET endpoint is defined for FR-072.

## Endpoint

### `PATCH /api/v1/leads/{id}/kyc/{kid}/resolve`

Resolves an open KYC exception on a specific `KYCVerification` record.

#### Path parameters
| Param | Type | Required | Description |
|---|---|---|---|
| `id` | UUID | Yes | `leads.lead_id` |
| `kid` | UUID | Yes | `kyc_verifications.kyc_verification_id` |

#### Request body (JSON)
```json
{
  "resolutionCode": "string",
  "remarks": "string",
  "evidenceRef": "string | null"
}
```

| Field | Type | Required | Constraints |
|---|---|---|---|
| `resolutionCode` | string | Yes | Must be one of the allowed codes (see Validation Logic) |
| `remarks` | string | Yes | 1–1000 characters |
| `evidenceRef` | string \| null | Conditional | Required when `resolutionCode` is `waiver` or `provider_down_manual` |

#### Response — 200 OK
```json
{
  "data": {
    "kycVerificationId": "uuid",
    "leadId": "uuid",
    "kycType": "pan",
    "status": "resolved",
    "exceptionType": "pan_mismatch",
    "resolutionCode": "re_verified",
    "updatedAt": "2026-06-09T10:00:00Z"
  },
  "meta": { "correlation_id": "corr_..." },
  "error": null
}
```

#### Error responses
| HTTP | code | detail.reason | Condition |
|---|---|---|---|
| 401 | `AUTH_REQUIRED` | — | Missing/invalid JWT |
| 403 | `FORBIDDEN` | — | Caller not KYC or BM, or out of branch scope |
| 403 | `FORBIDDEN` | — | `resolutionCode` = `provider_down_manual` and compliance flag not enabled |
| 404 | `NOT_FOUND` | — | `lead_id` not found / out of scope, or `kyc_verification_id` not found |
| 400 | `VALIDATION_ERROR` | — | Field validation failure (fields[] populated) |
| 409 | `CONFLICT` | — | `KYCVerification.status` is not `exception` (e.g. already `resolved` or `waived`) |
| 429 | `RATE_LIMITED` | — | Mutation rate limit exceeded (60/min per user) |
| 500 | `INTERNAL_ERROR` | — | Unhandled server error |

## Auth Check

**Step 1 — Authentication:** `JwtAuthGuard` (globally applied). Validates Bearer JWT; populates `req.user` with `{ userId, role, branchId, orgId, … }`. Unauthenticated → `AUTH_REQUIRED` (401).

**Step 2 — Authorization:** `@Requires('kyc_signoff', scopeResolver)` → `AbacGuard` → `EntitlementService.can(user, 'kyc_signoff', resource)`.

From `auth-matrix.json`:
- `KYC.kyc_signoff` = scope `B` (branch)
- `BM.kyc_signoff` = scope `B` (branch)

The `scopeResolver` extracts `branchId` from the `leads` row. The entitlement check verifies `user.branchId === lead.branch_id`. Out-of-scope → `FORBIDDEN` (403).

**Step 3 — Compliance flag (provider_down_manual):** If `resolutionCode === 'provider_down_manual'`, the service reads the org-level compliance configuration to verify the `kyc_manual_fallback_enabled` flag. If not set → `FORBIDDEN` (403). This is a post-auth business rule, not a guard, because it depends on request body data.

> **Ambiguity A-1:** The `kyc_manual_fallback_enabled` compliance flag is not mapped to a specific column in any table in `schema.sql`. The best-effort interpretation is that it is stored in `product_configs.sla_config` (JSONB) or as an org-level configuration row. This must be resolved with the product/data-model owner and backfilled into the relevant artefact before the coding agent implements this check. The coding agent must NOT invent a column.

## Data Operations

All Kysely queries use parameterised bindings. No raw string interpolation.

### 1. Fetch the lead (scope check)

```typescript
// apps/api/src/modules/kyc/kyc.repository.ts
const lead = await db
  .selectFrom('leads')
  .select(['lead_id', 'branch_id', 'kyc_status', 'version', 'org_id'])
  .where('lead_id', '=', leadId)
  .where('org_id', '=', user.orgId)
  .where('deleted_at', 'is', null)
  .executeTakeFirst();
// null → NOT_FOUND (404)
// branch_id mismatch after EntitlementService → FORBIDDEN (403)
```

### 2. Fetch the KYCVerification record

```typescript
const kyc = await db
  .selectFrom('kyc_verifications')
  .selectAll()
  .where('kyc_verification_id', '=', kycVerificationId)
  .where('lead_id', '=', leadId)
  .where('org_id', '=', user.orgId)
  .executeTakeFirst();
// null → NOT_FOUND (404)
// kyc.status !== 'exception' → CONFLICT (409)
```

### 3. Update the KYCVerification (within UnitOfWork transaction)

```typescript
await db
  .updateTable('kyc_verifications')
  .set({
    status:          'resolved',   // or 'waived' if resolutionCode is waiver
    resolution_code: dto.resolutionCode,
    updated_by:      user.userId,
    updated_at:      new Date(),
  })
  .where('kyc_verification_id', '=', kycVerificationId)
  .where('status', '=', 'exception')   // optimistic guard
  .execute();
```

The `WHERE status = 'exception'` clause acts as a lightweight concurrency guard: if a concurrent request already resolved the record, zero rows are updated and the service throws `CONFLICT` (409).

### 4. Check for remaining open exceptions (within same tx)

```typescript
const openCount = await db
  .selectFrom('kyc_verifications')
  .select(db.fn.count('kyc_verification_id').as('n'))
  .where('lead_id', '=', leadId)
  .where('status', '=', 'exception')
  .where('org_id', '=', user.orgId)
  .executeTakeFirstOrThrow();
```

If `openCount.n === 0`, call `LeadService.setKycStatus(leadId, derivedStatus, tx)` where `derivedStatus` is:
- `'verified'` if at least one `kyc_verifications` row for the lead is `success` and none are `failed`/`exception`
- `'waived'` if the resolved entry used a waiver code and no other exceptions remain
- Otherwise leave `kyc_status` unchanged (edge: other records still in `initiated`/`failed`)

> **Ambiguity A-2:** The precise derivation rule for `leads.kyc_status` after exception resolution (when multiple `kyc_verifications` rows exist with mixed statuses) is not fully specified by FR-072 alone; it depends on the overall KYC readiness logic owned by FR-071. Best-effort: status is `verified` when all mandatory checks are `success` or `waived`, `exception` when any are `exception`, `in_progress` otherwise. The coding agent must not hard-code this logic independently — it must call `LeadService.setKycStatus` which owns the derivation.

### 5. Emit outbox event (within same tx)

```typescript
await outboxService.emit(
  { eventCode: 'KYC_EXCEPTION', aggregateType: 'KYCVerification', aggregateId: kycVerificationId,
    payload: { leadId, kycVerificationId, resolutionCode: dto.resolutionCode, resolvedBy: user.userId } },
  tx
);
```

### 6. Emit audit intent (within same tx)

```typescript
await auditAppender.append(
  { action: 'kyc_exception', entityType: 'kyc_verifications', entityId: kycVerificationId,
    leadId, actorId: user.userId,
    detail: { resolutionCode: dto.resolutionCode, remarks: '[present]' } }, // never log raw remarks text if PII
  tx
);
```

`AuditChainConsumer` serialises the actual hash-chain write outside the tx.

## Validation Logic (Zod)

All validation runs at the controller boundary via `ValidationPipe(Zod)` before the service layer is invoked.

```typescript
// apps/api/src/modules/kyc/dto/resolve-kyc-exception.dto.ts
import { z } from 'zod';

const ALLOWED_RESOLUTION_CODES = [
  're_verified',          // re-verification passed
  'document_replaced',    // customer replaced document
  'name_variance_waiver', // minor name variance accepted
  'address_variance_waiver',
  'waiver',               // authorised full waiver (requires evidenceRef)
  'provider_down_manual', // manual capture during provider downtime (requires evidenceRef + compliance flag)
  'ckyc_manual_capture',  // manual CKYC alternative
  'duplicate_ckyc_resolved',
  'vcip_retaken',         // V-CIP retaken successfully
] as const;

export const ResolveKycExceptionDto = z.object({
  resolutionCode: z.enum(ALLOWED_RESOLUTION_CODES),
  remarks:        z.string().min(1).max(1000),
  evidenceRef:    z.string().max(255).nullable().optional(),
}).superRefine((data, ctx) => {
  if (
    (data.resolutionCode === 'waiver' || data.resolutionCode === 'provider_down_manual') &&
    !data.evidenceRef
  ) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      path: ['evidenceRef'],
      message: 'evidenceRef is required for waiver and provider_down_manual resolution codes.',
    });
  }
});
```

> **Ambiguity A-3:** The exhaustive list of allowed `resolution_code` values is not enumerated in `schema.sql` (the column is `VARCHAR(40)`, not an enum type) or in any contracts artefact. The list above is a best-effort derivation from the BRD §5.2.31 `kyc_exception` enum values. The product/compliance owner must confirm and backfill this list into `error-taxonomy.md` or a new shared enum in `@shared/enums` before the coding agent finalises the DTO.

| Field | Rule | Error message |
|---|---|---|
| `resolutionCode` | Required; must be one of `ALLOWED_RESOLUTION_CODES` | `"Resolution code is not valid."` |
| `remarks` | Required; 1–1000 chars | `"Remarks are required (max 1000 characters)."` |
| `evidenceRef` | Required when `resolutionCode` ∈ `{waiver, provider_down_manual}` | `"Evidence reference is required for waiver and provider downtime manual resolution."` |
| Path `id` | Valid UUID | `"Lead ID must be a valid UUID."` |
| Path `kid` | Valid UUID | `"KYC verification ID must be a valid UUID."` |

## Backend Flow (Step-by-Step)

```
PATCH /api/v1/leads/{id}/kyc/{kid}/resolve
```

1. **CorrelationMiddleware** — attaches or generates `X-Correlation-Id` on request and response.

2. **JwtAuthGuard** — validates access token; loads `{ userId, role, branchId, orgId }` onto `req.user`. If invalid → 401.

3. **AbacGuard / @Requires('kyc_signoff', scopeResolver)** — calls `EntitlementService.can(user, 'kyc_signoff', { branchId: lead.branch_id })`. Requires scope B; caller must share `branchId` with the lead. If denied → 403.

4. **ValidationPipe (Zod)** — parses path params (UUID format) and body against `ResolveKycExceptionDto`. Failure → 400 with `fields[]`.

5. **KycExceptionController.resolveException(id, kid, dto, user)** — delegates to `KycExceptionService.resolve(...)`.

6. **KycExceptionService.resolve(leadId, kycVerificationId, dto, user)**:

   a. **Fetch lead** — `KycRepository.findLead(leadId, orgId)`. Not found → 404.

   b. **Fetch KYCVerification** — `KycRepository.findVerification(kycVerificationId, leadId, orgId)`. Not found → 404.

   c. **Check status** — if `kyc.status !== 'exception'` → throw `CONFLICT` (409).

   d. **Compliance flag check** — if `dto.resolutionCode === 'provider_down_manual'`, read org-level config for `kyc_manual_fallback_enabled`. If false → throw `FORBIDDEN` (403).

   e. **Open UnitOfWork transaction**:
      ```
      uow.run(async (tx) => {
        // i.  Update kyc_verifications
        // ii. Recount open exceptions
        // iii. Conditionally call LeadService.setKycStatus
        // iv. Emit KYC_EXCEPTION outbox event
        // v.  Emit audit intent via AuditAppender
      })
      ```

   f. **Update `kyc_verifications`** (step 3 of Data Operations). Zero rows updated → `CONFLICT` (409).

   g. **Recount open exceptions** (step 4 of Data Operations). If count = 0 → call `LeadService.setKycStatus(leadId, derivedStatus, tx)`. `LeadService` writes `leads.kyc_status` and a `stage_history` row if the change triggers a stage unblock; all within the same `tx`.

   h. **Outbox emit** — `OutboxService.emit(KYC_EXCEPTION, ...)` within `tx`.

   i. **Audit intent** — `AuditAppender.append(kyc_exception, ...)` within `tx`.

   j. **Commit** — if any step throws, the UnitOfWork rolls back atomically. No partial state persists.

7. **ResponseEnvelopeInterceptor** — wraps the service return value in `{ data, meta, error: null }`.

8. **Notification side-effect (post-commit, non-blocking)** — `NotificationDispatchService.send(lead, 'KYC_EXCEPTION', 'in_app')` for RM/BM. This is a fire-and-forget Cloud Tasks dispatch; failure does not roll back the resolution.

## State Machine and Transaction Boundaries

### KYCVerification state machine (from `state-machines.md`)

```
initiated ──► success
         └──► failed ──► exception ──► resolved   ← FR-072 (resolution + evidence / KYC/BM)
                                   └──► waived     ← FR-072 (authorised waiver / KYC/BM)
```

**Invalid transitions** (→ `CONFLICT` 409):
- `resolved → exception` (re-opening a resolved record is not permitted; create a new verification)
- `waived → exception` (same)
- `initiated → resolved` (must pass through `exception`)
- `success → resolved` (already successful; no exception exists)

**Hand-off guard dependency** (from `state-machines.md`):
- `ready_for_handoff → handed_off` requires that no `KYCVerification` row for the lead has `status = 'exception'`. If any remain open, FR-081 returns `CONFLICT` with `detail.reason = KYC_EXCEPTION_OPEN`.

### Transaction boundary (UnitOfWork)

```
BEGIN
  UPDATE kyc_verifications SET status='resolved', resolution_code=... WHERE ... AND status='exception'
  SELECT count(*) FROM kyc_verifications WHERE lead_id=? AND status='exception'
  -- if count=0:
    UPDATE leads SET kyc_status=<derived>, version=version+1 ... (via LeadService.setKycStatus)
  INSERT INTO event_outbox (event_code='KYC_EXCEPTION', ...) 
  -- audit intent queued for AuditChainConsumer
COMMIT
```

If any statement fails → full rollback. `audit_logs` is written post-commit by the single-writer `AuditChainConsumer` from the queued intent; it is not in-transaction.

## Assumptions

1. `kyc_manual_fallback_enabled` compliance configuration exists at the org or product level (Ambiguity A-1 — exact column/table not in schema.sql; must be resolved before coding).
2. The `resolution_code` values are not a Postgres enum; the controller enforces allowed values via the Zod DTO. The list must be confirmed by compliance/product (Ambiguity A-3).
3. `leads.kyc_status` derivation logic lives exclusively in `LeadService.setKycStatus`; FR-072 only calls this method and does not replicate the derivation logic.
4. The KYC Workbench exception queue (list view) is served by FR-071's existing `GET /leads/{id}/kyc/{type}` endpoint filtered to `status=exception`; no new GET endpoint is in scope for FR-072.
5. SLA timer (`exception_sla_due_at`) is set by FR-071 when the exception is first created; FR-072 reads it for display but does not mutate it.
6. Escalation on SLA breach is handled by the `SlaEngine` background sweep (FR-104), not by this FR directly.

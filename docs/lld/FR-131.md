## Summary

FR-131 provides CRUD management for every master/config entity in the NBFC LMS. ADMIN and authorised Product Ops roles can create, list, and update: regions, branches, teams, partners, schemes, product configs, rejection reasons, allocation rules, SLA policies, business calendars, communication templates, DLA registry entries, and retention policies. All writes are funnelled through the FR-132 `ConfigurationVersion` governance layer (maker-checker pending → approved → active). Active masters may not be deleted; they must be deactivated (`is_active = false` or status change). Any change emits a `CONFIG_CHANGED` outbox event. The three canonical endpoints in `api-contract.yaml` use a `{masterResource}` path parameter to dispatch to the correct entity handler within a single `AdminController`.

### POST /api/v1/admin/{masterResource}

Create a new master record. Produces a `configuration_versions` row (`status=pending`) for FR-132 maker-checker flow. The master row itself is inserted in `draft`/`is_active=false` state; it becomes live only once the `configuration_versions` row is approved (FR-132). For simple entities without maker-checker enforcement (e.g. `regions`, `branches`) the record is created active immediately if the org has not enabled high-impact maker-checker (see ambiguities).

**Request:**
```
POST /api/v1/admin/rejection-reasons
Authorization: Bearer <jwt>
Content-Type: application/json

{
  "primaryReason": "out_of_area",
  "subReason": "pin not serviceable",
  "requiresRemarks": true
}
```

**Response 201:**

## Auth Check

| Guard | Detail |
|---|---|
| `JwtAuthGuard` (global) | Applied automatically; returns `AUTH_REQUIRED` (401) if token missing or expired |
| `AbacGuard` + `@Requires('configuration', scopeResolver)` | `EntitlementService.can(user, 'configuration', resource)` must return true |

**Scope rules from auth-matrix.json:**
- `ADMIN`: scope `A` (all org) — full CRUD on all master resource types.
- `HEAD`: scope `A` on `configuration` — full CRUD; no lead-data access.
- `BM`: scope `B` — can manage masters scoped to their own branch (e.g. branch-linked business_calendars, local SLA).
- `KYC`: scope `B` — configuration at branch level.
- `DPO`: scope `B` — configuration at branch level.
- `RM`, `SM`, `PARTNER`, `CUSTOMER`: no `configuration` capability → `FORBIDDEN` (403).

`scopeResolver` for `PATCH` additionally checks that BM/KYC/DPO actors only modify records that belong to their branch scope (e.g. a `business_calendar` with `branch_id = actor.branch_id`). Global records (no `branch_id`/`region_id`) require scope `A`.

## Data Operations (Kysely)

All queries use the Kysely `Db` instance from `core/db/`. Column names match `schema.sql` exactly.

### List (GET)

```typescript
// Example: rejection_reasons
const rows = await db
  .selectFrom('rejection_reasons')
  .selectAll()
  .where('org_id', '=', orgId)
  .$if(filterIsActive !== undefined, (qb) =>
    qb.where('is_active', '=', filterIsActive)
  )
  .orderBy('created_at', 'desc')
  .limit(limit)           // max 100; default 25
  .offset((page - 1) * limit)
  .execute();

const total = await db
  .selectFrom('rejection_reasons')
  .select(db.fn.countAll<number>().as('total'))
  .where('org_id', '=', orgId)
  .$if(filterIsActive !== undefined, (qb) =>
    qb.where('is_active', '=', filterIsActive)
  )
  .executeTakeFirstOrThrow();
```

Per-resource list queries follow the same pattern on their respective table. Allowed filter fields are allowlisted in `MasterResourceRegistry` (see File Locations).

### Create (POST) — within UnitOfWork transaction

```typescript
// Step 1: Insert master record
const record = await db
  .insertInto('rejection_reasons')
  .values({
    org_id: orgId,
    primary_reason: dto.primaryReason,
    sub_reason: dto.subReason ?? null,
    requires_remarks: dto.requiresRemarks,
    is_active: true,
    created_by: actorId,
    updated_by: actorId,
  })
  .returningAll()
  .executeTakeFirstOrThrow();

// Step 2: Insert configuration_versions row (maker-checker tracking)
const configVersion = await db
  .insertInto('configuration_versions')
  .values({
    org_id: orgId,
    config_type: 'rejection_reason',
    config_ref: record.rejection_reason_id,
    version: 1,
    maker_id: actorId,
    checker_id: null,
    status: 'pending',  // or 'active' for low-impact types per org setting
    diff: JSON.stringify({ op: 'create', after: record }),
    created_by: actorId,
    updated_by: actorId,
  })
  .returningAll()
  .executeTakeFirstOrThrow();

// Step 3: Emit CONFIG_CHANGED to event_outbox (same tx via OutboxService)
await outboxService.emit(
  { event_code: 'CONFIG_CHANGED', aggregate_type: 'rejection_reason', aggregate_id: record.rejection_reason_id, payload: { op: 'create', configType: 'rejection_reason' } },
  tx
);

// Step 4: Emit audit intent via AuditAppender
await auditAppender.emit({
  actor_id: actorId,
  action: 'config_change',
  entity_type: 'rejection_reason',
  entity_id: record.rejection_reason_id,
  detail: { op: 'create', config_version_id: configVersion.configuration_version_id },
}, tx);
```

### Update (PATCH) — within UnitOfWork transaction

```typescript
// Step 1: Load existing record; 404 if not found
const existing = await db
  .selectFrom('rejection_reasons')
  .selectAll()
  .where('rejection_reason_id', '=', id)
  .where('org_id', '=', orgId)
  .executeTakeFirst();
if (!existing) throw new NotFoundException();

// Step 2: In-use check for deactivation
if (dto.isActive === false) {
  await masterResourceRegistry.assertNotInUse('rejection_reason', id, db);
}

// Step 3: Apply update
const updated = await db
  .updateTable('rejection_reasons')
  .set({ ...patchFields, updated_by: actorId, updated_at: new Date() })
  .where('rejection_reason_id', '=', id)
  .where('org_id', '=', orgId)
  .returningAll()
  .executeTakeFirstOrThrow();

// Step 4: configuration_versions diff row (same pattern as create, op: 'update', before/after diff)
// Step 5: event_outbox CONFIG_CHANGED + audit (same as create)
```

### In-use check (deactivation guard)

Each resource type has a dedicated referential-integrity check. Examples:
- `rejection_reasons`: query `leads` where `rejection_reason_id = id` and `stage NOT IN ('handed_off','rejected')` — if count > 0, block with CONFLICT.
- `sla_policies`: query `leads` where `sla_policy_id = id` (if such column exists) — or check active SLA timer rows referencing the policy.
- `allocation_rules`: check `allocation_rules.is_active = true` referencing this rule, plus active leads in allocation queue.
- `branches`: check `users.branch_id = id` where `users.status = 'active'` — if count > 0, cannot deactivate.
- `business_calendars`: check `BusinessCalendarService` usage count (no direct FK; query via `SlaEngine` config).
- `communication_templates`: check `communication_logs` for in-flight sends referencing this `template_id`.
- `retention_policies`: check `RetentionEngine` active schedule referencing this policy_id; and `legal_hold = true` → CONFLICT with `detail.reason = 'LEGAL_HOLD'`.

## Backend Flow (step-by-step)

### GET /admin/{masterResource}

1. `CorrelationMiddleware` attaches `X-Correlation-Id`.
2. `JwtAuthGuard` validates JWT; loads actor onto request context.
3. `AbacGuard` + `@Requires('configuration', scopeResolver)` calls `EntitlementService.can(actor, 'configuration', null)` — denies with FORBIDDEN (403) if actor has no `configuration` capability.
4. `AdminController.listMaster(masterResource, paginationParams)` invoked.
5. `MasterResourceRegistry.resolve(masterResource)` — returns the entity handler or throws `VALIDATION_ERROR` if unknown.
6. `AdminMasterService.list(resourceType, { orgId, page, limit, filters })` — builds Kysely query on the resolved table; applies `org_id` scope; applies `is_active` filter if provided; applies `LIMIT` (max 100); executes count query for pagination total.
7. For BM/KYC/DPO actors: `scopeResolver` injects an additional `branch_id = actor.branch_id` filter on resources that are branch-scoped.
8. Results serialised through `ResponseEnvelopeInterceptor` → 200 with pagination meta.

### POST /admin/{masterResource}

1–3. Same middleware + auth as GET.
4. `AdminController.createMaster(masterResource, body)`.
5. `MasterResourceRegistry.resolve(masterResource)` → get schema + handler.
6. Zod `ValidationPipe` validates `body` against the resource-specific create DTO; VALIDATION_ERROR on failure.
7. `UnitOfWork.run(async (tx) => { ... })` opens a transaction:
   a. `AdminMasterService.create(resourceType, dto, actorId, orgId, tx)`:
      - Insert master row into the resource table (Kysely `insertInto`).
      - Insert `configuration_versions` row (`status='pending'` for high-impact resources; `'active'` for low-impact — see ambiguities).
   b. `OutboxService.emit({ event_code: 'CONFIG_CHANGED', ... }, tx)` — writes to `event_outbox` in same tx.
   c. `AuditAppender.emit({ action: 'config_change', entity_type: resourceType, ... }, tx)` — emits audit intent.
8. On commit: `ResponseEnvelopeInterceptor` → 201 with created resource + `configVersionId`.
9. On DB unique-constraint violation: caught → `CONFLICT` (409).
10. On FK-not-found: caught at step 7a → `VALIDATION_ERROR` (400) with field annotation.

### PATCH /admin/{masterResource}/{id}

1–3. Same middleware + auth.
4. `AdminController.updateMaster(masterResource, id, body)`.
5. `MasterResourceRegistry.resolve(masterResource)`.
6. Zod validates body against resource-specific patch DTO (all fields optional; at least one required).
7. `UnitOfWork.run(async (tx) => { ... })`:
   a. Load existing record — throw `NOT_FOUND` (404) if absent.
   b. Scope check: if actor is BM/KYC/DPO, verify record's `branch_id` (if applicable) matches actor's branch. Otherwise `FORBIDDEN` (403).
   c. If `isActive === false`: run `MasterResourceRegistry.inUseCheck(resourceType, id, db)` → `CONFLICT` (409) if in use; check `legal_hold` → `CONFLICT` + `LEGAL_HOLD` sub-reason.
   d. `AdminMasterService.update(resourceType, id, dto, actorId, orgId, tx)`:
      - Build Kysely `updateTable` with changed fields; set `updated_by`, `updated_at`.
      - Insert `configuration_versions` diff row.
   e. `OutboxService.emit({ event_code: 'CONFIG_CHANGED', ... }, tx)`.
   f. `AuditAppender.emit({ action: 'config_change', ... }, tx)`.
8. On commit → 200 with updated resource.
9. Stale concurrent update (between load and update): unique-constraint on `uq_allocation_rules_order` or FK violation → CONFLICT (409).

## State Machine + Transaction Boundaries

### ConfigurationVersion state (per state-machines.md)

```
POST/PATCH master record
    │
    ▼
configuration_versions.status = 'pending'   (maker submits)
    │
    ▼  (FR-132 checker approves)
configuration_versions.status = 'approved'
    │
    ▼  (FR-132 activates)
configuration_versions.status = 'active'    (live config applied)
    │
    ▼  (FR-132 rollback)
configuration_versions.status = 'rolled_back'
```

For FR-131, the create/update write always produces a `configuration_versions` row. The master record itself is inserted/updated immediately (for a clean read-experience). Whether the new state is "live" depends on the org's maker-checker flag for that config type (resolved by FR-132).

### Transaction boundary (UnitOfWork)

One `UnitOfWork.run(tx)` per create or update request encompasses:
1. Master entity INSERT/UPDATE.
2. `configuration_versions` INSERT.
3. `event_outbox` INSERT (via `OutboxService`).
4. `audit_logs` intent emit (via `AuditAppender`).

If any step fails, the entire transaction rolls back — no partial state. The `AuditChainConsumer` (single-writer) later processes the audit intent; post-commit.

## Assumptions

1. The org-level maker-checker enforcement flag (which config types require pending → checker approval vs immediate activation) is resolved by FR-132 and stored in an org config row not yet detailed in the schema. For FR-131, the `configuration_versions` row is always created (audit trail); whether `status` starts as `'pending'` or `'active'` is determined at runtime by the FR-132 service. FR-131 does not gate on this — it always delegates to `ConfigurationVersionService.submitChange(...)` (FR-132 service).
2. Scope check for BM actors on "global" resources (no `branch_id` column) — e.g. `rejection_reasons`, `allocation_rules`, `sla_policies` — is that only ADMIN/HEAD (scope A) can create/update them. BM's scope `B` `configuration` capability only reaches branch-scoped resources (`branches`, `business_calendars` where `branch_id = actor.branch_id`). This is consistent with auth-matrix `BM: configuration: B` but the exact set of BM-editable vs ADMIN-only resources is not explicitly listed in the contracts. Listed in ambiguities.

## Summary

FR-110 implements the purpose-wise consent ledger for M12 Compliance. Every customer data-processing action in the LMS maps to a declared `consent_purpose` (from the `@shared/enums` catalog). This FR owns the `consent_records` table, which is **strictly append-only** ‚Äî no row is ever updated or deleted.

**Three capture paths exist:**

1. **Staff-initiated (`POST /leads/{id}/consents`):** RM/BM/SM/KYC/PARTNER captures or records a consent event (grant, denial, withdrawal) on behalf of a lead. This is the primary path for in-person, branch, and partner-submitted consent.

2. **Customer self-service (`POST /c/{token}/consent`):** The customer grants or confirms consent via the tokenised micro-site link. This endpoint is `@Public()` and guarded by `CustomerLinkGuard` (opaque token + OTP step-up), not a JWT.

3. **Read path (`GET /leads/{id}/consents`):** Returns the full append-only history for a lead's consents, visible to all authorised roles (DPO sees all; others see within their scope).

**Side effects on write:**
- `leads.consent_status` is re-derived after each insert and updated via `LeadService.setConsentStatus()` within the same `UnitOfWork` transaction.
- A `CONSENT_WITHDRAWN` event is written to `event_outbox` when a withdrawal record is inserted.
- Audit intent is emitted via `AuditAppender` for every consent mutation.

**Consent gates:** The `los_handoff` consent purpose, if withdrawn, causes `StageGuardService` to fail the `ready_for_handoff ‚Üí handed_off` transition. This FR does not implement stage transitions ‚Äî it only maintains the ledger that the guard reads.

## Auth Check

### `GET /leads/{id}/consents` and `POST /leads/{id}/consents`

- Global `JwtAuthGuard` is in effect (no `@Public()`).
- `@Requires('consent_ledger', scopeResolver)` ‚Üí `EntitlementService.can(user, 'consent_ledger', { leadId: id })`.
- Scope resolver checks the lead against the caller's data scope:
  - `RM` (scope `O`): `lead.owner_id = user.user_id`
  - `SM` (scope `T`): `lead.owner_id` belongs to the same team as the caller
  - `BM` / `KYC` (scope `B`): `lead.branch_id = user.branch_id`
  - `HEAD` / `DPO` (scope `A`): any lead in the org
  - `PARTNER` (scope `P`): `lead.source_attribution.partner_id = user.partner_id`
  - `ADMIN`: `consent_ledger` capability is present (scope `A`) ‚Äî but `ADMIN` has no standing lead-content access; treat the same as DPO read.
- Out-of-scope lead ‚Üí `FORBIDDEN` (403). Lead not found ‚Üí `NOT_FOUND` (404).
- `DPO` sees all records unmasked (unless masking is required per ABAC condition `DPO.view_lead = M`; the consent record itself has no PII fields masked, but `ip_device` JSONB is treated as PII and masked for non-DPO roles).

### `POST /c/{token}/consent`

- Endpoint is `@Public()` ‚Äî `JwtAuthGuard` is bypassed.
- `CustomerLinkGuard` validates the opaque token: token exists in `customer_links`, `status = 'active'`, `expires_at > now()`. OTP step-up is required (same session as the link open). Resolves `lead_id` and `customer_profile_id` from the link record.
- A valid token + OTP grants access to capture consent only for the lead bound to that token.
- Expired/revoked token ‚Üí `NOT_FOUND` (404) (existence hidden per BRD ¬ß8.6).

## Data Operations

All queries use the `Db` Kysely instance. Exact column names match `docs/data-model/schema.sql` table `consent_records`.

### List consent records for a lead

```typescript
// apps/api/src/modules/compliance/consent.repository.ts
db.selectFrom('consent_records')
  .selectAll()
  .where('lead_id', '=', leadId)
  .where('org_id', '=', orgId)
  .$if(purpose != null, (q) => q.where('purpose', '=', purpose))
  .$if(state != null, (q) => q.where('state', '=', state))
  .orderBy('created_at', 'asc')   // chronological; ledger is read in order
  .limit(limit)                   // max 100; default 25
  .offset((page - 1) * limit)
  .execute()
```

A separate count query (same WHERE, no LIMIT) returns `total` for pagination meta.

### Insert a consent record (append-only)

```typescript
// Within UnitOfWork transaction (tx)
db.insertInto('consent_records')
  .values({
    consent_id:           generateUuid(),
    org_id:               orgId,
    lead_id:              leadId,
    customer_profile_id:  customerProfileId ?? null,
    purpose:              dto.purpose,
    data_category:        dto.data_category ?? null,
    state:                dto.state,
    channel:              channel,          // resolved from context or dto
    language:             dto.language ?? null,
    notice_version:       dto.notice_version,
    consent_text_version: dto.consent_text_version,
    actor:                actor,            // 'customer' | 'rm' | 'partner' | 'system'
    ip_device:            ipDevice ?? null, // JSONB; never logged
    expires_at:           dto.expires_at ?? null,
    superseded_by:        null,             // set by supersede path only
    created_at:           new Date(),
    updated_at:           new Date(),
  })
  .returningAll()
  .executeTakeFirstOrThrow()
```

No UPDATE or DELETE is ever issued against `consent_records`. Attempts are rejected at the service layer before reaching the DB.

### Mark prior record superseded (when superseding)

When a new consent record for the same `(lead_id, purpose)` has state `granted` and a previous `granted` record exists, the prior record's `superseded_by` column is set to the new `consent_id`:

```typescript
// Within the same UnitOfWork tx ‚Äî updating the prior record's superseded_by pointer only
db.updateTable('consent_records')
  .set({
    superseded_by: newConsentId,
    updated_at:    new Date(),
  })
  .where('consent_id', '=', priorConsentId)
  .where('org_id', '=', orgId)
  .execute()
```

Note: updating `superseded_by` is the one allowed mutation on `consent_records` ‚Äî it is a pointer added to the old row when a new row supersedes it. The `state` column is never changed.

### Re-derive leads.consent_status (via LeadService.setConsentStatus)

After inserting the new consent record (within the same tx), the service re-derives the summary:

```typescript
// Logic executed by ConsentService before calling LeadService
const allForLead = await consentRepo.findLatestPerPurpose(leadId, orgId, tx)
// latest = most recent non-superseded record per purpose
// derived rules:
//   any 'withdrawn' ‚Üí 'withdrawn'
//   all required purposes 'granted' ‚Üí 'captured'
//   some 'granted' but not all required ‚Üí 'partial'
//   none granted ‚Üí 'pending'
await leadService.setConsentStatus(leadId, derivedStatus, tx)
```

`LeadService.setConsentStatus` issues the `UPDATE leads SET consent_status = ?, updated_at = now() WHERE lead_id = ? AND org_id = ?` (no version bump ‚Äî this is a volatile system-managed field per architecture ¬ß11.2).

### Find latest non-superseded consent per purpose (helper query)

```typescript
db.selectFrom('consent_records')
  .selectAll()
  .where('lead_id', '=', leadId)
  .where('org_id', '=', orgId)
  .where('superseded_by', 'is', null)
  .orderBy('created_at', 'desc')
  .execute()
// then group by purpose in application code, take first per group
```

| `consent_text_version` | required; non-empty string, max 40 chars | "consent_text_version is required." |

**Business rules applied in service (not Zod):**
- `state` cannot be `expired` or `superseded` on a staff or customer POST ‚Äî these states are system-managed. Attempting to insert with `state = 'expired'` or `state = 'superseded'` ‚Üí `VALIDATION_ERROR` (400).
- A `withdrawn` record requires that at least one prior `granted` record exists for the same `(lead_id, purpose)`. Withdrawing a purpose that was never granted ‚Üí `VALIDATION_ERROR` (400) with message "Cannot withdraw consent that was never granted."

## Backend Flow

### Staff consent capture (`POST /leads/{id}/consents`)

1. `JwtAuthGuard` validates JWT; loads user + ABAC attributes onto request context.
2. `AbacGuard` ‚Üí `EntitlementService.can(user, 'consent_ledger', { leadId: id })` with scope resolver. Reject with `FORBIDDEN` (403) if out of scope.
3. `CaptureConsentDto` Zod parse; invalid fields ‚Üí `VALIDATION_ERROR` (400) with `fields[]`.
4. `ConsentService.capture(leadId, dto, actor, orgId, userId, tx)`:
   a. **Load lead** ‚Äî verify `lead_id` exists in org. Not found ‚Üí `NOT_FOUND` (404).
   b. **Validate state** ‚Äî reject `expired` / `superseded` states ‚Üí `VALIDATION_ERROR`.
   c. **Withdrawal pre-check** ‚Äî if `dto.state === 'withdrawn'`, query for a prior `granted` record for `(lead_id, purpose)`. None found ‚Üí `VALIDATION_ERROR`.
   d. **Supersede prior grant** ‚Äî if `dto.state === 'granted'`, find the most recent `granted` record for `(lead_id, purpose)` with `superseded_by IS NULL`. If found, mark it `superseded_by = newConsentId` within the same tx.
   e. **INSERT consent_records** ‚Äî append-only insert with resolved `actor`, `channel`, `ip_device`.
   f. **Re-derive `leads.consent_status`** ‚Äî call `LeadService.setConsentStatus(leadId, derivedStatus, tx)`.
   g. **AuditAppender.emit** ‚Äî `{ action: 'CONSENT_CAPTURED', entity: 'consent_records', entity_id: newConsentId, actor: userId, detail: { purpose, state } }` ‚Äî audit intent queued via single-writer consumer.
   h. **OutboxService.emit** ‚Äî if `dto.state === 'withdrawn'`: emit `CONSENT_WITHDRAWN` event with `{ lead_id, purpose }` in the same tx.
5. Commit UnitOfWork. Return `201` with `{ consent_id, lead_id, purpose, state, created_at, derived_consent_status }`.

### Customer consent grant (`POST /c/{token}/consent`)

1. `@Public()` bypasses `JwtAuthGuard`.
2. `CustomerLinkGuard` resolves `token` ‚Üí `customer_links` row. Checks `status = 'active'` and `expires_at > now()`. OTP step-up must have been completed (session flag set by `POST /c/{token}/otp`). Invalid/expired ‚Üí `NOT_FOUND`.
3. Resolve `lead_id`, `customer_profile_id`, `channel` from the `customer_links` row.
4. `CustomerConsentDto` Zod parse; invalid ‚Üí `VALIDATION_ERROR`.
5. `ConsentService.captureFromCustomer(leadId, customerProfileId, channel, dto, ipDevice, orgId)`:
   - Steps b‚Äìh as above with `actor = 'customer'` and `ip_device` captured from request headers (`X-Forwarded-For`, `User-Agent`). Note: `X-Forwarded-For` is never logged raw (PII/security).
6. Return `201`.

### Read consent history (`GET /leads/{id}/consents`)

1. `JwtAuthGuard` + `AbacGuard` (`consent_ledger` capability) ‚Äî same scope check as write path.
2. `PaginationParams` + optional filter params parsed.
3. `ConsentRepository.listForLead(leadId, orgId, filters, pagination)`.
4. `ip_device` JSONB field is masked (set to `null`) for all roles except `DPO` and `ADMIN` (treated as PII).
5. Return `200` with paginated envelope.

## State Machine and Transaction Boundaries

### ConsentRecord ‚Äî `consent_state` (from `state-machines.md`)

`consent_records` is append-only. No state transition occurs on existing rows. The "state machine" is the sequence of new rows appended for a purpose:

| New row `state` | Effect | Side effect in same tx |
|---|---|---|
| `granted` | New ledger entry; prior `granted` row for same purpose marked `superseded_by` | `leads.consent_status` re-derived; audit emitted |
| `denied` | New ledger entry | `leads.consent_status` re-derived; audit emitted |
| `withdrawn` | New ledger entry; requires prior `granted` row | `leads.consent_status` re-derived; `CONSENT_WITHDRAWN` outbox event; audit emitted |
| `expired` | System-only (future scheduled job); not accepted via API | n/a |
| `superseded` | System-only (set on prior row when a new `granted` row supersedes it); not accepted via API | n/a |

Invalid: `UPDATE` or `DELETE` on any `consent_records` row ‚Üí blocked at service layer ‚Üí `FORBIDDEN` (403). No DB trigger needed; enforcement is in `ConsentService`.

### Transaction boundary

All writes for a single consent capture are wrapped in `UnitOfWork.run(async (tx) => {...})`:

```
UnitOfWork.run(tx):
  ‚îú‚îÄ‚îÄ ConsentRepository.insert(consentRow, tx)          -- append new consent_records row
  ‚îú‚îÄ‚îÄ ConsentRepository.markSuperseded(priorId, newId, tx) -- if superseding; update superseded_by only
  ‚îú‚îÄ‚îÄ LeadService.setConsentStatus(leadId, status, tx)  -- derive + update leads.consent_status
  ‚îú‚îÄ‚îÄ AuditAppender.emit(intent, tx)                    -- audit intent to outbox (single-writer consumer appends later)
  ‚îî‚îÄ‚îÄ OutboxService.emit(CONSENT_WITHDRAWN, tx)         -- only if state = 'withdrawn'
```

If any step fails, the entire transaction rolls back. No partial consent record + no partial `leads.consent_status` update persists.

**Consent status derivation logic** (executed inside `ConsentService` before calling `LeadService`):

The required purposes for `captured` are configurable but default to: `lead_contact`, `product_eligibility`, `kyc`, `document_processing`, `los_handoff`.

```
latestPerPurpose = most recent non-superseded row per purpose for this lead

if any purpose in latestPerPurpose has state = 'withdrawn':
    derivedStatus = 'withdrawn'
elif all required purposes have state = 'granted':
    derivedStatus = 'captured'
elif any purpose has state = 'granted':
    derivedStatus = 'partial'
else:
    derivedStatus = 'pending'
```

## Ambiguities

None unresolvable from the available contracts. The following required a best-effort interpretation:

1. **Required purposes for `captured` derivation** ‚Äî The BRD states consent status is derived but does not enumerate the exact required purpose set. Best-effort default: `lead_contact`, `product_eligibility`, `kyc`, `document_processing`, `los_handoff`. This set should be confirmed in a `ProductConfig`-level configuration or explicit BRD addendum before production.

2. **Customer self-service withdrawal** ‚Äî The `CustomerConsentDto` restricts `state` to `granted` or `denied`. The BRD mentions withdrawal (AC-4) but does not specify a customer-initiated withdrawal flow via the tokenised micro-site. Best-effort: withdrawal via `/c/{token}/consent` is excluded; a separate rights-request flow (FR-112) covers customer withdrawal requests. If the product team intends an in-session withdrawal path, this LLD needs a new endpoint or state extension.

3. **`ip_device` PII masking for PARTNER role** ‚Äî The auth-matrix gives PARTNER scope `P`; the BRD says DPO sees all. It is unclear whether PARTNERs should see their own submissions' `ip_device` metadata. Best-effort: `ip_device` is masked (set to `null`) for all roles except `DPO` (scope A). If PARTNER self-inspection of `ip_device` is required, update the masking condition in `ConsentService.listForLead`.

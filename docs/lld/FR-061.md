## Summary

A customer, authenticated by an opaque token link (FR-060), raises a grievance or service request directly from the self-service micro-site. The endpoint:

1. Validates the token (active, not expired, OTP-verified).
2. Validates the request payload (category, description, optional attachment note).
3. Generates a unique `grievance_no` (`GRV-{YYYY}-{seq5}`) via `CodeGenerator`.
4. Looks up the active `SlaPolicy` for `applies_to = 'grievance'` and calculates `sla_due_at` via `SlaEngine`.
5. Inserts a `grievances` row with `status = 'open'` and `source = 'customer_link'`.
6. Emits `GRIEVANCE_CREATED` to `event_outbox` (same transaction).
7. Emits an audit entry via `AuditAppender`.

The response returns the created `grievance_no` and `sla_due_at` so the customer can record their reference.

A supplemental **read** endpoint allows the self-service UI to display the grievance officer contact from `dla_registry` before the form is submitted.

## Endpoint

### POST /api/v1/c/{token}/grievance

**operationId:** `customerGrievance`  
**Tags:** `SelfService`  
**Auth:** `@Public()` + `CustomerLinkGuard` (token + OTP step-up; never a JWT)

#### Path Parameter

| Name  | Type   | Required | Description              |
|-------|--------|----------|--------------------------|
| token | string | yes      | Opaque customer-link token |

#### Request Body (`application/json`)

```json
{
  "category": "service_delay",
  "description": "My loan application has been pending for 2 weeks with no update.",
  "attachmentNote": "Relevant email thread reference: REF-20260601"
}
```

| Field          | Type   | Required | Constraints                                              |
|----------------|--------|----------|----------------------------------------------------------|
| category       | string | yes      | One of `grievance_category` enum values                  |
| description    | string | yes      | 1–2000 characters                                        |
| attachmentNote | string | no       | 0–500 characters; free-text reference only (no binary)   |

#### Response 201 — Created

```json
{
  "data": {
    "grievanceId": "d3a1b2c4-...",
    "grievanceNo": "GRV-2026-00031",
    "status": "open",
    "sla_due_at": "2026-06-12T18:30:00+05:30",
    "message": "Your grievance has been registered. Reference: GRV-2026-00031."
  },
  "meta": { "correlation_id": "corr_abc123" },
  "error": null
}
```

#### Error Responses

| HTTP | Code              | Condition                                               |
|------|-------------------|---------------------------------------------------------|
| 400  | VALIDATION_ERROR  | category not in enum / description empty or > 2000 chars / attachmentNote > 500 chars |
| 404  | NOT_FOUND         | token does not exist or has been revoked                |
| 409  | CONFLICT          | token `status` not `active` / `otp_verified_at` is null / token `expires_at` in the past |
| 429  | RATE_LIMITED      | > 10 requests/min per IP on this endpoint               |
| 500  | INTERNAL_ERROR    | Unhandled exception                                     |

## Auth Check

| Layer | Guard / Decorator | Detail |
|-------|------------------|--------|
| Authentication | `@Public()` | Endpoint is listed in `public_endpoints`; `JwtAuthGuard` does not apply. |
| Token validation | `CustomerLinkGuard` | Resolves `token_hash = sha256(token)` → `customer_links` row. Fails with `NOT_FOUND` (404) if no row found. Fails with `CONFLICT` (409) if `status != 'active'` or `expires_at < now()` or `otp_verified_at IS NULL`. Sets `req.customerLink` (includes `lead_id`, `org_id`) on the request context. |
| Scope | Token-scoped | The resolved `lead_id` from `customer_links` is the only lead this request may touch. No cross-lead access. |
| Authorization note | No `AbacGuard` / `@Requires` | Token-scoped customer actions do not use `EntitlementService.can()`. The `CustomerLinkGuard` is the sole authz boundary. |

## Data Operations

All Kysely queries use parameterised bindings (no string interpolation). Every list query carries a `LIMIT`.

### 1. Token Resolution (CustomerLinkGuard)

```typescript
// Lookup by token hash (computed from raw token in guard)
const link = await db
  .selectFrom('customer_links')
  .select(['customer_link_id', 'lead_id', 'org_id', 'status', 'expires_at', 'otp_verified_at'])
  .where('token_hash', '=', sha256(rawToken))
  .executeTakeFirst();
```

Guard asserts `link` is defined, `link.status === 'active'`, `link.expires_at > new Date()`, `link.otp_verified_at !== null`.

### 2. SLA Policy Lookup

```typescript
const slaPolicy = await db
  .selectFrom('sla_policies')
  .select(['sla_policy_id', 'threshold_minutes', 'escalation_chain'])
  .where('org_id', '=', orgId)
  .where('applies_to', '=', 'grievance')
  .where('is_active', '=', true)
  .limit(1)
  .executeTakeFirst();
```

If no active policy found: `sla_due_at` is set to `null` and a `warn` log is emitted. Not a hard error — the grievance is still created (BRD does not block intake on missing SLA config).

### 3. Grievance Number Generation

`CodeGenerator.nextGrievanceNo(orgId, tx)` — inserts into a sequence-tracking table or uses a DB sequence to produce `GRV-{YYYY}-{seq5}`. The unique constraint `uq_grievances_no` is the safety net; a retry on unique-key violation (up to 3 times) is acceptable.

### 4. Insert grievances (inside UnitOfWork transaction)

```typescript
const grievance = await db
  .insertInto('grievances')
  .values({
    grievance_id:      generateUUID(),
    org_id:            link.org_id,
    grievance_no:      generatedNo,
    lead_id:           link.lead_id,
    source:            'customer_link',
    category:          dto.category,
    description:       dto.description,
    status:            'open',
    sla_due_at:        slaPolicy ? slaEngine.calcDueAt(slaPolicy.threshold_minutes) : null,
    created_by:        SYSTEM_USER_ID,   // no JWT; use org's system user
    updated_by:        SYSTEM_USER_ID,
  })
  .returning(['grievance_id', 'grievance_no', 'status', 'sla_due_at'])
  .executeTakeFirstOrThrow();
```

### 5. Emit outbox event (same tx)

```typescript
await outboxService.emit(
  { event_code: 'GRIEVANCE_CREATED', aggregate_type: 'grievance', aggregate_id: grievance.grievance_id,
    payload: { grievance_no: grievance.grievance_no, lead_id: link.lead_id, category: dto.category, source: 'customer_link' },
    schema_version: 1 },
  tx
);
```

### 6. Emit audit intent

```typescript
await auditAppender.append(
  { actor_id: SYSTEM_USER_ID, action: 'lead_create',   // closest available audit_action (no grievance_create in enum)
    entity_type: 'grievance', entity_id: grievance.grievance_id,
    lead_id: link.lead_id,
    detail: { grievance_no: grievance.grievance_no, category: dto.category, source: 'customer_link' } },
  tx
);
```

**Note on audit_action:** The `audit_action` enum does not contain a `grievance_create` value (see schema.sql line 112). The closest available action is `lead_create` (or a future addition). This is flagged as an ambiguity — see the Ambiguities section.

## Backend Flow (Step-by-Step)

```
1. HTTP POST /api/v1/c/{token}/grievance arrives.
2. CorrelationMiddleware assigns X-Correlation-Id; pino logs {method, path, correlation_id}.
3. ThrottlerGuard: public endpoint bucket — 10 req/min per IP. Exceeds → RATE_LIMITED (429).
4. JwtAuthGuard: @Public() — skipped.
5. CustomerLinkGuard:
   a. Compute sha256(rawToken).
   b. SELECT customer_links WHERE token_hash = $1 LIMIT 1.
   c. If not found → throw NOT_FOUND (404).
   d. If status != 'active' OR expires_at <= now() OR otp_verified_at IS NULL → throw CONFLICT (409).
   e. Attach { lead_id, org_id, customer_link_id } to request context.
6. ValidationPipe: parse request body through CreateGrievanceDtoSchema.
   On failure → VALIDATION_ERROR (400) with fields[].
7. SelfServiceController.createGrievance(token, dto, req.customerLink) invoked.
8. GrievanceService.createFromCustomerLink(customerLink, dto) called:
   a. Fetch active SlaPolicy (applies_to='grievance') for org_id. Warn if absent.
   b. Calculate sla_due_at via SlaEngine.calcDueAt(threshold_minutes).
   c. Generate grievance_no via CodeGenerator.nextGrievanceNo(orgId, tx).
   d. Open UnitOfWork transaction:
      i.  INSERT INTO grievances (values as above). Retry on unique-key collision (max 3).
      ii. OutboxService.emit('GRIEVANCE_CREATED', ..., tx).
      iii.AuditAppender.append(action, ..., tx).
   e. On success: commit tx.
   f. On error: rollback tx; re-throw (AllExceptionsFilter maps to INTERNAL_ERROR).
9. ResponseEnvelopeInterceptor wraps result in { data, meta, error: null }.
10. Return HTTP 201 with grievanceId, grievanceNo, status, sla_due_at, and a customer-facing message.
```

## State Machine

Entity: `grievances.status` — reference `docs/contracts/state-machines.md` §Grievance.

FR-061 produces only the **intake transition**: `(none) → open`.

| Trigger           | From  | To   | Guard                      | Side-effects in same tx          |
|-------------------|-------|------|----------------------------|----------------------------------|
| Customer submits  | —     | open | Token valid + OTP verified | INSERT grievances; GRIEVANCE_CREATED outbox; audit |

Subsequent transitions (`open → in_progress → escalated → resolved → closed`) belong to FR-114 (Grievance workflow — internal compliance console).

### Transaction Boundaries (UnitOfWork)

All writes in a single `uow.run(async (tx) => { ... })`:
- `grievances` INSERT
- `event_outbox` INSERT (via `OutboxService`)
- Audit intent emission (via `AuditAppender`)

On any failure, the entire unit of work rolls back. The customer receives `INTERNAL_ERROR (500)`.

## Ambiguities

1. **`audit_action` enum gap:** The `audit_logs.action` column is typed `audit_action` (enum). The current schema values (line 112) do not include `grievance_create`. The closest available action is `lead_create`. Resolution options: (a) reuse `lead_create` with `entity_type='grievance'` and `detail.source='customer_link'`; or (b) add `grievance_create` to the `audit_action` enum via a Flyway migration. This LLD uses option (a) as a best-effort; the data model team should confirm whether the enum should be extended before implementation.

2. **`created_by` / `updated_by` on `grievances`:** The `grievances` table has `created_by UUID NOT NULL REFERENCES users`. For customer-link submissions, there is no authenticated `user_id`. The LLD assumes the org's **system user** (a fixed `SYSTEM_USER_ID` seeded in the org) is used for `created_by`/`updated_by`. Confirm the system user's UUID from the seed data or `users` table bootstrap.

3. **`attachmentNote` vs file attachment:** BRD AC §1 mentions "optional attachment". The `grievances` table has no `attachment_ref` column (only `closure_proof_ref` for resolution). The LLD interprets "optional attachment" as a free-text note field for Phase 1 (no binary upload on this endpoint). If a file upload is required, a `documents` INSERT via FR-070 would be needed, requiring `PAYLOAD_TOO_LARGE`/`UNSUPPORTED_MEDIA` handling and GCS upload — this would elevate the FR to Tier 3. Confirm scope with the product owner before implementation.

4. **Owner assignment:** BRD AC §2 states grievances are "routed to configured owner". The `grievances.owner_id` column is nullable. FR-061 covers intake only; owner assignment routing logic (resolving from `sla_policies.escalation_chain` or a separate config) is not defined for the customer-link path. The LLD leaves `owner_id` as `null` on intake and defers owner assignment to FR-114 (Grievance workflow). Confirm whether an initial owner should be auto-assigned at intake and, if so, the resolution rule.

5. **Duplicate complaint edge case:** BRD states "duplicate complaint → linked to existing open grievance". The `grievances` table has no `linked_grievance_id` or similar column. FR-061 does not currently implement duplicate detection on intake. This check and the `CONFLICT` sub-reason would require a schema extension (a `parent_grievance_id` FK column) and is deferred to FR-114. The customer-link intake creates a new row unconditionally.

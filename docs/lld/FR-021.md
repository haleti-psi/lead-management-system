## Summary

FR-021 implements the authorised merge of two duplicate leads into a single master record without losing any historical data — attribution, consent, documents, or audit. The duplicate lead is not deleted; it is soft-archived with `master_lead_id` set and `duplicate_status = merged`. All child records (source attributions, documents, consent records, tasks) of the duplicate are re-parented to the master in one atomic transaction via `UnitOfWork`. The source attribution of the duplicate is marked `attribution_status = merged_into` so partner/campaign analytics remain accurate. An unmerge endpoint reverses the operation within a configurable time window; unmerge after the window is blocked.

The operation is Tier 3 because it:
- performs a coordinated multi-entity write across `leads`, `source_attributions`, `documents`, `consent_records`, `tasks`, `duplicate_matches`, `audit_logs`, and `event_outbox` inside a single `UnitOfWork` transaction,
- involves state-machine transitions on `leads.duplicate_status` and `source_attributions.attribution_status`,
- includes a configurable time-window business rule (unmerge),
- drives a UI dialog with field-precedence preview, and
- requires cross-branch ownership logic.

## Endpoints

### POST /api/v1/leads/{id}/merge

**Purpose:** Merge the lead at `{id}` (the duplicate) into a master lead. The lead at `{id}` becomes the duplicate; the body specifies the master.

#### Request

```http
POST /api/v1/leads/{id}/merge
Authorization: Bearer <access_token>
Content-Type: application/json
```

**Path parameter:**

| Parameter | Type | Constraint |
|---|---|---|
| `id` | UUID | The lead to be marked as duplicate. Must exist, not be deleted, not already have `duplicate_status = merged`. |

**Body (Zod-validated `MergeLeadDto`):**

```json
{
  "master_lead_id": "<UUID>",
  "reason": "<string, 1–500 chars>",
  "field_precedence": "master | duplicate | manual",
  "manual_overrides": {
    "owner_id": "<UUID | null>",
    "branch_id": "<UUID | null>"
  },
  "expected_version": "<integer>"
}
```

| Field | Type | Required | Rule |
|---|---|---|---|
| `master_lead_id` | UUID | Yes | Must reference an existing, non-deleted, non-merged lead different from `{id}` |
| `reason` | string | Yes | 1–500 characters |
| `field_precedence` | enum | Yes | `master`, `duplicate`, or `manual` |
| `manual_overrides` | object | Only when `field_precedence = manual` | `owner_id` and `branch_id` must be valid UUIDs when provided |
| `expected_version` | integer | Yes | Optimistic-lock version of the **duplicate** lead (the one at `{id}`) |

#### Response — 200 OK

```json
{
  "data": {
    "master_lead_id": "<UUID>",
    "duplicate_lead_id": "<UUID>",
    "merge_completed_at": "<ISO-8601>",
    "attribution_records_relinked": 1,
    "documents_relinked": 3,
    "consent_records_relinked": 2,
    "tasks_relinked": 1,
    "duplicate_match_resolved": true,
    "unmerge_allowed_until": "<ISO-8601 | null>"
  },
  "meta": { "correlation_id": "<string>" },
  "error": null
}
```

`unmerge_allowed_until` is `now() + MERGE_UNMERGE_WINDOW_HOURS` (env var, default 24 h). `null` if unmerge is disabled.

## Auth Check

### Merge

| Layer | Detail |
|---|---|
| Authentication | Global `JwtAuthGuard` — request must carry a valid JWT access token. Missing/expired → `AUTH_REQUIRED` (401). |
| Authorisation | `@Requires('edit_lead', scopeResolver)` → `AbacGuard` → `EntitlementService.can(user, 'edit_lead', duplicateLead)`. BM scope B (branch), SM scope T (team). The scope check applies to **both** the duplicate and the master lead; the actor must have `edit_lead` scope over both. If either is out of scope → `FORBIDDEN` (403). |
| Cross-branch merge | When the duplicate and master belong to different branches, the master's branch takes precedence for the merged record. The actor must have `edit_lead` scope over the master lead's branch (i.e., a BM from one branch cannot merge into another branch's master unless they hold cross-branch scope). |

### Unmerge

Same `JwtAuthGuard` + `AbacGuard` + `@Requires('edit_lead', scopeResolver)` on the unmerging lead. Additionally: the service validates that `now() <= unmerge_allowed_until`; if the window has passed → `FORBIDDEN` (403).

## Data Operations

All SQL via **Kysely** parameterised queries only. Exact table and column names from `docs/data-model/schema.sql`.

### Merge — reads (before opening transaction)

```typescript
// 1. Load duplicate lead (with version for optimistic lock)
const duplicateLead = await db
  .selectFrom('leads')
  .selectAll()
  .where('lead_id', '=', duplicateId)
  .where('org_id', '=', orgId)
  .where('deleted_at', 'is', null)
  .executeTakeFirstOrThrow();

// 2. Load master lead
const masterLead = await db
  .selectFrom('leads')
  .selectAll()
  .where('lead_id', '=', masterId)
  .where('org_id', '=', orgId)
  .where('deleted_at', 'is', null)
  .executeTakeFirstOrThrow();
```

### Merge — writes (inside `UnitOfWork.run()` transaction)

```typescript
await uow.run(async (tx) => {

  // Step 1: Mark duplicate lead as merged (optimistic lock on duplicate)
  const updated = await db.updateTable('leads')
    .set({
      duplicate_status: 'merged',
      master_lead_id: masterId,
      updated_by: actorId,
      version: duplicateLead.version + 1,
    })
    .where('lead_id', '=', duplicateId)
    .where('org_id', '=', orgId)
    .where('version', '=', expectedVersion)   // optimistic lock
    .executeTakeFirst();
  if (!updated.numUpdatedRows) throw new ConflictException(/* CONFLICT */);

  // Step 2: Apply field-precedence updates to master lead (bump master version)
  await LeadService.merge(masterId, duplicateId, reason, fieldPrecedenceResult, expectedMasterVersion, tx);
  // LeadService.merge() issues the UPDATE on leads (master) with version bump

  // Step 3: Re-parent source_attributions of duplicate → master
  await db.updateTable('source_attributions')
    .set({ attribution_status: 'merged_into', updated_by: actorId })
    .where('source_attribution_id', '=', duplicateLead.source_attribution_id)
    .where('org_id', '=', orgId)
    .execute();
  // The duplicate's source_attribution row itself is NOT deleted; it is re-linked via leads.source_attribution_id
  // No new source_attribution is created here; the existing row is flagged merged_into

  // Step 4: Re-parent documents → master
  await db.updateTable('documents')
    .set({ lead_id: masterId, updated_by: actorId })
    .where('lead_id', '=', duplicateId)
    .where('org_id', '=', orgId)
    .execute();

  // Step 5: Re-parent consent_records → master (append-only: INSERT superseded rows, then update lead_id)
  // consent_records is append-only: we INSERT new rows with superseded_by pointing to originals
  // then re-parent the originals by updating lead_id (the re-parent of lead_id is NOT an audit-chain event;
  // superseded_by linkage preserves history — see Ambiguities §1)
  await db.updateTable('consent_records')
    .set({ lead_id: masterId })
    .where('lead_id', '=', duplicateId)
    .where('org_id', '=', orgId)
    .execute();

  // Step 6: Re-parent tasks → master
  await db.updateTable('tasks')
    .set({ lead_id: masterId, updated_by: actorId })
    .where('lead_id', '=', duplicateId)
    .where('org_id', '=', orgId)
    .execute();

  // Step 7: Resolve duplicate_match records linking these two leads
  await db.updateTable('duplicate_matches')
    .set({ status: 'resolved', action: 'merged', action_by: actorId, action_reason: reason, updated_by: actorId })
    .where('org_id', '=', orgId)
    .where((eb) => eb.or([
      eb.and([eb('lead_id', '=', duplicateId), eb('matched_lead_id', '=', masterId)]),
      eb.and([eb('lead_id', '=', masterId), eb('matched_lead_id', '=', duplicateId)]),
    ]))
    .execute();

  // Step 8: Emit audit log (via AuditAppender within tx)
  await AuditAppender.append({
    action: 'lead_merge',
    entity_type: 'leads',
    entity_id: masterId,
    lead_id: masterId,
    actor_id: actorId,
    detail: {
      duplicate_lead_id: duplicateId,
      reason,
      field_precedence: fieldPrecedence,
      attribution_records_relinked: 1,
      documents_relinked: docCount,
      consent_records_relinked: consentCount,
      tasks_relinked: taskCount,
    },
  }, tx);

  // Step 9: Emit event outbox (via OutboxService within tx)
  await OutboxService.emit({
    event_code: 'LEAD_STAGE_CHANGED',
    aggregate_type: 'leads',
    aggregate_id: masterId,
    payload: { duplicate_lead_id: duplicateId, action: 'merged', actor_id: actorId },
  }, tx);

});
```

### Unmerge — writes (inside `UnitOfWork.run()` transaction)

```typescript
await uow.run(async (tx) => {

  // Validate: duplicate lead must be in merged state; window must not have passed
  // (window is stored in audit_logs.detail.unmerge_allowed_until from the merge event)

  // Step 1: Restore duplicate lead (clear master_lead_id, reset duplicate_status)
  await LeadService.unmerge(duplicateId, expectedMasterVersion, tx);
  // LeadService.unmerge() updates leads (duplicate): duplicate_status → 'none', master_lead_id → NULL, version++

  // Step 2: Restore source_attribution status
  await db.updateTable('source_attributions')
    .set({ attribution_status: 'original', updated_by: actorId })
    .where('source_attribution_id', '=', duplicateLead.source_attribution_id)
    .where('org_id', '=', orgId)
    .execute();

  // Step 3: Restore documents back to duplicate
  await db.updateTable('documents')
    .set({ lead_id: duplicateId, updated_by: actorId })
    .where('lead_id', '=', masterId)
    .where('org_id', '=', orgId)
    // Only restore documents that were originally from the duplicate (tracked via audit detail)
    .where('document_id', 'in', originalDocumentIds)
    .execute();

  // Step 4: Restore consent_records back to duplicate
  await db.updateTable('consent_records')
    .set({ lead_id: duplicateId })
    .where('lead_id', '=', masterId)
    .where('org_id', '=', orgId)
    .where('consent_id', 'in', originalConsentIds)
    .execute();

  // Step 5: Restore tasks back to duplicate
  await db.updateTable('tasks')
    .set({ lead_id: duplicateId, updated_by: actorId })
    .where('lead_id', '=', masterId)
    .where('org_id', '=', orgId)
    .where('task_id', 'in', originalTaskIds)
    .execute();

  // Step 6: Re-open duplicate_match records
  await db.updateTable('duplicate_matches')
    .set({ status: 'open', updated_by: actorId })
    .where('org_id', '=', orgId)
    .where((eb) => eb.or([
      eb.and([eb('lead_id', '=', duplicateId), eb('matched_lead_id', '=', masterId)]),
      eb.and([eb('lead_id', '=', masterId), eb('matched_lead_id', '=', duplicateId)]),
    ]))
    .execute();

  // Step 7: Audit + outbox
  await AuditAppender.append({ action: 'lead_merge', /* … detail.action: 'unmerged' … */ }, tx);
  await OutboxService.emit({ event_code: 'LEAD_STAGE_CHANGED', /* … */ }, tx);

});
```

**Note on originalDocumentIds / originalConsentIds / originalTaskIds:** The set of IDs re-parented during merge is stored in `audit_logs.detail.relinked_ids` at merge time so unmerge can restore exactly those rows without guessing. The service reads the most-recent `lead_merge` audit entry for the pair before executing unmerge. (See Ambiguities §2.)

## Validation Logic (Zod — `MergeLeadDto` / `UnmergeLeadDto`)

### MergeLeadDto

| Field | Zod rule | Error message |
|---|---|---|
| `master_lead_id` | `z.string().uuid()` | "master_lead_id must be a valid UUID" |
| `master_lead_id` ≠ path `id` | custom refinement | "master_lead_id must differ from the duplicate lead" |
| `reason` | `z.string().min(1).max(500)` | "Reason is required (max 500 characters)" |
| `field_precedence` | `z.enum(['master','duplicate','manual'])` | "field_precedence must be one of: master, duplicate, manual" |
| `manual_overrides.owner_id` | `z.string().uuid().optional()` — required when `field_precedence = manual` | "owner_id must be a valid UUID when field_precedence is manual" |
| `manual_overrides.branch_id` | `z.string().uuid().optional()` | "branch_id must be a valid UUID when provided" |
| `expected_version` | `z.number().int().positive()` | "expected_version must be a positive integer" |

### Service-layer validations (after DTO validation)

| Check | Failure response |
|---|---|
| Duplicate lead exists and `deleted_at IS NULL` | `NOT_FOUND` (404) |
| Master lead exists, `deleted_at IS NULL`, and `duplicate_status != 'merged'` | `NOT_FOUND` (404) for master if out-of-scope; `VALIDATION_ERROR` (400) if master is already merged |
| `duplicate_status` of the `{id}` lead is not `'merged'` | `CONFLICT` (409) — "Lead is already merged" |
| Actor has `edit_lead` scope over the duplicate | `FORBIDDEN` (403) |
| Actor has `edit_lead` scope over the master | `FORBIDDEN` (403) |
| Optimistic lock match (`version = expected_version`) | `CONFLICT` (409) — refresh and retry |
| `manual_overrides.owner_id` references a user in the master's branch/team | `VALIDATION_ERROR` (400) if owner is invalid for scope |

### UnmergeLeadDto

| Field | Zod rule | Error message |
|---|---|---|
| `reason` | `z.string().min(1).max(500)` | "Reason is required (max 500 characters)" |
| `expected_master_version` | `z.number().int().positive()` | "expected_master_version must be a positive integer" |

### Service-layer validations for unmerge

| Check | Failure response |
|---|---|
| Lead at `{id}` has `duplicate_status = 'merged'` | `VALIDATION_ERROR` (400) if not merged — "Lead is not in merged state" |
| `now() <= unmerge_allowed_until` | `FORBIDDEN` (403) — "Unmerge window has expired" |
| Actor has `edit_lead` scope over the merged lead | `FORBIDDEN` (403) |
| Optimistic lock on master (`expected_master_version`) | `CONFLICT` (409) |

All errors use the uniform envelope `{ data: null, meta: { correlation_id }, error: { code, message, retryable, fields?, detail? } }`.

| Scenario | HTTP | Code | `detail.reason` | `retryable` |
|---|---|---|---|---|
| JWT absent or expired | 401 | `AUTH_REQUIRED` | — | false |
| Actor lacks `edit_lead` scope over duplicate or master | 403 | `FORBIDDEN` | — | false |
| Unmerge window has passed | 403 | `FORBIDDEN` | — | false |
| Duplicate lead not found / out of scope | 404 | `NOT_FOUND` | — | false |
| Master lead not found / out of scope | 404 | `NOT_FOUND` | — | false |
| Duplicate already has `duplicate_status = merged` | 409 | `CONFLICT` | — | false |
| Optimistic-lock stale on duplicate | 409 | `CONFLICT` | — | false |

## Backend Flow (Step-by-Step)

### Merge

```
1. CorrelationMiddleware attaches X-Correlation-Id.
2. ThrottlerGuard checks mutation rate (60/min per user); excess → RATE_LIMITED (429).
3. JwtAuthGuard validates Bearer token; expired/missing → AUTH_REQUIRED (401).
4. AbacGuard + @Requires('edit_lead', scopeResolver) calls EntitlementService.can():
   a. Resolves duplicate lead from {id}.
   b. Checks BM scope B / SM scope T over duplicate.
   c. If pass, also checks the same scope over master_lead_id; if either fails → FORBIDDEN (403).
5. Controller extracts and validates request body via MergeLeadDto (Zod).
   Any field violation → VALIDATION_ERROR (400) with fields[].
6. MergeLeadService.merge(duplicateId, dto, actor) called.
7. Service reads duplicate lead; verifies not deleted, not already merged.
   Already merged → CONFLICT (409).
8. Service reads master lead; verifies not deleted, not already merged.
9. Service validates master ≠ duplicate.
10. Service resolves field-precedence map (which master/duplicate fields win per dto.field_precedence).
    For cross-branch merge: master's branch takes precedence for the resolved record.
11. uow.run(tx) opens a single Kysely transaction:
    a. UPDATE leads (duplicate): duplicate_status=merged, master_lead_id=masterId, version++ WHERE version=expected_version.
       Zero rows updated → CONFLICT (409); transaction rolls back.
    b. LeadService.merge(masterId, duplicateId, reason, fieldPrecedenceResult, expectedMasterVersion, tx):
       Updates leads (master) fields per precedence; bumps master version.
       Zero rows → CONFLICT (409); rollback.
    c. UPDATE source_attributions: attribution_status=merged_into for duplicate's attribution row.
    d. Count and UPDATE documents: lead_id=masterId WHERE lead_id=duplicateId.
    e. Count and UPDATE consent_records: lead_id=masterId WHERE lead_id=duplicateId.
    f. Count and UPDATE tasks: lead_id=masterId WHERE lead_id=duplicateId.
    g. UPDATE duplicate_matches: status=resolved, action=merged for all rows linking these two leads.
    h. AuditAppender.append(lead_merge, detail including relinked_ids arrays and unmerge_allowed_until) within tx.
    i. OutboxService.emit(LEAD_STAGE_CHANGED, {duplicate_lead_id, action:'merged'}) within tx.
12. Transaction commits atomically. Any DB exception → full rollback → INTERNAL_ERROR (500).
13. Post-commit (not in transaction, retryable):
    Service calls NotificationDispatchService if any notification rule triggers for the actor/branch.
14. Controller wraps result in ApiEnvelope and returns 200.
```

### Unmerge

```
1–4. Same correlation/throttle/auth/authz steps as merge.
5. Controller validates UnmergeLeadDto.
6. UnmergeLeadService.unmerge(duplicateId, dto, actor) called.
7. Service reads lead at {id}: must have duplicate_status=merged; else VALIDATION_ERROR (400).
8. Service reads the most-recent audit_logs row where action='lead_merge' AND entity_id=masterId
   AND detail->>'duplicate_lead_id' = duplicateId. Extracts unmerge_allowed_until and relinked_ids.
9. If now() > unmerge_allowed_until → FORBIDDEN (403).
10. uow.run(tx):
    a. LeadService.unmerge(duplicateId, reason, tx):
       UPDATE leads (duplicate): duplicate_status=none, master_lead_id=NULL, version++.
    b. UPDATE master leads version via LeadService mutator with expected_master_version check.
    c. RESTORE source_attributions: attribution_status=original for duplicate's row.
    d. RESTORE documents: lead_id=duplicateId WHERE document_id IN relinked_ids.documents.
    e. RESTORE consent_records: lead_id=duplicateId WHERE consent_id IN relinked_ids.consents.
    f. RESTORE tasks: lead_id=duplicateId WHERE task_id IN relinked_ids.tasks.
    g. UPDATE duplicate_matches: status=open for the pair.
    h. AuditAppender.append(lead_merge, detail={action:'unmerged', reason}).
    i. OutboxService.emit(LEAD_STAGE_CHANGED, {duplicate_lead_id, action:'unmerged'}).
11. Commit; any failure → full rollback → INTERNAL_ERROR (500).
12. Return 200 envelope with counts.
```

## State Machine & Transaction Boundaries

### State transitions for merge

| Entity | Field | Before → After |
|---|---|---|
| `leads` (duplicate) | `duplicate_status` | `none` / `flagged` / `linked` → `merged` |
| `leads` (duplicate) | `master_lead_id` | `NULL` → `<masterId>` |
| `source_attributions` (duplicate's row) | `attribution_status` | `original` / `reassigned` → `merged_into` |
| `duplicate_matches` | `status` | `open` → `resolved` |
| `duplicate_matches` | `action` | any → `merged` |

These transitions and all child re-parentings (documents, consent_records, tasks) happen inside **one `UnitOfWork` transaction**. If any step fails, the entire unit of work rolls back — no partial state persists (BRD §5.6.4).

### State transitions for unmerge

Reverses the above: `merged → none`, `merged_into → original`, `resolved → open`.

### Invalid transitions (→ CONFLICT 409)

- Merging a lead that already has `duplicate_status = merged`.
- Unmerging a lead that does not have `duplicate_status = merged`.
- Merging two leads where master is itself `duplicate_status = merged` (chained merge blocked).
- Unmerging after the window has expired (→ FORBIDDEN 403 instead).

### Optimistic locking

Both the duplicate and master `leads` rows are updated under `WHERE version = :v`. A concurrent write that increments the version will cause zero rows updated, triggering `CONFLICT` (409) and full transaction rollback. The UI presents a "refresh and retry" flow.

### Retention invariant (BRD §5.6.2)

A merged lead's `deleted_at` remains `NULL`. The retention engine (FR-115) must never purge a lead with `master_lead_id IS NOT NULL` or with child leads referencing it via `master_lead_id`. This is a retention-engine constraint, not enforced here, but noted for downstream FR-115 LLD.

## Ambiguities

1. **consent_records re-parent semantics.** `consent_records` is append-only at the row level. Re-parenting by updating `lead_id` on existing rows is necessary to keep the master as the single consent owner, but BRD §5.6.3 says "no UPDATE/DELETE." The BRD's merge description explicitly states "consents re-parent to master," implying the `lead_id` column update is an architectural exception scoped to merge, distinct from consent-state mutation (which remains append-only via `superseded_by`). **Best-effort resolution:** the `lead_id` UPDATE is permitted inside the merge transaction because it does not change the consent state (`consent_state`, `superseded_by`, `actor`); it is a FK relocation preserving row immutability for all other fields. This decision should be ratified in the data-model doc (DATA_MODEL.md addendum or consent_records migration note).

2. **Unmerge tracking of relinked_ids.** The BRD does not specify how unmerge identifies which documents/consents/tasks to restore (all documents on the master at unmerge time vs. only those moved during merge). **Best-effort resolution:** Store the list of relinked IDs (documents, consents, tasks) in `audit_logs.detail.relinked_ids` at merge time, and read that entry during unmerge. This avoids ambiguity about documents added to the master after the merge.

3. **MERGE_UNMERGE_WINDOW_HOURS env var.** The BRD says "configurable window" but the environment contract does not list this variable. **Best-effort resolution:** add `MERGE_UNMERGE_WINDOW_HOURS` (default `24`) to `docs/contracts/environment-contract.md` before implementation.

4. **tasks table re-parent.** The `tasks` table in `schema.sql` uses `ON DELETE CASCADE` for `lead_id`. Re-parenting tasks is a best-effort: tasks that are already `done` or `cancelled` are re-parented (historical record), but if the duplicate lead were ever soft-deleted after merge, the cascade would not fire (since `deleted_at` is used, not actual row deletion). This is consistent and safe but worth noting.

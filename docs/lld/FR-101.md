## Summary

FR-101 provides two capabilities:

1. **Template management (ADMIN):** Create and version communication templates keyed by `(org_id, code, channel, language, version)`. Templates are `draft ‚Üí active ‚Üí retired` lifecycle-governed by the `ConfigurationVersion` maker-checker pattern (FR-131 dependency). An `active` template is the one used for dispatch.

2. **Consent-gated dispatch (RM/BM/SM/KYC/PARTNER):** Send a templated message to a lead's recipient over a nominated channel. Before queuing the send, the service checks `NotificationPreference` and the active `ConsentRecord` for the required `consent_purpose`. The actual provider call is async via `IntegrationGateway` (`NotificationChannelPort`) + Cloud Tasks. The `communication_logs` row is written in the same transaction as the dispatch record; delivery-status updates arrive via provider webhook/polling and update `communication_logs.status`.

FR-101 does **not** own the `leads` table and does **not** call `LeadService` mutators. The only write it owns is `communication_logs`. Template configuration writes follow the maker-checker config path owned by M14/FR-131.

## Auth Check

### Template management (`GET /admin/templates`, `POST /admin/templates`)

- Global `JwtAuthGuard` is in effect (no `@Public()`).
- `@Requires('configuration', scopeResolver)` ‚Üí `EntitlementService.can(user, 'configuration', resource)`.
- Only `ADMIN` (scope `A`) has `configuration` capability.
- Any other role ‚Üí `FORBIDDEN` (403).

### Dispatch (`POST /leads/{id}/communications`)

- Global `JwtAuthGuard` is in effect.
- `@Requires('customer_comm', scopeResolver)` ‚Üí `EntitlementService.can(user, 'customer_comm', lead)`.
- The scope resolver checks that the lead `{id}` is within the caller's data scope:
  - `RM` (scope `O`): `lead.owner_id = user.user_id`
  - `SM` (scope `T`): `lead.owner_id` belongs to the same team
  - `BM` / `KYC` (scope `B`): `lead.branch_id = user.branch_id`
  - `ADMIN` (scope `A`): any lead
  - `PARTNER` (scope `P`): `lead.source_attribution.partner_id = user.partner_id`
- Out-of-scope lead ‚Üí `FORBIDDEN` (403) (existence not hidden for these roles; 403 not 404).
- Lead not found ‚Üí `NOT_FOUND` (404).

## Data Operations

All queries use the `Db` Kysely instance. Exact column names match `docs/data-model/schema.sql`.

### List templates

```typescript
// apps/api/src/modules/engagement/template.repository.ts
db.selectFrom('communication_templates')
  .selectAll()
  .where('org_id', '=', orgId)
  .$if(channel != null, (q) => q.where('channel', '=', channel))
  .$if(language != null, (q) => q.where('language', '=', language))
  .$if(category != null, (q) => q.where('category', '=', category))
  .$if(status != null, (q) => q.where('status', '=', status))
  .$if(productCode != null, (q) => q.where('product_code', '=', productCode))
  .orderBy('code', 'asc')
  .orderBy('version', 'desc')
  .limit(limit)  // max 100
  .offset((page - 1) * limit)
  .execute()
```

A separate count query (same WHERE, no LIMIT) returns `total` for the pagination meta.

### Insert template

```typescript
db.insertInto('communication_templates')
  .values({
    template_id: generateUuid(),
    org_id: orgId,
    code: dto.code,
    version: dto.version,
    channel: dto.channel,
    language: dto.language,
    category: dto.category,
    product_code: dto.product_code ?? null,
    body: dto.body,
    status: 'draft',
    created_by: userId,
    updated_by: userId,
  })
  .returningAll()
  .executeTakeFirstOrThrow()
```

Unique constraint `uq_comm_templates` (`org_id, code, channel, language, version`) will throw on duplicate ‚Üí catch and map to `CONFLICT` (409).

### Fetch active template for dispatch

```typescript
db.selectFrom('communication_templates')
  .selectAll()
  .where('template_id', '=', templateId)
  .where('org_id', '=', orgId)
  .where('status', '=', 'active')
  .executeTakeFirst()  // returns undefined if not found / not active
```

### Check NotificationPreference

```typescript
db.selectFrom('notification_preferences')
  .selectAll()
  .where('org_id', '=', orgId)
  .where('subject_type', '=', subjectType)  // 'customer' | 'user'
  .where('subject_ref', '=', subjectRef)
  .where('channel', '=', channel)
  .where('purpose', '=', consentBasis)
  .executeTakeFirst()
// if row found and opted_in = false ‚Üí blocked
// if row not found ‚Üí apply default: transactional = opted_in, marketing = opted_out
```

### Check ConsentRecord (customer-facing sends only)

```typescript
db.selectFrom('consent_records')
  .selectAll()
  .where('lead_id', '=', leadId)
  .where('purpose', '=', consentBasis)
  .where('state', '=', 'granted')
  .orderBy('created_at', 'desc')
  .limit(1)
  .executeTakeFirst()
// absent or non-granted ‚Üí CONSENT_MISSING
```

### Insert CommunicationLog (in UnitOfWork transaction)

```typescript

  .values({
    communication_log_id: generateUuid(),
```

```typescript
db.updateTable('communication_logs')
```

## Backend Flow

### Template creation (`POST /admin/templates`)

1. `JwtAuthGuard` validates JWT; extracts `user` from token.
2. `AbacGuard` ‚Üí `EntitlementService.can(user, 'configuration', null)` ‚Üí reject non-ADMIN with `FORBIDDEN`.
3. `CreateTemplateDto` Zod parse; reject invalid fields with `VALIDATION_ERROR`.
4. `TemplateService.create(dto, userId, orgId)`:
   a. Delegate to `TemplateRepository.insert(dto)`.
   b. Unique constraint violation ‚Üí `CONFLICT` (409).
   c. Return created `template_id` with `status = 'draft'`.
5. `AuditAppender.emit({ action: 'TEMPLATE_CREATED', entity: 'communication_templates', entity_id, actor: userId })` ‚Äî audit intent queued via outbox; `AuditChainConsumer` writes the chain row.
6. Response envelope `201`.

### Templated message dispatch (`POST /leads/{id}/communications`)

1. `JwtAuthGuard` validates JWT.
2. `AbacGuard` ‚Üí `EntitlementService.can(user, 'customer_comm', { leadId: id })` with scope resolver; reject with `FORBIDDEN` if out of scope.
3. `SendCommunicationDto` Zod parse; reject invalid with `VALIDATION_ERROR`.
4. `NotificationDispatchService.send(leadId, dto, userId, orgId)`:
   a. **Fetch lead** ‚Äî verify existence and scope (step 2 guarantees scope, but service re-checks `org_id`). Not found ‚Üí `NOT_FOUND`.
   b. **Fetch template** ‚Äî `TemplateRepository.findActiveById(dto.template_id, orgId)`. Not found or not `active` ‚Üí `NOT_FOUND`.
   c. **Channel consistency check** ‚Äî `template.channel !== dto.channel` ‚Üí `VALIDATION_ERROR`.
   d. **Category/consent alignment** ‚Äî `template.category === 'marketing'` requires `dto.consent_basis === 'marketing'`; mismatch ‚Üí `FORBIDDEN` + `CONSENT_MISSING`.
   e. **ConsentRecord gate** ‚Äî query `consent_records` for `(lead_id, purpose=dto.consent_basis, state='granted')`. Absent ‚Üí `FORBIDDEN` + `CONSENT_MISSING`.
   f. **NotificationPreference gate** ‚Äî query `notification_preferences` for `(subject_ref, channel, purpose)`. If found and `opted_in = false` ‚Üí `FORBIDDEN` + `CONSENT_MISSING`. If not found, apply default (transactional = allow, marketing = block).
   g. **UnitOfWork transaction:**
      - `INSERT INTO communication_logs` with `status = 'queued'`.
      - `AuditAppender.emit(...)` ‚Üí audit intent (single-writer chain consumer appends after commit).
   h. **Enqueue Cloud Tasks job** `DispatchCommunicationTask` with `{ communication_log_id, template_id, channel, recipient, template_body_rendered }` (post-commit; job handles the provider call).
5. Return `202` with `{ communication_log_id, status: 'queued' }`.

### Async provider send (Cloud Tasks worker)

1. Worker fetches `communication_logs` row (must be `queued`; idempotency check by log ID).
2. Renders `template.body` with lead/contact merge fields (safe substitution, no eval).
3. Calls `IntegrationGateway.call(NotificationChannelPort, { channel, recipient, body, providerTemplate, idempotencyKey: logId }, ...)`.
4. On success ‚Üí `UPDATE communication_logs SET status='sent', provider_ref=..., sent_at=now()`.
5. On provider error ‚Üí `UPDATE communication_logs SET status='failed', failure_reason=...`; Cloud Tasks retries (exp backoff, max 3); after max retries ‚Üí failover to secondary provider if configured, otherwise final `failed`.
6. Delivery confirmation (provider webhook) ‚Üí `UPDATE communication_logs SET status='delivered'`.

## State Machine

**`CommunicationLog.delivery_status`** (`queued ‚Üí sent ‚Üí delivered / failed`)

| From | To | Trigger | Notes |
|---|---|---|---|
| `queued` | `sent` | Provider accepts the message | `provider_ref` + `sent_at` recorded |
| `sent` | `delivered` | Provider delivery receipt | Final positive terminal |
| `queued` / `sent` | `failed` | Provider permanent error or max retries exhausted | `failure_reason` recorded; retryable=true on 503 |

Invalid: `delivered ‚Üí *` or `failed ‚Üí *` (terminal states; re-send creates a new `communication_logs` row).

**`CommunicationTemplate.config_status`** (`draft ‚Üí active ‚Üí retired`)

Activation and retirement happen via the FR-131 maker-checker path; this FR only creates `draft` rows. The service rejects dispatch against non-`active` templates.

## Transaction Boundaries

The dispatch flow uses `UnitOfWork.run(async (tx) => {...})` to wrap:
- `INSERT INTO communication_logs` (status=`queued`)
- Audit intent emit (append-only, enlisted in the same `tx`)

Cloud Tasks enqueue is done **after** the transaction commits (post-commit hook pattern). If the transaction rolls back (e.g., DB error after INSERT), the Cloud Tasks job is not enqueued and no partial state persists.

The provider call itself is **outside** the request transaction ‚Äî it runs inside the Cloud Tasks worker. This ensures the HTTP endpoint meets the `‚â§ 800 ms` write-path NFR.

## External Service Calls

Via `IntegrationGateway` + `NotificationChannelPort` (three adapters: `SmsAdapter`, `WhatsappAdapter`, `EmailAdapter`).

**Environment variables consumed:**
- `SMS_PROVIDER_URL`, `SMS_PROVIDER_KEY`, `SMS_SENDER_ID`, `TRAI_DLT_ENTITY_ID`
- `WHATSAPP_BSP_URL`, `WHATSAPP_BSP_KEY`, `WHATSAPP_WABA_ID`
- `EMAIL_PROVIDER_URL`, `EMAIL_PROVIDER_KEY`, `EMAIL_FROM`

**India-specific gating:**
- SMS: `TRAI_DLT_ENTITY_ID` + registered sender header + pre-approved template body (OD-17). Free-form SMS not permitted.
- WhatsApp: `WHATSAPP_WABA_ID` (BSP) + per-template Meta approval. Free-form messages not permitted.

**Failure handling:**

| Failure | Detection | Response |
|---|---|---|
| Provider 4xx/5xx | HTTP status from adapter | Mark log `failed`, log `failure_reason`; Cloud Tasks retry |
| Provider timeout (> 10 s) | fetch timeout | Same as 5xx |
| Unapproved template body | Provider rejects | Mark log `failed` + `failure_reason='unapproved_template'`; no free-form fallback |
| Channel-level failover | After N retries on primary | Route to secondary provider if `SMS_SECONDARY_PROVIDER_URL` configured |
| Max retries exhausted | Cloud Tasks DLQ | Final `failed` status; operational alert |

In dev/test: `MockChannelAdapter` (`COMM_MOCK=true`) captures all sends without hitting real providers.

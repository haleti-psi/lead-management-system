## Summary

FR-071 orchestrates identity-verification checks (PAN, CKYC, DigiLocker, Aadhaar OTP, V-CIP) for a lead already in `kyc_in_progress` stage. The endpoint receives a KYC type in the URL path, gates on `kyc` consent, calls the provider through `IntegrationGateway` (idempotency + retry + circuit-breaker), stores only the masked/tokenised response in `kyc_verifications`, updates `lead_identities` (pan_token, ckyc_id, aadhaar_ref_token) where the provider returns identity enrichment, writes a `data_sharing_logs` row, recomputes `leads.kyc_status`, and emits `KYC_EXCEPTION` to the outbox on provider failure. Raw Aadhaar numbers and biometrics are **never stored** (BRD §2.4).

Tier 3 because: external-service call with failure modes, multi-entity transactional write (kyc_verifications + data_sharing_logs + lead_identities + leads via LeadService + audit + outbox), full KYC state machine, and provider-down exception queue.

## Endpoint

### `POST /api/v1/leads/{id}/kyc/{type}`

**Operation ID:** `runKyc`
**Tags:** KYC
**x-frs:** FR-071

#### Path Parameters

| Parameter | Type | Constraint | Description |
|---|---|---|---|
| `id` | UUID | required | Lead UUID |
| `type` | string | required; one of `pan`, `ckyc`, `digilocker`, `aadhaar_otp`, `vcip`, `manual` | KYC check type (maps to `kyc_type` enum) |

#### Request Body (`application/json`)

```json

  "idempotencyKey": "idem_<uuid>",
  "consentId": "<uuid>"
```

## Auth Check

1. **Authentication:** `JwtAuthGuard` (global) — validates JWT access token; returns `AUTH_REQUIRED` (401) if missing or expired.
2. **Authorization:** `AbacGuard` + `@Requires('verify_doc', scopeResolver)` where `scopeResolver` resolves to the lead's `branch_id`.
   - Permitted roles: `KYC` (scope `B` — branch KYC queue), `BM` (scope `B`).
   - `EntitlementService.can(user, 'verify_doc', { branchId: lead.branch_id })` must return `true`; otherwise `FORBIDDEN` (403).
   - RM has `verify_doc` scope `O` (preliminary) but KYC orchestration is restricted to KYC/BM — RM is excluded at the capability-condition level (auth-matrix `KYC.edit_lead: KYC fields only`).
3. **Consent gate:** Before calling the provider, verify `consent_records` contains an active `granted` row with `purpose = 'kyc'` for this lead. Absence → `FORBIDDEN` (403) + `detail.reason = CONSENT_MISSING`.

## Data Operations

All Kysely queries use parameterised values only. Every list query has a LIMIT clause.

### Step 1 — Load and validate lead

```typescript
// Fetch lead with lock (skip-locked on concurrent duplicate requests)
const lead = await db
  .selectFrom('leads')
  .select([
    'lead_id', 'stage', 'kyc_status', 'branch_id', 'version',
    'lead_identity_id', 'product_config_id', 'org_id'
  ])
  .where('lead_id', '=', leadId)
  .where('deleted_at', 'is', null)
  .executeTakeFirst();
```

Must exist and `stage = 'kyc_in_progress'`; otherwise `NOT_FOUND` (404) or `CONFLICT` (409) with appropriate sub-reason.

### Step 2 — Verify consent

```typescript
const consent = await db
  .selectFrom('consent_records')
  .select(['consent_id', 'state', 'expires_at'])
  .where('lead_id', '=', leadId)
  .where('purpose', '=', 'kyc')
  .where('state', '=', 'granted')
  .where((eb) => eb.or([
    eb('expires_at', 'is', null),
    eb('expires_at', '>', new Date())
  ]))
  .orderBy('created_at', 'desc')
  .limit(1)
  .executeTakeFirst();
```

No active consent → `FORBIDDEN` (403) + `detail.reason = CONSENT_MISSING`.

### Step 3 — Idempotency check

```typescript
const existingLog = await db
  .selectFrom('integration_logs')
  .select(['integration_log_id', 'status'])
  .where('idempotency_key', '=', idempotencyKey)
  .where('integration', '=', kycTypeToIntegrationKind(kycType))
  .limit(1)
  .executeTakeFirst();
```

If `existingLog` found and `status = 'success'` → load the matching `kyc_verifications` row and return `200` with `detail.reason = IDEMPOTENT_REPLAY`.

### Step 4 — Call provider via IntegrationGateway (outside transaction)

```typescript
const providerResult = await integrationGateway.call(
  kycPort,
  buildProviderRequest(kycType, leadIdentity, requestBody),
  {
    idempotencyKey,
    integration: kycTypeToIntegrationKind(kycType),
    leadId,
    correlationId
  }
);
// Returns { success: boolean, reference?: string, maskedResponse?: object, exceptionType?: kyc_exception }
// Throws IntegrationUnavailableError on provider 5xx/timeout after retries exhausted
```

Provider call is made **before** the write transaction so the transaction is short and not held open waiting for a network call.

### Step 5 — Write transaction (UnitOfWork)

```typescript
await uow.run(async (tx) => {

  // 5a. Insert kyc_verifications row
  const kycVerif = await db
    .insertInto('kyc_verifications')
    .values({
      kyc_verification_id: generateUuid(),
      org_id: lead.org_id,
      lead_id: leadId,
      kyc_type: kycType,
      provider: providerResult.provider,
      status: providerResult.success ? 'success' : 'failed',
      reference: providerResult.reference ?? null,
      masked_response: providerResult.maskedResponse ?? null,
      exception_type: providerResult.exceptionType ?? null,
      integration_log_id: providerResult.integrationLogId,
      created_by: actorId,
      updated_by: actorId
    })
    .returningAll()
    .executeTakeFirstOrThrow();

  // 5b. If failed: mark exception status on same row
  // (already set above via status='failed'; FR-072 owns transition to 'exception' after queue)

  // 5c. Update lead_identities with enrichment from provider (when PAN success)
  if (kycType === 'pan' && providerResult.success) {
    await db
      .updateTable('lead_identities')
      .set({
        pan_token: providerResult.panToken,       // tokenised; never raw
        pan_masked: providerResult.maskedPan,
        updated_by: actorId,
        updated_at: new Date()
      })
      .where('lead_identity_id', '=', lead.lead_identity_id)
      .execute();
  }

  // For ckyc type: update ckyc_id on lead_identities
  if (kycType === 'ckyc' && providerResult.success && providerResult.ckycId) {
    await db
      .updateTable('lead_identities')
      .set({
        ckyc_id: providerResult.ckycId,
        updated_by: actorId,
        updated_at: new Date()
      })
      .where('lead_identity_id', '=', lead.lead_identity_id)
      .execute();
  }

  // For aadhaar_otp: update aadhaar_ref_token (tokenised; raw Aadhaar never stored)
  if (kycType === 'aadhaar_otp' && providerResult.success && providerResult.aadhaarRefToken) {
    await db
      .updateTable('lead_identities')
      .set({
        aadhaar_ref_token: providerResult.aadhaarRefToken,
        updated_by: actorId,
        updated_at: new Date()
      })
      .where('lead_identity_id', '=', lead.lead_identity_id)
      .execute();
  }

  // 5d. Insert data_sharing_log (external data pull)
  await db
    .insertInto('data_sharing_logs')
    .values({
      data_sharing_log_id: generateUuid(),
      org_id: lead.org_id,
      lead_id: leadId,
      recipient: providerResult.provider,
      purpose: 'kyc',
      data_category: 'identity',
      consent_id: requestBody.consentId,
      status: 'shared',
      created_by: actorId,
      updated_by: actorId
    })
    .execute();

  // 5e. Recompute and update leads.kyc_status via LeadService (sole lead writer)
  const newKycStatus = await computeLeadKycStatus(leadId, tx);
  await leadService.setKycStatus(leadId, newKycStatus, tx);

  // 5f. Emit outbox event on exception
  if (!providerResult.success) {
    await outboxService.emit(
      'KYC_EXCEPTION',
      { leadId, kycVerificationId: kycVerif.kyc_verification_id, exceptionType: kycVerif.exception_type },
      tx
    );
  }

  // 5g. Audit intent (consumed by single-writer AuditChainConsumer)
  await auditAppender.append(
    {
      action: 'kyc_response',
      subject_type: 'kyc_verifications',
      subject_id: kycVerif.kyc_verification_id,
      actor_id: actorId,
      meta: { kycType, status: kycVerif.status, exceptionType: kycVerif.exception_type ?? null }
    },
    tx
  );

  return kycVerif;
});
```

### `computeLeadKycStatus` helper

Queries all `kyc_verifications` for the lead (LIMIT 100) and derives `leads.kyc_status`:
- Any row `status = 'exception'` and not resolved → `'exception'`
- All required checks `status = 'success'` or `status = 'waived'` → `'verified'`
- At least one `status IN ('initiated', 'success', 'failed')` → `'in_progress'`
- All `status = 'waived'` → `'waived'`
- No rows → `'not_started'`

Path param `type` is validated against the `kyc_type` enum before reaching the service; invalid value → `VALIDATION_ERROR` (400) with `fields: [{ field: "type", issue: "Must be one of: pan, ckyc, digilocker, aadhaar_otp, vcip, manual" }]`.

| Field | Rule | Error message |
|---|---|---|
| `type` (path) | `kyc_type` enum | Must be one of: pan, ckyc, digilocker, aadhaar_otp, vcip, manual |
| `pan` (body) | `^[A-Z]{5}[0-9]{4}[A-Z]{1}$` when type=pan | PAN format invalid (AAAAA9999A) |
| `consentId` | UUID format | consentId must be a valid UUID |
| `idempotencyKey` | max 120 chars | idempotencyKey must be ≤ 120 characters |
| `aadhaarOfflineXml` | non-empty base64 string; raw Aadhaar digits never accepted | Aadhaar offline XML required |
| `digilockerCode` | non-empty string | DigiLocker OAuth code required |

## Error Cases

| Condition | Code | HTTP | `detail.reason` |
|---|---|---|---|
| Missing/invalid JWT | `AUTH_REQUIRED` | 401 | — |
| Role without `verify_doc` for this lead's branch | `FORBIDDEN` | 403 | — |
| No active `kyc` consent granted | `FORBIDDEN` | 403 | `CONSENT_MISSING` |
| Lead not found or soft-deleted | `NOT_FOUND` | 404 | — |
| Lead stage not `kyc_in_progress` | `CONFLICT` | 409 | — |
| Invalid `type` path parameter | `VALIDATION_ERROR` | 400 | — |
| PAN format invalid (when type=pan) | `VALIDATION_ERROR` | 400 | — |
| Provider 5xx / timeout (after retries) | `UPSTREAM_UNAVAILABLE` | 503 | — |
| Stale optimistic lock on leads.version | `CONFLICT` | 409 | — |
| Replayed idempotency key (success) | 200 with original data | 200 | `IDEMPOTENT_REPLAY` |
| Rate limit exceeded | `RATE_LIMITED` | 429 | — |

## Backend Flow (Step-by-Step)

```
POST /api/v1/leads/{id}/kyc/{type}
  |
  1. CorrelationMiddleware — attach/generate X-Correlation-Id
  2. JwtAuthGuard — validate access token → load user + ABAC attributes
  3. AbacGuard @Requires('verify_doc', branchScopeResolver)
     → EntitlementService.can(user, 'verify_doc', {branchId}) → 403 if denied
  4. ZodValidationPipe → validate path param `type` + request body
     → VALIDATION_ERROR (400) + fields[] on failure
  5. KycController.runKyc(leadId, type, body, user)
  6. KycService.runVerification(leadId, kycType, body, actor):
     6a. Load lead (404 if not found / soft-deleted)
     6b. Assert lead.stage === 'kyc_in_progress' → 409 CONFLICT if not
     6c. Assert consent active for purpose='kyc' → 403 FORBIDDEN + CONSENT_MISSING
     6d. Idempotency check via Redis (integration_logs.idempotency_key)
         → if existing success found: return IDEMPOTENT_REPLAY (200) immediately
     6e. EntitlementService.can — scope guard (branchId match)
     6f. Build provider request (strip/never include raw Aadhaar digits)
     6g. IntegrationGateway.call(KycPort, providerReq, {idempotencyKey, ...})
         → KycPort dispatches to adapter (PanAdapter | CkycAdapter | DigiLockerAdapter | AadhaarAdapter | VcipAdapter)
         → On 5xx/timeout after max retries: throw IntegrationUnavailableError
            → catch in service → set providerResult.success=false, exceptionType='provider_down'
         → On mismatch result: success=false, exceptionType = (pan_mismatch | name_mismatch | ...)
     6h. UnitOfWork.run(tx => ...):
         i.   INSERT kyc_verifications (status = success | failed)
         ii.  UPDATE lead_identities (pan_token, ckyc_id, or aadhaar_ref_token) when success
         iii. INSERT data_sharing_logs (recipient=provider, purpose=kyc, consent_id)
         iv.  computeLeadKycStatus(leadId, tx) → recompute from all kyc_verification rows
         v.   LeadService.setKycStatus(leadId, newStatus, tx) — sole lead writer
         vi.  If failed: OutboxService.emit('KYC_EXCEPTION', {...}, tx)
         vii. AuditAppender.append(kyc_verification_run, tx)
         → On any failure: full rollback (no partial state)
     6i. If providerResult.success=false due to IntegrationUnavailableError:
         → Return UPSTREAM_UNAVAILABLE (503) + kycVerificationId (exception created for FR-072)
     6j. Return 200 with KycVerificationDto (masked fields only)
  7. ResponseEnvelopeInterceptor — wrap in { data, meta, error }
  8. MaskingInterceptor — apply role masking (pan_masked vs unmasked based on DPO/KYC scope)
```

## State Machine + Transaction Boundaries

### KYCVerification state transitions (state-machines.md §KYCVerification)

```
initiated (implicit at request start)
  |
  +-- provider success  --> status = 'success'   [writes in tx, step 6h]
  |
  +-- provider mismatch --> status = 'failed'    [writes in tx, step 6h]
  |                          exception_type set
  |
  +-- provider_down     --> status = 'failed'    [writes in tx, step 6h]
                            exception_type = 'provider_down'
                            KYC_EXCEPTION emitted
                            --> FR-072 transitions to 'exception' via queue
```

### Lead.kyc_status transitions (derived, recomputed in tx)

```
not_started --> in_progress   (first KYC check initiated for this lead)
in_progress --> verified      (all required checks = success/waived)
in_progress --> exception     (any check = failed/exception and unresolved)
exception   --> verified      (all exceptions resolved via FR-072)
any         --> waived        (authorised waiver covers all checks — FR-072)
```

### Transaction boundary (UnitOfWork step 6h)

One Kysely transaction encompasses:
- `INSERT kyc_verifications`
- `UPDATE lead_identities` (conditional)
- `INSERT data_sharing_logs`
- `LeadService.setKycStatus` → `UPDATE leads SET kyc_status=?, updated_at=?, updated_by=? WHERE lead_id=? AND version=?`
- `INSERT event_outbox` (KYC_EXCEPTION if failed)
- Audit intent enqueue

**Commit or full rollback — no partial state.**

## External Service Calls via IntegrationGateway

All calls use `IntegrationGateway.call(port, request, opts)` per integration-map.md.

### Port: `KycPort`

| KYC Type | Adapter (prod) | Adapter (test/dev) | `integration` enum value |
|---|---|---|---|
| `pan` | `PanAdapter` | `MockKycAdapter` | `pan` |
| `ckyc` | `CkycAdapter` | `MockKycAdapter` | `ckyc` |
| `digilocker` | `DigiLockerAdapter` | `MockKycAdapter` | `digilocker` |
| `aadhaar_otp` | `AadhaarAdapter` | `MockKycAdapter` | `aadhaar` |
| `vcip` | `VcipAdapter` | `MockKycAdapter` | `vcip` |
| `manual` | no external call | — | — |

For `manual` type, no provider call is made; `kyc_verifications` row is inserted with `status='success'`, `provider=null`, `masked_response=null`, and an `audit_logs` entry records the manual authorisation.

### Provider response → masked storage

| Provider returns | What LMS stores | What LMS never stores |
|---|---|---|
| PAN validation result | `masked_response.panStatus`, `masked_response.nameMatch`, `pan_masked` in lead_identities | Raw PAN (only tokenised `pan_token`) |
| CKYC record | `masked_response` (redacted fields), `ckyc_id` in lead_identities | Raw CKYC document content beyond reference |
| DigiLocker e-doc | `reference` (document URN), `masked_response.docType` | Document bytes (stored in GCS via FR-070, not here) |
| Aadhaar OTP offline | `aadhaar_ref_token` in lead_identities | **Raw Aadhaar number or biometrics — never** |
| VCIP result | `masked_response.livenessResult`, `reference` (recording ref) | Recording bytes |

### Failure handling

| Failure mode | Detection | Response |
|---|---|---|
| Provider 5xx / timeout | HTTP error or timeout > configured threshold | `IntegrationUnavailableError` → `kyc_verifications.status='failed'`, `exception_type='provider_down'`, `KYC_EXCEPTION` outbox; Cloud Tasks retry (exp backoff, max 3); return `UPSTREAM_UNAVAILABLE` (503) |
| Provider mismatch result | Provider returns mismatch code | `kyc_verifications.status='failed'`, `exception_type` set (pan_mismatch / name_mismatch / address_mismatch / ...); `KYC_EXCEPTION` outbox; return 200 with status='failed' for the UI to show exception (not a 503) |
| Provider circuit-breaker open | IntegrationGateway detects repeated failure | Same as 5xx path; operational queue; Cloud Monitoring alert |
| Compliance-disabled manual fallback | `provider_down` + no compliance fallback flag | `exception_type='provider_down'`; manual fallback only if compliance-enabled (FR-072) |

- Success: `StatusChip status="success"` + masked PAN displayed in `<MaskedField>`

## Assumptions

1. The `kyc` consent purpose is always captured as part of the lead's consent ledger (FR-110) before KYC orchestration begins; the stage guard `documents_pending → kyc_in_progress` in FR-052 / StageGuardService should enforce this.
2. CKYC, DigiLocker, Aadhaar OTP, and V-CIP adapters are Phase 1.5 and will be built against the `KycPort` interface later; only `PanAdapter` and `MockKycAdapter` are MVP-Must.
3. The `idempotencyKey` is sourced from the HTTP header `Idempotency-Key` (per architecture §4) and forwarded in the request body DTO for convenience; the IntegrationGateway also reads it.
4. `computeLeadKycStatus` queries a maximum of 100 `kyc_verifications` rows (NFR-17 LIMIT enforcement); this is safe because the number of KYC check types is bounded (6).
5. V-CIP (recording assignment, liveness) is Phase 1.5 only; the `vcip` type endpoint exists but the `VcipAdapter` returns `UPSTREAM_UNAVAILABLE` with a "phase not enabled" message in MVP.

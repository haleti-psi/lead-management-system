## Summary

A PARTNER user (DSA/dealer/connector/OEM/aggregator/referral) submits a lead through a limited
Partner Console (or the same `POST /partners/leads` API) and tracks **only their own** submitted
leads. The endpoint is a partner-scoped facade over the standard FR-010 capture flow:

- The partner is resolved from `users.partner_id` (the authenticated PARTNER user). The partner's
  `partner_id` is **forced** onto the new lead's `source_attributions.partner_id`; the partner can
  never submit on behalf of another partner.
- A **suspended/expired** partner cannot submit (`FORBIDDEN`).
- `source` is forced to a partner channel (`DSA` or `Dealer` per the partner `type`); `creator_channel`
  / `channel_created_by` = `partner`.
- The create runs the standard pipeline: validate against the active `ProductConfig.field_schema`,
  run **duplicate detection (FR-020)**, then atomically insert `lead_identities` +
  `source_attributions` + `leads` (stage `captured`) + `lead_product_details` stub + `stage_history`
  + `audit_logs` + `event_outbox(LEAD_CREATED)` in **one transaction** via `UnitOfWork`, with `leads`
  written **only** through `LeadService.create`.
- A **strong duplicate** blocks creation and returns a **masked** `CONFLICT`
  (`detail.reason = DUPLICATE_BLOCKED`) with **no** other-customer PII or IDs — generic
  "already exists" only.
- `Idempotency-Key` replays return the original `201` result (no duplicate).
- `GET /partners/leads` lists the partner's own leads, paginated, with PII **masked**.

Partner-submitted documents are **out of scope of this FR** — the BRD routes them to the standard
M8 KYC/document queue via the document-upload endpoints (FR-060/070); FR-091 only creates the lead
and its attribution. (See Ambiguities.)

## Tier Specification

### Endpoint(s)

All paths are prefixed `/api/v1`. Uniform envelope `{ data, meta, error }` (architecture §4.4).
Both endpoints are protected (`JwtAuthGuard` global, no `@Public()`).

#### 1. `POST /partners/leads` — `partnerCreateLead`

Request body (`PartnerLeadCreateDto`; subset of `LeadCreate` — partner cannot set owner/branch scope
freely, and `source.partner_code` is ignored/forced):

```jsonc
{
  "product_code": "CV",                         // enum, required
  "identity": {                                 // required
    "name": "Ramesh Kumar",                     // required
    "mobile": "9876543210",                     // required, ^[6-9]\d{9}$
    "email": "ramesh@example.com",              // optional
    "preferred_language": "hi"                  // optional, lang enum
  },
  "sub_source": "showroom-A",                   // optional, <=80 chars
  "pin_code": "560001",                          // optional, ^[0-9]{6}$
  "requested_amount": 800000,                    // optional, >= 0
  "product_detail": { "asset": "tata-ace" }      // optional, validated vs ProductConfig.field_schema
}
```

> `source.source` and `source.partner_code` from the generic `LeadCreate` schema are **NOT accepted**
> from a partner; the server derives both. Any client-supplied `source`/`partner_code`/`owner` field
> is ignored (stripped by Zod `.strip()`), not honoured.

Headers: optional `Idempotency-Key` (string).

Response `201 Created` (envelope `data` = created lead, **masked**):

```jsonc
{
  "data": {
    "lead_id": "…uuid…",
    "lead_code": "LD-2026-000123",
    "stage": "captured",
    "product_code": "CV",
    "consent_status": "pending",
    "kyc_status": "not_started",
    "duplicate_status": "none",
    "name_masked": "Ramesh xxxxx",
    "mobile_masked": "98xxxxxx10",
    "created_at": "2026-06-09T10:00:00.000Z"
  },
  "meta": { "correlation_id": "…" },
  "error": null
}
```

Idempotent replay (same `Idempotency-Key`, same partner): returns the **original** `201` body
(`detail.reason = IDEMPOTENT_REPLAY` is an internal marker; the response is the original payload, no
new lead). Status remains `201` for the replay of a create per error-taxonomy "original result".

Query: `PaginationParams` — `page` (default 1, min 1), `limit` (default 25, max 100). Optional

Response `200 OK` (envelope `data` = array of masked partner-leads):

```jsonc
{
  "data": [
    {
      "lead_id": "…",
      "lead_code": "LD-2026-000123",
      "stage": "assigned",
      "product_code": "CV",
      "duplicate_status": "none",
      "name_masked": "Ramesh xxxxx",
      "mobile_masked": "98xxxxxx10",
      "created_at": "2026-06-09T10:00:00.000Z"
    }

  "meta": { "correlation_id": "…", "pagination": { "page": 1, "limit": 25, "total": 1 } },

```

> **Limited status only (AC2):** a partner sees `stage`, `duplicate_status`, `created_at` and masked
> identity. Internal-only fields (`score`, `score_reasons`, `owner_id`, internal notes, other
> partners' data) are **never** serialized for PARTNER. The `Lead` projection for PARTNER omits them.

### Auth Check

Global `JwtAuthGuard` enforces authentication (missing/expired → `AUTH_REQUIRED` 401).
Authorization via `AbacGuard` + `@Requires(...)` → `EntitlementService.can(user, capability, resource)`.

| Endpoint | Decorator | Capability | Scope | Resource | Resolver / ownership rule |
|---|---|---|---|---|---|
| `POST /partners/leads` | `@Requires('create_lead', partnerScope)` | `create_lead` | `P` | `leads` | `user.role === 'PARTNER' && user.partner_id != null`; the new lead's `source_attributions.partner_id` is forced to `user.partner_id`. |
| `GET /partners/leads` | `@Requires('view_lead', partnerScope)` | `view_lead` | `P` | `leads` | every returned row satisfies `source_attributions.partner_id = user.partner_id`. |

- `partnerScope` resolver returns `{ partnerId: user.partner_id }`; `EntitlementService.can` confirms
  `PARTNER.create_lead === 'P'` / `PARTNER.view_lead === 'P'` from `auth-matrix.json`.
- A non-PARTNER role (RM/BM/SM/HEAD/KYC/DPO/ADMIN/CUSTOMER) calling these endpoints → `FORBIDDEN` (403)
  (`auth-matrix` grants no PARTNER-scope `P` capability to them on this route).
- A PARTNER whose `users.partner_id` is null → `FORBIDDEN` (403) (mis-provisioned partner user).
- **Partner-active gate (business authorization, AC + FR-090):** after the ABAC pass, the service
  loads the partner row and rejects `status IN ('suspended','expired')` (and `valid_until < today`)
  with `FORBIDDEN` (403). This is a domain check (not ABAC) but uses the same `FORBIDDEN` code.

### Data Operations

All queries are **Kysely** typed parameterised builders against the exact schema columns. Every
list/read query carries `org_id` + partner scope and a `LIMIT`.

**A. Resolve & gate the partner (read, before the tx):**
```ts
const partner = await db.selectFrom('partners')
  .select(['partner_id', 'type', 'status', 'valid_until', 'branch_id', 'mapped_rm_id'])
  .where('org_id', '=', user.org_id)
  .where('partner_id', '=', user.partner_id)
  .executeTakeFirst();
// null -> FORBIDDEN; status in (suspended,expired) or valid_until < today -> FORBIDDEN
```

**B. Load active ProductConfig (read, validation input):**
```ts
const cfg = await db.selectFrom('product_configs')
  .select(['product_config_id', 'field_schema', 'pan_required_at'])
  .where('org_id', '=', user.org_id)
  .where('product_code', '=', dto.product_code)
  .where('status', '=', 'active')
  .orderBy('version', 'desc')
  .limit(1)
  .executeTakeFirst();
// null -> VALIDATION_ERROR (field: product_code, issue: "no active product configuration")
```

**C. Idempotency pre-check (read, if Idempotency-Key present):**
```ts
const replay = await db.selectFrom('integration_logs')
  .select(['lead_id', 'request_ref'])
  .where('org_id', '=', user.org_id)
  .where('idempotency_key', '=', idempotencyKey)
  .where('integration', '=', 'partner_intake')   // see integration_kind enum / Ambiguities
  .limit(1)
  .executeTakeFirst();
// found -> reload that lead, return original 201 (no new write)
```

**D. Duplicate detection (FR-020 via `DuplicateService`, read):** runs inside the tx against
`lead_identities` (mobile/PAN/email/etc.). Strong match → block; weak/medium → flagged.

**E. Atomic create (inside one `UnitOfWork.run(tx => …)`):**
1. Insert `lead_identities` (name, mobile, email, preferred_language, org_id, created_by/updated_by = `user.user_id`).
2. Insert `source_attributions`:
   - `source` = `DSA` if `partner.type === 'DSA'` else `Dealer` (both satisfy the
     `ck_source_attr_partner` CHECK requiring `partner_id`);
   - `partner_id` = `partner.partner_id` (**forced**);
   - `sub_source` = dto.sub_source ?? null; `creator_channel` = `'partner'`;
     `attribution_status` = `'original'`.
3. `LeadService.create(input, tx)` — **the only writer of `leads`** — inserts the lead:
   `lead_code` (via `CodeGenerator`, `LD-{YYYY}-{seq6}`), `stage='captured'`, `product_code`,
   `product_config_id` = cfg.product_config_id, `branch_id` = partner.branch_id (nullable),
   `pin_code`, `source_attribution_id`, `lead_identity_id`, `channel_created_by='partner'`,
   `requested_amount`, `consent_status='pending'`, `kyc_status='not_started'`,
   `duplicate_status` = derived from step D, `version=1`, `created_by/updated_by = user.user_id`.
   `LeadService.create` also writes, **in the same tx**: the initial `stage_history`
   (`from_stage=null, to_stage='captured', actor_id=user.user_id`), `audit_logs(lead_created)`
   intent via `AuditAppender`, and `event_outbox(LEAD_CREATED, aggregate_type='lead', aggregate_id=lead_id)`
   via `OutboxService.emit(..., tx)`.
4. Insert `lead_product_details` stub (`lead_id`, `product_config_id`, `attributes` = dto.product_detail ?? `{}`, `validation_status='incomplete'`).
5. If `Idempotency-Key` present: insert `integration_logs` row (`integration='partner_intake'`,
   `direction='inbound'`, `lead_id`, `idempotency_key`, `correlation_id`, `status='success'`) — the
   partial unique index `uq_integration_idempotency` enforces single-use.

**F. List (read):**
```ts
const rows = await db.selectFrom('leads as l')
  .innerJoin('source_attributions as sa', 'sa.source_attribution_id', 'l.source_attribution_id')
  .innerJoin('lead_identities as li', 'li.lead_identity_id', 'l.lead_identity_id')
  .select(['l.lead_id','l.lead_code','l.stage','l.product_code','l.duplicate_status',
           'l.created_at','li.name','li.mobile'])
  .where('l.org_id', '=', user.org_id)
  .where('sa.partner_id', '=', user.partner_id)     // P-scope, non-negotiable
  .where('l.deleted_at', 'is', null)
  .$if(!!stage, qb => qb.where('l.stage', '=', stage))
  .orderBy('l.created_at', 'desc')
  .limit(limit).offset((page - 1) * limit)
  .execute();
// total via a parallel count() query with the same WHERE (no LIMIT on count, but bounded by scope)
```
`name`/`mobile` are masked by the `MaskingService` interceptor before serialization (PARTNER role).

### Validation Logic (Zod — `PartnerLeadCreateDto`)

Reuses `MobileSchema`, `PinSchema` from shared utilities; `.strip()` removes any client attempt to
set `source`, `partner_code`, `owner_id`, `branch_id`, `stage`.

| Field | Rule | Message (→ `VALIDATION_ERROR.fields[]`) |
|---|---|---|
| `product_code` | required; ∈ `@shared/enums` product_code | "Select a valid product." |
| `identity.name` | required; 1–150 chars | "Name is required." |
| `identity.mobile` | required; `MobileSchema` `^[6-9]\d{9}$` | "Enter a valid 10-digit mobile number." |
| `identity.email` | optional; email; ≤255 | "Enter a valid email address." |
| `identity.preferred_language` | optional; ∈ `lang` enum | "Unsupported language." |
| `sub_source` | optional; ≤80 chars | "Sub-source is too long." |
| `pin_code` | optional; `PinSchema` `^[0-9]{6}$` | "Enter a valid 6-digit PIN code." |
| `requested_amount` | optional; number ≥ 0 | "Amount must be zero or more." |
| `product_detail` | optional; object; validated vs active `ProductConfig.field_schema` | per-field schema messages |
| (server) active `ProductConfig` exists for `product_code` | else `VALIDATION_ERROR` | "No active configuration for this product." |
| (server) PAN timing | if `cfg.pan_required_at === 'at_capture'` then PAN required in identity | "PAN is required for this product." |

> Partner-active and partner-scope are **authorization** checks → `FORBIDDEN`, not validation.

### Error Cases (codes from `error-taxonomy.md` only)

| Condition | Code | HTTP | `detail.reason` | Notes |
|---|---|---|---|---|
| Missing/expired JWT | `AUTH_REQUIRED` | 401 | — | global `JwtAuthGuard` |
| Non-PARTNER role, or PARTNER with null `partner_id` | `FORBIDDEN` | 403 | — | ABAC deny |
| Partner `status` ∈ {suspended, expired} or `valid_until` past | `FORBIDDEN` | 403 | — | AC + FR-090 gate; generic message, no partner internals leaked |
| Field/payload invalid; no active ProductConfig; PAN-timing | `VALIDATION_ERROR` | 400 | — | carries `fields[]` |
| product_detail fails `field_schema` | `VALIDATION_ERROR` | 400 | — | `fields[]` per attribute |
| Strong duplicate detected | `CONFLICT` | 409 | `DUPLICATE_BLOCKED` | **masked** — message "A lead with these details already exists." No matched lead_id/PII/owner returned |
| Stale `Idempotency-Key` collision on a different in-flight body | `CONFLICT` | 409 | — | unique-index violation surfaces as conflict (rare) |
| Idempotency-Key replay (same key) | (original `201`) | 201 | `IDEMPOTENT_REPLAY` (internal) | returns original lead, no new row |
| Mid-write failure (DB/outbox/audit) | `INTERNAL_ERROR` | 500 | — | whole `UnitOfWork` rolls back; alerted; no stack/SQL leaked |
| Mutation rate exceeded (60/min/user) | `RATE_LIMITED` | 429 | — | `ThrottlerGuard` (mutation tier); `Retry-After` |

> FR-091 raises **no** `UPSTREAM_UNAVAILABLE` itself (no synchronous external provider call on the
> create path — duplicate detection is internal; eligibility/LOS happen later in the pipeline).

### Backend Flow (`POST /partners/leads`)

1. `JwtAuthGuard` authenticates → `req.user` (role, user_id, org_id, partner_id).
2. `ThrottlerGuard` (mutation tier, 60/min/user) checks rate.
3. `AbacGuard` + `@Requires('create_lead', partnerScope)` → `EntitlementService.can(user, 'create_lead', {resource:'leads', partnerId:user.partner_id})`. Deny → `FORBIDDEN`.
4. Controller validates body with `PartnerLeadCreateDto` (Zod). Fail → `VALIDATION_ERROR`.
5. `PartnerLeadService.submit(user, dto, idempotencyKey)`:
   a. Resolve partner (op A); gate active/valid → else `FORBIDDEN`.
   b. If `Idempotency-Key` present, idempotency pre-check (op C); hit → reload + return original.
   c. Load active `ProductConfig` (op B); null → `VALIDATION_ERROR`. Validate `product_detail` vs `field_schema`; PAN timing.
   d. `UnitOfWork.run(async tx => { … })` (op E): duplicate detection (op D) → strong → throw `CONFLICT(DUPLICATE_BLOCKED)` (rolls back, nothing persisted); else insert identity + attribution → `LeadService.create` (lead + stage_history + audit + outbox) → product_detail stub → idempotency log.
6. Serialize created lead through `MaskingService` (PARTNER projection) → `201` envelope.
7. Post-commit: outbox publisher relays `LEAD_CREATED` to Pub/Sub (downstream allocation FR-030 etc.); not part of the request transaction.

### State Machine (Lead — `lead_stage`)

- Transition realised: **create → `captured`** (the entry state; `stage_history.from_stage = null`,
  `to_stage = 'captured'`). Per `state-machines.md`, the create writes `leads` + `stage_history` +
  `audit_logs` + `event_outbox` in **one** transaction. No further transition is performed by this FR;
  the lead then follows the standard pipeline (allocation moves `captured → assigned`).
- No optimistic-lock concern on create (version starts at 1); `expectedVersion` applies to later
  mutators, not `create`.

### Transaction Boundaries

One `UnitOfWork.run(tx => …)` wraps **all** writes of a single submission:
`lead_identities`, `source_attributions`, `leads` (via `LeadService.create`), `stage_history`,
`audit_logs` (intent), `event_outbox(LEAD_CREATED)`, `lead_product_details`, and the
`integration_logs` idempotency row. Owner-writes honoured: only `LeadService` writes `leads`, enlisted
in the ambient `tx`. Any failure (duplicate block, constraint, outbox/audit error) rolls back the
**entire** unit — no partial Lead/identity/attribution/audit/outbox state can persist. Notification
and Pub/Sub relay are **post-commit** (retryable; do not roll back the create).

### External Service Calls

**None synchronous on this path.** Duplicate detection is an internal `DuplicateService` read; KYC/LOS
are downstream. Therefore no `IntegrationGateway` provider call and no `UPSTREAM_UNAVAILABLE` here.
(The `event_outbox(LEAD_CREATED)` relay to Pub/Sub is the only async fan-out, handled by the outbox
publisher, not this request.)

## Assumptions

1. **PARTNER user → partner mapping** is via `users.partner_id` (schema line 200, FK to `partners`).
   The lead's `source_attributions.partner_id` is forced to this value; client-supplied
   `partner_code`/`source` are stripped.
2. **Forced source value:** the `lead_source` enum has no literal "Partner"; partner leads use `DSA`
   (partner.type `DSA`) or `Dealer` (all other partner types), both of which satisfy the schema CHECK
   `ck_source_attr_partner` (partner_id required). `creator_channel`/`channel_created_by = 'partner'`
   captures the partner channel precisely.
3. **Masked duplicate (AC3/edge case):** strong duplicate returns `CONFLICT` +
   `detail.reason=DUPLICATE_BLOCKED` with a generic message and **no** matched-lead identifiers,
   owner, or PII — satisfying "no PII leak". Weak/medium duplicates do not block; the lead is created
   with `duplicate_status='flagged'` and the partner sees only the generic flag.
4. **Idempotency channel:** the idempotency replay record is stored in `integration_logs` with
   `integration='partner_intake'`, reusing the existing `uq_integration_idempotency` partial unique
   index (schema line 1193), consistent with architecture §11.5. (See Ambiguities for the enum value.)
5. **Limited status (AC2):** the PARTNER lead projection deliberately omits `score`, `score_reasons`,
   `owner_id`, internal notes, and any other partner's data.

## Ambiguities (best-effort notes; flagged for back-write into this LLD)

- **`integration_kind` enum value for partner intake.** This FR records the idempotency replay in
  `integration_logs.idempotency_key`, but the `integration_kind` enum membership is defined in
  `schema.sql` §1 and not quoted in this FR's read set. Best-effort: use a value semantically equal to
  "partner_intake"/"intake"; if no such member exists, store the idempotency key on the lead-create
  path via the same Redis + `integration_logs` mechanism FR-010/FR-140 use, with whatever member
  FR-140's LLD standardises. The exact enum literal must be confirmed against the `integration_kind`
  definition and FR-140's LLD, then written back here.
- **Partner document submission (AC4).** The BRD says "partner-submitted documents go to the
  KYC/document queue," but FR-091's endpoints (`GET/POST /partners/leads`) include **no** document
  upload route, and document upload is owned by M8 (FR-060/070, `documents` table). This LLD treats
  document submission as **out of scope** of FR-091 (the partner uploads via the standard
  document-upload endpoints once the lead exists). If a dedicated partner document-upload route is
  intended, it must be added to `api-contract.yaml` and a separate LLD/FR — do not invent it here.

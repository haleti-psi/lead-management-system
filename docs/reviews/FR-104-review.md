# FR-104 — Stage 8 Per-FR Review

**Verdict:** REJECT

> FR-104 (SLA Configuration & Escalation Engine) is largely well-implemented: auth/ABAC guards are correctly layered, the owner-writes port pattern is properly used for lead/KYC/grievance writes, all permitted error codes are used, all list queries have LIMIT ≤ 100, no console.* or `any` casts appear, and the unit test suite (engine, service, repository, guard, DTO, controller metadata) is thorough. Two issues prevent approval: (1) a silently swallowed JSON parse error in the repository violates the project non-negotiables, and (2) the `list` endpoint passes no user/org context to the service, making scope-B filtering (explicitly required by the LLD for BM/KYC/DPO and tested by TC-003) impossible in the current code path — even in a single-org build this is a spec conformance gap. Integration and e2e tests are correctly deferred per the project-wide decision.

## Findings

### MAJOR — `apps/api/src/modules/engagement/sla-policy.repository.ts:184-189`

The `safeParse` helper catches JSON.parse errors and silently returns `null` (`catch { return null; }`). This violates the project non-negotiable 'no silently swallowed errors (`catch {}`)'. A corrupted JSONB `condition` or `escalation_chain` in the DB would be silently treated as no-condition/empty-chain, potentially misrouting SLA policy resolution without any log or alert.

**Fix:** Log the parse error before returning null: `catch (err) { this.logger.warn({ err, raw }, 'Failed to parse JSONB field from sla_policies'); return null; }`. The repository must inject the PinoLogger (same pattern as SlaEngine / BusinessCalendarService) to make the logger available in this module-level helper, or promote `safeParse` to an instance method.

### MAJOR — `apps/api/src/modules/engagement/sla-policy.controller.ts:33-36 and apps/api/src/modules/engagement/sla-policy.service.ts:55-65`

The `list` endpoint passes no user/org context to `SlaPolicyService.list()`. The LLD §Auth Check states scope-B callers (BM/KYC/DPO) must have results 'filtered to their org/branch scope', and TC-003 asserts 'results filtered to BM's org/branch scope; no cross-org rows'. The service delegates straight to `repo.list()` which always queries `ORG_ID_DEFAULT` with no per-caller scoping, making it impossible to implement the scope-B filtering the spec requires.

**Fix:** Add `@CurrentUser() user: AuthUser` and `@Req() req: AbacRequestContext` parameters to the `list` handler (mirroring `create`), then pass `{ user, effectiveScope: req[EFFECTIVE_SCOPE_KEY] }` to `service.list()`. The service should apply org/branch predicate based on `effectiveScope` — scope A queries the full org, scope B adds a branch predicate using `user.branchId` (or equivalent). Update `repo.list()` to accept an optional `branchId` filter.

### MINOR — `apps/api/src/modules/engagement/sla-policy.repository.ts:93-105`

The `insertPolicy` call does not include a `sla_policy_id` (UUID) in the values object — the column is expected to be auto-generated by the DB default. The LLD pseudocode explicitly calls `generateUUID()` for `sla_policy_id`. If the DB column has no default (or the generated types do not make it optional), this INSERT will fail at runtime.

**Fix:** Verify `schema.sql` confirms `sla_policy_id uuid DEFAULT gen_random_uuid()`. If no default exists, add `sla_policy_id: generateUUID()` to the `insertInto('sla_policies').values({...})` call (matching the LLD pseudocode and the `insertConfigVersion` pattern which correctly omits the PK only if the type marks it generated).

# Council Evaluation: Lead Management System for NBFCs — BRD Version 5

*Date: 2026-06-05 · Method: 5-advisor adversarial council (Proponent, Contrarian, First Principles Thinker, Outsider, Executor) → anonymous peer review → chairman synthesis → targeted second pass → verdict · **Second pass triggered on 3 fundamental clashes** · 26 agents · ~1.98M tokens*

**Subject:** `docs/requirements/Lead_Management_System_BRD_Version_5.md` (2,728 lines; 16 sections; 42-entity data model; 48 FRs with LLDs) — a BRD that doubles as an "AI-buildable build contract" for a fleet of parallel coding agents.

> **Framed question.** Whether to proceed to build on this spec (pipeline Stage 2 data model → parallel code generation) **as-is**, revise it first, or rethink. A good outcome is a clear, actionable verdict: is this BRD genuinely build-ready as a contract for autonomous/parallel AI agents, and what are the highest-leverage things to fix or de-risk before committing engineering effort? Evaluated across product/market viability, regulatory soundness, technical/architectural buildability, data-model & contract integrity, scope realism, and risk.

## Chairman's Verdict

### Verdict: **REVISE BEFORE BUILD** — *scoped pre-dispatch hardening, not a rewrite*

**Do NOT rewrite, and do NOT dispatch the agent fleet yet. Run a short (days, not weeks) pre-dispatch hardening pass that converts the named seam ambiguities into frozen, version-pinned shared contracts, then dispatch.**

The foundation is genuinely strong (42-entity field-level model, §5.4 ownership matrix, §5.5 single enum catalog, §14.6 arbiter, transactional outbox FR-141, read-only LOS boundary) — so this is not major rework. Four of five advisors — and the Proponent's own second-pass position ("build_ready **conditional on resolving the seam-contracts before the consumer waves dispatch**") — converged that the decisive variable is **not defect magnitude** (the fixes are surgical) **but detectability before divergence**: these defects sit on the shared critical path of 6–8 agents, where §14.8 is a presence/coverage check that is structurally blind to them, and the §14.6 arbiter is *reactive* — it fires on symbol collisions, not silent semantic divergence. A *missing* shared contract is not arbitrable: the first agent either invents it (violating §4.8.1) or stops at `AMBIGUITY.md`; either way you pay serial rework at merge.

**Freeze, in this order, before any A2/A4/A5/A6/A7 dispatch:**

1. **The `LeadService` write-command interface** (`transitionStage` / `assignOwner` / `setScore` / `setHotFlag` / `recordKyc` / `markHandedOff`, each taking `expectedVersion`) — the true serialization point for the 14 Lead-writer FRs in §5.4.
2. **An explicit unit-of-work / transaction-propagation pattern** in the architecture doc, with `AuditLog`/`EventOutbox`/`DataSharingLog` declared as append-only sinks any owner may INSERT within the caller's transaction — this dissolves the §4.8.3-vs-§5.6.4 owner-writes / atomicity contradiction on a single PostgreSQL connection (NestJS modular monolith, §4.1).
3. **A `StageHistory` read-model** (or a pinned `AuditLog.detail` stage-transition schema) owned by ONE FR and consumed by both FR-053 and FR-120/121 so "reconciles with reports" is one codepath.
4. **Rule-engine seeds + a frozen JSONB grammar** for FR-011/030/031 and the `field_schema` dialect.
5. **The tokenization / `pan_token` owner** and the deterministic-token decision that FR-020 dedup silently assumes.
6. **The `consent_purpose` × `lead_stage` gate map** (OD-06) and a `StageGuardService` owner.
7. **A versioned LMS-owned LOS mock contract** (`los-mock.openapi.yaml` + `LosPort`) so FR-080/081/082 build behind a port, with real integration sequenced last.

Also: assign **M4/Allocation** to an agent (it is in no §14.3 agent), resolve the **`AuditLog` owner** (named M13 in §5.1, A1 in §14.3, A10 via M13), and **correct the §14.8 "0 unresolved gaps" claim** — true for coverage, false for integration; strip the LOS external leg out of it.

> **Separately and non-negotiably:** you physically cannot dispatch Monday regardless of the council, because the repo holds only the V5 BRD — there is no `schema.sql`, no `docs/contracts/`, no `manifest.json` for phase-executor or any coding agent to read.

### Where the Council Agrees (highest-confidence findings)

1. **The BRD is unusually disciplined and high-quality for the category** — 42-entity field-level model, single §5.5 enum catalog, §5.4 ownership matrix, §14.6 arbiter, transactional outbox (FR-141), append-only patterns, and a deliberately clean read-only LOS boundary (§1.1, §2.4). All five advisors concede this; the foundation needs zero rethink.
2. **The LOS hand-off (FR-080/081/082) is the headline feature (BO-04) yet its wire contract is undefined and third-party-owned.** §2.5.1 only *assumes* the LOS "will expose" APIs; `ProductConfig.eligibility_mapping` (line 711) specifies the LMS side of the arrow but leaves the LOS payload schema blank. Can be scaffolded against a mock now, but not integration-verified.
3. **The §14.8 "Unresolved gaps: 0" reconciliation is a self-certified per-FR *presence* check, not a buildability/consistency check, and is overstated** — 16 open decisions in Appendix B/§16.2, FR-131 configurable consent purposes vs the hardcoded §5.5 enum, and no column for transaction-ownership or service-interface completeness.
4. **No first-class `StageHistory` entity exists in §5.1**, yet §12 funnel/TAT reports, dashboards (FR-053), and reconciliation depend on stage-reached/dwell math reconstructed independently from the hash-chained `AuditLog` (built for tamper-evidence, not analytics).
5. **The MVP is over-scoped** (~48 FRs including retention engine, audit explorer, config maker-checker, export approval, differentiator reports) and should be re-tiered toward a genuine validation MVP, with provider integrations deferred behind the IntegrationGateway seam.
6. **Critical external dependencies are unstarted, multi-week, and approval-gated on the critical path** — LOS API spec/sandbox, KYC/CKYC/DigiLocker/PAN vendor selection (OD-08), AA/GST sandboxes, India messaging registration (TRAI DLT / SMS templates / WhatsApp WABA), and legal/DPO sign-off on consent text and retention periods.

### Where the Council Clashed (and how each resolved)

**1. Go/no-go: surgical-patches-while-proceeding vs revise-before-build.** → **`revise_before_build`, but SCOPED — not foundation-level.** The foundation holds (41/42 entities have a §5.4 owner; catalogs frozen). The fixes are surgical in content and size; the decisive axis is *sequencing*, not magnitude — the seam-contracts sit on the shared critical path where §14.8 is blind and the §14.6 arbiter is reactive, and the §14.4 reconcile-before-reporting checkpoint fires *after* FR-053 has already merged its own metric math. The Proponent's own second pass collapsed into this verdict.

**2. Is the parallel-agent thesis (owner-writes + arbiter + outbox) a differentiator or the spec's deepest contradiction?** → **Both, at different layers.** A *genuine, rare* differentiator at the **data-contract layer** (single enum catalog, frozen schema/error codes, contract-version pinning, arbiter — these defeat the canonical parallel-agent failure). **Not yet real at the transaction/service-interface layer**: owner-writes (§4.8.3) conflated with mandated atomic multi-owner writes (§5.6.4) is unbuildable until a unit-of-work contract and the `LeadService` mutator signatures exist. *Retractions noted: Contrarian withdrew "guaranteed 409 storms"; First Principles withdrew the "logical impossibility" framing; Outsider withdrew "God-service bottleneck."*

**3. Is the undefined LOS hand-off de-riskable by internal spec work, or an immovable Gate-0 blocker?** → **Neither extreme — both poles converged on the Executor's mock-first plan.** Not a Gate-0 stop (the seam is LMS-owned via IntegrationGateway/IntegrationLog and the inbound webhook the LMS *defines*), so the scaffold and all failure branches are buildable now behind a versioned mock + `LosPort`. But internal completeness de-risks the **scaffold, not the guarantee** (AC1 semantic guards, two-party idempotency, FR-082 HMAC/status semantics are LOS-defined). Build behind the mock now; sequence real wire integration **last**; add the LOS wire contract to Appendix B; remove FR-081's all-Y row from §14.8; split done-definition into *scaffold-complete* vs *integration-verified*.

**4. (non-fundamental) How to remediate Lead write-contention?** → Do both, but the **`LeadService` mutator interface is the mandatory pre-dispatch deliverable**; additionally split system-managed volatile fields (`sla_*_due_at`, `is_hot`, `score`) from the human-edited optimistic-lock fields so background writes don't raise false 409s.

**5. (non-fundamental) How deep to cut MVP scope?** → Cut deeper than the Proponent but staged: a ~15-FR validation core (auth, capture, dedup, allocation, tasks/SLA, consent ledger, documents, one LOS hand-off behind the mock, core funnel); defer the compliance/analytics platform to fast-follow.

### Blind Spots Caught (only via anonymous peer review — none individually named these)

- **No business-time model.** No `BusinessCalendar`/`Holiday`/`WorkingHours` entity exists, yet the entire SLA/TAT layer (SLAPolicy §5.2.38, FR-104, OD-05) and the headline KPIs (BO-03 first-contact adherence, BO-05 median doc TAT < 5 **business days**) are defined in business hours/days. Five timers would each invent their own clock → primary success metrics non-reproducible. **(Not addressed in second pass.)**
- **DPDPA erasure vs immutable audit — the compliance crown jewel turns on itself.** Right-to-erasure / storage-limitation is mutually exclusive with the append-only, hash-chained, PII-bearing `AuditLog` and the "retention-exempt" `ConsentRecord` (IP/device/actor evidence). You cannot purge a data principal without breaking the tamper-evidence the chain exists to provide; FR-112/115 only do time-based purge + legal-hold. No crypto-shredding / PII-out-of-chain design. **(Not addressed in second pass.)**
- **No downstream erasure propagation.** The praised one-way read-only LOS boundary means a later consent withdrawal/erasure cannot be propagated to the LOS once a lead is handed off (`DataSharingLog` only records that PII left), yet DPDPA obliges the fiduciary to cause downstream processors to erase. **(Not addressed.)**
- **India messaging regulatory infrastructure entirely absent.** TRAI DLT principal-entity registration, SMS sender-header + per-content-template registration, WhatsApp BSP/WABA + per-template Meta approval — multi-week, approval-gated, *per-template*. So the FR-060 first-contact link and OTP can't be free-form; the very first outbound message is un-sendable, biting earlier than KYC and compounding the consent-bootstrap deadlock. **(Not addressed.)**
- **Build-vs-buy was never asked.** For a largely commoditised lead-management category, generating and AI-reviewing a 42-entity regulated platform was never weighed against configuring an incumbent Indian lending CRM (LeadSquared, Salesforce FSC); several claimed differentiators may be configuration, not a moat. The council debated HOW to build, never WHETHER. **(Not addressed.)**
- **Hash-chain vs stateless scaling.** `AuditLog`'s `prev_audit_hash` chain needs a serial/serializing writer, but §5.4 instructs *all* modules to append under NFR-05 "stateless horizontal scaling" with no single serializing worker/queue specified — breaks tamper-evidence under concurrent appends.

### Idea Evolution

First pass split 1–4: the Proponent rated it `build_ready_with_fixes` (holes are surgical); Contrarian, First Principles, Outsider, and Executor rated it `revise_before_build` (contradictions live at cross-FR seams §14.8 cannot catch). Peer review surfaced blind spots none had individually named (business-time clock, DPDPA-vs-append-only-audit, downstream erasure, India DLT/WhatsApp). The second pass produced genuine convergence by separating two axes the first pass conflated: defect **magnitude** (all five agreed: surgical, schema excellent, no model rethink) vs dispatch **sequencing** (the fixes define shared write/transaction contracts 6–8 agents encode on day one). The Proponent moved from "patch in flight" to "build_ready conditional on resolving three seam-contracts *before* the consumer waves" — functionally joining the revise camp. Dissenters narrowed "foundation-level revise" to a bounded, time-boxed pre-dispatch seam-reconciliation pass, explicitly **not** a re-authoring. Several overstatements were retracted; the LOS clash collapsed onto a single mock-first plan.

### First Principles Thinker — leaning: `revise_before_build`

As a BRD it is genuinely excellent: coherent thesis (§16.4 origination control tower), a real LOS boundary (§1.1, §2.4), regulation-aware design (FR-110/111/113/115, NFR-09), and a largely normalized 42-entity model. But "build-ready for parallel agents without further clarification" is overstated, and the framing optimizes the wrong variable. The **binding constraints are external**: the LOS API (§2.5.1 only assumes it "will expose" APIs, yet FR-080/081/082 invent the LMS side), KYC/comm vendor procurement and sandboxes (§2.5.3), legal sign-off on consent text/retention, NBFC master data and the seven product field-schemas (FR-041 are placeholders), and field-RM adoption. **A more complete spec moves none of these.** Meanwhile real internal contradictions undercut "0 gaps" (§14.8): §16.2 lists 16 OPEN decisions; FR-131 says consent purposes are admin-configurable while §5.5 hardcodes `consent_purpose` as a fixed enum and FR-110 enforces "purpose ∈ enum"; §4.1 fixes the stack but OD-16 lists it as open. The parallel-agent contract is incomplete precisely at the inter-agent boundary: named services (EntitlementService §4.7, AllocationService/ScoringService §6, IntegrationGateway FR-140, plus implied AuditService/OutboxService) have no signatures, and §4.2's owner-writes rule collides with the mandated single-transaction multi-entity writes (Lead+AuditLog+EventOutbox in FR-010; §5.6.4). Model ambiguities (Lead.product_id vs product_config_version with no Product master; reopen with no stored prior_stage; StageHistory derived from hash-chained AuditLog yet §12.5 funnels depend on it) will be resolved inconsistently across agents. Finally, ~48 MVP-labeled FRs constitute a v2 platform, not an MVP — 6–8 months of build gated by undefined externals.

### Outsider — leaning: `revise_before_build`

Coming in cold, this is an unusually well-organised spec: a real enum catalog (§5.5), an error-code catalog (§8.4), a glossary (§15) that pre-empts most jargon (LOS, DSA, V-CIP, CKYC, DPDPA, FOIR, idempotency, transactional outbox), and a 0-gap reconciliation table (§14.8). For ~70% of FRs (auth FR-001/002, tasks FR-100, lists FR-050, dashboard FR-053, audit explorer FR-123) a non-NBFC engineer could build directly. But the NBFC-critical core is specified in prose, not contract, and "0 gaps" is overstated.
